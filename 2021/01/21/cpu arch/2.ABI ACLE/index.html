    ABI、ACLE 与 原子操作

    Alex Wu
    2021-01-21
    CPU Arch

    1. ARM ABI（Application Binary Interface）

    1.1 定义与作用

    ARM ABI 是一套二进制接口规范，定义了在 ARM 架构上程序模块之间如何交互的底层规则，包括：

    • 函数调用约定（参数传递、返回值处理）
    • 寄存器使用规则
    • 栈帧布局
    • 数据对齐要求
    • 异常处理机制

    1.2 常见 ARM ABI 类型

    • 32位架构：
      • armeabi：基础 ARM EABI（Embedded ABI）
      • armeabi-v7a：支持 ARMv7 指令集，包含 Thumb-2 和 VFP
    • 64位架构：
      • arm64-v8a：AArch64 架构的标准 ABI
    • 浮点处理：
      • gnueabi：软浮点（软件模拟浮点运算）
      • gnueabihf：硬浮点（硬件浮点单元）

    1.3 ABI 的重要性

    ABI 确保了不同编译单元、库和可执行文件之间的二进制兼容性，是链接和运行时正确执行的基础。

    2. ACLE（ARM C Language Extensions）

    2.1 定义与目的

    ACLE 是由 ARM 公司制定的一套 C/C++ 语言扩展规范（文档编号 IHI0053），旨在让开发者能够：

    • 安全、可移植地访问 ARM 特有硬件功能
    • 避免手写内联汇编
    • 在保持代码可读性的同时获得接近汇编的性能

    2.2 ACLE 的核心组成部分

    2.2.1 Intrinsics（内建函数）

    • NEON/SIMD Intrinsics：<arm_neon.h>，用于向量化计算
    • SVE Intrinsics：<arm_sve.h>，用于可伸缩向量扩展
    • Crypto Intrinsics：硬件加速的加密操作

    2.2.2 系统级操作函数

    定义在 <arm_acle.h> 中：

    • 内存屏障：__dmb(), __dsb(), __isb()
    • 原子操作原语：__ldrex(), __ldrexd(), __strex(), __strexd()
    • 低功耗指令：__wfi(), __wfe()
    • CPU 信息获取：__get_cpuid()

    2.2.3 专用数据类型

    • NEON 类型：int8x8_t, float32x4_t 等
    • SVE 类型：svint32_t, svfloat64_t 等

    2.2.4 特性检测宏

    • __ARM_FEATURE_NEON
    • __ARM_FEATURE_SVE
    • __ARM_FEATURE_CRYPTO
    • __ARM_FEATURE_CRC32

    2.3 工具链支持

    ACLE 被主流编译器广泛支持：

    • GCC：通过 -march/-mfpu/-mcpu 选项控制功能启用
    • Clang：完全兼容 GCC 的 ACLE 实现，Android NDK 默认使用
    • ARM Compiler：ARM 官方编译器，支持最完整及时

    3. 原子操作：标准库 vs 底层原语

    3.1 两种实现方式的关系

    • C11/C++11 标准原子操作：高层抽象，跨平台
    • ARM ACLE 原子原语：底层实现，特定于 ARM 架构
    • 关系：标准原子操作在 ARM 平台上底层就是通过 LDREX/STREX + 内存屏障实现的

    3.2 使用建议

    优先使用标准原子操作（C11/C++11）当：

    • 开发用户态应用程序
    • 需要跨平台支持
    • 不需要对硬件行为做极端精细控制

    考虑使用 ARM ACLE 原语当：

    • 开发操作系统内核
    • 在无标准库的裸机环境
    • 需要实现标准库未提供的特殊同步原语

    3.3 关键区别对比

    维度 标准原子操作 ARM ACLE 原语
    可移植性 跨平台 仅 ARM
    易用性 高层语义，简单易用 需手动处理重试、内存屏障
    内存模型 完整支持 memory_order 需手动插入屏障指令
    适用场景 应用层、通用库 内核、裸机、特殊同步原语

    4. 总结

    1. 理解分层：ABI 是二进制层面的规范，ACLE 是源代码层面的扩展
    2. 优先标准：在可能的情况下，优先使用 C11/C++11 标准原子操作
    3. 条件编译：利用 ACLE 特性检测宏编写可移植的优化代码
    4. 工具链选择：根据项目需求选择合适的编译器，现代 GCC/Clang 对 ACLE 支持良好
    5. 性能验证：不要假设手写底层代码一定更快，现代编译器优化能力很强
