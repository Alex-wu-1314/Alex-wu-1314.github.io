    Cortex-A55 微架构与性能分析

    Alex Wu
    2021-01-09
    CPU Arch

    一、Cortex-A55 基础微架构特性

    1. 双发射（Dual-Issue）超标量架构

    Cortex-A55 采用双发射超标量架构，每个时钟周期最多可分发（Issue）两条独立指令，理论最大 IPC（Instructions Per Cycle）为 2。

    然而，实际性能往往低于理论上限，原因包括：

    • 数据依赖关系：指令间的 RAW（Read After Write）真依赖导致流水线等待；
    • 缓存行为：L1/L2 缓存 miss 引发访存延迟；
    • 资源争用：功能单元端口有限，多条指令竞争同一执行资源。

    2. 支持有限乱序执行（Out-of-Order Execution）

    尽管 Cortex-A55 定位为”小核”（Little Core），但它具备基本的乱序执行（Out-of-Order Execution，OoO）能力，主要体现在以下硬件支持：

    硬件结构 作用
    寄存器重命名（Register Renaming） 消除 WAW/WAR 假依赖，暴露更多指令级并行
    保留站（Reservation Station） 缓存等待操作数的指令，就绪后乱序发射
    重排序缓冲区（ROB） 跟踪乱序执行状态，保证顺序提交
    非阻塞缓存（Non-blocking Cache） 允许 cache miss 时继续执行后续无关指令

    二、现代超标量处理器的指令生命周期（Cortex-A55）

    一条指令从取指到最终生效，在 Cortex-A55 流水线中经历以下 7 个阶段：

    Fetch → Decode → Rename → Dispatch → Execute → Write-back → Retire

    阶段 功能说明
    1. 取指（Fetch） 从 L1I 缓存读取指令；若 miss 则触发内存子系统访问，可能导致前端停顿。
    2. 译码（Decode） 解析指令格式，识别操作码与操作数；分配 ROB 条目、保留站槽位等资源。
    3. 重命名（Rename） 将逻辑寄存器（如 X1）映射到物理寄存器（如 P10），消除 WAW/WAR 假依赖。
    4. 分发（Dispatch / Issue） 将指令放入保留站，等待源操作数就绪。Cortex-A55 支持 2-wide issue。
    5. 执行（Execute） 当所有操作数就绪，由 ALU/FPU/AGU 等功能单元执行计算或地址生成。
    6. 写回（Write-back） 执行结果写入物理寄存器堆（PRF）或 ROB，供后续指令读取（旁路网络）。
    7. 提交（Retire / Commit） 按程序顺序将结果正式写入架构状态（逻辑寄存器或内存），对外可见。

    三、“已执行（Executed）” vs “已提交（Retired）” 的关键区别

    理解”已执行（Executed）”与”已提交（Retired）”的区别，是掌握乱序处理器模型的核心。

    维度 已执行（Executed） 已提交（Retired）
    定义 指令已在功能单元完成计算，结果写回 PRF/ROB。 指令结果已按程序顺序正式成为架构状态（如更新 X0–X30 或内存）。
    发生时机 可早于或晚于程序顺序（乱序完成）。 严格按程序顺序进行（保证精确异常和一致性）。
    对程序可见性 不可见（仅为中间状态） 可见（构成程序最终行为）
    异常处理 若执行出错（如除零），仅标记异常，不立即处理。 提交时若存在异常，则触发精确异常（precise exception）。
    资源释放 物理寄存器仍被占用（直到 retire）。 物理寄存器释放回空闲池；ROB 条目回收。
    举例 ADD X1, X2, X3 在 cycle 10 完成计算 → executed。 该指令在 cycle 15 按序提交 → X1 正式更新 → retired。

    重要原则：

    • CPU 可以乱序执行，但必须顺序提交。
    • 只有 retired 的指令才对软件状态产生影响。
    • 性能分析中，“retire bandwidth” 是衡量有效吞吐的关键指标。

    四、寄存器重命名机制详解

    1. 发生时机

    • 运行时刻（硬件自动完成），非编译行为。

    2. 逻辑寄存器 vs 物理寄存器

    • 逻辑寄存器（X0–X30）：ISA 定义，编译器使用。
    • 物理寄存器（P0–P95+）：微架构内部，数量更多，用于重命名。

    3. 重命名目的

    • 消除假依赖（WAW、WAR），暴露真实 ILP。
    • 使无依赖指令可并行执行，提升 IPC。

    4. 硬件支持

    • 重命名表（Map Table）
    • 物理寄存器堆（PRF）
    • 重排序缓冲区（ROB）跟踪指令状态

    五、关键性能瓶颈指标及其形成机制

    1. Pipeline Bubbles（流水线气泡）

    • 成因：依赖、分支预测失败、资源冲突导致流水线插入 NOP。
    • 分类：
      • 前端气泡（取指停顿）
      • 后端气泡（执行停顿）
    • 表示：STALL_FRONTEND, STALL_BACKEND

    2. Stall Cycles due to Cache Misses

    • 成因：L1D/L1I miss 后需访问慢速存储，若指令依赖该数据则 stall。
    • 注意：乱序执行可掩盖部分 miss。
    • 表示：L1D_CACHE_REFILL, BUS_ACCESS, Miss Rate, Stall Cycles

    3. Register Bank Conflicts

    • 成因：物理寄存器堆端口带宽不足，多指令争用同一 bank。
    • 在 A55 中：通常影响较小，但在高密度整数代码中可能出现。
    • 表示：无直接 PMU 事件，可通过 RESOURCE_STALLS 或 retire bandwidth 下降推断。

    六、关注的核心性能指标体系

    在性能分析中，应围绕以下三层指标构建诊断框架：

    1. 顶层指标（Overall Throughput）

    指标 说明
    IPC（Instructions Per Cycle） 最核心吞吐指标。A55 理论上限为 2。低 IPC 表明存在瓶颈。
    CPI（Cycles Per Instruction） IPC 的倒数，常用于学术分析。

    2. 瓶颈定位指标（Stall Breakdown）

    指标 用途
    % Frontend Stall Cycles 判断是否受限于取指（如 I-cache miss、分支错误）
    % Backend Stall Cycles 判断是否受限于执行（如数据依赖、cache miss）
    L1D Miss Rate 衡量数据局部性好坏
    Memory Stall Cycles 量化内存子系统对性能的影响

    3. 微架构效率指标（Microarchitectural Efficiency）

    指标 说明
    Issue Bandwidth 每周期分发的指令数（反映前端供给能力）
    Retire Bandwidth 每周期提交的指令数（反映有效吞吐）
    Gap between Issue and Retire 若 issue 高但 retire 低，可能因 ROB 满、长延迟操作（如除法）、或频繁 flush（如分支错误）
    ROB Occupancy 高 occupancy 表示后端执行慢，可能 cache miss 或依赖链长

    分析口诀：

    • IPC 低？ → 看 stall 分布
    • Backend stall 高？ → 查 cache miss / 资源冲突
    • Issue 高但 retire 低？ → 查 ROB 满 / 异常 flush / 长延迟指令

    事件名 含义
    INST_RETIRED 已提交的指令数（用于计算 IPC）
    CYCLES 总周期数

    BUS_ACCESS 总线访问次数（反映内存压力）

    注意：并非所有 Cortex-A55 实现都暴露全部事件，需查阅 TRM（Technical Reference Manual）。

    八、总结

    主题 核心要点
    指令生命周期 Fetch → Decode → Rename → Dispatch → Execute → Writeback → Retire
    Executed vs Retired 执行是乱序的中间状态；提交是顺序的最终状态
    寄存器重命名 运行时硬件技术，逻辑→物理映射，消除假依赖
    性能瓶颈来源 前端（取指）、后端（执行）、内存（cache miss）、资源（端口/ROB）
    核心指标 IPC、Stall %、Issue/Retire Bandwidth、Miss Rate
    分析流程 测 IPC → 分 stall → 查具体事件 → 优化代码/数据布局

    使用建议：在嵌入式系统性能调优中，优先通过 perf 获取 IPC + stall breakdown，再结合代码特征判断瓶颈类型。对于 Cortex-A55，常见瓶颈为 L1D cache miss 和 分支预测失败，而非寄存器冲突。
