    1. 概述

    在密码学中，密钥封装机制（Key Encapsulation Mechanism，简称 KEM）是一种公钥密码体制，它允许发送方生成一个短的随机密钥，并以保密的方式将其传输给接收方，即使存在窃听者或中间人攻击也能保持安全。现代公钥加密标准（如 NIST FIPS 203、RFC 9180 HPKE）通常都以 KEM 为基础构建。

    KEM 的核心思想是：

    • 发送方知道接收方的公钥，通过封装算法（Encapsulation）同时生成一个短的随机密钥 $K$ 及其对应的封装密文 $C$；
    • 接收方知道对应的私钥，通过解封装算法（Decapsulation）从封装密文 $C$ 中恢复出相同的随机密钥 $K$；
    • 任何不知道私钥的人无法从封装密文中获得关于密钥的任何信息，即使他们已经窃听了通信或向接收方提交了其他封装以研究接收方的反应。

    KEM 与传统公钥加密方案的核心区别在于：传统公钥加密方案允许发送方从某个消息空间中选择任意消息进行加密，而 KEM 则由算法本身为发送方随机生成一个短密钥。这使得 KEM 比完整的公钥加密方案更容易设计和安全性分析。

    2. 三大核心算法

    一个 KEM 由以下三个算法组成：

    算法 符号 输入 输出
    密钥生成（Key Generation） $\text{Gen}()$ 无 公钥 $pk$、私钥 $sk$
    封装（Encapsulation） $\text{Encap}(pk)$ 公钥 $pk$ 随机密钥 $K$、封装密文 $C$
    解封装（Decapsulation） $\text{Decap}(sk, C)$ 私钥 $sk$、封装密文 $C$ 密钥 $K$，或失败符号 $\bot$

    关于密钥长度：

    • 在理论密码学的渐近分析框架下，算法是关于安全参数 $\lambda$ 的概率多项式时间算法，密钥长度是安全参数的函数。
    • 在实际密码学中，每种算法通常使用固定长度的密钥。例如：
      • ML-KEM 固定使用 256 位密钥；
      • RFC 9180 中的算法根据配置使用 256、384 或 512 位密钥；
      • 任意长度的密钥可通过密钥派生函数（KDF）从 $K$ 进一步派生。

    3. 正确性与失败处理

    正确性：KEM 是正确的（Correct），当且仅当对于任意由 $\text{Gen}()$ 生成的密钥对 $(pk, sk)$，以高概率满足：

    $$\text{Decap}(sk, C) = K \quad \text{其中} \quad (K, C) \leftarrow \text{Encap}(pk)$$

    解封装失败发生在以下场景：输入的封装密文 $C$ 并非由合法的 $\text{Encap}$ 产生，而是被篡改或恶意构造。KEM 处理失败有两种方式：

    • 显式拒绝（Explicit Rejection）：当解封装失败时，返回特殊的失败符号（如错误码或异常），明确告知调用方解封装失败。
    • 隐式拒绝（Implicit Rejection）：当解封装失败时，不返回失败符号，而是返回一个由私钥 $sk$ 和密文 $C$ 伪随机派生的随机密钥。这种方式可以防止攻击者通过观察是否失败来获得额外信息，ML-KEM 采用此方式。

    4. 安全性定义：IND-CCA

    KEM 的安全性通过自适应选择密文攻击下的不可区分性（Indistinguishability under Adaptive Chosen-Ciphertext Attack，IND-CCA）来量化。直观上，IND-CCA 衡量的是：攻击者在获得一个随机密钥 $K$ 和一个封装密文 $C^*$ 之后，能否区分 $K$ 是由 $C^*$ 封装的密钥，还是一个独立随机选取的密钥——即攻击者的能力是否超过掷硬币猜测。

    IND-CCA 游戏流程如下：

    1. 运行 $\text{Gen}()$ 生成密钥对 $(pk, sk)$；
    2. 将公钥 $pk$ 公开给攻击者；
    3. 攻击者可以向解封装预言机查询任意封装密文（自适应选择密文查询阶段 1）；
    4. 运行 $\text{Encap}(pk)$ 随机生成挑战密钥 $K_0$ 和封装密文 $C^*$，同时独立随机生成另一个密钥 $K_1$；
    5. 抛一枚公平硬币，得到结果 $b \in {0, 1}$；
    6. 将挑战对 $(K_b, C^*)$ 公开给攻击者；
    7. 攻击者可以继续查询解封装预言机，但不能查询 $C^*$（自适应选择密文查询阶段 2）；
    8. 攻击者输出猜测 $b’$，若 $b’ = b$ 则赢得游戏。

    攻击者的 IND-CCA 优势定义为：

    $$\text{Adv}^{\text{IND-CCA}} = \left| \Pr[b’ = b] - \frac{1}{2} \right|$$

    即攻击者正确区分封装密钥与随机密钥的概率超过公平猜测的部分。一个安全的 KEM 要求此优势对于任何高效攻击者都可忽略不计。

    5. KEM 与 DEM 的组合：混合公钥加密

    KEM 本身只解决了密钥传输问题，要加密任意长度的消息，需要将 KEM 与数据封装机制（Data Encapsulation Mechanism，DEM）组合，构成混合公钥加密（Hybrid Public-Key Encryption，HPKE）方案。

    DEM 即一个对称认证加密方案，其安全要求是对单条发送方加密消息的选择密文攻击下的不可区分性（IND-CCA for a single message）。

    混合加密方案的构造：

    给定安全的 KEM（算法 $\text{Gen}/\text{Encap}/\text{Decap}$）和安全的 DEM $E$：

    加密（发送方对消息 $m$ 用公钥 $pk$ 加密）：

    1. 运行 $\text{Encap}(pk)$，得到随机密钥 $K$ 及封装密文 $C_1$；
    2. 用 $K$ 作为对称密钥，对消息 $m$ 进行认证加密，得到密文 $C_2 \leftarrow E_K(m)$；
    3. 发送密文对 $(C_1, C_2)$。

    解密（接收方用私钥 $sk$ 解密）：

    1. 运行 $\text{Decap}(sk, C_1)$，恢复密钥 $K$，若失败则中止；
    2. 用 $K$ 解密 $C_2$，恢复消息 $m$，若失败则中止。

    安全性结论：只要 KEM 满足 IND-CCA，且 DEM 对单条消息满足 IND-CCA，则该混合加密方案整体满足公钥设定下的 IND-CCA。

    重要提示：上述匿名公钥加密方案不提供发送方认证——任何持有公钥的人都可以向接收方发送消息。若需要证明发送方身份，必须配合数字签名等机制。此外，必须使用认证加密（而非仅提供 IND-CPA 安全的加密）；若使用未认证的加密，攻击者可以在传输过程中选择性地篡改密文，不仅在技术上违反 IND-CCA，还可能在实践中危及机密性（参见 EFAIL 攻击）。

    6. KEM 与公钥加密方案的关系

    特性 传统公钥加密（PKE） KEM
    发送方的角色 选择任意消息加密 随机生成短密钥
    消息长度 受限（如 RSA 通常仅适合加密短消息） 输出固定长度密钥
    设计与分析难度 较高（需处理 padding、消息编码等） 较低（无需 padding，直接派生密钥）
    实际用途 几乎总是用于加密对称密钥（即退化为 KEM 使用方式） 直接为混合加密提供对称密钥

    理论上，公钥加密方案可以转换为 KEM（随机选取密钥并加密），但设计安全的 KEM 比设计安全的 PKE 更简单。因此，大多数现代公钥加密标准直接基于 KEM 构建，而非反之。

    典型例子：RSAES-PKCS1-v1_5、RSAES-OAEP、ElGamal 加密在实践中几乎都用于加密短随机密钥（混合加密），而非直接加密任意消息。

    7. KEM 在认证密钥协商中的应用

    KEM 还可用于带前向安全性（Forward Secrecy）的认证密钥协商协议，如 TLS：

    • 客户端和服务端各自生成临时 KEM 密钥对；
    • 双方交换经签名的封装密文；
    • 会话结束后立即删除临时私钥。

    这样，即使长期私钥日后泄露，攻击者也无法解密历史会话，实现了前向安全性。

    8. 典型 KEM 构造

    不同的 KEM 依赖不同的数学难题来保证安全性。以下介绍三种典型构造。

    8.1 RSA-KEM

    背景：传统 RSA 加密（”教科书 RSA”）因无随机化而完全不安全——同一明文总产生相同密文，无法抵抗已知明文攻击。各种随机化 padding 方案（如 RSAES-PKCS1-v1_5、RSAES-OAEP）试图修复这一问题，但有些已被证明存在漏洞（如 Bleichenbacher 攻击）。

    RSA-KEM 构造（更简单且具有更紧归约）：

    • 密钥生成：生成 $n$ 位半素数 $n = pq$，计算 $d = e^{-1} \pmod{\lambda(n)}$。公钥为 $(n, e)$，私钥为 $(n, d)$。
    • 封装 $\text{Encap}(pk)$：
      1. 均匀随机选取整数 $r \in \mathbb{Z}/n\mathbb{Z}$；
      2. 计算封装密文 $C = r^e \bmod n$；
      3. 通过密钥派生函数计算密钥 $K = \text{KDF}(r)$；
      4. 返回 $(K, C)$。
    • 解封装 $\text{Decap}(sk, C)$：
      1. 计算 $r = C^d \bmod n$；
      2. 返回 $K = \text{KDF}(r)$。

    RSA-KEM 直接基于 RSA 困难问题（整数因子分解），比 RSAES-OAEP 等 padding 方案具有更紧的安全性归约，且无需复杂的 padding 逻辑。

    量子安全性：RSA 安全性基于整数因子分解的困难性，而运行 Shor 算法的量子计算机可以高效分解大整数，因此 RSA-KEM 不具备量子安全性。

    8.2 Elgamal-KEM / DH-KEM

    背景：传统 ElGamal 加密定义在有限域乘法子群上，可以证明对选择密文攻击是不安全的（例如，攻击者可以通过提交变形密文 $(u, 2v)$ 来恢复 $2m$，进而还原 $m$）。此外，ElGamal 加密在椭圆曲线上的推广需要将消息可逆编码为曲线点，增加了复杂性。

    DH-KEM 构造（也称 Elgamal-KEM）：

    • 密钥生成：选取生成元 $g$，随机选取私钥 $x$，计算公钥 $h = g^x$。
    • 封装 $\text{Encap}(pk)$：
      1. 均匀随机选取 $r$；
      2. 计算 $u = g^r$（封装密文）；
      3. 计算 $K = \text{KDF}(h^r)$（由共享秘密派生密钥）；
      4. 返回 $(K, u)$。
    • 解封装 $\text{Decap}(sk, u)$：
      1. 验证 $u$ 在合法子群中，否则返回 $\bot$；
      2. 计算 $K = \text{KDF}(u^x)$（因为 $u^x = g^{rx} = h^r$）；
      3. 返回 $K$。

    DH-KEM 与认证加密组合后本质上等价于集成加密方案（IES/ECIES）。由于 KEM 只需对群元素进行单向哈希（无需可逆消息编码），它可以自然地推广到椭圆曲线群（如 ECIES、RFC 9180 中的 DHKEM(X25519) 等），在相同安全级别下更为紧凑高效。

    量子安全性：DH-KEM 安全性基于离散对数问题，同样易受 Shor 算法攻击，不具备量子安全性。

    8.3 ML-KEM（后量子）

    ML-KEM（Module-Lattice-Based KEM）是 NIST 于 2024 年 8 月发布的后量子 KEM 标准（FIPS 203），其安全性基于模格上的带错误学习问题（Module Learning With Errors，Module-LWE）的计算困难性。

    该问题目前已被研究数十年，尚未发现量子计算机能够高效求解的算法（即使对具有 Shor 算法能力的量子计算机也不例外）。

    FIPS 203 规定了三个参数集（安全性从低到高，性能从快到慢）：

    参数集 对应安全级别 密钥/密文大小（大致）
    ML-KEM-512 NIST Level 1（≈AES-128） 较小
    ML-KEM-768 NIST Level 3（≈AES-192） 中等
    ML-KEM-1024 NIST Level 5（≈AES-256） 较大

    ML-KEM 固定输出 256 位的封装密钥，采用隐式拒绝处理解封装失败。

    9. KEM 组合器（KEM Combiner）

    KEM 组合器是将两个 KEM（KEM₁ 和 KEM₂）组合成一个新 KEM 的方案，使得只要 KEM₁ 或 KEM₂ 中有任意一个是安全的，组合后的 KEM 就是安全的。

    这种方案的典型应用是构建量子-经典混合 KEM：

    • 将经典 KEM（如基于 X25519 的 DH-KEM）与后量子 KEM（如 ML-KEM）组合；
    • 若量子计算机未来被证明能破解经典 KEM，后量子 KEM 仍能保障安全；
    • 若后量子 KEM 未来被发现存在经典漏洞，经典 KEM 仍能提供保护。

    注意区分混合 KEM（hybrid KEM，指结合量子安全与经典 KEM 的组合器方案）与混合密码系统（hybrid cryptosystem，指将公钥密码与对称密码结合的系统），两者是不同的概念。

    相关标准：ETSI TS 103 744 提供了量子安全混合密钥交换的技术规范。
