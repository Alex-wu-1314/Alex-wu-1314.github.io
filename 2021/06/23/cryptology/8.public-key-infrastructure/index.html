    一、概述

    公钥基础设施（Public Key Infrastructure，PKI）是一套用于创建、管理、分发、使用、存储和撤销数字证书，以及管理公钥加密所需的角色、策略、硬件、软件和程序的综合体系。

    PKI 的核心目的是为一系列网络活动（如电子商务、网络银行和机密邮件）提供安全的电子信息传输。在简单密码已不足以完成身份认证的场景下，PKI 能提供更严格的身份证明，并对传输信息进行完整性验证。

    在密码学领域，PKI 是一种将公钥与实体身份（如个人或组织）绑定的机制。这种绑定通过证书颁发机构（Certificate Authority，CA）完成注册和证书颁发来实现。根据绑定的保证级别不同，该过程可由自动化系统或人工监督完成。当绑定通过网络进行时，需使用安全的证书注册或证书管理协议（如 CMP）。

    关键概念： 术语”可信第三方”（Trusted Third Party，TTP）也常用于指代证书颁发机构（CA）。在很多场合，PKI 本身也被用作 CA 实现的同义词。

    二、核心目标：CIA 三要素

    PKI 提供”信任服务”——即信任实体（人或计算机）的行为或输出。信任服务目标涵盖以下一项或多项能力：

    2.1 保密性（Confidentiality）

    保证任何实体不能恶意或无意地以明文查看数据载荷。数据通过加密变为秘密，即使被读取也呈现为乱码。

    PKI 用于保密性最典型的场景是传输层安全（TLS）协议。TLS 保障传输中数据（即传输过程中）的安全。经典示例是：使用网页浏览器通过输入密码登录某个基于互联网的服务网站时，TLS 对该连接进行加密保护。

    2.2 完整性（Integrity）

    保证一旦实体对传输数据进行了任何细微的篡改，该篡改将是显而易见的，因为数据完整性已被破坏。

    通常，防止完整性被破坏（防篡改）并非最重要的，但一旦完整性被破坏，必须有明确的证据（可检测篡改）。这是重中之重。

    2.3 真实性（Authenticity）

    保证实体能够：

    • 确定其所连接的目标是什么（服务器证书认证，即连接到 Web 服务器时通常采用的方式）；
    • 在连接到受保护服务时证明自身的合法性（客户端证书认证，例如使用持有数字证书和私钥的智能卡登录时所用方式）。

    三、PKI 的组成结构

    PKI 系统由以下几个关键组成部分构成：

    3.1 证书颁发机构（Certificate Authority，CA）

    CA 是 PKI 的核心。其主要职能是对数字证书进行数字签名并发布与给定用户绑定的公钥。这一过程使用 CA 自身的私钥完成，因此对用户公钥的信任依赖于对 CA 公钥有效性的信任。

    当 CA 是独立于用户和系统之外的第三方时，也被称为注册机构（Registration Authority，RA）。

    证书颁发机构的层次结构：

    根 CA 处于信任链的顶端，通常离线保存以防攻击。中间 CA 代表根 CA 颁发证书，这种分层结构降低了根 CA 被攻击的风险。

    3.2 注册机构（Registration Authority，RA）

    RA 是 CA 可委托的、用于保证有效正确注册的 PKI 角色，负责接受数字证书申请并认证发出请求的实体。

    IETF 的 RFC 3647 对 RA 的定义如下：

    负责以下一项或多项功能的实体：对证书申请人进行识别和认证、批准或拒绝证书申请、在特定情况下启动证书撤销或暂停、处理订阅者撤销或暂停其证书的请求、批准或拒绝订阅者续期或重新申请密钥的请求。然而，RA 不签署或颁发证书（即 RA 代表 CA 执行某些任务）。

    注意： RA 没有 CA 的签名权限，只负责管理证书的审核和分发。在 Microsoft PKI 体系中，RA 的功能由 Microsoft 证书服务网站或 Active Directory 证书服务提供；而在 Microsoft 独立 CA 的情况下，RA 功能不存在，因为所有控制程序都基于托管 CA 的系统的管理和访问程序。

    3.3 验证机构（Validation Authority，VA）

    第三方验证机构（VA）可以代表 CA 提供实体信息。实体在每个 CA 域内必须根据该实体的相关信息具有唯一可识别性。

    3.4 中央目录（Central Directory）

    中央目录是密钥被索引和存储的安全位置。

    3.5 证书管理系统（Certificate Management System）

    该系统负责管理已存储证书的访问权限，以及待颁发证书的交付。

    3.6 证书策略（Certificate Policy）

    证书策略阐明了 PKI 对其程序的要求，目的是让外部人员能够分析 PKI 的可信度。

    3.7 数字证书（Digital Certificate）

    X.509 标准定义了最常用的公钥证书格式。数字证书将公钥映射到实体，并由 CA 安全地将这些证书存储于中央存储库中，需要时可撤销。

    一张标准数字证书通常包含：

    • 证书持有人的可识别信息（姓名、序列号、有效期等）
    • 公钥副本（用于加密数据和验证数字签名）
    • CA 的数字签名（证明证书真实性）

    四、证书的撤销机制

    证书可能在到期之前被撤销，这意味着它不再有效。如果没有撤销机制，攻击者可以利用被攻击或错误颁发的证书，直到其到期为止。因此，撤销是公钥基础设施的重要组成部分。

    撤销由颁发证书的 CA 执行，CA 生成经密码认证的撤销声明。

    4.1 撤销信息的分发权衡

    在向客户端分发撤销信息时，存在以下权衡：

    • 及时性（发现撤销的时间窗口，即攻击者利用被攻击证书的窗口）
    • 查询撤销状态的资源消耗
    • 隐私问题

    当撤销信息不可用（由于意外或攻击）时，客户端必须做出选择：

    • 硬性失败（fail-hard）：将证书视为已撤销处理（牺牲可用性）
    • 软性失败（fail-soft）：将其视为未撤销处理（允许攻击者绕过撤销）

    4.2 主要撤销机制

    由于撤销检查的成本以及来自可能不可靠的远程服务的可用性影响，Web 浏览器会限制它们将执行的撤销检查，并在执行时采取软性失败策略。

    证书撤销列表（CRL）

    • CRL 的带宽消耗过高，不适合常规使用。

    在线证书状态协议（OCSP）

    • OCSP 存在连接延迟和隐私问题。

    其他方案

    • 已有其他方案被提出，但尚未成功部署以实现硬性失败检查。

    自 2020 年 9 月起，TLS 证书有效期缩短至 13 个月。

    五、信任模型

    5.1 CA 信任模型

    在这一信任关系模型中，CA 是受信任的第三方，同时被证书的主体（所有者）和依赖证书的一方信任。

    CA 市场格局

    根据 2015 年 Netcraft 报告（行业监控活跃 TLS 证书的标准），TLS 生态系统虽然竞争激烈，但主要由少数几家主要 CA 主导：

    三家证书颁发机构（Symantec、Sectigo、GoDaddy）占公共 Web 服务器上所有颁发的 TLS 证书的四分之三。

    然而，由于证书颁发管理中出现的重大问题，所有主要浏览器从 2017 年开始逐步撤销对 Symantec 颁发证书的信任，并于 2021 年完成。

    单点登录（SSO）中的 CA 应用

    这种方式涉及一个在单点登录系统中充当离线证书颁发机构的服务器。SSO 服务器将数字证书颁发到客户端系统，但从不存储它们。用户可以使用临时证书执行程序等操作。这种解决方案通常与基于 X.509 的证书配合使用。

    六、替代方案

    6.1 信任网（Web of Trust）

    信任网是公钥信息公开认证问题的替代方法，使用自签名证书和对这些证书的第三方证明。

    “信任网”这一单数术语并不意味着存在单一的信任网或共同的信任点，而是指任意数量的潜在不相交”信任网”之一。该方法的实现示例包括：

    • PGP（Pretty Good Privacy）
    • GnuPG（OpenPGP 的实现，PGP 的标准化规范）

    信任网的优点之一（如 PGP 中的实现）是：它可以与域中所有方完全信任的 PKI CA（例如公司内部 CA）互操作，该 CA 作为可信引荐人愿意为证书提供担保。

    信任网的局限性： PKI 的价值取决于控制证书颁发的标准和实践，引入 PGP 或个人建立的信任网可能会显著降低企业或域的 PKI 实现的可信度。

    Phil Zimmermann 的原始愿景（1992年）：

    随着时间推移，你会从其他人那里积累密钥，你可能希望将他们指定为可信引荐人。其他每个人都会各自选择自己的可信引荐人。每个人都会逐渐积累并随着其密钥分发来自其他人的认证签名集合，期望任何收到它的人都会信任其中至少一两个签名。这将导致一个去中心化、容错的所有公钥的信任网络的出现。

    —— PGP 创始人 Phil Zimmermann，1992年，PGP 2.0 版手册

    6.2 简单公钥基础设施（SPKI）

    SPKI（Simple Public Key Infrastructure）是克服 X.509 和 PGP 信任网复杂性的三项独立工作的成果。

    SPKI 的主要特点：

    • 不将用户与人关联，因为受信任的是密钥，而不是人
    • 不使用信任概念，因为验证者同时也是颁发者（SPKI 术语中称为”授权循环”）
    • 授权是其设计的核心组成部分

    SPKI 特别适合不依赖第三方进行证书授权和证书信息的 PKI 集成，一个典型示例是办公室中的空气隔离网络（air-gapped network）。

    6.3 去中心化公钥基础设施（DPKI）

    去中心化标识符（DIDs）消除了对集中式标识符注册表以及层次化 PKI 中标准密钥管理的集中式证书颁发机构的依赖。

    在 DID 注册表为分布式账本的情况下，每个实体可以充当自己的根权威机构。这种架构被称为去中心化 PKI（DPKI）。

    七、PKI 的历史沿革

    7.1 早期发现（1969年）

    PKI 的概念和使用最早由英国情报机构 GCHQ 的科学家于 1969 年通过 James Ellis 的工作发现。此后，Clifford Cocks 等人在加密算法和密钥分发方面取得了重要发现。

    由于 GCHQ 的工作高度保密，这些研究成果被保密，直到 1990 年代中期才被公开承认。

    7.2 公开披露与奠基（1976年）

    1976 年，Diffie、Hellman、Rivest、Shamir 和 Adleman 公开披露了安全密钥交换和非对称密钥算法，彻底改变了安全通信的面貌。

    随着高速数字电子通信（互联网及其前身）的进一步发展，用户之间安全通信以及确认通信对方真实身份的需求变得显而易见。

    7.3 密码协议的发展

    随后，各种密码协议被发明和分析，新的密码原语得以有效使用。随着万维网（WWW）的发明及其迅速传播，认证和安全通信的需求变得更加迫切——商业原因（如电子商务、通过 Web 浏览器在线访问专有数据库）便已足够充分。

    Taher Elgamal 等人在 Netscape 开发了 SSL 协议（即 Web URL 中的 “https”），包括密钥建立、服务器认证（v3 之前为单向）等功能，由此为 Web 用户/站点建立了 PKI 结构。

    7.4 商业化与法律化（1990年代）

    供应商和企业家看到了巨大的市场潜力，纷纷成立公司并开始推动 PKI 的法律认可和责任保护：

    • 美国律师协会（ABA）技术项目发布了对 PKI 运营某些可预见法律方面的广泛分析（参见 ABA 数字签名指南）
    • 此后不久，多个美国州（犹他州于 1995 年率先）及世界各地的其他司法管辖区开始制定法律和法规
    • 消费者群体对隐私、访问和责任问题提出了质疑，在某些司法管辖区比其他地区得到了更多重视

    7.5 发展放缓与现状

    到 21 世纪初，底层密码工程明显不易正确部署：

    • 操作程序（手动或自动）难以正确设计，即使设计正确，执行时也难以达到工程所要求的完美
    • 现有标准不足
    • PKI 供应商找到了市场，但并非 1990 年代中期所设想的那个市场，增长比预期更缓慢，方向也有所不同

    PKI 在政府实施中取得了最大成功，迄今为止最大的 PKI 实施是美国国防信息系统局（DISA）的 PKI 基础设施，用于通用访问卡（Common Access Cards）计划。

    八、PKI 的应用场景

    各种类型的 PKI，来自不同供应商，有多种用途，包括提供公钥和用户身份绑定，用于：

    8.1 加密通信

    • 电子邮件加密与发件人认证（如 OpenPGP 或 S/MIME）
    • 文档加密与认证（如 XML Signature 或 XML Encryption 标准，适用于 XML 编码文档）

    8.2 身份认证

    • 用户对应用程序的认证（如智能卡登录、通过 SSL/TLS 的客户端认证）
    • 实验性用途包括 Enigform 和 mod_openpgp 项目中的数字签名 HTTP 认证

    8.3 安全通信协议引导

    • 引导安全通信协议，如互联网密钥交换（IKE）和 SSL/TLS
    • 在这两种情况下，安全通道（”安全关联”）的初始建立使用非对称密钥（即公钥）方法，而实际通信则使用更快的对称密钥（即密钥）方法

    8.4 移动签名

    移动签名是使用移动设备创建的电子签名，依赖于与位置无关的电信环境中的签名或认证服务。

    8.5 物联网（IoT）安全

    物联网要求相互信任的设备之间进行安全通信。PKI 使设备能够获取和更新 X.509 证书，用于在设备之间建立信任并使用 TLS 加密通信。

    9.2 Let’s Encrypt 与免费证书

    一些人认为，为通过 SSL/TLS 保护网站和通过代码签名保护软件而购买证书对小型企业来说是一项昂贵的工作。然而，Let’s Encrypt 等免费替代方案的出现改变了这一现状。

    HTTP/2 与 PKI 的强绑定：

    HTTP/2 理论上允许不安全连接，但实际上，主要浏览器公司已明确表示只会在 PKI 保护的 TLS 连接上支持该协议。包括 Chrome、Firefox、Opera 和 Edge 在内的 Web 浏览器 HTTP/2 实现仅通过 TLS 使用 ALPN 扩展支持 HTTP/2，这意味着网站所有者为获得 HTTP/2 的速度优势，将被迫购买由企业控制的 SSL/TLS 证书。

    十、PKI 面临的挑战与批评

    10.1 浏览器预装中间证书的风险

    目前，大多数 Web 浏览器都预装了由证书颁发机构颁发和签署的中间证书，这些中间证书由所谓的根证书认证的公钥进行认证。这意味着浏览器需要携带大量不同的证书提供商，增加了密钥被攻击的风险。

    当密钥已知被攻击时，可以通过撤销证书来修复，但这种攻击不容易检测，可能是一个巨大的安全漏洞。浏览器必须发布安全补丁来撤销由被攻击的根证书颁发机构颁发的中间证书。

    10.2 可用性与安全性的权衡

    • CRL（证书撤销列表）带宽消耗过高
    • OCSP 存在连接延迟和隐私泄露问题
    • 浏览器倾向于软性失败而非硬性失败，为攻击者留下了时间窗口

    10.3 单点故障

    整个 PKI 体系依赖于对根 CA 的信任。一旦某个 CA 被攻击，整个 PKI 的安全性都可能受到影响。Symantec 事件（2017–2021年被主流浏览器逐步撤销信任）就是一个典型案例。

    10.4 管理复杂性

    PKI 的价值取决于控制证书颁发的标准和实践。底层密码工程并不容易正确部署，操作程序难以完美执行。

    十一、相关协议与标准

    PKI 生态系统中涉及众多相关协议与标准：

    协议/标准 全称 用途
    X.509 - 最常用的公钥证书格式标准
    CMP Certificate Management Protocol 证书管理协议
    CMC Certificate Management over CMS 基于 CMS 的证书管理
    SCEP Simple Certificate Enrollment Protocol 简单证书注册协议
    EST Enrollment over Secure Transport 通过安全传输进行注册
    ACME Automatic Certificate Management Environment 自动证书管理环境（Let’s Encrypt 使用）
    RPKI Resource Public Key Infrastructure 资源公钥基础设施
    OCSP Online Certificate Status Protocol 在线证书状态协议
    TLS/SSL Transport Layer Security 传输层安全协议
