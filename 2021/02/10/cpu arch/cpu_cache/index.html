    理解 CPU 缓存的组织与结构

    Alex Wu
    2021-02-10
    CPU Arch

    软件性能深受内存访问效率的影响。内存访问延迟背后的故事是分层的：从 CPU 缓存出发，经过虚拟内存转换，最终抵达物理 DRAM。每一层都引入了各自的开销与优化挑战。

    本文专注于 CPU 缓存，内存层级中仅次于 CPU 寄存器的第二快层级。我们将深入探讨 CPU 缓存的结构设计、数据放置与查找机制，以及这些机制如何影响代码的执行速度。

    为什么需要 CPU 缓存？

    CPU 缓存的引入，是为了弥合高速处理器与低速主内存之间的巨大速度鸿沟。如果不解决这一差距，CPU 将频繁停顿，等待数百个时钟周期才能获取到数据。

    到 1960 年代末，这种速度失配已成为重大瓶颈。计算机架构师提出了三种可能的解决方案：

    方案一：让主内存更快

    • 磁芯存储器和早期 DRAM 等技术速度太慢，无法跟上 CPU 的节奏。
    • 更快的内存，SRAM 要么价格过高，要么难以扩展到大容量。

    方案二：更高效地利用 CPU

    • 流水线和指令乱序执行等技术正在兴起。
    • 这些技术虽然有助于隐藏延迟，但并不能消除延迟，内存瓶颈依然存在。

    方案三：引入小而快的缓冲区（缓存）

    • SRAM 虽然成本高、难以扩展到大容量，但非常适合在 CPU 旁边实现小型、高速的存储层。
    • 对操作系统、Web 服务器、机器学习、数据库等真实负载的研究表明：一小部分内存承载了绝大多数的访问。这种访问的集中性使得缓存即便容量有限也能发挥极大作用（即局部性原理）。

    尽管三个方向都在持续发展，但在性能、成本和复杂性之间，缓存提供了最佳的权衡，并从 1970 年代起成为 CPU 架构的基础组成部分。

    内存层级：缓存处于哪个位置

    现代计算机的存储系统是一个金字塔形的层级结构，越靠近 CPU 的层级速度越快、容量越小、成本越高：

    关键机制：

    • L1 和 L2 缓存：每个 CPU 核心私有，延迟极低，是最频繁命中的层级。
    • L3 缓存：容量更大，所有核心共享，用作核间数据共享的缓冲区。
    • 当 CPU 需要数据时，按顺序依次检查：L1 → L2 → L3。若未找到（缓存未命中），则回退到速度慢得多的主内存（DRAM）。
    • 层级越低，容量越大、每比特成本越低，但延迟也越高。

    一个直观的类比：寄存器是你桌面上正在使用的便签，L1 是触手可及的抽屉，L2/L3 是办公室书柜，DRAM 是楼道里的文件柜，而磁盘是地下室的档案室。

    缓存的组织与结构

    缓存以固定大小的块存储数据，这些块称为缓存行（Cache Line），通常为 64 字节。多条缓存行被归为一个组（Set），每个组能容纳多少条缓存行取决于缓存的关联度（Associativity），也称为”路数（Ways）”。

    这一设计类似于哈希表：每个组就像一个哈希桶，组内的多条缓存行则像用于解决哈希冲突的链式条目。

    直接映射（1-Way 关联）

    每个内存地址只映射到一个组中的一条固定缓存行。查找时只需检查那一条缓存行，就能判断目标数据是否存在。

    • 硬件实现简单，查找速度快
    • 冲突率高，容易产生”缓存抖动”（两个地址反复驱逐对方）

    N 路组相联（如 2-Way、4-Way）

    每个组包含 N 条缓存行。一个内存地址可以存放在该组的任意一条缓存行中，因此查找时需要检查组内所有缓存行。

    • 冲突率低，灵活性高，是现代 CPU 最常用的设计（通常 4-Way 或 8-Way）
    • 硬件复杂度略高，查找需并行比较 N 个标签

    全相联（Fully Associative）

    没有组的划分，任意内存地址都可以存放在任意缓存行中。查找时必须检查所有缓存行。

    • 冲突率最低，灵活性最高
    • 硬件成本极高，仅用于极小容量的缓存（如 TLB）

    内存地址如何映射到缓存

    CPU 缓存的工作方式非常类似于哈希表。正如哈希表用键来高效地存储和检索值，缓存用内存地址来快速存储和查找数据。

    物理地址的三段分解

    对于一个物理地址，缓存硬件将其拆分为三个字段：

    以一个典型的 32KB、8-Way 组相联、64 字节缓存行的 L1 缓存为例：

    • 组数 = 32KB / (8 × 64B) = 64 个组
    • Offset 位：log₂(64) = 6 位（定位缓存行内的字节）
    • Index 位：log₂(64) = 6 位（选择组）
    • Tag 位：剩余高位（用于匹配）

    Offset（偏移量）→ 提取正确的字节

    哈希表类比：获取与键关联的值中的特定部分。

    每条缓存行存储 64 字节连续内存。通过 Index 和 Tag 确认缓存命中后，CPU 使用 Offset 位定位缓存行中的确切字节。

    Index（索引）→ 找到正确的组

    哈希表类比：用哈希函数找到正确的桶。

    缓存被划分为多个组，地址的 Index 位用于选择对应的组，告诉 CPU 在哪个组中进行搜索。

    Tag（标签）→ 在组内匹配条目（处理冲突）

    哈希表类比：进入桶后，比较桶中存储的键以解决冲突。

    在 N 路组相联缓存中，当 CPU 访问某个内存地址时：

    1. 用 Index 定位到组
    2. 将传入地址的 Tag 与该组内所有 N 条缓存行的 Tag 并行比较
    3. 同时检查每行的 Valid 位（有效位）是否置位
    4. Tag 匹配 + Valid = 1 → 缓存命中（Cache Hit）
    5. 无匹配 → 缓存未命中（Cache Miss） ，需从下级存储获取数据

    缓存未命中时发生了什么

    如果组内没有标签匹配，就发生了缓存未命中，CPU 必须从较慢的主内存中获取整个数据块。由于 64 位数据总线每个 CPU 周期传输 8 字节，将 64 字节数据块加载进缓存行通常需要约 8 个内存总线周期，加上内存控制器延迟，总代价可达数十至数百个 CPU 时钟周期。

    缓存更新：未命中时的处理流程

    Step 1：从主内存（或下级缓存）获取数据

    CPU 从主内存读取以对齐地址为起点的整个数据块（64 字节）。例如，若 CPU 请求地址 0x1A3F，则缓存加载从对齐基地址 0x1A00 起始的完整 64 字节块（低 6 位清零）。

    Step 2：定位组内槽位

    数据块被加载到由 Index 位确定的缓存组中。若有空闲行（Valid = 0），直接存入。

    Step 3：必要时淘汰现有缓存行

    若组已满，缓存使用替换策略（如 LRU）决定淘汰哪条缓存行。若被淘汰的缓存行是”脏”的（Dirty bit = 1，即被 CPU 修改过但尚未写回内存），需先将其内容写回下级存储（Write-Back 策略）。

    Step 4：更新元数据

    将所选缓存行的 Tag 更新为新数据块的地址，设置 Valid = 1，清除 Dirty = 0。

    Step 5：填充缓存层级

    L1 缓存未命中不仅填充 L1，获取的数据块还会插入到所有相关缓存层级（L1、L2、L3）。

    注：部分现代 CPU 使用”排他性缓存”设计，某一缓存行只存在于某一层级中，避免重复存储。

    Step 6：恢复访问

    更新完成后，CPU 使用 Offset 位在缓存行中访问目标字节，与命中时的操作一致。

    不同组相联类型的比较

    特性 直接映射（1-Way） N-Way 组相联 全相联
    查找复杂度 O(1)，只查 1 行 O(N)，查 N 行 O(所有行)
    冲突率 高 中（随 N 增大而降低） 极低
    硬件成本 最低 中等 最高
    典型使用场景 简单嵌入式系统 现代 CPU L1/L2/L3 TLB、微小缓存
    现代主流 N 值 — 4-Way、8-Way、16-Way —
    抖动风险 高 低 极低

    现代 CPU 的典型配置：

    • L1 Data Cache：32–64 KB，8-Way，64B 缓存行
    • L2 Cache：256 KB – 1 MB，8-Way 或 16-Way
    • L3 Cache：数 MB 至数十 MB，16-Way 或更高

    缓存替换策略

    策略 描述 优点 缺点
    LRU（最近最少使用） 淘汰最久未被使用的数据块 命中率高，符合局部性原理 高关联度下硬件开销大
    PLRU（伪 LRU） LRU 的近似实现，用二叉树位近似跟踪 硬件效率高，广泛用于现代 CPU 非精确，极少情况次优
    FIFO（先进先出） 淘汰存在时间最长的数据块 实现简单 可能淘汰仍频繁使用的数据
    随机（Random） 随机选择淘汰对象 硬件极简 可预测性差，性能不稳定
    LFU（最不常使用） 淘汰历史访问次数最少的数据块 对热点数据友好 需维护计数器，实现复杂

    实践中：现代 CPU（如 Intel Core、AMD Zen）的 L1/L2 通常使用 PLRU，L3 使用更复杂的自适应策略（如 Intel 的 SHiP 或 RRIP）。

    缓存结构如何影响性能

    缓存的设计旨在利用内存访问的两个关键原则：时间局部性和空间局部性。

    时间局部性（Temporal Locality）

    程序倾向于重复访问最近访问过的数据。

    支撑时间局部性的缓存机制

    • 保留近期访问的数据：频繁使用的值（如循环计数器、累加变量）保留在 L1 缓存中，访问速度最快。
    • 智能替换策略：LRU/PLRU 策略保留近期访问的数据块，淘汰最不可能被复用的数据。
    • 多级缓存层级：若某数据块因 L1 压力而被淘汰，它可能仍存在于 L2 或 L3，充当备用层。
    • 预取与复用预测：现代 CPU 能根据历史访问模式预测复用行为，提前保留数据。

    示例：循环中的局部变量

    变量 sum 和 i 在每次迭代中都被更新，因为时间局部性，它们始终驻留在寄存器或 L1 缓存中。

    破坏时间局部性的陷阱

    • 工作集过大：若程序使用的数据超出缓存容量，旧数据会在被复用前遭到淘汰。
    • 访问模式不规则：频繁在不相关的内存区域之间跳转（如随机链表遍历），使缓存无法预测复用行为。

    空间局部性（Spatial Locality）

    程序还倾向于访问相邻的内存地址。这正是缓存加载整条缓存行（64 字节）的原因，读取一个字节时，附近的字节很可能很快也会被访问。

    支撑空间局部性的缓存机制

    • 缓存行存储数据块：CPU 访问某个内存地址时，缓存从主内存加载整条 64 字节缓存行，预期其邻近字节很快会被访问。
    • 基于块的内存传输：访问相邻地址（如 a[i+1]、a[i+2] 等）不需要额外的内存访问，它们已在同一条或相邻缓存行中。
    • 硬件预取（Hardware Prefetching）：现代 CPU 能自动检测顺序访问模式，提前获取下一条缓存行，隐藏内存延迟。

    示例：数组遍历（最优）

    一旦加载一条缓存行，后续 15 个 int 元素已在其中（64B / 4B = 16 个元素）。

    示例：矩阵按行访问（行优先，最优）vs 按列访问（破坏空间局部性）

    对于一个 1024×1024 的 int 矩阵，按列访问每次跨越 4KB，远大于缓存行大小，缓存命中率极低。

    破坏空间局部性的陷阱

    1. 大步长跨步访问

    2. 结构体中夹杂大量填充字段

    3. 数据结构未对齐

    若结构体因内存未对齐而横跨多条缓存行，即使只访问一个字段也可能触发多次内存访问。

    总结

    CPU 缓存是现代处理器性能的核心支柱。理解其组织结构，有助于我们写出更高效的代码：

    核心概念 要点
    缓存行（64 字节） 内存访问的最小单位，一次加载邻近数据
    组相联 平衡冲突率与硬件复杂度，N-Way 是现代主流
    Tag / Index / Offset 地址分解机制，类比哈希表的键查找
    替换策略 LRU/PLRU 优先保留近期数据
    时间局部性 重复访问同一数据 → 让热点数据驻留缓存
    空间局部性 顺序访问连续内存 → 最大化缓存行利用率
    实战优化 数组优先、SoA 布局、循环分块、避免伪共享

    缓存友好的代码并不意味着”底层魔法”，而是理解数据如何在内存中流动，让你的访问模式与硬件设计协同工作。
