    P2P / SFU / MCU：三种 WebRTC 部署架构对比

    Alex Wu
    2021-11-19
    webrtc

    1. 为什么需要讨论部署架构

    WebRTC 原生是为点对点（P2P）通信设计的：两个浏览器通过 ICE/DTLS/SRTP 直接交换媒体，不需要任何媒体服务器介入。

    这在 1 对 1 通话场景中工作得很好，但当参与者增多时，问题立刻暴露出来。每个参与者需要同时向其他所有人发送和接收媒体流，上行带宽与 CPU 随参与者数量爆炸式增长。

    为了解决这个扩展性问题，工程实践中形成了三种主流的媒体拓扑：

    • P2P（Peer-to-Peer，端到端）/ Mesh（网状）：所有参与者直接两两相连
    • SFU（Selective Forwarding Unit，选择性转发单元）：媒体服务器只路由，不处理
    • MCU（Multipoint Control Unit，多点控制单元）：媒体服务器解码、混合、重新编码

    2. P2P / Mesh 架构

    2.1 工作原理

    在纯 P2P 架构中，每个参与者与其他所有参与者建立直接的 RTCPeerConnection。对于 N 个参与者的会话，连接数为：

    $$\text{连接总数} = \frac{N \times (N-1)}{2}$$

    每个参与者需要：

    • 上行：向其他 N-1 个参与者各发送一路流
    • 下行：从其他 N-1 个参与者各接收一路流
    • CPU：对本地流进行 N-1 次编码（或至少维护 N-1 条发送通道）

    2.2 带宽与资源消耗分析

    以每路视频流 1 Mbps（720p 30fps）、音频 50 kbps 为例：

    参与者数 N 连接数 每人上行 每人下行 每人总带宽
    2 1 ~1.05 Mbps ~1.05 Mbps ~2.1 Mbps
    3 3 ~2.1 Mbps ~2.1 Mbps ~4.2 Mbps
    4 6 ~3.15 Mbps ~3.15 Mbps ~6.3 Mbps
    5 10 ~4.2 Mbps ~4.2 Mbps ~8.4 Mbps
    8 28 ~7.35 Mbps ~7.35 Mbps ~14.7 Mbps

    可以看到，P2P 的上行带宽需求与 N 线性增长，而普通家庭宽带的上行通常只有 5–20 Mbps，4 人以上就已捉襟见肘。

    2.3 核心优势

    天然端到端加密（E2EE）：由于媒体直接在两端之间流动，DTLS-SRTP 的加密完全由两端控制，不存在任何第三方解密节点。这是 P2P 架构在安全性上无可替代的优势。

    基础设施成本几乎为零：不需要任何媒体服务器，唯一可能需要的是 TURN 服务器（当直连无法建立时作为中继）。信令服务器的开销也极小。

    最低延迟：媒体沿最短路径传输，端到端延迟通常在 50–100 ms 级别，是三种架构中延迟最低的。

    实现简单：基本的 WebRTC API 即可完成，无需部署额外的媒体服务器组件。

    2.4 核心缺点

    上行带宽瓶颈：上行带宽是 P2P 最严重的限制。由于互联网的不对称性（下行 >> 上行），即使在带宽充足的场景下，多人同时发送多路高清流也很快会耗尽上行容量。

    客户端 CPU 压力：浏览器需要对本地流进行 N-1 路并行编码。随着参与者增多，CPU 占用急剧上升，在移动设备和低配置电脑上尤为明显。

    不可扩展：连接数以 O(N²) 速度增长，无论是对客户端还是网络，实际可用上限通常在 4–6 人。

    信令复杂度线性增加：加入或退出一个参与者，需要在其与所有现有参与者之间重新进行 ICE 协商，信令开销随 N 线性增长。

    2.5 适用场景

    • 1 对 1 通话：最理想场景，充分发挥低延迟、E2EE 的优势
    • 小型私密会议（2–4 人）：对安全性有极高要求（医疗、法律咨询等）且参与者带宽充足
    • 开发调试：快速验证 WebRTC 基本流程，无需搭建媒体服务器

    3. SFU 架构（选择性转发单元）

    3.1 工作原理

    SFU 是当前最主流的 WebRTC 多方通信架构。其核心思想极其简洁：服务器只路由，不处理。

    每个参与者只向 SFU 发送一路（或多路，用于 Simulcast）上行流，SFU 根据每个订阅者的网络状况与订阅请求，选择性地将其他参与者的流转发给该订阅者。

    SFU 的关键特性是：RTP 包在转发过程中不被解码、不被修改（少数情况如 SRTP 重加密除外），服务器仅做路由决策。

    3.2 带宽消耗对比（SFU vs P2P）

    以每路 1 Mbps 视频 + 50 kbps 音频为例：

    参与者数 N P2P 每人上行 SFU 每人上行 上行节省
    3 ~2.1 Mbps ~1.05 Mbps 50%
    5 ~4.2 Mbps ~1.05 Mbps 75%
    10 ~9.45 Mbps ~1.05 Mbps 89%
    20 ~19.95 Mbps ~1.05 Mbps 95%

    SFU 将每个客户端的**上行带宽需求从 O(N) 降为 O(1)**，这是其相对于 P2P 最核心的改进。

    下行带宽与 P2P 相似（仍需接收 N-1 路流），但这通常不是瓶颈，因为互联网下行通常远大于上行。

    3.3 SFU 的核心能力

    3.3.1 Simulcast（同步多播）

    Simulcast 是 SFU 最重要的自适应码率手段。发送端同时编码多个分辨率版本（例如 720p / 360p / 180p），分别作为独立的 RTP 流（使用不同 SSRC）上行到 SFU。

    SFU 根据每个接收者的实时网络状况，为其选择并转发最合适的分辨率层，而无需重新编码。当某个接收者的带宽下降时，SFU 只需切换转发的层，切换延迟通常在一个关键帧周期内（约 100–300 ms）。

    3.3.2 SVC（可伸缩视频编码）

    SVC（Scalable Video Coding）是另一种自适应策略，但原理不同：编码器产生单一码流，其中包含多个可独立解码的质量层（时间层 / 空间层）。SFU 通过丢弃高层数据包来降低转发码率，而无需发送方产生多路独立流。

    VP9 的 SVC 支持（K-SVC 模式）在 Chrome 中已较为成熟，AV1 也原生支持 SVC。但 Safari/iOS 的 SVC 支持历来较为滞后，这导致大多数生产环境仍以 Simulcast 为主。

    3.3.3 带宽估计与拥塞控制

    SFU 通常集成或协助 Google Congestion Control（GCC）机制：

    • 接收端通过 Transport-CC RTCP 反馈精细的包到达时间戳
    • SFU 将这些 RTCP 包透传给对应的发送端
    • 发送端的带宽估计模块（BWE）据此调整编码码率

    部分高级 SFU（如 mediasoup）还实现了发送端带宽估计（Sender-Side BWE），在服务器侧聚合全局视角，做出更优的拥塞控制决策。

    3.3.4 灵活的布局控制

    由于每个参与者收到的是独立的视频轨，客户端可以自由决定：

    • 以何种布局（Grid、Speaker View、Spotlight）展示
    • 是否订阅某个参与者的流（节省下行带宽）
    • 对某路流进行本地后处理（缩放、滤镜等）

    这是 SFU 相对 MCU 的重要优势，布局由客户端控制，而非服务器固定输出。

    3.3.5 服务端录制

    SFU 可以在不解码的情况下，将 RTP 流直接写入录制管道（如输出为 .webm 文件），录制服务器可作为一个”哑订阅者”连接到 SFU。这比 MCU 录制更高效，也支持录制原始独立轨道。

    3.4 SFU 的局限性

    客户端下行压力依然存在：N-1 路独立流仍需客户端各自解码和渲染。参与者超过 20–30 人时，即使有 Simulcast 降低分辨率，客户端的解码 CPU 占用也会成为瓶颈。

    E2EE 实现较为复杂：标准 WebRTC 的 DTLS-SRTP 密钥在 SFU 服务器上终止（SFU 需要解密 SRTP 才能读取 RTP Header 以做路由决策）。因此，传统 SFU 架构不是真正的端到端加密。要实现 E2EE，需要在应用层对媒体内容进行额外加密（如使用 WebRTC Insertable Streams API / Encoded Transform），SFU 仅能转发其无法解密的密文包。这种方案已有生产落地（如 Zoom 的 E2EE 会议），但实现复杂度显著提高。

    单节点水平扩展有挑战：单个 SFU 进程通常绑定在单个 CPU 核心上（如 mediasoup 的 Worker 模型）。支持大规模会议需要实现 SFU 级联（Cascading）或集群（Clustering）架构，这增加了系统复杂度。

    运维成本：相比 P2P，需要部署和维护媒体服务器，带宽成本从客户端转移到服务器侧（服务器需承载所有参与者的上行流和下行路由）。

    3.5 主流开源 SFU

    项目 语言 特点

    3.6 适用场景

    • 视频会议（5–100 人）：最典型场景，Zoom、Teams、Google Meet 等均以 SFU 为核心
    • 在线教育：多人课堂，教师流与学生流分别路由
    • 互动直播（低延迟场景）：主播上行，大量观众下行，结合订阅控制
    • 实时协作平台：游戏连麦、多人音视频协作

    4. MCU 架构（多点控制单元）

    4.1 工作原理

    MCU 代表了与 SFU 截然不同的哲学：服务器承担所有复杂性，客户端只处理一路流。

    每个参与者将媒体流上行到 MCU，MCU 完成以下操作：

    1. 解密（SRTP → 原始 RTP）
    2. 解码（压缩视频 → 原始像素）
    3. 合成布局（将多路视频帧拼接为一张画面）
    4. 混音（将多路音频波形叠加）
    5. 重新编码（原始像素/波形 → 压缩视频/音频）
    6. 重新加密（原始 RTP → SRTP）
    7. 发送（向每个参与者发送合成后的单路流）

    4.2 带宽消耗分析

    维度 P2P SFU MCU
    客户端上行 (N-1) × 流 1 × 流 1 × 流
    客户端下行 (N-1) × 流 (N-1) × 流 1 × 合成流
    客户端解码数 N-1 路 N-1 路 1 路
    服务器带宽 无（或仅 TURN） N 路上行 + N*(N-1) 路下行路由 N 路上行 + N 路合成下行
    服务器 CPU 极低 低（仅路由） 极高（解码 + 合成 + 重编码）

    MCU 的带宽效率是：客户端下行从 O(N) 降为 **O(1)**，代价是服务器承担了所有的计算成本。

    4.3 核心优势

    客户端负担恒定
    无论房间里有多少参与者，客户端始终只需解码和渲染一路流。这对以下场景至关重要：

    • 老旧设备（CPU/GPU 处理能力有限）
    • 移动端（节省电量与发热）
    • 极低带宽环境（农村网络、卫星链路）

    固定布局输出
    MCU 可以在服务端生成固定的”九宫格”画面，所有参与者看到完全相同的合成视图。这在以下场景有明显意义：

    • 录制与合规：录制内容与参与者所见完全一致，满足”所见即所录”的审计要求
    • 与传统系统集成：单路合成流可直接接入传统视频会议终端（H.323/SIP）、IPTV 系统、CDN 推流等

    跨编解码器转码
    MCU 可以在参与者使用不同编解码器时进行转码（如一端使用 VP9，另一端只支持 H.264），这是 SFU 原生无法做到的（SFU 只透传，不转码）。

    遗留系统兼容性
    企业中大量存在支持 H.264/H.323/SIP 的传统视频会议终端，MCU 可作为 WebRTC 世界与传统 VTC 世界之间的桥梁。

    4.4 核心缺点

    服务器 CPU/GPU 成本极高
    解码 + 合成 + 重编码是计算密集型操作，尤其对于高清视频。参与者越多，服务器计算量线性增加。规模化 MCU 部署通常需要专用 GPU 加速（NVENC/Quick Sync 等硬件编码器）。

    无法实现真正的 E2EE
    这是 MCU 的根本性安全局限：MCU 必须解密并解码媒体流才能合成，这意味着服务器可以看到所有参与者的原始媒体内容。对于医疗会诊、法律顾问、机密商务会议等高隐私场景，MCU 存在固有的信任风险。

    布局僵化，灵活性差
    合成布局由服务器决定，客户端通常无法自定义（例如放大特定参与者、切换不同视图模式等）。虽然可以通过信令让服务端切换布局，但实时性和灵活性远不如 SFU。

    引入额外延迟
    解码 → 合成 → 重编码的流水线不可避免地引入额外延迟（通常在 100–200 ms 量级），这在低延迟要求场景（如实时游戏）中是不可接受的。

    水平扩展困难
    MCU 的计算高度集中，难以简单地水平扩展——你不能把一个视频合成任务拆分到多台机器上。典型解决方案是”级联 MCU”（Cascading MCU），但架构复杂度成倍上升。

    4.5 主流 MCU 实现

    项目 特点
    Kurento 基于 GStreamer，支持服务端媒体处理管道，Twilio Video 早期基础
    FreeSWITCH 主要面向音频会议/呼叫中心场景，视频能力相对有限
    Ant Media Server 同时支持 SFU 和 MCU 模式，GPU 加速
    Frozen Mountain 商业 MCU，企业级支持
    OpenMCU-ru 开源 MCU 实现

    值得注意的是，Kurento 的开发在 2023 年后基本停滞，许多原本基于 Kurento 的服务（如早期 Twilio Video）已迁移至 SFU 架构。这一趋势折射出整个行业从 MCU 向 SFU 的迁移。

    4.6 适用场景

    • 大规模广播型会议（参与者设备性能差异大）：客户端负担恒定
    • 传统视频会议终端互通：与 H.323/SIP 设备对接
    • 合规录制场景：所有人看到相同画面的合规审计
    • 超低带宽客户端：只能承受一路流的极端网络环境
    • 电话会议 / 纯音频场景：音频混合计算量远低于视频

    实际的生产系统往往不是纯粹的单一架构，而是根据场景动态组合。

    6.1 P2P + SFU 动态切换

    7.1 决策树

    7.2 关键决策因素

    在选择架构时，以下因素是核心考量：

    参与者规模与增长预期：这是最主要的决策维度。P2P 适合 ≤4 人，SFU 适合 5–100 人，更大规模需要集群或混合方案。

    客户端设备能力：如果目标用户大量使用移动设备、老旧电脑或处于低带宽网络环境，MCU 的”客户端只收 1 路”特性能有效保障体验。

    端到端加密需求：涉及医疗、法律、金融等高隐私场景，P2P 是唯一天然支持 E2EE 的方案；SFU 可通过 Insertable Streams 实现应用层 E2EE，但增加实现复杂度；MCU 无法实现真正的 E2EE。

    延迟要求：实时游戏、在线拍卖等对延迟极为敏感的场景，优先 P2P 或低延迟 SFU；MCU 的转码延迟通常不可接受。

    录制与合规：需要”所见即所录”的合规录制 → MCU；需要独立轨道录制（后期剪辑、AI 分析等）→ SFU。

    传统系统集成：需要与 H.323/SIP 终端对接 → MCU 是最自然的选择。

    基础设施成本：P2P ≈ 零服务器成本；SFU 带宽成本可预测（按参与者上行计算）；MCU CPU/GPU 成本高且随参与者线性增长。

    8. 小结

    三种架构不存在绝对的优劣，而是对不同约束条件的不同权衡：

    P2P 是 WebRTC 的本源，适合小规模、高安全性场景。它以客户端带宽和 CPU 换取零服务器成本和天然 E2EE。

    SFU 是当前行业主流。它用一台路由服务器解决了 P2P 的上行瓶颈，同时保持低服务器成本、灵活布局和较低延迟。Simulcast 和 SVC 是其核心的自适应码率机制。绝大多数现代视频会议产品（Zoom、Teams、Google Meet、Discord）的核心都是 SFU。

    MCU 将计算复杂性从客户端转移到服务器，以高昂的服务器成本换取客户端的极简体验。它在遗留系统集成和特定合规场景中仍有不可替代的价值，但在隐私性和灵活性上存在固有局限。

    在实际工程中，混合架构往往是最佳答案：用 P2P 优化 1 对 1 通话成本，用 SFU 支撑多方会议，用 MCU 或 CDN 覆盖广播场景，每层架构各司其职，共同构成一个可扩展的实时通信系统。
