    WebRTC Framework：整体架构概览与浏览器内部模块分层

    Alex Wu
    2021-11-07
    webrtc

    1. 什么是 WebRTC

    WebRTC（Web Real-Time Communication）是一套面向浏览器与移动端的开放标准，允许应用在无需插件或第三方软件的情况下，在对等节点（Peer）之间直接传输音频、视频与任意数据。

    从标准化角度，WebRTC 由两大组织共同推进：

    • W3C WEBRTC 工作组：负责定义浏览器端 JavaScript API（RTCPeerConnection、MediaStream、RTCDataChannel 等），最终产出为 WebRTC 1.0 W3C Recommendation。
    • IETF RTCWEB 工作组：负责定义底层协议、数据格式、安全机制等，核心规范汇总于 RFC 8825 — Overview: Real-Time Protocols for Browser-Based Applications（H. Alvestrand，January 2021）。

    RFC 8825 本身是一份”适用性声明”（applicability statement）：它不规定任何新协议，而是指明一个合规的 WebRTC 实现必须遵循哪些已有规范。这一设计思路决定了 WebRTC 天然是协议的组合体，而非一个单一协议。

    2. WebRTC 不是一个协议，而是一套协议栈

    在理解具体模块之前，必须建立一个关键认知：WebRTC 是由十余个 IETF/W3C 规范共同组成的协议套件。以下是最核心的几份规范：

    RFC / 规范 职责
    RFC 8825 WebRTC 协议总览与协调文档
    RFC 8826 WebRTC 安全考量
    RFC 8827 WebRTC 安全架构（DTLS 指纹、证书模型）
    RFC 8829 (JSEP) JavaScript 会话建立协议（SDP Offer/Answer 模型）
    RFC 8830 RTCPeerConnection 接口规范
    RFC 8831 WebRTC DataChannel
    RFC 8832 DataChannel 建立协议（DCEP/SCTP）
    RFC 8834 WebRTC 中 RTP 的使用规范
    RFC 8835 WebRTC 传输层规范（ICE/DTLS/SRTP 组合方式）
    RFC 8836 拥塞控制需求
    RFC 8854 前向纠错（FEC）需求
    RFC 7742 视频编解码与处理要求
    RFC 7874 音频编解码与处理要求
    W3C Media Capture getUserMedia / MediaStream API

    这套协议栈的最终目标，用 RFC 8825 的话来说：

    “…allow an implementation to communicate with another implementation using audio, video, and data sent along the most direct possible path between the participants.”

    即：在参与方之间，沿最直接的路径传输音视频与数据。

    整个架构有两个关键分层边界：

    1. Web App ↔ Web API：这是应用开发者的工作区域，通过标准 JavaScript API 控制媒体与连接，无需了解底层协议细节。
    2. Web API ↔ Native C++ API：这是浏览器厂商（Chrome、Firefox、Safari 等）的实现边界，Native C++ API 为浏览器提供了一个统一的接入点以实现上层 Web API。

    4. 第一层：Web App

    这是应用开发者直接接触的层面。无论是视频会议应用、在线教育平台还是实时协作工具，开发者通过标准的 JavaScript API 与 WebRTC 交互。

    重要说明：WebRTC 标准不规定信令协议。开发者可以自由选择 WebSocket、SIP、XMPP、HTTP REST，甚至电子邮件来交换 SDP。这是 WebRTC 设计上的刻意留白，目的是保持灵活性。

    5. 第二层：Web API

    W3C 将浏览器侧的 WebRTC 能力抽象为三个核心 JavaScript 接口：

    5.1 MediaStream（媒体流）

    MediaStream 由 W3C Media Capture and Streams 规范定义，对应 getUserMedia() API，用于从设备获取音视频流。

    关键特性：

    • 一个 MediaStream 对象由一个或多个 MediaStreamTrack 组成（例如一条音频轨、一条视频轨）。
    • 同一个 MediaStream 中的多条轨道是时间同步的。
    • 输入源可以是物理设备（麦克风、摄像头）、屏幕共享，或者来自网络对端的远程流。
    • 输出可以是本地 <video>/<audio> 元素、Canvas 处理，或者通过 RTCPeerConnection 发送给远端。

    获取到的流在送入网络之前，会由浏览器的 VoiceEngine / VideoEngine 自动完成降噪、回声消除、编码等处理。

    5.2 RTCPeerConnection（对等连接）

    RTCPeerConnection 是 WebRTC 的核心接口，负责：

    • 管理信令状态机（Offer/Answer 模型，由 RFC 8829 JSEP 定义）
    • 驱动 ICE Agent 完成 NAT 穿透与候选收集
    • 协商 DTLS 握手（建立加密信道）
    • 复用 SRTP 传输媒体数据、SCTP/DTLS 传输 DataChannel

    RTCPeerConnection 背后隐藏了 ICE、STUN、TURN、DTLS、SRTP 等所有协议的复杂性。

    5.3 RTCDataChannel（数据通道）

    RTCDataChannel 提供了浏览器之间传输任意二进制或文本数据的能力，底层基于 SCTP over DTLS over ICE/UDP（由 RFC 8831 定义）。

    与媒体流不同，DataChannel 支持：

    • 可靠（Reliable）或不可靠（Unreliable）传输模式
    • 有序或无序传输
    • 可配置的最大重传次数或最大生命周期

    典型使用场景：游戏状态同步、实时聊天、文件传输、白板协作。

    6. 第三层：WebRTC Native C++ API

    这一层是浏览器厂商实现 WebRTC 的 C++ 接入点，它将 Web API 的高层语义映射到下层的 VoiceEngine、VideoEngine 和 Transport 模块。

    Web 应用开发者通常不需要直接面对这一层，但理解它对于调试性能问题、构建 Native 客户端（Android/iOS 原生 SDK）至关重要。

    7. 第四层：核心引擎模块

    7.1 VoiceEngine（音频引擎）

    VoiceEngine 是音频媒体链的完整实现框架，覆盖从声卡采集到网络发送、从网络接收到扬声器播放的全链路。

    7.1.1 音频编解码器

    WebRTC 强制要求实现以下编解码器（RFC 7874）：

    Opus（核心推荐，RFC 6716）

    • 采样率：8 kHz 至 48 kHz 自适应
    • 码率：6 kbps 至 510 kbps 可变
    • 帧长：2.5 ms 至 60 ms 可选
    • 同时支持语音（SILK 内核）和音乐（CELT 内核）两种模式，可在通话中无缝切换
    • 低延迟特性（最低约 5 ms 算法延迟）使其成为 WebRTC 音频的事实标准

    G.711（PCMA/PCMU）（RFC 7478 要求的最低兼容性）

    • 64 kbps 固定码率
    • 主要用于与传统 PSTN/VoIP 系统互通

    历史遗留（已较少使用）：

    • iSAC：16/32 kHz 采样，12–52 kbps 自适应码率，Google GIPS 时代的宽带编解码器
    • iLBC（RFC 3951/3952）：8 kHz 采样，15.2 kbps（20ms 帧）或 13.33 kbps（30ms 帧），窄带语音编解码器

    7.1.2 NetEQ

    NetEQ 是 VoiceEngine 中的动态抖动缓冲与错误隐藏算法，由 Google 开源自 GIPS。

    它的核心职责是：

    • 通过自适应缓冲平滑网络抖动（Jitter），在保持低延迟的同时维持播放连续性
    • 在发生丢包时执行包丢失隐藏（PLC，Packet Loss Concealment），通过外推或插值生成替代音频，避免出现明显的卡顿或静音

    NetEQ 持续监测网络抖动统计，动态调整缓冲区深度，在延迟与音质之间寻找最优平衡。

    7.1.3 AEC（声学回声消除，Acoustic Echo Cancellation）

    AEC 解决的是扬声器播出的声音被麦克风重新采集这一问题，即”回声”。

    工作原理：

    1. 以”远端参考信号”（即扬声器输出）作为参考
    2. 通过自适应滤波器（LMS/NLMS/RLS 等算法）估计并消除麦克风信号中的回声成分
    3. 持续更新滤波器参数以适应动态变化的声学环境（房间响应、设备移动等）

    这一模块在无耳机场景（如手机免提、视频会议大屏）下至关重要。

    7.1.4 NR（噪声抑制，Noise Reduction）

    NR 组件基于频谱减法或深度学习等方法，消除背景噪声（风扇声、键盘声、环境杂音等），提升语音清晰度。

    7.2 VideoEngine（视频引擎）

    VideoEngine 是视频媒体链的完整实现框架，覆盖从摄像头采集到编码发送、从接收解码到屏幕渲染的全链路。

    7.2.1 视频编解码器

    VP8（WebM Project，早期默认）

    • 由 Google 开源，完全免专利
    • 专为低延迟 RTC 场景设计
    • 支持 Temporal Scalability（时间可伸缩性）

    VP9

    • VP8 的后继者，相同质量下码率降低约 50%
    • 完全开源免专利
    • 现代浏览器广泛支持

    H.264（AVC）

    • 硬件编解码支持最为成熟（移动端尤为重要）
    • 专利授权复杂，但已广泛部署
    • WebRTC 规范中作为可选项（RFC 7742 规定浏览器可选择支持）

    AV1

    • 新一代开源免费视频编解码器
    • 压缩率超越 VP9 和 H.265，现代浏览器逐步支持中

    7.2.2 视频抖动缓冲（Video Jitter Buffer）

    与音频的 NetEQ 类似，视频抖动缓冲对网络传输的视频包进行重排序和缓存，隐藏网络抖动与丢包对视频质量的负面影响。

    视频抖动缓冲还负责配合 NACK/PLI 机制触发重传请求或关键帧请求。

    7.2.3 图像质量增强（Image Enhancements）

    包括去噪（Denoising）、亮度/对比度自动调整等，通常在采集端完成，以提升编码质量。

    7.3 Transport / Session（传输与会话层）

    这是 WebRTC 中协议密度最高的区域，覆盖连接建立、NAT 穿透、加密和实际数据传输。

    7.3.1 整体协议栈分层

    7.3.2 RTP Stack

    RTP（实时传输协议，RFC 3550）是媒体数据在网络上传输的核心载体：

    • 每个 RTP 数据包携带：序列号（Sequence Number）、时间戳（Timestamp）、同步源标识符（SSRC）、媒体负载（Payload）
    • 配套的 RTCP（RTP Control Protocol）负责传输质量反馈（接收方报告 RR、发送方报告 SR、丢包率、抖动统计等）
    • WebRTC 中 RTP 的具体使用方式由 RFC 8834 规范

    WebRTC 在 RTP 之上还扩展了多个重要机制：

    • BUNDLE（RFC 9143）：将音频、视频、DataChannel 复用在同一个 UDP 五元组上，减少 ICE 候选对数量
    • RTP Header Extensions：携带额外元数据，如绝对发送时间（abs-send-time）、传输序号（transport-wide-cc）等
    • RTCP Feedback：NACK（重传请求）、PLI（图片丢失指示，触发关键帧）、REMB/Transport-CC（带宽估计反馈）

    7.3.3 ICE / STUN / TURN

    ICE（Interactive Connectivity Establishment，RFC 8445）是 WebRTC 解决 NAT 穿透的核心框架：

    • 每个端点收集本地候选（host）、服务器反射候选（srflx，通过 STUN 获取）和中继候选（relay，通过 TURN 获取）
    • 双端通过信令交换候选列表
    • ICE Agent 对候选对（Candidate Pair）进行连通性检测（Connectivity Check），选出最优路径
    • 支持 Trickle ICE（RFC 8838）：候选边收集边发送，无需等待完整列表

    STUN（Session Traversal Utilities for NAT，RFC 5389）：轻量级协议，用于获取 NAT 映射的公网地址。

    TURN（Traversal Using Relays around NAT，RFC 5766）：当直连失败（如对称型 NAT 或严格防火墙）时，通过 TURN 服务器中继媒体数据。

    7.3.4 Session Management（JSEP）

    会话管理由 JSEP（JavaScript Session Establishment Protocol，RFC 8829） 定义，它规定了 WebRTC 中 SDP Offer/Answer 的交互模型：

    • Caller 调用 createOffer() 生成描述本端能力的 SDP
    • SDP 通过信令通道传输给 Callee
    • Callee 调用 createAnswer() 回复本端能力
    • 双端分别 setLocalDescription / setRemoteDescription 完成协商

    JSEP 刻意将会话协商（SDP 交换）与信令传输（WebSocket 等）解耦，应用开发者可以自由选择信令机制。

    7.3.5 DTLS 与 SRTP

    WebRTC 强制要求所有媒体传输必须加密（RFC 8827）：

    • DTLS（Datagram TLS，RFC 6347 / RFC 9147）：在 UDP 之上运行的 TLS 变体，用于握手并协商 SRTP 密钥。WebRTC 使用自签名证书（non-PKI），证书指纹通过 SDP 交换并绑定到信令通道，防止中间人攻击。
    • SRTP（Secure RTP，RFC 3711）：在 RTP 基础上添加对称加密（默认 AES-128-CM）和消息认证（HMAC-SHA1），密钥由 DTLS 握手协商产生（DTLS-SRTP，RFC 5764）。

    9. WebRTC 架构设计哲学

    理解了各层模块后，可以提炼出 WebRTC 架构的几个核心设计原则：

    9.1 安全优先（Security by Default）
    RFC 8827 明确要求：WebRTC 中的媒体传输必须加密，不存在”可选加密”。DTLS 握手与 SRTP 加密是强制的，不可绕过。

    9.2 NAT 友好（NAT Traversal First-Class）
    ICE/STUN/TURN 是架构的一等公民。RFC 8825 将 NAT 穿透视为互联网现实的组成部分，而非例外情况。

    9.3 信令解耦（Signaling Agnostic）
    WebRTC 不规定信令协议，这是有意为之。它使 WebRTC 能够与现有的 SIP、XMPP 基础设施集成，同时又不被绑定在任何特定协议上。

    9.4 浏览器内嵌（Browser-Native）
    所有处理——音视频采集、编解码、网络传输、加密，均在浏览器内完成，应用开发者通过简洁的 JavaScript API 控制，无需关心底层细节。

    9.5 UDP 优先（UDP First）
    WebRTC 以 UDP 为基础传输层，以满足实时通信对低延迟的要求。在 UDP 被阻断时，ICE 可降级为 TCP（RFC 8835 允许 TCP 作为候选传输方式），TURN over TLS/TCP 作为最后的保底方案。

    11. 小结

    WebRTC 的整体架构可以用一句话概括：通过精心设计的分层体系，将十余个 IETF/W3C 规范组合成一套能在浏览器中直接运行的实时通信平台。

    从开发者视角，WebRTC 提供了三个简洁的 JavaScript API（MediaStream、RTCPeerConnection、RTCDataChannel），屏蔽了 ICE、DTLS、SRTP、RTP、NetEQ 等大量底层复杂性。

    从架构视角，每一层都有其精确的职责划分：

    • VoiceEngine：从采集到发送的音频全链路处理
    • VideoEngine：从采集到发送的视频全链路处理
    • Transport/Session：NAT 穿透（ICE）、加密（DTLS/SRTP）、媒体传输（RTP/RTCP）、数据传输（SCTP）
