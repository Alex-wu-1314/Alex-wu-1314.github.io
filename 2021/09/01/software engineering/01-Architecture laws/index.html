    软件工程之架构篇：系统运行的客观规律

    Alex Wu
    2021-09-01
    software engineering

    引言

    软件架构的本质是在不确定性与约束之间做出取舍。我们总希望系统是可控的，API 有明确的契约，抽象能完美隐藏细节，分布式环境像单机一样可靠。但现实反复证明，这些期望都是幻觉。

    下面九条定律，从不同维度揭示了系统运行的客观规律。它们不是教条，而是无数工程师在构建和运维复杂系统时反复验证的经验。理解它们，不是为了找到完美的解决方案，因为完美并不存在，而是为了在面对不可避免的取舍时，做出清醒的选择。

    Hyrum’s Law — 当用户足够多时，所有可观察行为都会被依赖

    当 API 的用户足够多时，你承诺什么并不重要：系统的所有可观察行为都会被某些人所依赖。

    Overview

    Hyrum’s Law 描述的是一种在实践中反复出现的现象：软件文档化接口与实现细节之间的边界会逐渐模糊。不论你在 API 契约中承诺了什么，只要使用的人足够多，他们就会依赖那些你从未正式支持的行为。随着时间的推移，实现本身就变成了接口，每一个可观察的特性、性能表现、错误码，都成了某个用户工作流的基石。这极大地限制了系统演进的自由度，因为哪怕是最”内部”的改动，也可能破坏那些依赖旧行为的人。

    这条定律由 Google 工程师 Hyrum Wright 在 2011-2012 年左右首次提出。他注意到，即使是对 Google 核心库的改动，理论上不应产生可见影响，也总会破坏某个项目，因为那个项目依赖了未文档化的行为。他的同事 Titus Winters 随后将这个观察命名为”Hyrum’s Law”。

    Examples

    Microsoft Windows 的兼容性负担：Windows 系统中有许多未文档化的行为和 Bug，大量第三方应用实际上依赖它们运行。当微软在更新中修复或改变这些行为时，那些应用就会崩溃。为此，微软不得不反复回退修改，或刻意维持这些”怪异”行为以保证兼容性。

    API 返回顺序的隐性契约：某个库的文档声明返回的是无序列表，但实际上它返回的是有序列表。一些用户开始依赖这个有序行为。当库在后续版本中真的改为无序时，那些用户的代码就会出问题。

    API 版本控制的困境：即使是最小版本号的变更，从某些用户的角度看也可能是破坏性变更。异步属性替换同步属性、错误信息的措辞调整、响应时间的变化，任何可观察到的差异都可能成为隐性的依赖点。

    Takeaways

    1. 随着用户数量的增长，系统的每一个行为都会变成依赖点。意料之外的副作用或 Bug 也可能成为某些人工作流中的关键”特性”。
    2. Hyrum’s Law 提醒维护者：任何改动都可能破坏某人的系统。消费者可能以你预期不到的方式集成了你的 API，包括依赖时序、错误消息、格式化等细节。
    3. 你的软件的实际契约不仅是官方 API 或规格说明，而是在现实中被观察到的精确行为。这个契约甚至可以是某种非正式的约定，比如用户已经习惯的 UI 交互方式。

    Gall’s Law — 复杂系统都从可运行的简单系统演化而来

    一个能运行的复杂系统，毫无例外地是从一个能运行的简单系统演化而来的。

    Overview

    John Gall 观察到，能成功运行的复杂系统，无一例外是从成功的简单系统起步的。如果你试图从一开始就创建一个复杂系统，它通常不会工作，因为有太多尚未验证的未知因素。正确的做法是，从一个简单的架构开始，在真实世界的使用中学习，然后逐步增加能力或复杂度，确保每一步变更仍然工作。

    当你从一个小型、可运行的核心开始，你就能尽早验证假设。每一次新能力的增量添加都会在现实中被测试，迫使系统不断适应。建立最小可行产品（MVP），一个简单的可运行内核，然后迭代增长，而不是采用一次性的”大爆炸”式交付，是这条定律最直接的工程实践指导。

    Examples

    Facebook 的演化路径：Facebook 最初只是一个面向哈佛学生的简单网站，本质上是一个基础的用户资料系统。它在那个小规模下能够运行，然后逐步扩展，添加功能，承载更多用户。如果 Mark Zuckerberg 在 2004 年试图从零直接构建今天的 Facebook，它几乎必然会被自身的复杂性压垮。

    微服务的起点建议：在微服务时代，受 Gall’s Law 影响的一条常见建议是：不要从微服务开始。先构建一个能工作的单体应用（更简单的架构），随着系统增长，再识别出可以拆分为独立服务的模块。这些模块因为有过在单体架构中存在和运行的历史，其需求更明确，隔离后成功运行的概率也更高。

    Takeaways

    1. 不要试图从零构建复杂系统。先做一个能运行的简单版本，然后迭代并增加复杂度。这种方式为实验和创新提供了发射平台。
    2. 成功的大型系统通常是有机生长的。一个从头开始全面设计的复杂系统常常失败，因为你无法预见所有交互。
    3. 逐步生长的系统能更好地适应和处理复杂性，因为它们在每个阶段都经过了测试和打磨。过度复杂的初始设计在遇到意外情况时容易崩溃。

    The Law of Leaky Abstractions — 所有非平凡抽象都是有漏洞的

    所有非平凡的抽象，在某种程度上，都是有漏洞的。

    Overview

    这条定律认为，在复杂系统中，我们为隐藏复杂性而创建的抽象层，最终不可避免地会在某些场景下失效，将底层的复杂性暴露给程序员。这并不是说抽象不好，正相反，抽象是必不可少的。但定律提醒我们，必须做好准备，迎接它们失效的时刻。

    Joel Spolsky 在 2002 年的一篇博客文章中提出了这条定律。他以 TCP 为例做了经典阐释：TCP 在不可靠的 IP 上抽象出可靠连接，但在网络状况糟糕时，这个抽象会泄露，导致超时。

    Examples

    ORM 的”干净代码”陷阱：对象关系映射（ORM）让你可以将数据库记录当作普通对象操作。大多数时候它工作得很好，但偶尔你会遇到性能极差的场景，不得不去思考底层生成的 SQL 查询究竟是什么。抽象层”泄露”了数据库的内部细节。

    垃圾回收的抽象漏洞：Java 和 Python 等高级语言通过垃圾回收抽象掉了手动内存管理。但内存泄露仍然会发生（由于对象持有引用无法释放），你也会遇到 GC 暂停影响性能的情况。开发者突然需要了解垃圾回收的内部运作原理，”无限内存”的抽象失效了。

    网络的不可靠性泄露：Web 开发者通常将 HTTP 请求看作一个快速的操作，抽象掉了网络细节。然而，当延迟飙升或数据包丢失时，网络的现实就会透过抽象层渗入你的应用。

    Takeaways

    1. 无论抽象层设计得多好（库、框架等），总会有边界情况需要依赖内部细节才能处理。
    2. 开发者需要明白，使用高级工具并不代表可以完全不了解底层发生的事情，至少应当掌握基础知识。
    3. “泄露”意味着你可能会遇到性能问题、Bug 或迫使你关注底层系统（如网络、OS）的行为。
    4. 在创建抽象层时，应当尽量最小化泄露，并文档化可能失效的场景。

    Tesler’s Law (Conservation of Complexity) — 复杂度只能转移，不能消除

    每个应用都有一份固有的、不可简化的复杂度，这份复杂度只能被转移，不能被消除。

    Overview

    Tesler’s Law，也被称为复杂度守恒定律，指出对于任何系统而言，都存在一定量的复杂度是无法被消除的，它内在于问题本身。关键在于由谁来处理它。如果你设计了让用户使用非常简单的软件，很可能意味着软件内部在做大量复杂的工作。如果你让软件内部极为简单，它可能就把复杂度倾倒在了用户身上（比如需要大量手工操作）。Tesler’s Law 鼓励设计者尽可能将复杂度从用户体验中移开。

    这条定律的命名来自 Larry Tesler，他曾参与 Apple Lisa 和早期 GUI 概念的工作，在 1980 年代提出了这一原则。他观察到，你不可能让一切都很简单，却没有人来承担复杂度，这是设计中的一个根本性取舍。

    Examples

    会议排期工具：安排会议这件事本质上就很复杂（需要找到共同可用时间）。Calendly 等工具将这种复杂度隐藏在用户界面背后（软件代劳），而通过邮件链来约定时间则将复杂度完全放在了用户身上，让用户自己去摸索。两种方案都面对同一份固有复杂度，区别只在于谁承担它。

    数据库与应用的取舍：设计数据库时，有时将逻辑下推到数据库（如存储过程）可以简化应用代码，但增加了数据库层的复杂度；反过来则将复杂度留在应用层。复杂度没有消失，只是被重新分配了。

    Takeaways

    1. 好的设计通常要求开发者吸收大部分复杂度（通过智能默认设置、算法等手段），让用户的交互更简单。
    2. 如果你的 UI 要求用户管理大量设置或步骤，那你把复杂度守恒在了错误的地方（用户侧）。让产品内部管理复杂性通常是更好的选择。
    3. 好的设计通过内部消化复杂性来隐藏它，而不是消除它。

    CAP Theorem — 分布式系统的三选二困境

    一个分布式系统只能同时保证一致性、可用性和分区容错性这三者中的两个。

    Overview

    CAP 定理指出，分布式系统无法同时提供三项保证：一致性（所有节点看到相同的数据）、可用性（每个请求都能收到响应）和分区容错性（系统在网络故障时仍能运行）。在实践中，网络分区不可避免，因此系统必须是分区容错的。这意味着在设计分布式架构时必须选择：当网络分区发生时，是优先保障一致性还是可用性。

    CAP 定理是一个有效的思维起点，虽然它是一个简化的模型，并不能覆盖设计空间中的所有维度。这个定理由 Eric Brewer 在 2000 年提出，后来 Gilbert 和 Lynch 在 2002 年给出了严格的形式化证明。

    Examples

    DNS：AP 系统：域名系统被设计为 AP（可用且分区容错）。即使某些名称服务器被网络隔离，它们仍然会回复请求（可用性），哪怕这些信息在区域同步完成前可能略有滞后。

    MongoDB：CP 系统：CP（一致且分区容错）数据库，在网络分区期间会阻塞写操作，以保证所有副本保持一致。

    Cassandra：AP 系统：AP（可用且分区容错）数据库，即使副本之间暂时出现分歧，也会继续响应查询。

    Takeaways

    1. 一个分布式系统只能同时保证三者中的两个。当网络健康时你可以三者兼顾，但一旦发生网络分区，你就必须放弃其中一个。
    2. 当网络分裂发生时，你面临的选择是：保持一致性（所有节点数据一致，但部分请求可能失败），或保持可用性（所有请求都有应答，但数据可能略微过期）。两者无法兼得。
    3. 实际的数据库产品都需要站队。MongoDB 倾向于一致性（分区时阻塞写入以保证所有副本同步），Cassandra 倾向于可用性（即使副本出现短暂分歧也保持响应）。

    Second-System Effect — 第二个系统总是过度设计的

    小型、成功的系统之后，往往会跟着一个过度工程化、臃肿的替代品。

    Overview

    一个小而精的系统被成功构建并运行良好。然后，第二个版本的项目启动了，设计者从第一个项目的各种约束中解放出来，试图构建一个规模更大、更复杂的系统。他们会添加各种增强功能，考虑所有边界情况，集成所有心愿清单上的特性。结果往往是一个过于复杂且进度严重滞后的系统。

    Fred Brooks 在 IBM 目睹了这一现象：IBM OS/360 之前有一些简单但成功的操作系统，而 OS/360 本身由于试图做太多事而陷入延迟和困境。他在 1975 年的《人月神话》中创造了这个词。

    Examples

    创业公司从 V1 到 V2 的陷阱：一个创业公司构建了一个极简的 Web 服务，取得了巨大成功。对于版本 2，团队计划全面重写，增加微服务架构、大量配置选项、插件框架和用户从未要求过的功能。项目的复杂度急剧膨胀，交付日期一推再推。

    Netscape 的陨落：1990 年代的 Netscape Navigator 在早期大获成功后，决定重写他们的浏览器，最终推出了 Mozilla Suite（Netscape Communicator）。重写耗时数年，在此期间 Internet Explorer 迎头赶上。新的套件被认为过于复杂，Netscape 丧失了其领先地位。

    Takeaways

    1. 第一个系统因为资源约束或经验不足而保持精简。设计其继任者时，人们倾向于一股脑地加入最初被舍弃的所有特性，导致功能膨胀。
    2. 过度自信在其中推波助澜。有了第一次成功，团队认为自己能处理更大的目标，却低估了复杂性。
    3. 第二系统效应常常导致过度工程化，更多的模块、更泛化的设计、超出必要的功能，最终损害了性能和可维护性。

    Fallacies of Distributed Computing — 分布式系统的八大谬误

    一组新入行的分布式系统设计师常犯的八个错误假设。

    • The network is reliable: 网络是可靠的，认为网络传输永远不会丢包、出错、中断。
    • Latency is zero：延迟为零，认为数据在网络中传输不需要任何时间。
    • Bandwidth is infinite：带宽是无限的，认为网络可以无限制地传输数据，不会出现拥塞。
    • The network is secure：网络是安全的，认为网络不会被攻击、窃听或篡改。
    • Topology doesn’t change：网络拓扑不会变化，认为设备不会上下线、网络结构不会发生变化。
    • There is one administrator：只有一个管理员，认为整个网络由一个人 / 一个机构完全控制和管理。
    • Transport cost is zero：传输成本为零，认为网络传输数据不需要任何成本（流量、硬件、运维等）。
    • The network is homogeneous：网络是同构的，认为网络中的所有设备、协议、技术都是统一标准的。

    Overview

    分布式计算的八大谬误是一份”不要假设什么”的检查清单。它们都源自将分布式系统当作本地系统来对待的思维。开发者可能写出调用远程服务就像调用本地函数一样的代码，忽略了延迟和失败。这些错误假设会导致严重问题：网络断开时未处理的错误、由于忽略延迟导致的低性能、未能对远程调用进行认证引发的安全漏洞。

    这些谬误由 Sun Microsystems 的 L. Peter Deutsch 在 1994 年左右提出（James Gosling 等人后来增补），最初是七条，Gosling 后来补充了第八条（”网络是同构的”）。Deutsch 观察到，许多工程师，尤其是习惯了本地计算的工程师，潜意识里会假设网络总是可靠的。

    Examples

    分布式缓存的延迟陷阱：开发者构建分布式缓存系统时假设”网络延迟为零”，于是设计缓存每次查询都从远程节点获取数据。实际运行中，系统因高延迟而性能糟糕。

    忽视安全的代价：系统假设”网络是安全的”，在服务间发送未加密的敏感数据，或不对来自其他服务的输入进行验证。一旦网络被攻破，就会造成安全事件。

    拓扑变化的冲击：没有为拓扑变化做规划的系统，在机器意外增减时往往会崩溃。

    Takeaways

    1. 网络会丢包、有延迟、带宽有限、而且可能不安全。一个正确构建的分布式系统必须通过重试、超时、安全措施和动态发现来应对这些问题。
    2. 这些谬误常常以微妙的 Bug 形式表现出来。假设延迟为零可能导致在本地工作良好的频繁远程调用，在网络上变得痛苦地缓慢。
    3. 将这些谬误纳入考量，意味着采用防御性设计：使用缓存（带宽和延迟不是完美的），构建冗余（网络不可靠），并处理动态成员关系（拓扑会变化）。

    Law of Unintended Consequences — 改变复杂系统，准备好迎接意外

    每当你改变一个复杂系统，准备好迎接意外。

    Overview

    意外后果定律指出，结果并不完全可预测。系统具有复杂的相互依赖关系和人的因素，这可能导致各种意外。添加一个新功能可能因为与某个不相关模块的交互而意外降低性能。简化 UI 可能导致后端负载飙升，因为用户比预期更频繁地使用该功能。它提醒工程师，系统是复杂的，而我们的心智模型是不完整的；在实施变更时，需要预见到”未知的未知”会浮现出来。

    社会学家 Robert K. Merton 在 20 世纪普及了这个术语，他识别了意外后果的五个来源：无知、错误、即时利益、基本价值观，以及自我挫败的预言。

    Examples

    日志功能的意外灾难：启用一个新的日志功能用于调试，结果因为日志迅速写满磁盘导致系统崩溃，一个与”提升稳定性”这一初衷截然相反的悖谬结果。

    算法的连锁反应：社交网络调整算法以提升用户参与度，但意外地放大了愤怒情绪和错误信息。

    Bug 修复引发回归：修复某个模块的 Bug，却意外导致另一个模块出现回归问题，因为那个模块恰好依赖了被修复的 Bug 行为。

    Takeaways

    1. 无论你规划得多周密，对复杂系统的任何重大变更都可能产生你无法预料的结果。
    2. 后果可能是意外之喜（惊喜）、意外之弊（副作用），或悖谬之果（行为反而让原始问题更加恶化）。
    3. 在软件工程中，这常常表现为一次修复或新功能在别处引入了 Bug 或性能问题。

    Zawinski’s Law — 每个程序都试图扩张，直到它能收发邮件

    每个程序都试图不断扩张，直到它能读取邮件。那些不能如此扩张的，终将被那些能扩张的程序所取代。

    Overview

    Zawinski’s Law 是一条半玩笑式的观察，描述了软件演化中的功能蔓延现象：应用持续不断获取新功能，直到它能做一切事情，甚至包括那些完全超出其原始范畴的事。随着应用吸引更多用户，它会面临越来越大的压力去增加更多能力。一个基础的笔记应用后来可能加入聊天或分享功能。”读取邮件”是 Zawinski 挑选的典型例子，因为在 90 年代中期，用户常常需要退出当前应用再单独打开邮件程序。

    Jamie Zawinski（也称 jwz）在 1995 年前后于 Netscape 工作期间提出了这条定律。他是 Netscape Navigator 的核心程序员之一，后来为浏览器添加了集成的邮件阅读器。

    Examples

    Netscape 自身的扩张史：从一个轻量浏览器成长为 Netscape Communicator，一个集浏览器、邮件客户端、新闻组阅读器和网页编辑器于一体的庞大套件。它变得臃肿缓慢，最终为 Firefox 的崛起铺平了道路，后者刻意回归到一个纯粹快速的浏览器。不过有趣的是，Firefox 后来也因插件和主题变得越来越重。

    Slack 的平台化：Slack 起初的口号是”杀死邮件”，但现在集成了语音通话、视频会议、文件共享、Bot 和应用插件。它现在试图成为一个一站式的 workplace hub，远远超出了简单的即时通讯。

    GitHub 的不断扩展：从最初托管代码，扩展到 Issue 跟踪、Wiki、项目看板、Discussion、CI 流水线和 Package Registry。

    Takeaways

    1. 功能蔓延无法避免。软件会不断积累更多功能，导致软件臃肿。
    2. 一个精干、极简的应用一旦开始流行，就会持续添加功能，直到变得和竞品一样复杂。
    3. 程序会扩张，因为用户（以及产品经理）在不断要求”再多一个功能”。持续的压力来自为阻止用户流失而必须加入流行的能力。
    4. 每个新功能都会增加复杂度，让产品对用户来说更令人困惑。开发者应当守护工具的核心焦点，抵制平台化蔓延。

    结语：架构的艺术在于取舍

    回顾这九条定律，一个贯穿始终的主题是：软件架构中没有银弹，只有取舍。

    • Hyrum’s Law 告诉我们任何改动都有代价；
    • Gall’s Law 揭示了复杂必须逐步生长；
    • Leaky Abstractions 戳破了”完全透明”的幻象；
    • Tesler’s Law 指出复杂度永不会消失，只能重新分配；
    • CAP Theorem 在分布式系统的根本层面上画出了物理边界；
    • Second-System Effect 警示了傲慢的代价；
    • Fallacies of Distributed Computing 列出了我们对网络最荒谬的幻想；
    • Law of Unintended Consequences 提醒我们永远无法完全预见结果；
    • Zawinski’s Law 则用幽默揭示了熵增在软件开发中的真实面貌。

    理解这些定律，并不意味着能避免它们，许多定律本身就是不可规避的客观限制。但当面对架构决策、技术选型、系统演化时，它们会成为手中的罗盘，帮助在迷雾中看清哪些路是走不通的，哪些代价是必须承受的，哪些取舍是值得做的。而这，正是架构师真正的修养所在。
