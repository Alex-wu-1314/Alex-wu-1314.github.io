    软件工程之规划篇：关于进度、时间与目标的博弈

    Alex Wu
    2021-09-27
    software engineering

    引言

    为什么我们明明预留了缓冲时间，项目却依然延期？为什么一个看似快完成的项目，最后总是陷入漫长的收尾阶段？为什么一旦制定了 KPI，团队的行为就会开始变得诡异？

    这一切的背后，其实都藏着客观规律。下面六条规律是关于“规划、时间与目标”的经典软件工程定律，理解它们，将不再对失控的项目进度感到困惑，并学会如何在不确定性的博弈中找到平衡。

    1. Premature Optimization (Knuth’s Principle) —— 过早优化是万恶之源

    Overview

    Premature optimization is the root of all evil.

    — Donald Knuth

    Donald Knuth 的这句名言可能是软件开发领域被引用最多、也被误用最多的一句话。它的核心含义是：在尚未明确性能瓶颈之前，就为了追求效率而牺牲代码的清晰性和正确性，是极其危险的。 根据 Knuth 的原意，开发者应该“忘记”约 97% 场景下的微小效率问题，将精力聚焦于清晰的设计和正确的功能实现。

    过早优化的代价并非仅仅是浪费了优化本身的时间，更糟的是它会显著增加代码的复杂度与可读性负担。正如原作者指出的，优化过的代码通常更复杂、更难读，如果这些优化发生在一个最终被证明不是性能热点的模块上，你就是在为一个根本不需要的成本买单。

    扩展阅读：类似关于保持简单与克制的理念，我们在后续的 《设计篇》 中还会探讨 KISS 原则和 YAGNI 原则，它们与 Knuth 的观点一脉相承。

    Examples

    一位开发者花了整整两天时间，用底层 C 语言编写了一个充满精巧位操作的高性能例程。最终性能分析显示，这个函数在整个程序生命周期中只被调用了一次（启动时），消耗了总运行时间的 0.001%。两天的时间被白白浪费，而代码库中却多了一段难以理解的复杂逻辑。

    另一个典型案例是，开发者在数据量极小的场景下，为了理论上的 O(log N) 查找效率而使用了自定义树结构，然而实际数据规模可能只需要一次简单的线性遍历就足够快。正确的做法是：先用最简单的方式实现，通过性能剖析找到真正的热点（比如那个消耗 80% 时间的循环），然后集中精力优化那个部分，而不是凭直觉四处“预优化”。

    Takeaways

    1. 关注真正的瓶颈：绝大多数代码并不在性能关键路径上，执着于全局微优化浪费时间并增加维护成本。
    2. 97% 法则：Knuth 建议，应忘记约 97% 场景下的微小效率问题，专注于清晰的设计和正确的功能。
    3. 复杂度账单：优化后的代码通常更复杂或更难读。如果过早优化，即使最终这些优化并非必要，复杂度成本也已经付出。
    4. 正确的顺序：先让代码正确运行（Make it work），再让它变快（Make it fast），最后才是让它变漂亮（Make it pretty）。

    2. Parkinson’s Law —— 工作会膨胀以填满所有可用时间

    Overview

    Work expands to fill the time available for its completion.

    — Cyril Northcote Parkinson

    这是一条跨越管理学的经典定律。对于软件开发者而言，这意味着：如果你给一个开发者两周时间去做一个两天就能完成的任务，这个任务往往会不紧不慢地拖到两周才结束。 多余的时间并不会被用来进行更深层次的思考，而是被用于“镀金”，过度打磨一些无关紧要的细节、反复修改实现方案、或者单纯地拖延到截止日期前才开始动手。

    需要注意的是，Parkinson’s Law在实际中常与Hofstadter’s Law形成一种微妙的张力：如果你将时间压缩得太紧，会触发Hofstadter’s Law的延迟诅咒；但如果放得太松，Parkinson’s Law就会让团队效率低迷。

    延伸思考：与Parkinson’s Law密切相关的，是我们在 《团队篇》 中会讲到的 Ringelmann 效应（社会惰化），即随着团队规模扩大，个人产出反而下降。过松的时间线正是加剧这种惰化的温床。

    Examples

    当一个开发者被通知“这个模块不急，大概一两个月做完就行”，他大概率不会提前完成，而是会不自觉地放慢节奏，甚至花大量时间去尝试多种实现方式、或者在命名和代码格式化等非核心事项上反复斟酌。但如果同样的任务被设定了一个明确且紧张的一周截止日（配合敏捷 Sprint 的时间盒机制），开发者往往刚好能在一周内交付。

    另一个生动的场景：一场会议如果预定了全天的时间，它往往会真的占满一整天；但如果同样的议程被限制在两个小时，参与者反而能集中注意力高效完成讨论。

    Takeaways

    1. 时间膨胀效应：如果任务被给予过长的时间线，人们倾向于将所有时间消耗殆尽（或拖延到最后一刻）。
    2. 警惕“镀金”：多余的时间往往被用于过度打磨，添加并非严格必要的“锦上添花”式改进。
    3. 用“紧迫感”反制：适当收紧但符合现实的截止日期，通过“截止日期驱动开发”（Deadline-Driven Development）来抵消时间膨胀。

    3. The Ninety-Ninety Rule —— 前 90% 的代码用 90% 的时间，后 10% 也用 90%

    Overview

    The first 90% of the code accounts for the first 90% of development time. The remaining 10% accounts for the other 90%.

    — Tom Cargill, Bell Labs

    这是一条带有黑色幽默却直击灵魂的定律。它指出了软件开发中一种普遍存在的“乐观偏差”：当核心功能跑通时，所有人都以为项目即将收尾；但实际上，那些剩余的集成联调、边界情况处理、性能调优和 Bug 修复，往往会消耗与前期开发同样多，甚至更多的时间。 原作者特意强调，“基本完成”只是时间意义上的真正过半，离交付还差得远。

    令人惊讶的是，这条定律与中国古话“行百里者半于九十”在哲理上高度一致，一百里的路程，即使走完了九十里，也只能算作才走了一半。软件工程中，最后 10% 的工作（真正的打磨与稳定）往往最消磨士气。

    Examples

    一个典型场景：团队花了三个月完成了新 App 的所有核心功能，所有人都认为“已经完成了 90%”，乐观地预估一个月后就能发布。然而，集成测试阶段暴露了大量模块间通讯问题、特定边界输入导致崩溃、内存占用过高等。最终，这个“最后的 10%”又花了整整三个月。

    在 Web 开发中同样如此：基本页面和登录功能可能两天就能搭好，但跨浏览器兼容性修复、响应式设计适配、无障碍访问改进以及性能优化，这些看起来“顺便就能搞定”的收尾工作，往往会再消耗与主体开发等量甚至更长的时间。

    Takeaways

    1. 最后一步远超预期：项目的最后部分（打磨、边界情况、集成、Bug 修复）所花精力往往远超预期，可能与前期开发总量相当。
    2. “基本完成”是错觉：表面上的收尾工作，包含大量棘手问题和未知因素。
    3. “快做完了”≈ 才做了一半：达到“基本完成”的状态容易滋生乐观情绪，但这条法则警告我们，宣称 90% 完成度在时间上往往只意味着真正过半。

    4. Hofstadter’s Law —— 永远比你预期的时间长，即使考虑了这条定律本身

    Overview

    It always takes longer than you expect, even when you take into account Hofstadter’s Law.

    — Douglas Hofstadter

    这是一条带有递归幽默的定律，出自侯世达 1979 年的著作《哥德尔、埃舍尔、巴赫：集异璧之大成》。它所揭示的真相是：人类在估算复杂创造性工作时天生不擅长，而软件开发的复杂度恰恰在于，实施过程中永远存在预先不可见的“未知之未知”。 即使你已经意识到自己是个糟糕的估算者，并且为计划预留了缓冲时间，现实往往还是会超出这个缓冲。

    原网站特别指出，这条定律与Parkinson’s Law构成了一对危险的互补关系，如果时间线太紧，Hofstadter’s Law会惩罚你；如果时间线太松，Parkinson’s Law会消磨你。

    Examples

    一个团队为新功能计划了一个月的工期，因为有经验的项目经理特意将预估改为六周以“考虑意外”。结果三个月过去了，功能仍然没有上线。原因是什么？第三方 API 的集成比想象中困难得多，核心开发者在关键时刻生病请假，而且最初的技术方案被证明行不通，必须推倒重来。

    原作者给出的经验法则是：如果一个任务号称 2 分钟完成，那就当即动手；如果需要几分钟，按 1 小时来预估；如果需要几小时，按 1 天来预估。

    Takeaways

    1. 人类天生不擅长估算：即便已经意识到自己不擅长估算，我们仍然会系统性低估任务耗时。
    2. 未知之未知：实施过程中总会出现事先无法预见的复杂性和未知因素，不断拉长时间线。
    3. 缓冲也会被耗尽：尽管应该在日程中加入应急缓冲，但即便是缓冲时间也常常不够用。
    4. 不要过度乐观：承认延迟必然发生，并将其纳入项目时间线与期望管理之中。

    5. Goodhart’s Law —— 当一个指标变成目标，它就不再是好指标

    Overview

    When a measure becomes a target, it ceases to be a good measure.

    — Charles Goodhart

    这条出自经济学家查尔斯·古德哈特的定律，在软件工程领域同样振聋发聩。它的本质含义是：一旦你将某个度量指标设为团队追求的目标，人们就会找到各种办法去“优化”这个指标本身，而不是去优化指标本应反映的真实价值。其结果是，指标与真实价值之间的关联被彻底切断。

    在软件开发管理中，这是一个极其常见的陷阱。管理者渴望“数据驱动”，于是设置了代码行数、故事点数、Bug 关闭量、代码覆盖率等指标作为考核目标。然而，度量指标只是你真正关心的事物的代理变量，一旦把它当成目标，团队就会开始以牺牲真实目标为代价来追求这个代理变量。

    Examples

    一个经典的案例是：管理层将“代码行数”作为绩效衡量标准，结果开发者开始大量编写冗长的啰嗦代码，甚至将同一段代码复制粘贴到多个文件中，以便让数据更好看。最终，代码库变得臃肿不堪，而度量指标却一片大好。

    另一场景：QA 团队以“执行的测试用例数量”为考核目标，于是测试人员倾向于执行大量简单但价值不高的测试，而真正需要深入验证的核心场景反而被跳过了。同样，当代码覆盖率成为硬性指标后，开发者编写了大量不包含实质性断言的“空壳”单元测试，以满足覆盖率门槛。

    Takeaways

    1. 指标是代理变量：度量指标只是你真正关心的事物（生产力、质量等）的代理变量，而非事物本身。
    2. 指标一旦成为目标就会被操纵：一旦将特定指标设为考核目标，人们便会设法在该指标上取得好成绩，哪怕损害原初目的。
    3. 度量是洞察工具，不是目标本身：指标应用于洞察和趋势分析，必须结合上下文与定性判断综合使用。
    4. 组合使用多维度指标：多个指标的组合可以避免因单一焦点而被操纵。

    6. Gilb’s Law —— 任何需要量化的东西都能被测量

    Overview

    Anything you need to quantify can be measured in some way better than not measuring it.

    — Tom Gilb

    Goodhart’s Law可能会让人产生一种错觉：既然度量指标会被操纵，那我们干脆别度量了。Gilb’s Law正是对这类“度量虚无主义”的有力回击。这条定律的核心思想是：即使是一个粗略的、间接的度量，也比完全没有任何数据反馈要好，只要你清楚意识到这个度量的局限性。

    原作者强调，哪怕是对一个模糊的概念，比如“代码质量”，你仍然可以找到一系列间接指标（圈复杂度、Lint 警告数、缺陷率等）作为部分反映它的晴雨表。 这些度量数据单独拿出来确实不完美，但组合在一起就能描绘出一个大致的图景，为改进提供起点。

    与 《质量篇》 的呼应：技术债务如果没有被量化追踪，就只是模糊的抱怨而已。当你开始统计圈复杂度上升趋势或缺陷率变化，债务才有了可见度，这正是后续 Technical Debt 一节的核心观点，也与 Gilb 定律中“先测量再改进”的哲学高度一致。

    Examples

    “开发者生产力”向来被视为最难度量的指标之一（代码行数不准确，故事点数也存在主观不一致）。但你可以使用部署频率和变更前置时间这些来自 DORA（DevOps 研究与评估）体系的代理指标。它们固然无法全面反映生产力，但当部署频率突然下降时，你至少能意识到流水线可能出了问题，这比一无所知要好得多。

    类似地，追踪技术债务也没有完美的度量方法，但将代码复杂度分数、线上事故率、开发者满意度调查等指标放在一起追踪，就能提供你单凭直觉无法获得的客观可见性。正如彼得·德鲁克所言：“我们无法改进我们没有度量的东西。”

    Takeaways

    1. 有数据优于没数据：对某个现象拥有一定的数据或度量指标，总比完全盲目要好，前提是你能理解该指标的局限性。
    2. 不要因为不完美而放弃度量：与Goodhart’s Law形成互补——古德哈特警告指标的滥用，吉尔布则提醒不应因噎废食。
    3. 先起步，再迭代：从一个基础度量开始，在使用过程中持续完善。度量的行为本身就能帮助团队聚焦和识别趋势。
    4. 近似测量好过零测量：即便是近似的或间接的测量，也比完全不测量要好。

    结语

    这六条定律并非孤立存在，而是构成了一张互相制衡的网络，理解它们之间的动态关系，才能真正做好软件项目规划：

    张力关系 定律 A 定律 B 核心矛盾
    时间估算的“松紧悖论” Parkinson’s Law Hofstadter’s Law 时间太松则膨胀浪费，太紧则必然延期的递归困境
    度量的“进退两难” Goodhart’s Law Gilb’s Law 度量会被操纵，但不度量等于盲目飞行
    效率投入的“优先级陷阱” 过早优化 九十-九十法则 前期在非瓶颈上过度优化，后期真正的瓶颈却得不到资源

    核心启示：

    1. 接受不确定性，但不要放弃规划。 Hofstadter’s Law和Parkinson’s Law告诉我们，完美预测是不可能的，但合理的约束（如敏捷中的时间盒）可以显著减少无效的时间膨胀。
    2. 度量是手段，不是目的。 遵循Gilb’s Law去建立数据意识，同时用Goodhart’s Law时刻审视你的指标是否已经“异化”。永远用多维度指标和定性判断来补充定量数据。
    3. 先做对，再做快，最后才做美。 Knuth 和九十-九十法则共同指向了一个结论，软件工程的最大敌人往往是“过早的完美主义”和“对收尾难度的低估”。把有限的精力留给剖析后发现的真正瓶颈，而不是凭直觉四处救火。
