    1. 概述与背景

    实时传输协议（RTP）负责将媒体数据从发送端搬运到接收端，但它本身并不提供任何质量反馈机制，它既不知道接收端收到了多少包，也不清楚网络的抖动和时延状况。为此，IETF 在同一份规范里为 RTP 配套设计了 RTCP（RTP Control Protocol，RTP 控制协议），二者共同定义于 RFC 3550（2003 年，废止并替代了 1996 年的 RFC 1889）。

    RTCP 是一个带外（out-of-band）控制协议，它自身不传输任何媒体数据，只传输关于会话质量的统计信息与控制指令。从架构上看，每一路 RTP 媒体流都应该有一路并行的 RTCP 控制流与之对应，构成一个完整的”媒体传输 + 质量反馈”闭环。

    设计哲学：RTCP 的带宽消耗被严格限制在整个会话带宽的 5% 以内（其中 25% 的 RTCP 带宽保留给发送方），并通过动态调整发送间隔来适应不同规模的会话，确保在数千人的多播会话中协议本身不会成为网络瓶颈。

    2. RTCP 的核心职责

    RFC 3550 §6 定义了 RTCP 的四大核心功能：

    2.1 传输质量反馈（QoS Feedback）

    这是 RTCP 最主要的职责。接收端周期性地向发送端汇报丢包率、累计丢包数、到达间隔抖动（interarrival jitter）等统计数据，发送端可以据此做出自适应调整，例如：

    • 降低编码码率；
    • 切换编解码器；
    • 触发拥塞控制算法（如 GCC）。

    2.2 持久标识符（CNAME）

    RTP 使用 SSRC（Synchronization Source Identifier）来区分不同的媒体流，但 SSRC 可能在会话过程中发生冲突或重分配。RTCP 的 SDES（Source Description） 报文携带了 CNAME（Canonical End-Point Identifier），为每个端点提供一个在整个应用实例生命周期内唯一的持久标识符。

    CNAME 还是多路媒体流（音频、视频）做唇音同步（lip-sync） 的关键：接收端通过 CNAME 将不同的 RTP 流关联到同一个端点，再结合 SR 中的 NTP 时间戳与 RTP 时间戳的对应关系，实现精确的音视频同步。

    2.3 会话成员数量估计（Session Size Estimation）

    由于所有参与者都发送 RTCP 包，每个端点都可以通过监听来独立估计当前会话的规模，进而动态调整自己的 RTCP 发送间隔，防止大型多播会话被 RTCP 流量淹没。

    2.4 最小化会话控制（Minimal Session Control）

    这是可选功能，例如通过 BYE 报文通知其他参与者某个源已离开会话，或通过 APP 报文传递应用特定的控制信息。

    所有 RTCP 报文都以一个固定的 4 字节公共头开始（RFC 3550 §6.1）：

    各字段含义如下：

    字段 位宽 说明
    V（Version） 2 固定为 2，标识 RTP/RTCP 版本
    P（Padding） 1 若为 1，报文末尾有填充字节（最后一个字节指示填充数量），通常用于加密对齐

    5. 复合 RTCP 包（Compound RTCP Packet）

    RFC 3550 §6.1 明确规定，RTCP 报文必须以复合包（compound packet） 的形式发送，即将多个 RTCP 单包拼接在同一个底层 UDP 数据报中，每个单包的边界由各自的 length 字段界定。

    为什么这样设计？ 三点原因：

    1. 让接收统计信息（SR/RR）尽量频繁地被发送，提高反馈分辨率。
    2. 新加入的接收端能尽快获取发送端的 CNAME，用于媒体同步。
    3. 将 SR/RR 置于首位可以提高包头校验成功率，有效过滤被误投的 RTP 数据包。

    RFC 4585 的例外：针对实时反馈场景（如立即请求关键帧），RFC 4585 引入了最小复合 RTCP 包（Minimal Compound RTCP），只需要包含 RR 和 SDES(CNAME) 即可，以减少反馈时延。

    6. 标准报文类型详解

    6.1 SR：Sender Report（发送方报告）

    PT = 200，适用于既发送也接收的参与者。SR 由两部分组成：Sender Info Block 和零到多个 Report Block。

    Sender Info Block 各字段说明：

    • NTP timestamp（64 bits）：发送该 SR 时的绝对挂钟时间（wall-clock time），格式为网络时间协议（NTP）时间戳，单位为自 1970-01-01 00:00 UTC 以来的秒数（高 32 位）和秒的小数部分（低 32 位，精度约 232 皮秒）。这是音视频同步（lipsync） 的基础。

    • RTP timestamp（32 bits）：与 NTP 时间戳时刻对应的 RTP 时间戳，接收端通过两者的映射关系可以将不同媒体流（音频、视频）的 RTP 时间戳换算到同一绝对时间轴，从而实现精确的唇音同步。

    • sender’s packet count（32 bits）：自会话开始（或 SSRC 最后一次改变）以来，本发送端发出的 RTP 数据包总数。接收端可用它与自身统计的收包数做差，得出丢包总量。

    • sender’s octet count（32 bits）：已发送的 RTP 有效负载字节总数（不含 RTP 头）。

    Report Block（接收报告块），SR 和 RR 共用同一结构，每个 Block 针对一路被接收的 RTP 流：

    各字段含义：

    字段 位宽 含义
    fraction lost 8 自上次 Report Block 以来的近期丢包率：floor(丢包数/期望包数 × 256)，值域 0–255，对应 0%–~100%
    cumulative packets lost 24 累计丢失包数（有符号，从会话开始统计）。序列号回绕等情况可能导致该值为负，上限为 0x7FFFFF
    extended highest seq 32 高 16 位为序列号循环计数，低 16 位为收到的最大序列号
    interarrival jitter 32 到达间隔抖动估计，单位为 RTP 时间戳单位（见第 9 节算法详解）
    last SR (LSR) 32 最近收到的 SR 中 NTP 时间戳的中间 32 位（高 16 位为秒数低 16 位，低 16 位为秒小数高 16 位）；未收到过 SR 则为 0
    delay since last SR (DLSR) 32 从收到最近一个 SR 到发送此 Report Block 之间的时延，单位为 1/65536 秒

    6.2 RR：Receiver Report（接收方报告）

    PT = 201，适用于只接收不发送的参与者，或在 SR 中 Report Block 数量超过 31 个时追加使用。结构比 SR 简单，去掉了 Sender Info Block，只有公共头和 Report Block 列表。

    6.3 SDES：Source Description（源描述）

    PT = 202，携带关于媒体源的文本描述信息。SDES 由一个 3 层结构组成：RTCP 头 → 一或多个 Chunk → 每个 Chunk 包含一或多个 Item。

    SDES Item 类型：

    类型码 名称 必须性 说明
    0 END 必须 Chunk 结束标记，后跟填充至 32 位对齐
    1 CNAME 强制 规范端点标识符，格式建议为 user@host；必须在每个复合包中携带
    2 NAME 可选 用户真实姓名
    3 EMAIL 可选 电子邮件地址
    4 PHONE 可选 电话号码
    5 LOC 可选 地理位置
    6 TOOL 可选 应用程序名称和版本
    7 NOTE 可选 状态注释
    8 PRIV 可选 私有扩展，前缀为应用标识前缀

    WebRTC 中 CNAME 的生成：RFC 8834 §4.9 规定，WebRTC 实现必须为 CNAME 使用足够随机的值（至少 96 bits 熵），以保证隐私性，防止被用于跨会话跟踪。

    6.4 BYE：离会通知

    PT = 203，当一个端点结束发送时，通知其他参与者移除对应的 SSRC 状态。

    规范指出，接收到 BYE 后不应立即删除该 SSRC 的状态，而应先打上标记，等待一段时间后再清理，以防止迟到的 RTP 数据包引发状态重建。

    6.5 APP：应用自定义

    PT = 204，为应用程序提供扩展 RTCP 的机制，包含一个 4 字节的 ASCII 应用名称和应用自定义的数据体。由于缺乏互操作标准，APP 在实践中使用较少，更规范的做法是定义独立的 RTCP 扩展报文类型（如 RFC 4585 的 RTPFB/PSFB）。

    RTCP 的发送间隔不是固定的，而是自适应的，其目的是将所有参与者的 RTCP 总带宽控制在会话带宽的 5% 以内。RFC 3550 §6.2 描述了完整算法，核心逻辑如下：

    为了防止所有参与者同时发送（synchronization）导致的短暂流量爆发，实际发送时间在 [0.5T, 1.5T] 区间内均匀随机化。这一机制被称为 “定时器重新考量（Timer Reconsideration）” 。

    7.4 Reverse Reconsideration

    当参与者数量突然减少时（例如大量 BYE 报文同时到来），每个参与者都应提前重新计算下次发送时间，避免发送间隔因为参与者数量减少而收窄，导致下一次发送时间变得合理但当前等待时间过长。

    7.5 WebRTC/RFC 4585 的简化

    在 WebRTC 等点对点或 SFU 场景中，参与者数量有限，复杂的自适应间隔算法意义不大。RFC 4585（AVPF）取消了强制的 5 秒最小间隔限制，允许根据网络反馈需要立即发送（Early RTCP） 反馈消息（但有防止反馈洪泛的退避机制）。

    9. 抖动（Jitter）的计算

    到达间隔抖动（interarrival jitter）是衡量网络延迟稳定性的关键指标。RFC 3550 Appendix A.8 给出了推荐的在线算法，采用指数加权滑动平均（非真正的标准差）：

    9.1 算法定义

    设：

    • Si：包 i 的 RTP 发送时间戳（单位：RTP 时间戳的 tick）
    • Ri：包 i 在接收端的实际到达时间（也转换为 RTP 时间戳单位）
    • Di：包 i 与包 i-1 之间的相对传输时延差

    这里 1/16 的权重系数使得算法对短时波动不过于敏感，同时能跟踪趋势变化。

    9.2 实现要点

    • 计算前需要将接收时间戳换算到与 RTP 时间戳相同的单位（例如，音频 48 kHz 流的 1 tick = 1/48000 秒）。
    • 序列号不连续（丢包）时，仍然基于实际到达的包对计算，跳过的包不参与计算。
    • 初始值 J(0) = 0。

    10. 扩展报文：RFC 4585 反馈消息（RTCP-FB）

    RFC 4585（Extended RTP Profile for RTCP-Based Feedback，RTP/AVPF）为 RTCP 引入了即时反馈消息机制，定义了两类新的报文类型：

    10.1 RTPFB（PT = 205）：传输层反馈

    这类反馈与具体的媒体编码格式无关，关注传输层事件：

    Generic NACK（FMT = 1）（RFC 4585）

    通知发送方哪些包丢失，请求重传：

    • PID（16 bits）：丢失包的起始序列号
    • BLP（16 bits）：位图，每一位对应 PID+1 到 PID+16，为 1 表示对应偏移的包也丢失

    一个 NACK 包可以包含多个 PID+BLP 对，高效地报告批量丢包。

    TMMBR / TMMBN（FMT = 3/4）（RFC 5104）

    临时最大媒体流码率请求（Temporary Maximum Media Stream Bit Rate Request），接收端向发送端请求限速：

    Transport-CC（FMT = 15）（draft-ietf-rmcat-transport-cc）

    传输层拥塞控制反馈，接收端将每个收到的 RTP 包的精确到达时间戳反馈给发送端，发送端在本端运行 GCC 或 NADA 算法估算可用带宽（详见 Transport-CC）。

    10.2 PSFB（PT = 206）：负载特定反馈

    这类反馈依赖于特定的媒体编码，主要用于视频：

    PLI（FMT = 1）（RFC 4585）：Picture Loss Indication

    接收端告知发送端，由于丢包导致当前帧无法解码，请求发送一个关键帧（Intra Frame）：

    PLI 没有 FCI（Feedback Control Information）字段，结构极其简单。

    FIR（FMT = 4）（RFC 5104）：Full Intra Request

    与 PLI 语义不同：FIR 请求发送端发出一个完整的关键帧，但原因不是丢包，而是有新的接收者加入（如 SFU 新成员入会），或者解码器状态被重置。RFC 5104 明确禁止在丢包时使用 FIR，这种场景应当使用 PLI。

    SLI（FMT = 2）（RFC 4585）：Slice Loss Indication

    报告视频切片级别的丢包，指示哪些宏块区域的数据丢失，适用于 H.261 等支持切片级修复的编码器。

    RPSI（FMT = 3）（RFC 4585）：Reference Picture Selection Indication

    接收端向发送端指示哪个参考帧仍然有效，帮助编码器选择合适的参考帧以避免错误传播。

    REMB（FMT = 15）（draft-alvestrand-rmcat-remb）：Receiver Estimated Maximum Bitrate

    11. 扩展报文：RTCP XR（RFC 3611）

    RFC 3611（2003 年）定义了 RTCP XR（Extended Reports，扩展报告），PT = 207，提供比 SR/RR 更精细的质量度量，主要面向 VoIP 监控、多播质量推断等场景。

    XR 包由报告块（Report Block）列表组成，每个块有自己的 BT（Block Type） 字段：

    RFC 3611 定义的 7 种块类型：

    BT 名称 用途
    1 Loss RLE Report Block 以游程编码描述详细的包丢失分布
    2 Duplicate RLE Report Block 描述重复包的分布
    3 Packet Receipt Times Report Block 报告各包的实际到达时间戳
    4 Receiver Reference Time Report Block 接收端的 NTP 时间戳，配合 BT=5 计算接收端 RTT
    5 DLRR Report Block 对 BT=4 的响应，包含 LRR 和 DLRR 字段
    6 Statistics Summary Report Block 序列号、丢包、重复、抖动、TTL 等统计摘要
    7 VoIP Metrics Report Block 专为 VoIP 设计的综合指标块（含 MOS、回声、抖动缓冲等）

    BT=4/5 的非发送方 RTT 测量：在纯接收端（不发送 RTP 数据因此没有 SR）之间测量 RTT 的场景，标准 SR/RR 机制失效，BT=4（Receiver Reference Time）+ BT=5（DLRR）提供了等效机制：

    12.1 强制要求的 RTCP 特性

    根据 RFC 8834（WebRTC 中 RTP 的使用），WebRTC 实现必须支持：

    特性 RFC 说明

    13. 安全：SRTCP

    RTCP 数据包的安全保护通过 SRTCP（Secure RTCP） 实现，与 SRTP 一同定义于 RFC 3711。

    SRTCP 在 RTCP 复合包的末尾附加：

    • E 位：0 表示该包未加密（但仍有认证），1 表示已加密。允许对 RTCP 进行选择性加密，例如仅认证不加密以支持 RTCP 监控代理。
    • SRTCP Index：单调递增的 31 位计数器，防止重放攻击。
    • Authentication Tag：默认为 80-bit HMAC-SHA1，保护包的完整性和来源。

    WebRTC 中，SRTCP 密钥通过 DTLS 握手（RFC 5764 定义的 DTLS-SRTP 扩展）协商，无需单独的 SDP 密钥交换。

    15. 小结

    RTCP 作为 RTP 的”神经系统”，承担着整个实时通信系统的质量感知与反馈控制职责：

    • SR/RR 提供周期性的传输质量统计（丢包、抖动、RTT），是拥塞控制算法的核心输入；
    • SDES 中的 CNAME 是多媒体流同步（唇音同步）的基础，也是端点识别的持久锚点；
    • RTCP-FB（RFC 4585/5104） 将 RTCP 从”被动上报”扩展到”主动干预”，NACK、PLI、FIR 直接参与流的修复和控制；
    • Transport-CC 将 RTCP 的反馈粒度提升到每个数据包级别，成为现代 WebRTC 拥塞控制（GCC）的核心数据来源；
    • rtcp-mux（RFC 5761） 在 WebRTC 中将 RTCP 流与 RTP 流合并到单一端口，配合 ICE 和 DTLS-SRTP，构成完整的安全实时传输栈。

    理解 RTCP 的工作机制，是深入分析 WebRTC 通话质量问题、调优拥塞控制策略、实现自定义 SFU 的必备基础。
