    1. 协议概述与背景

    RTP（Real-time Transport Protocol，实时传输协议） 是 IP 网络上音视频实时传输的核心协议，由 IETF 音视频传输工作组（Audio-Video Transport Working Group）开发。

    标准化历程：

    时间 里程碑
    1977 IETF RFC 741 发布，早期实时传输研究
    1992 IETF 开始正式开发 RTP
    1996 RFC 1889 发布，RTP 初版标准
    2003 RFC 3550 发布，取代 RFC 1889，成为当前正式标准（Internet Standard 64）

    当前权威规范为 RFC 3550，由 Columbia University 的 Henning Schulzrinne 等人主导撰写。

    RTP 的核心职责是为具有实时特征的数据提供端到端传输服务，包括：

    • 载荷类型标识（Payload Type Identification）：标明携带的媒体格式
    • 序列号（Sequence Numbering）：检测丢包与乱序
    • 时间戳（Timestamping）：支持播放同步与抖动补偿
    • 传输监控（Delivery Monitoring）：配合 RTCP 实现质量反馈

    重要认知：RTP 本身不保证及时投递或服务质量（QoS），也不进行资源预留。它依赖下层协议（通常是 UDP）提供基础传输能力，依赖 RTCP 进行质量监控。

    2. 设计哲学与架构原则

    2.1 应用层分帧（Application-Layer Framing）

    RTP 遵循 ALF（Application-Layer Framing） 原则：协议功能在应用层实现，而非操作系统协议栈。这使得 RTP 可以根据不同媒体类型灵活定制行为，而不需要修改协议核心。

    2.2 为何选择 UDP 而非 TCP

    实时多媒体应用优先考虑及时性而非可靠性：

    音频丢失一帧（例如 20 ms）通过 PLC（丢包隐藏）处理，对用户影响极小；但因 TCP 重传导致的数百毫秒延迟对实时通话则是不可接受的。

    2.3 协议框架而非完整实现

    RTP 是一个协议框架（Protocol Framework），刻意保持不完整。一个完整的 RTP 实现需要三个文档层次：

    1. RFC 3550：RTP 核心规范（本文重点）
    2. Profile 文档：定义载荷类型码表与应用特定扩展（如 RFC 3551 for A/V）
    3. Payload Format 文档：定义特定编解码器的打包方式（如 RFC 6184 for H.264）

    3. RTP 报文结构详解

    3.1 固定头部字段

    每个 RTP 数据包都以最小 12 字节的固定头部开始：

    （来源：RFC 3550, Section 5.1）

    各字段含义如下：

    V（Version）：2 bits

    当前版本号固定为 2。值 0 由早期的 “vat” 音频工具使用，值 1 由 RTP 草稿初版使用。接收方应丢弃版本号不为 2 的包。

    P（Padding）：1 bit

    填充标志位。若为 1，表示报文末尾附有填充字节（用于满足加密算法的块对齐要求）。填充区的最后一个字节记录填充字节总数（含自身）。在 SRTP 场景下，加密块对齐往往需要此字段。

    X（Extension）：1 bit

    扩展标志位。若为 1，表示固定头部之后、载荷之前有头部扩展（详见 3.2 节）。

    CC（CSRC Count）：4 bits

    指示跟随固定头部的 CSRC 标识符数量（0-15 个）。

    M（Marker）：1 bit

    标记位，含义由 Profile 定义。

    • 视频流：通常在帧的最后一个包设为 1，标记帧边界。接收端可据此判断一帧视频数据是否完整到达。
    • 音频流：部分实现在静默期后的第一个说话包设为 1，接收端可据此调整播放延迟。

    PT（Payload Type）：7 bits

    载荷类型，取值 0-127，标识 RTP 载荷的编码格式。接收方根据 PT 值选择对应的解码器。详见第 5 节。

    Sequence Number（序列号）：16 bits

    每发送一个 RTP 包，序列号加 1，回绕后从 0 重新开始（最大值 65535）。初始值应随机选取（安全考虑，避免猜测攻击）。

    接收端用序列号：

    • 检测丢包：序列号出现跳跃
    • 检测乱序：序列号逆序到达
    • NACK 反馈：告知发送端具体哪些包丢失

    Timestamp（时间戳）：32 bits

    反映 RTP 数据包中第一个字节的采样时刻，由单调递增的时钟驱动。时间戳不代表绝对时间，仅在同一流内部有意义，须通过 RTCP SR（Sender Report）与 NTP 时间关联后才能换算为真实时间。

    时间戳的关键特性：

    • 时钟频率（Clock Rate）因媒体类型而异：
      • 音频（如 Opus）：通常为 48000 Hz，每个采样周期时间戳增加 1
      • 视频（如 H.264/VP8）：通常为 90000 Hz，即使视频帧率只有 30fps，时间戳分辨率也足够精细
    • 同一帧的所有包时间戳相同：一帧视频可能被拆分为多个 RTP 包，它们共享相同的时间戳
    • 初始值随机：防止安全攻击，初始时间戳不应从 0 开始
    • 静默期间时间戳仍线性增长：区分”静默期”与”丢包”

    特殊案例：G.722 的采样率为 16000 Hz，但 RTP 时钟频率是 8000 Hz。这是历史错误——RFC 1890 错误地记录了此值，为向后兼容而保留（见 RFC 3551 §4.5.2）。

    SSRC（Synchronization Source Identifier）：32 bits

    同步源标识符，随机生成的 32 位值，在一个 RTP 会话内唯一标识一条媒体流。

    SSRC 的用途：

    • 接收端通过 SSRC 区分不同的媒体流
    • RTCP 报文通过 SSRC 与对应 RTP 流关联
    • 同一 RTP 会话中，一个参与者可以有多个 SSRC（如同时发送音频和视频）

    若两个源恰好选择了相同的 SSRC（碰撞概率低但存在），RFC 3550 Section 8 定义了碰撞检测与解决机制：检测到碰撞后，源必须发送 RTCP BYE 并选择新的 SSRC。

    CSRC List（Contributing Source Identifiers）：0-15 × 32 bits

    贡献源列表，仅在 Mixer（混音器） 插入时出现，最多 15 个。记录被混合进当前包的各原始源 SSRC。

    典型用途：多方音频会议中，Mixer 将多路音频混合后发出，CSRC 列表记录哪些参与者的声音被包含在内，接收端可据此在 UI 上显示”当前发言人”。

    3.2 头部扩展（Header Extension）

    当 X 位为 1 时，固定头部（含 CSRC 列表）后紧跟头部扩展区：

    • defined by profile（16 bits）：Profile 自定义标识，或 RFC 8285 中的 magic number
    • length（16 bits）：扩展数据长度（单位：32 位字，不含该 4 字节头部）

    原始的 RFC 3550 头部扩展每个包只能携带一条扩展信息，且 ID 分配混乱。RFC 8285（详见 8.1 节）通过”紧凑头部扩展”解决了这个问题。

    3.3 载荷（Payload）

    头部扩展之后（若有）是媒体载荷数据：已编码的音频帧、视频 NAL 单元片段等。载荷的具体打包格式由对应的 Payload Format 规范定义，与 RTP 核心无关。

    4. 关键机制解析

    4.1 序列号与丢包检测

    序列号使接收端能够：

    检测丢包：若收到序列号 100、101、103，则 102 丢失。接收端可通过 RTCP NACK 请求发送端重传。

    处理乱序：序列号逆序到达时，接收端在 Jitter Buffer 中等待并重排。

    计算丢包率（算法见 RFC 3550 Appendix A.3）：

    序列号 16 位意味着最多标识 65536 个包，回绕后从 0 开始。接收端必须处理回绕（wraparound）逻辑，通常采用扩展序列号（Extended Sequence Number）保存历史状态。

    4.2 时间戳与媒体同步

    播放时序控制

    接收端根据时间戳决定何时播放一帧数据，而非收到即播。这是 Jitter Buffer 工作的基础：

    接收端维护一个目标延迟，将各包放入 Jitter Buffer，按时间戳顺序取出，平滑抖动。

    音视频唇形同步（Lip Sync）

    音频流和视频流分别有各自的 SSRC 和时间戳，时钟频率也不同（音频 48000 Hz，视频 90000 Hz）。跨流同步依赖 RTCP SR（Sender Report）：

    RTCP SR 同时携带：

    • NTP 时间戳（绝对墙钟时间）
    • 对应时刻的 RTP 时间戳

    接收端通过两条 SR（一条音频、一条视频）建立映射关系，将不同时钟频率的 RTP 时间戳统一到 NTP 时间轴，实现唇形同步。

    不连续传输（DTX / Silence Suppression）

    静默期间，发送端可以停止发送音频包（节省带宽），但时间戳必须保持线性连续增长。这样接收端可区分”静默期（时间戳跳跃但没有丢包）”与”真实丢包（时间戳连续但序列号跳跃）”。

    4.3 SSRC 与流复用

    在 WebRTC 的 SFU 架构中，一个客户端可能同时发送多条流：

    流 SSRC PT 时钟频率
    音频（Opus） 随机值 A 111（动态） 48000
    视频（H.264） 随机值 B 96（动态） 90000
    视频 RTX 随机值 C 97（动态） 90000
    视频低分辨率（Simulcast） 随机值 D 96 90000

    接收端通过 SSRC 区分这些流，并通过 MID/RID 扩展头（见 9.4 节）将其映射到 SDP 协商的 m= 行。

    SSRC 碰撞检测：若两端恰好生成相同的 SSRC（32 位随机值的碰撞概率约为 1/2^32），RFC 3550 Section 8 定义了解决方案：

    1. 检测到碰撞后，发送 RTCP BYE 包
    2. 选择新的 SSRC 重新加入会话
    3. 循环检测直至 SSRC 唯一

    4.4 CSRC 与混音器

    Mixer 将多路流合并后：

    • 使用自己的 SSRC（如 100）发送混合包
    • 在 CSRC 列表中记录参与混合的各源 SSRC（最多 15 个）

    接收端可通过 CSRC 列表配合 RFC 6465（Mixer-to-Client Audio Level）扩展获取每个参与者的音量，在 UI 中显示谁在发言，即使所有包的 SSRC 都是 Mixer 的 SSRC。

    WebRTC 实践：现代 WebRTC 部署普遍使用 SFU（Selective Forwarding Unit）而非 Mixer。SFU 转发原始 RTP 包，保留原 SSRC，因此 CSRC 机制在 WebRTC 中实际上很少被用到。

    5. 载荷类型（Payload Type）体系

    5.1 静态载荷类型（Static PT）

    RFC 3551 为常见媒体格式预分配了固定 PT 值（0-95）：

    PT 名称 类型 时钟率（Hz） 备注
    0 PCMU 音频 8000 G.711 μ-law
    8 PCMA 音频 8000 G.711 A-law
    9 G722 音频 8000 实际采样 16kHz，历史原因时钟 8kHz
    14 MPA 音频 90000 MPEG Audio
    26 JPEG 视频 90000 JPEG 视频
    31 H261 视频 90000 H.261
    32 MPV 视频 90000 MPEG 视频
    34 H263 视频 90000 H.263
    96-127 动态 任意 协商 通过 SDP 分配

    RFC 3551 明确声明不再新增静态载荷类型，所有新编码格式只能通过动态 PT 协商。

    5.2 动态载荷类型（Dynamic PT）

    PT 96-127 保留给动态分配，通过 SDP a=rtpmap 属性在会话建立时绑定：

    在这个 SDP 示例中：

    • PT 111 → Opus，48000 Hz，2 声道
    • PT 96 → VP8，90000 Hz 时钟
    • PT 97 → RTX（重传流），关联 PT 96（VP8）

    5.3 时钟频率（Clock Rate）说明

    时钟频率决定时间戳的增长速率，不一定等于编解码器采样率：

    • 视频编解码器统一使用 90000 Hz：高分辨率时间戳便于精确的 RTCP NTP 对齐
    • Opus 固定使用 48000 Hz：即使实际以更低采样率工作
    • G.722 使用 8000 Hz：历史错误，实际采样 16000 Hz

    6. RTP Profile 机制

    Profile 是连接 RTP 核心与具体应用的桥梁。主要 Profile：

    Profile RFC 特点
    RTP/AVP RFC 3551 音视频基础 Profile
    RTP/SAVP RFC 3711 AVP + SRTP 加密
    RTP/AVPF RFC 4585 AVP + 快速 RTCP 反馈
    RTP/SAVPF RFC 5124 SAVP + AVPF（WebRTC 强制要求）

    WebRTC 强制使用 RTP/SAVPF（RFC 8834 Section 4.4），即所有 RTP 和 RTCP 包必须通过 SRTP/SRTCP 加密，且支持快速 RTCP 反馈（不受标准 5% 带宽上限约束）。

    7. 中间件：Mixer 与 Translator

    Mixer（混音器）

    Mixer：

    • 接收多路 RTP 流
    • 将它们混合（音频叠加，视频合成）
    • 以新 SSRC 发出合并后的单路流
    • 在 CSRC 列表记录贡献源

    Mixer 会终结原始流的时序信息，并重新生成时间戳。RTCP 也由 Mixer 接管。

    Translator（转译器）

    Translator：

    • 转发 RTP 流，保留原始 SSRC
    • 可变更载荷类型（如 H.264 → VP8 转码）
    • 可调整采样率、码率等
    • 不混合多条流

    SFU 本质上是一种 Translator，它选择性转发各端的 RTP 流，保留 SSRC 不变。

    8. RTP 扩展机制

    8.1 RFC 8285 紧凑头部扩展

    原始 RFC 3550 头部扩展只支持单条扩展信息。RFC 8285（取代 RFC 5285）定义了支持多条扩展的紧凑格式。

    One-Byte Header 格式（推荐）

    当扩展数据 ≤ 16 字节时使用，magic number 为 0xBEDE：

    • ID（4 bits）：扩展类型 ID（1-14，0 是填充，15 保留）
    • len（4 bits）：数据长度减 1（即 0 表示 1 字节数据，15 表示 16 字节数据）

    8.2 RTX 重传格式（RFC 4588）

    RTX（RTP Retransmission）是 RTP 层的重传机制，使用独立的 SSRC 和 PT 发送重传包：

    • OSN（Original Sequence Number，2 字节）：被重传包的原始序列号
    • 重传包时间戳 = 原始包时间戳（便于接收端正确排序）
    • 重传包有自己的序列号（在重传流内递增）

    RTX 的优势：

    • 无头阻塞：放弃超时未到的重传，不影响后续包播放
    • 与原始流解耦：不干扰原始流的时序分析

    9. RTP 在 WebRTC 中的使用

    9.1 强制使用 SRTP（RFC 8834）

    RFC 8834（Media Transport and Use of RTP in WebRTC）明确规定：

    WebRTC endpoints MUST NOT send packets using the basic RTP/AVP profile or the RTP/AVPF profile; they MUST employ the full RTP/SAVPF profile to protect all RTP and RTCP packets that are generated.

    即 WebRTC 中所有 RTP 包实际上都是 SRTP 包，密钥通过 DTLS-SRTP 协商（详见本系列 DTLS 和 SRTP 篇）。SRTP 保留与 RTP 相同的头部布局，仅对载荷加密并附加认证标签（Authentication Tag）。

    9.2 RTCP-MUX（RFC 5761）

    传统上 RTP 和 RTCP 使用相邻端口（RTP 用偶数端口，RTCP 用奇数端口 +1）。WebRTC 要求使用 RTCP-MUX，将 RTP 和 RTCP 合并到同一个 UDP 端口传输：

    具体区分规则见 RFC 7983：通过检查包的首字节范围来判断是 RTP、RTCP、DTLS 还是 STUN 包。

    RTCP-MUX 的好处：减少 NAT 打孔的端口数量，简化防火墙配置，降低延迟。

    9.3 BUNDLE 多路复用

    WebRTC 进一步支持 BUNDLE（RFC 8843），将多条 RTP 流（音频、视频、屏幕共享）合并到同一个传输层连接（同一 ICE/DTLS 流量）中。

    接收端通过 MID 头部扩展将包路由到对应 SDP m= 行处理。

    9.4 常用 WebRTC RTP 头部扩展

    以下扩展在现代 WebRTC 实现中被广泛使用：

    URI 作用
    urn:ietf:params:rtp-hdrext:ssrc-audio-level 携带音频电平（RFC 6464），接收端实时知道发言者音量
    urn:ietf:params:rtp-hdrext:sdes:mid Media Identification，标识包属于哪个 SDP m= 行，支持 BUNDLE

    10. 安全性考量

    10.1 SRTP 强制加密

    如前所述，WebRTC 强制要求 SRTP。原始 RTP 存在的安全问题：

    • 窃听：未加密的载荷可直接还原媒体内容
    • 重放攻击：攻击者重发旧包干扰通话
    • 包注入：伪造 RTP 包注入噪声媒体

    SRTP（RFC 3711）提供：

    • 载荷加密（AES-CM 或 AES-GCM）
    • 消息认证（HMAC-SHA1 或 GCM 内置认证）
    • 重放保护（Replay Protection Window）

    10.2 SSRC 随机化

    RFC 3550 要求 SSRC 随机生成，防止攻击者预测 SSRC 值注入伪造包。

    10.3 序列号随机初始值

    初始序列号也应随机选取，防止针对初始值的攻击。

    11. 协议栈位置与交互关系

    RTP 与各组件的交互：

    • SDP ↔ RTP：SDP 协商 PT 映射、时钟频率、头部扩展 ID 等，是 RTP 的”配置文件”
    • RTP ↔ RTCP：RTCP 是 RTP 的控制面，两者共享 SSRC 标识，RTCP 提供统计反馈
    • RTP ↔ SRTP：SRTP 是加密版 RTP，头部不变，载荷加密
    • RTP ↔ ICE/DTLS：ICE 提供传输通道，DTLS 协商 SRTP 密钥
    • RTP ↔ Jitter Buffer：接收端 Jitter Buffer 利用 RTP 时间戳和序列号进行重排和平滑

    小结

    RTP 是实时媒体传输的基石，其设计哲学高度务实：

    1. 最小化头部开销：12 字节固定头部，大部分字节用于媒体数据
    2. 灵活可扩展：通过 Profile 和 Payload Format 适配任意媒体类型，头部扩展机制支持新特性
    3. 不求完美，追求及时：接受丢包，优先低延迟
    4. 关注点分离：数据传输（RTP）与控制反馈（RTCP）分离，与下层传输解耦

    在 WebRTC 中，RTP 被强制以 SRTP 形式使用，配合 RTCP-MUX、BUNDLE 等优化，构成了浏览器实时通信的核心媒体传输层。理解 RTP 的报文结构、序列号/时间戳机制和扩展体系，是深入理解 WebRTC 媒体质量保障（BWE、Jitter Buffer、NACK/FEC）的前提。
