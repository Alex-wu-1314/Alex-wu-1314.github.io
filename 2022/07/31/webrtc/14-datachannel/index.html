    1. 概述：为什么需要 DataChannel

    WebRTC 的初期印象往往是”浏览器视频通话”，但其规范同时定义了一套完整的任意数据对等传输机制：DataChannel。

    1.1 与现有方案的对比

    特性 WebSocket HTTP/AJAX DataChannel
    传输方向 双向（Client ↔ Server） 单向请求/响应 双向（Peer ↔ Peer）
    拓扑 必须经过服务器 必须经过服务器 真正 P2P，无需中间服务器
    底层协议 TCP TCP SCTP over DTLS over UDP
    可靠性 仅可靠有序 仅可靠有序 可配置（可靠/部分可靠/不可靠）
    加密 TLS（可选） TLS（可选） DTLS 强制
    延迟 受 TCP 队头阻塞影响 受 TCP 队头阻塞影响 低，基于 UDP，无队头阻塞

    1.2 核心价值

    DataChannel 的设计目标来自 RFC 8831 定义的实际需求：

    • 实时游戏：通过一个或多个不可靠数据通道发送位置和对象状态信息
    • P2P 文件传输：不经过服务器，直接在浏览器间传输文件
    • 实时文本聊天：作为音视频通话的配套文字频道
    • 协同编辑：低延迟同步编辑操作
    • IoT / 遥测：传输传感器数据或控制指令

    单个 RTCPeerConnection 最多可同时创建 65,534 个 DataChannel，这为多路复用提供了充足空间。

    2. 协议栈与架构

    DataChannel 的协议栈是一个精心设计的分层结构，每一层解决不同的问题：

    与媒体传输（RTP/SRTP）类似，DataChannel 共享同一个 RTCPeerConnection，因此可以复用相同的 ICE 候选和 DTLS 握手，无需额外的连接建立开销。在实现层面，SCTP 关联和媒体传输可以共享同一个 UDP 端口，这是通过 DTLS 层的多路复用完成的。

    3. SCTP 在 WebRTC 中的角色

    3.1 为什么选择 SCTP

    WebRTC DataChannel 选择 SCTP（Stream Control Transmission Protocol，RFC 4960）而非 TCP 或 UDP，原因在于 SCTP 天然具备以下特性，精确匹配 DataChannel 的需求：

    多流（Multi-streaming）：SCTP 在单个关联（Association）内支持多个独立的流（Stream），每个流可以独立配置可靠性和顺序性。WebRTC 的每个 DataChannel 正是对应一个 SCTP 流。这样，一个大文件的传输不会阻塞同一连接上的游戏状态消息。

    消息边界保留：与 TCP 的字节流不同，SCTP 是面向消息的协议，发送方发送一条消息，接收方就完整收到一条消息，无需应用层自己处理粘包/拆包。

    原生可靠性配置：通过 PR-SCTP（Partial Reliability SCTP，RFC 3758），SCTP 支持部分可靠传输，可以按重传次数或超时时间限制重传。

    头部阻塞规避：由于 SCTP 运行在 UDP 之上，不同流之间相互独立，避免了 TCP 的队头阻塞（Head-of-Line Blocking）。

    3.2 WebRTC 对 SCTP 的裁剪

    WebRTC 并未使用 SCTP 的全部特性。具体地，WebRTC 不使用 SCTP 的多宿主（Multi-homing）功能，路径选择交由 ICE 负责。WebRTC 使用的 SCTP 关键扩展如下：

    RFC 扩展 用途
    RFC 3758 PR-SCTP 部分可靠传输
    RFC 6525 Re-configuration 动态增加流数量；重置单个流（关闭 DataChannel）
    RFC 8260 I-DATA / 消息交织 避免大消息独占关联，支持流优先级调度

    3.3 SCTP 的 PPID（Payload Protocol Identifier）

    SCTP 用 PPID 区分不同上层协议的数据。WebRTC 中使用以下五个 PPID：

    PPID 值 名称 含义
    50 WebRTC DCEP DataChannel 建立协议消息
    51 WebRTC String DataChannel 字符串消息（UTF-8）
    53 WebRTC Binary DataChannel 二进制消息
    56 WebRTC String Empty 空字符串消息（SCTP 不支持发送空消息，故特殊处理）
    57 WebRTC Binary Empty 空二进制消息

    注意：PPID 在 JavaScript API 层面不可直接访问，由浏览器底层自动处理。

    3.4 消息大小与交织

    RFC 8831 指出，SCTP 基础协议（RFC 4960）不支持用户消息交织。这意味着发送一条大消息时，会独占整个 SCTP 关联，阻塞其他流的消息。为此：

    • 应用应当启用 RFC 8260 定义的消息交织扩展（I-DATA chunk）
    • 若交织扩展不可用，发送方应当将单条消息大小限制在 16 KB 以下
    • 最大消息大小可通过 SDP 的 max-message-size 属性（RFC 8841）协商，默认值为 64 KB；设为 0 表示不限大小

    4. DataChannel 建立协议（DCEP）

    DCEP（Data Channel Establishment Protocol）由 RFC 8832 定义，是在 SCTP 关联建立后，用于协商单个 DataChannel 参数的应用层协议。

    4.1 带内协商（In-band Negotiation）

    这是默认方式，由浏览器自动完成，无需开发者干预。

    流 ID 的奇偶性规则（RFC 8832）：为避免双方同时打开时的流 ID 冲突：

    • DTLS 客户端一方：使用偶数流 ID（0, 2, 4, …）发送 DATA_CHANNEL_OPEN
    • DTLS 服务端一方：使用奇数流 ID（1, 3, 5, …）发送 DATA_CHANNEL_OPEN

    开启方可以在收到 DATA_CHANNEL_ACK 之前就开始发送用户数据，降低握手延迟。

    4.2 DATA_CHANNEL_OPEN 消息格式

    各字段含义：

    字段 说明
    Message Type 固定为 0x03（OPEN）
    Channel Type 通道可靠性类型（见下一节）
    Priority 优先级，建议取值：128（低）、256（普通）、512（高）、1024（极高）
    Reliability Parameter 若为 REXMIT 类型则为最大重传次数；若为 TIMED 则为超时毫秒数
    Label 通道名称（UTF-8，可为空）
    Protocol 子协议名（参照 WebSocket 子协议注册表，RFC 6455）

    4.3 带外协商（Out-of-band Negotiation）

    当开发者在 createDataChannel() 中指定 negotiated: true 并手动提供 id 时，双方跳过 DCEP 握手，直接在应用层约定参数。此方式更灵活，可以让两端的通道拥有不同属性，或通过 SDP 的 a=dcmap 属性（RFC 8864）进行声明式协商。

    5. 可靠性模型与传输语义

    这是 DataChannel 相比 WebSocket 最核心的差异化能力。DataChannel 支持三大类传输语义，通过 createDataChannel() 的选项控制。

    5.1 Channel Type 枚举（来自 RFC 8832）

    Channel Type 值 可靠性 顺序性 Reliability Parameter
    0x00 DATA_CHANNEL_RELIABLE 完全可靠 有序 忽略
    0x80 DATA_CHANNEL_RELIABLE_UNORDERED 完全可靠 无序 忽略
    0x01 DATA_CHANNEL_PARTIAL_RELIABLE_REXMIT 部分可靠（按重传次数） 有序 最大重传次数
    0x81 DATA_CHANNEL_PARTIAL_RELIABLE_REXMIT_UNORDERED 部分可靠（按重传次数） 无序 最大重传次数
    0x02 DATA_CHANNEL_PARTIAL_RELIABLE_TIMED 部分可靠（按超时） 有序 超时毫秒数
    0x82 DATA_CHANNEL_PARTIAL_RELIABLE_TIMED_UNORDERED 部分可靠（按超时） 无序 超时毫秒数

    5.2 三种模式详解

    模式一：完全可靠有序（TCP-like，默认）

    适用场景：聊天消息、文件传输、信令数据。消息保证送达且按序交付，但在高丢包网络下延迟可能较高（类似 TCP 的行为）。

    模式二：完全可靠无序

    消息保证全部送达，但不保证顺序。适用于每条消息相互独立、无序关系的场景，例如独立的事务请求。相比有序模式，避免了队头阻塞：一条消息的丢失不会阻塞后续消息的交付。

    模式三：部分可靠（按重传次数）

    maxRetransmits: 0 相当于纯 UDP 语义：发送后不等待 ACK，不重传。适合游戏中的位置更新、鼠标坐标等”旧数据没有意义”的场景。

    模式四：部分可靠（按超时）

    发送方在 maxPacketLifeTime 毫秒后停止重传该消息。适合对新鲜度敏感但又比纯 UDP 稍微可靠一些的数据。

    注意：maxRetransmits 和 maxPacketLifeTime 不能同时设置，二者互斥（RFC 8832）。

    9. 安全性

    9.1 强制 DTLS 加密

    RFC 8831 明确要求：DataChannel 的所有数据必须通过 DTLS 加密传输。DTLS 保护完整的 SCTP 数据包，提供以下安全保证：

    • 机密性：数据内容对中间人不可见
    • 完整性：数据不可被篡改
    • 源认证：通过证书指纹（a=fingerprint）验证对端身份

    DTLS 的安全性等同于 HTTPS/TLS，因为它基于相同的 TLS 密码套件。

    9.2 P2P 传输的隐私优势

    由于 DataChannel 是真正的点对点传输（直连时），数据不经过任何服务器。即使是应用的后端服务器也无法监听传输内容。这使得 DataChannel 非常适合端到端加密的私密通信场景。

    当然，若 ICE 无法直连而需要 TURN 中继，数据会经过 TURN 服务器，但 DTLS 层的加密保证了 TURN 服务器也无法解密内容。

    9.3 应用层安全注意事项

    DTLS 保护的是传输层安全，应用层仍需注意：

    • 对接收的消息进行严格的输入验证，不要盲目信任对端数据
    • 使用结构化消息格式（如带类型字段的 JSON 或自定义二进制协议），防止注入攻击
    • 若在 DataChannel 上实现子协议，应在 protocol 字段中声明，便于双方验证

    小结

    WebRTC DataChannel 将 SCTP 的多流、消息边界保留、可配置可靠性等特性，通过 DTLS 加密后搭载在 ICE 建立的 UDP 路径上，最终以简洁的 Web API 暴露给开发者。

    其核心价值在于三点：

    真正的 P2P：数据直接在对等端间传输，不经过应用服务器，降低延迟并节省带宽成本。

    灵活的可靠性：从完全可靠有序（类 TCP），到不可靠无序（类 UDP），到各种部分可靠模式，开发者可以精确匹配业务需求。游戏状态同步用不可靠通道降低延迟，文件传输用可靠通道保证完整性。

    与媒体的无缝集成：DataChannel 与音视频轨道共享同一个 RTCPeerConnection，复用 ICE 和 DTLS 基础设施，无额外连接建立开销。

    在实际工程中，常见的坑是忽视背压控制，务必通过 bufferedAmount 和 bufferedAmountLowThreshold 实现发送节流，以及消息分片，单条消息保持在 16 KB 以内以获得最佳跨浏览器兼容性。
