    1. 背景：为什么需要反馈机制

    RTP 本身是一个无连接的单向传输协议，不提供任何内置的可靠性保证。在真实网络中，数据包丢失不可避免。对于视频流而言，H.264 / VP8 / H.265 等编解码器大量使用帧间预测（P 帧、B 帧），一旦参考帧中的某个 RTP 包丢失，后续所有依赖该参考帧的帧都将无法正常解码，在屏幕上表现为马赛克、花屏或画面静止。

    传统的 RTCP RR（接收者报告）虽然可以周期性地汇报丢包率，但其反馈粒度是”统计级别”而非”包级别”，且发送间隔往往以秒计，无法驱动快速的错误修复。

    为此，IETF 定义了 AVPF（Audio-Visual Profile with Feedback） 扩展，允许接收端以近实时的方式向发送端发送细粒度反馈，由此催生了 NACK、PLI、FIR 等机制。

    2. RTCP 反馈框架：RFC 4585 / AVPF

    RFC 4585（2006 年，标准轨道）是 NACK、PLI 等反馈消息的基础规范，正式名称为 *Extended RTP Profile for RTCP-Based Feedback (RTP/AVPF)*。它在标准 AVP Profile（RFC 3551）的基础上增加了：

    1. 立即反馈模式（Immediate Feedback Mode）：当会话规模较小时，允许接收端在检测到丢包后几乎立即发出 RTCP 反馈包，而不必遵守严格的最小发送间隔限制。
    2. 早期反馈模式（Early RTCP Mode）：介于立即模式和常规 RTCP 轮询之间，允许提前发送（但仍受一定带宽约束）。
    3. 反馈消息分类：
      • 传输层反馈（Transport Layer FB，PT=205）：与编解码器无关，如 Generic NACK。
      • 载荷特定反馈（Payload-Specific FB，PT=206）：与编解码器相关，如 PLI、FIR、SLI 等。
      • 应用层反馈（Application Layer FB）：供应用自定义扩展使用。

    所有 RTCP 反馈包共享一个通用头部格式：

    其中：

    • V=2：RTP 版本号，固定为 2。
    • P：填充位。
    • FMT（5 bits）：反馈消息类型，在 PT=205 和 PT=206 两个命名空间下分别编号。
    • PT：205 = RTPFB（传输层反馈），206 = PSFB（载荷特定反馈）。
    • SSRC of packet sender：反馈消息发送方的 SSRC（即接收端）。
    • SSRC of media source：被反馈的媒体流的 SSRC（即发送端）。
    • FCI：各消息类型自定义的控制信息。

    3. NACK —— 丢包重传请求

    Generic NACK（Negative ACKnowledgement） 是 RFC 4585 定义的唯一一条传输层反馈消息，类型标识为 PT=205，FMT=1。

    3.1 工作原理

    接收端的检测依据是 RTP 序列号的连续性。当收到序列号为 N+2 的包，而 N+1 的包从未到达时，即可判断 N+1 丢失，随即发送 NACK。发送端接收到 NACK 后，从其重传缓冲区（retransmission buffer） 中取出对应包进行重传。若该包已经超出缓冲区保留时长（由 rtx-time SDP 参数控制），则发送端可以选择忽略 NACK。

    重要约束：发送端收到 NACK 后不保证一定重传，是否重传是实现层面的决策，取决于缓冲区状态和网络拥塞估计。

    3.2 报文格式（RFC 4585 §6.2.1）

    NACK 的 FCI 字段由一个或多个 NACK Item（每项 32 bits） 组成，格式如下：

    字段 长度 含义
    PID（Packet ID） 16 bits 丢失包的 RTP 序列号
    BLP（Bitmask of Lost Packets） 16 bits PID 之后 16 个序列号的丢失位掩码

    整个 RTCP 反馈包的 length 字段值为 2 + n，其中 n 为 FCI 中 NACK Item 的个数。

    3.3 PID 与 BLP 的位掩码编码

    BLP 的每一位对应 PID+1 到 PID+16 的包是否丢失。以最低有效位（bit 1）对应 PID+1，最高有效位（bit 16）对应 PID+16：

    • 若 bit i = 1，表示序列号 PID+i 的包也丢失了，同样需要重传。
    • 若 bit i = 0，不代表 PID+i 已被接收，只是接收端此时尚未报告它丢失。

    即 BLP = 0x000B（二进制 0000 0000 0000 1011）。

    这种设计使得单条 NACK Item 可以紧凑地表达最多 17 个连续丢包（PID 本身 + BLP 覆盖的 16 个），大幅减少了突发丢包场景下的 RTCP 带宽占用。

    3.4 RTX：重传包的独立载荷格式（RFC 4588）

    标准的 RTP 重传使用 RFC 4588 定义的 RTX（RTP Retransmission）格式。重传包通过一个独立的 SSRC 发送，其载荷头部携带原始包的序列号（OSN，Original Sequence Number）：

    RTX 流与原始流使用不同的载荷类型（payload type）和 SSRC，通过 SDP 中的 apt（associated payload type）参数关联：

    rtx-time=3000 表示发送端会在重传缓冲区中保留已发包长达 3000 毫秒。接收端在决定是否发送 NACK 时，应将此值纳入考量，避免请求一个发送端已不再持有的包。

    3.5 SDP 协商

    双端均需在 SDP offer/answer 中声明支持，才能使用 NACK。

    3.6 适用场景与局限性

    维度 说明
    最佳场景 低 RTT（< 100ms）、中低丢包率（< 5%）、视频流
    局限性 需要至少一个 RTT 才能完成重传，RTT 过高时包可能重传到达时已错过解码窗口
    音频适用性 较差。Opus 等音频编解码器有内置的 PLC，NACK 带来的延迟代价往往大于收益
    与 FEC 的关系 FEC（前向纠错）主动冗余，NACK 被动请求；高丢包时 FEC 更优，低丢包时 NACK 更节约带宽

    4. PLI —— 图像丢失指示

    PLI（Picture Loss Indication） 定义于 RFC 4585 §6.3.1，类型标识为 PT=206（PSFB），FMT=1。

    4.1 工作原理

    当接收端检测到一帧或多帧视频数据丢失，导致解码器无法维持正常解码状态时，发送 PLI 通知发送端。发送端收到 PLI 后，可以（MAY）生成一个新的关键帧（I 帧/IDR 帧）并发送给接收端，以重置解码上下文。

    PLI 是一种”礼貌性请求”——发送端可以忽略它，也可以根据自身的拥塞控制策略决定何时生成关键帧。这与 FIR 的强制语义形成对比。

    4.2 报文格式（RFC 4585 §6.3.1）

    PLI 的 FCI 字段为空（0 字节）。整个 PLI 报文结构如下：

    length=2 表示报文长度为 (2+1) × 4 = 12 字节（length 字段计算的是 4 字节字数，不含首个 32 bits，故 length=2 表示共 3 个 32 bits = 12 bytes）。

    PLI 是所有 RTCP 反馈消息中结构最简单的一种，因为触发条件本身（”画面丢失”）不需要携带额外参数。

    5. FIR —— 全帧内刷新请求

    FIR（Full Intra Request） 定义于 RFC 5104（2008 年，标准轨道），正式名称为 *Codec Control Messages in the RTP Audio-Visual Profile with Feedback (AVPF)*。类型标识为 PT=206（PSFB），FMT=4。

    注意：FIR 定义在 RFC 5104 而非 RFC 4585，但两者都使用 PT=206 的 PSFB 命名空间。

    5.1 工作原理

    FIR 向发送端发出强制性命令，要求其必须生成并发送一个解码器刷新点（decoder refresh point），即关键帧。与 PLI 不同，发送端不得以”当前网络拥塞”等理由拒绝 FIR——接到 FIR 必须响应（除非刚刚在 2×RTT 时间窗口内已响应过）。

    FIR 的设计语义是：系统状态需要从全新的关键帧重新建立解码上下文，而不仅仅是弥补丢失帧。RFC 5104 §4.3.1.2 明确规定：

    “FIR SHALL NOT be sent as a reaction to picture losses — it is RECOMMENDED to use PLI [RFC4585] instead. FIR SHOULD be used only in situations where not sending a decoder refresh point would render the video unusable for the users.”

    典型的 FIR 使用场景：

    • 多点会议中，新参与者加入时需要完整的第一帧。
    • SFU 切换发言人时，新订阅方需要立即开始解码新的流。
    • 某些录制、转码场景中需要从关键帧重新建立时间轴。

    5.2 报文格式（RFC 5104 §4.3.1）

    FCI 中每个条目（8 字节）包含：

    • SSRC（32 bits）：被请求发送关键帧的媒体源 SSRC。
    • **Seq nr.**（8 bits）：FIR 序列号，每次新的 FIR 请求递增。用于抑制重复响应。
    • Reserved（24 bits）：保留，置 0。

    5.3 FMT 字段与序列号

    FIR 使用 FMT=4，而不是 PLI 的 FMT=1，两者同属 PT=206 命名空间。

    FIR 报文中的 8 位序列号（Seq nr.）是一个关键设计——发送端收到与上一次响应的 FIR 序列号相同的重复请求时，若其在 2×RTT 内已生成过关键帧，可以安全忽略，以避免在突发性重复 FIR 场景下不必要地产生大量关键帧。

    5.4 SDP 协商

    FIR 在 SDP 中通过 ccm fir 而非 nack 参数族声明，因为它属于编解码器控制消息（Codec Control Message）而不是反向确认机制。

    6. PLI vs FIR：语义差异与选用原则

    尽管 PLI 和 FIR 在实践中通常都会导致发送端生成一个新关键帧，但两者在语义和协议约束上存在本质区别：

    维度 PLI FIR
    定义规范 RFC 4585 RFC 5104
    PT / FMT 206 / 1 206 / 4
    SDP 属性 a=rtcp-fb:N nack pli a=rtcp-fb:N ccm fir
    触发原因 接收端画面丢失（因丢包） 需要全新的解码起点（与丢包无关）
    强制程度 发送端 MAY 响应，可忽略 发送端 MUST 响应（有重复抑制窗口）
    使用限制 RFC 5104 明确建议用于丢包场景 明确禁止用于丢包恢复
    FCI 大小 0 字节 8 字节/条目
    序列号 无 有（用于抑制重复响应）
    典型场景 丢包导致解码失败，需要重建参考帧 新用户加入会议、切换发言人

    选用原则（基于 RFC 5104 §4.3.1.2）：

    • 因丢包引发的解码失败 → 优先使用 PLI，其次 NACK。
    • 因系统状态切换（新订阅者、流切换）需要强制关键帧 → 使用 FIR。
    • 在实现了两者的系统中，许多发送端对 PLI 和 FIR 的响应方式在实践上并无区别，但协议层面的语义区分依然重要，有助于 SFU/MCU 等中间节点作出正确的信令决策。

    8. SFU 场景下的实践

    在 SFU（Selective Forwarding Unit）架构中，NACK/PLI/FIR 的处理比 P2P 复杂得多：

    8.1 NACK 的 SFU 处理

    SFU 通常会终结（terminate） 来自下行订阅者的 NACK，自行维护一个已转发包的缓冲区，并在收到 NACK 时直接从缓冲区重传，而不是透明地将 NACK 转发给上行发布者。这样可以：

    • 避免重复的 NACK 请求淹没上行发布者（N 个订阅者同时 NACK 同一个包）。
    • 降低端到端 RTT（SFU 的缓存命中比转发 NACK 到发布者更快）。

    8.2 PLI/FIR 的 SFU 聚合

    当有 N 个订阅者同时发送 PLI 时，若 SFU 直接转发所有 PLI，将迫使发布者连续生成多个大体积关键帧，严重浪费带宽。正确做法是：

    1. PLI 聚合（PLI Aggregation）：SFU 在一段时间窗口内将多个 PLI 合并为一个，再向发布者发送一次 PLI。
    2. FIR 用于新订阅者：当新订阅者加入时，SFU 可以向发布者发送一个 FIR，而不是将关键帧请求错误地表达为 PLI。
    3. 缓存关键帧：SFU 可以缓存最近一次发布者发出的关键帧，在新订阅者加入时直接转发，无需再次请求。

    在 Simulcast（多路流）场景下，每一路流（高清/中清/低清）都有独立的 SSRC，NACK/PLI/FIR 必须针对具体流的 SSRC 发送，SFU 在层切换时还需要向新选中的层的发布者发送 FIR 以请求关键帧。

    10. 常见问题与调试技巧

    Q1：为什么 NACK 了但没有收到重传？

    • 发送端的重传缓冲区（rtx-time）可能太短，目标包已被清出缓冲区。
    • 发送端实现可能未开启 NACK 响应（需检查 SDP 是否正确协商了 a=rtcp-fb:N nack）。
    • RTT 过高，NACK 往返时间超过了解码缓冲的容忍窗口，发送端选择忽略。
    • 网络拥塞，发送端主动降低发送码率，放弃部分 NACK 重传。

    Q2：PLI 风暴（PLI Storm）

    在 SFU 场景下，如果多个订阅者同时发送 PLI，而 SFU 未做聚合直接转发，发布端会收到大量 PLI，导致频繁生成关键帧，每个关键帧体积可能是普通帧的 10 倍以上，迅速耗尽带宽并触发更多丢包，从而形成正反馈循环。解决方案是 SFU 实施 PLI 节流（throttling），最小间隔通常设置为 1~3 秒。

    Q3：如何用 Wireshark 抓包分析？

    在 Wireshark 中：

    • 过滤 RTCP 反馈包：rtcp.pt == 205 (RTPFB) 或 rtcp.pt == 206 (PSFB)
    • NACK：rtcp.pt == 205 && rtcp.rtpfb.fmt == 1
    • PLI：rtcp.pt == 206 && rtcp.psfb.fmt == 1
    • FIR：rtcp.pt == 206 && rtcp.psfb.fmt == 4

    也可以在 Chrome 中访问 chrome://webrtc-internals，在 Stats 面板中查看：

    • RTCOutboundRtpVideoStream.nackCount：发送端收到 NACK 的次数
    • RTCInboundRtpVideoStream.nackCount：接收端发送 NACK 的次数
    • RTCInboundRtpVideoStream.pliCount：接收端发送 PLI 的次数
    • RTCInboundRtpVideoStream.firCount：接收端发送 FIR 的次数

    Q4：NACK 与 FEC 如何搭配？

    两者可以同时启用，互为补充：

    • FEC 在发送时主动添加冗余包，对突发丢包有更快的恢复速度（无需等待 RTT）。
    • NACK 在 FEC 无法覆盖的丢包场景下兜底，请求重传。

    在 WebRTC 中，通常根据网络质量动态调整两者的比例：丢包率高时增加 FEC 冗余，丢包率低时依赖 NACK 节省带宽。
