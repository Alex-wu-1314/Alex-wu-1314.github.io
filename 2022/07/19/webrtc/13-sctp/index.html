    1. 前言：为什么 WebRTC 需要 SCTP？

    WebRTC 的媒体数据（音频、视频）走 SRTP over UDP 这条路，但当我们需要传输任意的应用层数据，比如游戏状态、文件片段、聊天消息、二进制帧，就需要另一套机制。

    最直觉的选择是 TCP 或 WebSocket，但它们有致命问题：

    • TCP 的队头阻塞（Head-of-Line Blocking）：一个丢失的包会阻塞整个字节流，即使后续数据已经到达。
    • TCP 是字节流：应用层需要自行分帧，增加实现复杂度。
    • WebSocket 基于 TCP：继承了上述所有问题。
    • UDP 没有可靠性保证：需要应用层自行处理重传、排序、拥塞控制。

    SCTP（Stream Control Transmission Protocol，流控制传输协议）正好填补了这个空白。它将 TCP 的可靠性与 UDP 的消息边界特性融合在一起，并额外提供多流、部分可靠性等高级功能，使其成为 WebRTC DataChannel 底层传输的理想选择。

    2. SCTP 的起源与标准化

    SCTP 最初由 IETF SIGTRAN（Signaling Transport）工作组设计，目标是将 PSTN（公共交换电话网）的 SS7 信令消息承载在 IP 网络上。2000 年 10 月，SCTP 作为协议号 132 正式发布（RFC 2960）。

    主要标准演进：

    RFC 时间 内容
    RFC 2960 2000年 SCTP 初始规范
    RFC 3758 2004年 PR-SCTP 部分可靠性扩展
    RFC 4960 2007年 取代 RFC 2960，SCTP 修订版
    RFC 9260 2022年 当前最新版本，取代 RFC 4960
    RFC 8261 2017年 SCTP over DTLS 封装规范
    RFC 8831 2021年 WebRTC DataChannel 架构规范
    RFC 8832 2021年 DCEP 数据通道建立协议
    RFC 8841 2021年 SDP 中 SCTP-over-DTLS 的 Offer/Answer

    注意：RFC 9260 是当前权威规范，但 WebRTC 实现文档（如 RFC 8831）仍引用 RFC 4960，两者在核心机制上是兼容的。

    3. SCTP 核心特性

    3.1 多流（Multi-streaming）

    这是 SCTP 最重要的特性，也是解决 TCP 队头阻塞的核心手段。

    在一个 SCTP 关联（Association） 中可以同时维护多个独立的逻辑流（Stream）。每个流有自己独立的序列号空间（Stream Sequence Number，SSN）。一个流上的丢包不会阻塞其他流的数据交付。

    WebRTC 规范要求每端支持至少 1024 个流，理论上限为 65534 个 DataChannel（奇偶编号分别由 offer 方和 answer 方使用，0 号预留）。

    3.2 消息边界保留（Message-oriented）

    SCTP 以消息（Message） 为单位交付数据，而非字节流。发送方提交一个完整的应用层消息，接收方收到的也是这个完整消息，不需要应用层自己解帧。

    如果消息超过路径 MTU（PMTU），SCTP 会自动在发送端分片（Fragment），并在接收端重组（Reassemble），这对应用层透明。

    理论上，单条 SCTP 消息可以达到数 GB，但实际场景应根据 PMTU 和内存情况合理控制消息大小。

    3.3 可配置的可靠性（Partial Reliability）

    标准 SCTP 提供全可靠传输（类似 TCP）。PR-SCTP（RFC 3758）扩展允许发送方为每条消息单独配置可靠性策略：

    策略 说明 WebRTC DataChannel 类型
    完全可靠（Full Reliability） 无限重传直到确认 DATA_CHANNEL_RELIABLE
    按重传次数限制（RTX） 最多重传 N 次后放弃 DATA_CHANNEL_PARTIAL_RELIABLE_REXMIT
    按时间限制（Timed） 超过 T ms 后放弃 DATA_CHANNEL_PARTIAL_RELIABLE_TIMED

    每种策略还可以叠加无序交付（Unordered）选项，对应 _UNORDERED 后缀。

    3.4 多宿（Multi-homing）

    SCTP 允许一个端点绑定多个 IP 地址，在主路径失效时自动切换到备用路径，实现传输层的故障转移。

    在 WebRTC 场景中，由于 SCTP 运行在 DTLS 之上（SCTP over DTLS），而 DTLS 连接已经由 ICE 完成了路径选择，多宿特性在 WebRTC 中并未被使用（RFC 8261 明确说明 SCTP 此时无法感知底层 IP 地址）。

    3.5 抗 SYN Flood 的四次握手

    TCP 三次握手在服务端回复 SYN-ACK 后即分配连接资源，容易遭受 SYN Flood 攻击。SCTP 采用四次握手 + 有状态 Cookie 机制，服务端在收到第三个包（COOKIE ECHO）之前不分配任何资源，有效防御此类攻击。

    4.1 分层功能视图

    RFC 9260 定义了 SCTP 传输服务的内部功能层次（自上而下）：

    4.2 关键概念

    关联（Association）

    SCTP 中的”连接”称为关联，是两个 SCTP 端点之间共享状态的总称，等价于一个会话。关联由两端的验证标签（Verification Tag）唯一标识。

    流（Stream）

    关联内的单向逻辑信道，每个流有独立的 Stream Sequence Number（SSN）空间。注意：SCTP 的流是单向的，WebRTC DataChannel 是双向的，因此底层实际使用两个方向上对称的流 ID。

    TSN（Transmission Sequence Number）

    全局单调递增的 32 位序列号，标识每个 DATA Chunk。TSN 跨越所有流，用于确认和重传控制。

    SSN（Stream Sequence Number）

    每条流内部的 16 位有序序列号，仅在有序传输时使用，用于流内消息的顺序交付。

    PPID（Payload Protocol Identifier）

    每个 DATA Chunk 携带的 32 位标识，指示上层协议类型。WebRTC 使用 PPID 区分 DCEP 控制消息和用户数据。

    5. SCTP 数据包结构

    5.1 公共包头（Common Header）

    每个 SCTP 数据包以 12 字节的公共包头开始：

    • Source Port / Destination Port：16 位端口号，与 TCP/UDP 类似。
    • Verification Tag：32 位随机验证标签，在关联建立时协商，防止伪造包注入。
    • Checksum：CRC32c 校验和（比 TCP 的 16 位 Internet Checksum 更强）。

    公共包头之后跟随一个或多个 Chunk。一个 SCTP 数据包可以携带多个 Chunk（Chunk Bundling），提升效率。

    5.2 Chunk 结构

    每个 Chunk 都有统一的格式：

    • Chunk Type（8 bit）：最高 2 位决定收到未知 Chunk 类型时的处理策略（丢弃/报错/忽略）。
    • Chunk Flags（8 bit）：各 Chunk 类型有不同含义的标志位。
    • Chunk Length（16 bit）：包含头部在内的 Chunk 总长度（字节），不含填充。
    • Chunk Value：Chunk 具体内容，4 字节对齐（不足时补零填充）。

    5.3 主要 Chunk 类型详解

    DATA Chunk（Type = 0x00）

    承载用户数据的核心 Chunk：

    Chunk Flags 含义：

    • U（Unordered）：置 1 时无序交付，忽略 SSN。
    • B（Beginning fragment）：消息第一个分片。
    • E（Ending fragment）：消息最后一个分片。

    当 B=1 且 E=1 时表示消息未分片，一个 Chunk 承载完整消息。

    INIT Chunk（Type = 0x01）

    发起关联建立请求，包含：

    • Initiate Tag：本端的 Verification Tag，对端后续包必须使用此值。
    • a_rwnd：本端的接收窗口大小（Advertised Receiver Window Credit）。
    • Outbound Streams（OS）：本端期望使用的出向流数量。
    • Inbound Streams（MIS）：本端能接受的最大入向流数量。
    • Initial TSN：初始传输序列号（随机值）。
    • Optional Parameters：扩展参数，如 IP 地址列表、Cookie Preservative、支持的扩展等。

    INIT ACK Chunk（Type = 0x02）

    INIT 的响应，额外包含：

    • State Cookie：服务端生成的有状态 Cookie（包含完整关联参数的签名结构），发回给客户端，客户端须在 COOKIE ECHO 中原样返回。服务端在此阶段不保存任何状态。

    SACK Chunk（Type = 0x03）

    选择性确认（Selective Acknowledgement），关键字段：

    • Cumulative TSN Ack：已连续收到的最高 TSN（期望下一个 TSN = 此值 + 1）。
    • a_rwnd：当前接收窗口大小（动态更新）。
    • Gap Ack Blocks：已收到但尚有空洞的 TSN 范围列表，帮助发送端精确定位丢失包。
    • Duplicate TSNs：重复收到的 TSN 列表（用于诊断）。

    SACK 不会延迟超过 500ms，通常每两个 DATA 包 ACK 一次（类似 TCP 的延迟 ACK）。

    HEARTBEAT / HEARTBEAT ACK Chunk（Type = 0x04 / 0x05）

    用于路径存活检测。发送方携带一个不透明的信息块（Heartbeat Info），接收方原样回送。WebRTC 场景中此机制维持连接保活。

    COOKIE ECHO Chunk（Type = 0x0A）

    客户端将 INIT ACK 中收到的 State Cookie 原样发回，服务端验证后分配资源并完成四次握手。

    COOKIE ACK Chunk（Type = 0x0B）

    服务端对 COOKIE ECHO 的确认，关联建立完成。

    ABORT Chunk（Type = 0x06）

    立即终止关联，不等待对端响应，类似 TCP RST。

    SHUTDOWN / SHUTDOWN ACK / SHUTDOWN COMPLETE（Type = 0x07 / 0x08 / 0x0E）

    有序关闭序列的三个 Chunk，确保数据全部交付后再关闭关联。

    FORWARD TSN Chunk（Type = 0xC0）

    PR-SCTP 扩展（RFC 3758）新增，允许发送方指示接收方跳过某些 TSN（用于已放弃重传的消息），推进接收窗口。

    6. 连接生命周期

    6.1 关联建立：四次握手 + Cookie 机制

    Cookie 机制的安全意义：

    服务端在收到 COOKIE ECHO 并验证 Cookie 的 HMAC 签名之前，完全不分配内存资源。Cookie 内包含了关联所需的全部参数（双方 Verification Tag、初始 TSN 等），服务端通过验证 Cookie 即可重建完整状态，有效防御 SYN Flood 式的资源耗尽攻击。

    6.4 心跳保活机制

    关联建立后，SCTP 定期向对端发送 HEARTBEAT Chunk，对端必须回复 HEARTBEAT ACK。若在超时时间内未收到响应，则标记路径失效。在 WebRTC 场景中，心跳间隔通常为 30 秒。

    7. 可靠性与重传机制

    7.1 TSN 与 SACK

    SCTP 使用 TSN + SACK 实现可靠传输，机制与 TCP 的序列号 + ACK 类似，但有两个关键区别：

    1. TSN 是全局的，不绑定任何特定流，用于确认和重传控制。
    2. SACK 携带 Gap Ack Blocks，允许精确报告非连续的已收数据范围，发送方可以精确知道哪些 TSN 丢失，无需等到超时即可触发快速重传。

    发送方维护一个发送窗口（cwnd），受拥塞控制算法动态调整。接收方通过 SACK 中的 a_rwnd 字段通告可用的接收缓冲区大小，实现流量控制。

    7.2 T3-rtx 重传定时器

    每个目标地址维护一个 T3-rtx 定时器（Retransmission Timer）：

    • 发送 DATA Chunk 时启动定时器。
    • 收到该 Chunk 的 SACK 时停止定时器。
    • 定时器超时则认为数据包丢失，触发重传，并将 cwnd 减半（慢启动）。
    • RTO（Retransmission Timeout）使用指数退避（Exponential Backoff），初始值通常为 1 秒，最大不超过 60 秒。

    SCTP 同样支持快速重传：收到 4 个 Gap Ack Block 指示某 TSN 缺失时，立即重传该 TSN，无需等待 T3-rtx 超时。

    7.3 PR-SCTP：部分可靠性扩展（RFC 3758）

    标准 SCTP 会无限重传直到确认，这对实时应用（如游戏状态同步）不合适，一个已过时的帧即便最终送达也没有意义。PR-SCTP 扩展了两种放弃策略：

    按重传次数（RTX Policy）：发送方为每条消息设置最大重传次数。超过次数后，发送方发送 FORWARD TSN Chunk，通知接收方跳过该 TSN，推进确认点。

    按时间（Timed Policy）：发送方为每条消息设置生存时间（TTL）。超时后同样发送 FORWARD TSN 放弃该消息。

    这个机制使 WebRTC DataChannel 能够配置为”发完即忘”的不可靠信道，非常适合低延迟场景。

    8. 拥塞控制与流量控制

    8.1 流量控制（Flow Control）

    SCTP 使用基于接收窗口的流量控制：

    • 接收端在每个 SACK 中通告 a_rwnd（可用接收缓冲区大小）。
    • 发送端在途未确认数据量（飞行中字节数，in-flight bytes）不得超过 min(cwnd, a_rwnd)。
    • 当接收端缓冲区满时，a_rwnd = 0，发送端停止发送（Zero Window Probe 探测除外）。

    8.2 拥塞控制（Congestion Control）

    SCTP 的拥塞控制算法与 TCP Reno/NewReno 类似，每个目标地址独立维护：

    慢启动（Slow Start）：

    • 初始 cwnd = min(4 * PMTU, max(2 * PMTU, 4380))（字节）。
    • 每收到一个 SACK，cwnd += min(outstanding_bytes, PMTU)（指数增长）。
    • 直到 cwnd >= ssthresh，进入拥塞避免。

    拥塞避免（Congestion Avoidance）：

    • 每个往返时间（RTT）内，cwnd += PMTU * PMTU / cwnd（线性增长）。

    丢包响应：

    • T3-rtx 超时：ssthresh = max(cwnd/2, 4*PMTU)，cwnd = PMTU，重回慢启动。
    • 快速重传（收到 4 个 Gap）：ssthresh = max(cwnd/2, 4*PMTU)，cwnd = ssthresh，进入拥塞避免。

    ECN（显式拥塞通知）：

    SCTP 支持 ECN（RFC 3168），网络设备可以在 IP 头部标记拥塞信号，SCTP 端点据此调整发送速率，避免实际丢包。

    9.2 RFC 8261 规范要点

    RFC 8261 详细定义了 SCTP over DTLS 的封装规则：

    SCTP 的无感知性：由于 SCTP 运行在 DTLS 之上，它感知不到底层 IP 地址。SCTP 的多宿特性因此无法在 WebRTC 中使用。ICE 负责处理多路径和 NAT 穿透，DTLS 连接建立在 ICE 选择的最优路径上，SCTP 只看到一个单一的 DTLS 传输。

    端口号：在 SCTP over DTLS 模式下，SCTP 的源/目的端口通常由实现内部管理，常见实现使用固定端口（如 5000），对外不感知。

    PMTU 处理：SCTP 需要感知路径 MTU。在 DTLS 封装下，实际可用 PMTU 需减去 DTLS 和 UDP/IP 报头的开销（通常约 60 字节），SCTP 应使用调整后的 PMTU 进行分片决策。

    校验和：RFC 8261 建议在 SCTP over DTLS 场景下，可以禁用 SCTP 的 CRC32c 校验和（因为 DTLS 已提供完整性保护），但这是可选优化。

    9.3 SDP 协商（RFC 8841）

    在 WebRTC 信令阶段，SCTP-over-DTLS 通过 SDP 进行协商。RFC 8841 定义了两个新的 SDP proto 值：

    • UDP/DTLS/SCTP：SCTP over DTLS over UDP（最常用）
    • TCP/DTLS/SCTP：SCTP over DTLS over TCP（用于 UDP 不可用时的降级）

    SDP 示例：

    关键 SDP 属性：

    • a=sctp-port：SCTP 在 DTLS 层内使用的端口号（通常为 5000）。
    • a=max-message-size：支持的最大消息大小（字节），0 表示无限制（受实现限制）。

    10. SCTP 与 DataChannel 的映射关系

    10.1 DataChannel 到 SCTP Stream 的映射

    每个 WebRTC DataChannel 对应一个 SCTP 流（Stream），且是双向的（实现上在两个方向使用相同的 Stream ID）。

    Stream ID 的分配遵循奇偶规则（RFC 8832）：

    • Offer 发起方使用偶数 Stream ID（0, 2, 4, …）。
    • Answer 响应方使用奇数 Stream ID（1, 3, 5, …）。
    • 这避免了两端同时创建 DataChannel 时的 ID 冲突。

    10.2 PPID：Payload Protocol Identifier

    WebRTC 使用以下 PPID 值（已在 IANA 注册）区分不同类型的数据：

    PPID 值 名称 说明
    50 WebRTC DCEP DCEP 控制消息（通道建立）
    51 WebRTC String DataChannel 字符串消息（非空）
    53 WebRTC Binary DataChannel 二进制消息（非空）
    56 WebRTC String Empty 空字符串消息
    57 WebRTC Binary Empty 空二进制消息

    空消息使用单独 PPID 是因为 SCTP DATA Chunk 要求 User Data 字段非空，通过 PPID 区分来表达零长度消息的语义。

    10.3 DCEP：数据通道建立协议（RFC 8832）

    SCTP 本身没有”通道名称”概念，DataChannel 的 label 和 protocol 等元数据由 DCEP 在带内传输（In-band）。DCEP 只有两个消息：

    DATA_CHANNEL_OPEN（Message Type = 0x03）：

    Channel Type 枚举值（对应 PR-SCTP 策略）：

    值 名称 语义
    0x00 DATA_CHANNEL_RELIABLE 有序、全可靠
    0x80 DATA_CHANNEL_RELIABLE_UNORDERED 无序、全可靠
    0x01 DATA_CHANNEL_PARTIAL_RELIABLE_REXMIT 有序、按重传次数限制
    0x81 DATA_CHANNEL_PARTIAL_RELIABLE_REXMIT_UNORDERED 无序、按重传次数限制
    0x02 DATA_CHANNEL_PARTIAL_RELIABLE_TIMED 有序、按时间限制
    0x82 DATA_CHANNEL_PARTIAL_RELIABLE_TIMED_UNORDERED 无序、按时间限制

    DATA_CHANNEL_ACK（Message Type = 0x02）：

    接收方打开通道后，回复此单字节消息作为确认，通道正式激活。

    DCEP 消息通过 PPID=50 的 SCTP DATA Chunk 传输，使用所对应流的 Stream ID。

    11. SCTP 与 TCP/UDP 的横向对比

    特性 TCP UDP SCTP
    传输模型 字节流 数据报 消息（保留边界）
    可靠性 全可靠 不可靠 可配置（全可靠 / 部分可靠）
    有序性 全局有序 无序 流内有序（可选无序）
    队头阻塞 有（全局） 无 无（流间独立）
    连接建立 三次握手 无连接 四次握手 + Cookie
    多路复用 无（需多个 socket） 无 有（多流）
    多宿支持 无 无 有（WebRTC 中未使用）
    消息大小限制 无（字节流） 64KB（IPv4） 无限制（自动分片）
    拥塞控制 有 无 有（类 TCP）
    流量控制 有 无 有
    SYN Flood 防护 弱（SYN Cookie 可选） — 强（Cookie 机制）
    WebRTC 中用途 信令（WebSocket） 媒体（SRTP） DataChannel

    12. 实现与调试

    12.1 主流实现库

    usrsctp：由 Michael Tuexen 维护的用户空间 SCTP 实现，是 WebRTC 生态中最重要的 SCTP 库。Chromium、Firefox 和大多数 WebRTC 原生库都基于 usrsctp。

    • 项目地址：https://github.com/sctplab/usrsctp
    • 支持 PR-SCTP（RFC 3758）、流重置（RFC 6525）等扩展。

    内核实现：Linux 内核（net/sctp）、FreeBSD（SCTP 参考实现）、Windows 均有内核级 SCTP 支持，但 WebRTC 通常使用用户空间实现以跨平台。

    12.2 Wireshark 抓包分析

    由于 WebRTC 的 SCTP 运行在 DTLS 之上，Wireshark 默认无法解密。调试步骤：

    1. 导出 DTLS 会话密钥（通过 SSLKEYLOGFILE 环境变量或 WebRTC 内部接口）。
    2. 在 Wireshark 中配置 Pre-Master Secret 文件路径（Edit → Preferences → Protocols → TLS）。
    3. 解密后 Wireshark 能识别 SCTP over DTLS 协议，可以查看 INIT/INIT ACK/DATA/SACK 等 Chunk 详情。

    也可以使用 chrome://webrtc-internals 中的统计数据监控 DataChannel 的收发状态：

    • dataChannelMessagesSent / dataChannelMessageReceived
    • dataChannelBytesSent / dataChannelBytesReceived

    14. 小结

    SCTP 在 WebRTC DataChannel 中扮演着承上启下的关键角色。理解它的工作原理，有助于我们在以下场景做出正确决策：

    选择 DataChannel 类型时：了解 PR-SCTP 的两种策略（RTX vs Timed）有助于根据应用需求精确配置可靠性。游戏状态同步适合 Timed 策略，文件传输适合全可靠模式，控制消息适合有序可靠。

    排查延迟问题时：DataChannel 的延迟抖动往往来自 SCTP 的重传定时器（T3-rtx）或拥塞控制导致的发送暂停，结合 webrtc-internals 统计数据可以定位瓶颈。

    理解 DataChannel 与媒体通道的关系时：媒体走 SRTP over UDP，DataChannel 走 SCTP over DTLS over UDP，两者共享同一个 ICE 通道（通过 DTLS 多路复用），但传输语义完全独立。
