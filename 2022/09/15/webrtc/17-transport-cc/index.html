    1. 背景与动机

    1.1 为什么需要 Transport-CC

    WebRTC 早期的拥塞控制采用 REMB（Receiver Estimated Maximum Bitrate） 机制。在 REMB 体系下，接收端承担了延迟梯度（delay gradient）的计算与带宽估计的核心逻辑，再通过 RTCP 消息将估算结果上报给发送端。

    这种架构存在几个根本性缺陷：

    信息损耗：接收端在本地完成估算，发送端只能得到一个聚合后的”最大码率”数值，丢失了每个数据包的细粒度到达时序信息。发送端无法独立判断拥塞的具体形态。

    版本同步困难：拥塞控制算法分布在发送端和接收端两侧。每次算法迭代升级，必须同时协调两端的实现版本，在多厂商互联互通的场景下极为复杂。

    流粒度过粗：REMB 以单条媒体流（RTP stream）为单位反馈，而网络拥塞本质上发生在传输层（即整条 UDP 连接），与流边界无关。当音视频同时传输时，各流的独立反馈难以全面刻画共享瓶颈的真实状态。

    丢包检测延迟：流 A 的包丢失，必须等到流 A 的下一个包才能发现。若流 B 先到达，这条信息就白白浪费了。

    Transport-CC 正是为了解决上述问题而提出的。它将复杂的估算逻辑全部集中到发送端，接收端只做一件事：如实记录每个包的到达时间，并原样上报。

    1.2 标准化历程

    Transport-CC 最初以 IETF Internet-Draft 形式提出：

    • draft-holmer-rmcat-transport-wide-cc-extensions-01（2015 年）：由 Google 工程师 Stefan Holmer、Magnus Flodman、Erik Språng 起草，定义了 RTP 头扩展（传输层序列号）和 RTCP 反馈报文（RTPFB，PT=205，FMT=15）。
    • 该草案虽未正式进入 RFC 轨道，但已被 Chrome、Firefox、Safari 等主流浏览器以及绝大多数 WebRTC SFU（如 mediasoup、Janus、Pion）广泛实现。
    • transport-wide-cc-02（实验性扩展）：Google 在 webrtc.org 源码中进一步演进，增加了”按需反馈”机制，以 http://www.webrtc.org/experiments/rtp-hdrext/transport-wide-cc-02 为扩展 URI 发布。

    2. Transport-CC 概述

    2.1 核心设计思想

    整个流程分为三个层次：

    1. 标注层：每个出站 RTP 包在头部携带一个传输层序列号（transport-wide sequence number）。
    2. 反馈层：接收端周期性地将包的到达状态（是否到达、到达时间增量）封装进 RTCP 反馈报文，发回给发送端。
    3. 估算层：发送端结合本地的发送时间记录和收到的反馈，在发送侧完整地执行拥塞检测与带宽估计。

    2.2 关键属性

    属性 描述
    作用范围 传输层（Transport-wide），覆盖同一 DTLS/ICE 连接上的所有 RTP 流
    反馈方向 接收端 → 发送端（via RTCP）
    逻辑集中点 发送端（Sender-side BWE）
    依赖 RFC RFC 3550（RTP）、RFC 4585（RTCP Feedback）、RFC 5285（RTP Header Extensions）
    核心草案 draft-holmer-rmcat-transport-wide-cc-extensions-01

    3. RTP 头扩展：传输层序列号

    3.1 语义

    每个通过同一传输连接（如通过 BUNDLE 复用的单一 DTLS 连接）发出的 RTP 包，都被打上一个统一的、单调递增的 16 位序列号。这个序列号跨越所有媒体流，不区分音频流、视频流或 DataChannel（尽管 DataChannel 走 SCTP，不在此列）。

    与每条 RTP 流自有的 RTP 序列号不同，transport-wide 序列号的语义是：

    • 唯一标识一次”传输事件”，即一个 UDP 数据报的发出。
    • 允许发送端在不知道接收端具体接收了哪条流的情况下，精准追踪任意包的命运。

    3.2 头扩展格式

    按照 RFC 5285 的一字节头（One-Byte Header）格式：

    字段说明：

    • 0xBE 0xDE：RFC 5285 的 Magic Cookie，标识使用一字节头格式。
    • ID：4 bits，SDP 协商确定的扩展 ID（通常为 5）。
    • L=1：4 bits，数据长度为 L+1 = 2 字节。
    • transport-wide sequence number：16 bits 无符号整数，每发一个包递增 1，在整个传输连接范围内单调递增（溢出后回绕到 0）。

    3.3 提前检测丢包

    传输层序列号的一个额外好处是提前发现丢包。假设：

    • 流 A 先发出序列号 100、101、102；
    • 流 B 随后发出序列号 103；
    • 接收端收到 100、102、103，未收到 101。

    在旧体系（REMB）下，流 A 的丢包只有在流 A 的下一个包（假设为序列号 104）到达时才能察觉。但在 Transport-CC 体系下，流 B 的包（序列号 103）到达时，接收端便可确认序列号 101 丢失，并在下一次 RTCP 反馈中上报，从而加速丢包检测与重传触发。

    4. RTCP 反馈报文格式详解

    Transport-CC 的反馈报文是一个 RTCP Transport Feedback（RTPFB）消息：

    • Payload Type（PT）= 205（RTPFB，由 RFC 4585 定义）
    • Feedback Message Type（FMT）= 15

    4.1 报文总体结构

    4.2 各字段含义

    通用 RTCP 头部

    字段 大小 说明
    V 2 bits RTP 版本，固定为 2
    P 1 bit Padding 标志
    FMT 5 bits Feedback 类型，固定为 15
    PT 8 bits Payload Type，固定为 205
    length 16 bits 报文长度（单位为 32-bit 字，不含前 4 字节）
    SSRC of packet sender 32 bits 发出本 RTCP 报文的实体的 SSRC（接收端）
    SSRC of media source 32 bits 被反馈的媒体源 SSRC（传输层反馈中可为任一媒体流的 SSRC）

    Transport Feedback 专属字段

    字段 大小 说明
    base sequence number 16 bits 本次反馈中第一个被描述的传输层序列号
    packet status count 16 bits 本次反馈描述的包总数，从 base sequence number 开始连续计数
    reference time 24 bits 有符号整数，绝对参考时间，单位 64ms。接收端自选基准时刻，用于计算首个 recv delta
    fb pkt. count 8 bits 反馈报文计数器，每发一个反馈包递增 1，用于检测反馈包丢失
    packet chunk 16 bits × N 数据包状态块（详见第 5 节）
    recv delta 8 or 16 bits × M 对每个”已收到”状态的包，追加其相对于上一已收包的到达时间增量（详见第 6 节）

    reference time 的重要性：由于 recv delta 是相对值（差分编码），若某个反馈报文丢失，后续报文的增量将无法还原为绝对时间。reference time 提供了一个绝对锚点，即便中间有反馈报文丢失，发送端也能将增量序列重新对齐到正确的时间轴。

    5. 数据包状态编码

    5.1 状态符号（Packet Status Symbol）

    每个包用一个 2-bit 符号描述其状态：

    符号值 含义
    00 Packet not received（未收到）
    01 Packet received, small delta（已收到，时间增量较小，用 1 字节表示）
    10 Packet received, large or negative delta（已收到，时间增量较大或为负，用 2 字节表示）
    11 保留（Reserved）

    注意：状态为”未收到”的包不一定已经丢失，可能只是还未到达（乱序）。发送端应等待足够长的时间再做判断。

    5.2 Run Length Chunk（游程编码块）

    当连续多个包具有相同状态时，使用游程编码（RLE）压缩：

    • T = 0：标识为 Run Length Chunk
    • S（2 bits）：重复的状态符号
    • Run Length（13 bits）：重复次数，最大 8191

    示例：0|00|0000000011101 表示 221 个”未收到”的包。

    5.3 Status Vector Chunk（状态向量块）

    当包的状态各不相同时，使用向量编码，每个 chunk 可描述 7 或 14 个包：

    • T = 1：标识为 Status Vector Chunk
    • S = 0：每个 symbol 占 1 bit（仅区分”收到”与”未收到”，共 14 个包）
    • S = 1：每个 symbol 占 2 bit（完整 4 态，共 7 个包）

    两种 chunk 格式可在同一反馈报文中混用，由实现自行选择最紧凑的编码方式。

    6. 接收时间增量（Receive Delta）

    对每个状态为”已收到”的包，紧随所有 packet chunk 之后，顺序追加该包相对于上一个已收到包的到达时间增量（receive delta）。

    6.1 Small Delta（1 字节）

    对应状态符号 01，时间增量范围：[0, 63.75ms]，分辨率 250μs（0.25ms）：

    值 = 实际增量（μs）/ 250

    6.2 Large or Negative Delta（2 字节）

    对应状态符号 10，时间增量范围：[-8192ms, +8191.75ms]，分辨率同为 250μs：

    有符号 16-bit 整数，值 = 实际增量（μs）/ 250。

    负增量表示乱序到达（后发先至）。

    6.3 时间重建示例

    发送端拿到这些到达时间后，与本地记录的 send_time[seq] 相减，得到各包的单向延迟（One-Way Delay，OWD），再进一步分析延迟变化趋势。

    7. 发送端带宽估计流程

    拿到 RTCP Transport Feedback 后，发送端执行以下流程：

    包组（Packet Group）划分：发送端按发送时间将包分组（窗口约 5ms），同组内包被当作一个整体来计算组间延迟变化，以降低单包抖动的影响。

    单向延迟变化公式：

    $$d(i) = (A_i - S_i) - (A_{i-1} - S_{i-1})$$

    其中：

    • $A_i$ = 第 i 组最后一个包的到达时间
    • $S_i$ = 第 i 组第一个包的发送时间

    8. Trendline Filter：取代 Kalman Filter

    GCC 第一代（REMB 时代）使用 Kalman Filter（卡尔曼滤波器） 对接收端侧的延迟梯度进行估计。Transport-CC 将拥塞检测迁移到发送端后，WebRTC 团队将其替换为更简洁的 Trendline Filter（趋势线滤波器）。

    8.1 核心思路

    Trendline Filter 本质上是对累积延迟变化序列做线性回归，用回归直线的斜率来判断网络队列的增长趋势：

    1. 计算第 i 包组的累积延迟（accumulated delay）：

    $$\text{accumulated_delay}i = \sum{k=1}^{i} d(k)$$

    1. 对累积延迟序列施加指数平滑：

    $$\text{smoothed_delay}i = \alpha \cdot \text{smoothed_delay}{i-1} + (1 - \alpha) \cdot \text{accumulated_delay}_i$$

    其中 $\alpha$ 为平滑系数（典型值接近 0.9）。

    1. 以到达时间为 x 轴，以 smoothed_delay 为 y 轴，对最近 N 个数据点（默认约 20 个包组）做最小二乘线性回归，求出斜率 $k$。

    2. 斜率 $k$ 即为 Trendline 的输出，反映排队延迟的增长速率：

      • $k > 0$ ：延迟在增长，队列在积压，可能发生 overuse；
      • $k \approx 0$：延迟稳定；
      • $k < 0$ ：延迟在下降，队列在清空，underuse。

    8.2 Overuse Detector 与动态阈值

    Trendline 的斜率 $k$ 被送入 Overuse Detector，与一个动态阈值 $\gamma$ 比较：

    条件 信号
    $k > \gamma$ Overuse（过载）
    $-\gamma \le k \le \gamma$ Normal（正常）
    $k < -\gamma$ Underuse（欠载）

    阈值 $\gamma$ 并非固定值，而是根据历史信号动态调整：拥塞时适当放宽，以避免震荡；空闲时收紧，提高灵敏度。这一设计被称为 Adaptive Threshold（自适应阈值）。

    相比 Kalman Filter，Trendline Filter 的实现更简单、参数更少、对信号噪声的鲁棒性更好，且由于全部逻辑在发送端，便于快速迭代和 A/B 测试。

    8.3 Rate Controller（AIMD）

    Overuse Detector 的输出驱动一个 AIMD（Additive Increase Multiplicative Decrease）控制器：

    • Overuse → 乘法减少：$A_r = \beta \cdot \hat{R}$（$\beta \approx 0.85$），快速降低发送码率。
    • Normal → 加法增加：缓慢探测可用带宽。
    • Underuse → 保持或小幅增加。

    9. 与 REMB 的对比

    维度 REMB Transport-CC（TWCC）
    估算位置 接收端（Receiver-side） 发送端（Sender-side）
    反馈内容 单一聚合估算值（最大码率） 每个包的到达时间增量
    反馈粒度 较粗（帧级） 精细（包级）
    算法可升级性 差（需同步两端） 好（仅需升级发送端）
    跨流感知 无（单流独立反馈） 有（传输层统一视角）
    SFU 友好度 低 高（SFU 作为发送端，完全掌控估算）
    标准化状态 从未正式进入 RFC 草案（被广泛实现）
    并用限制 可与 Transport-CC 并存 不可与 CCFB（RFC 8888）同时使用

    9.1 REMB 的遗留价值

    尽管 Transport-CC 在设计上更优越，REMB 并未完全消失。在以下场景中 REMB 仍然有意义：

    • 对端不支持 Transport-CC 扩展时的兼容回退。
    • 某些简单部署场景中，REMB 的实现成本更低。
    • 作为 Transport-CC 的辅助参考信号（双机制并行）。

    在实际的 Chrome 实现中，当双方协商了 Transport-CC，发送端会优先使用基于 Transport Feedback 的带宽估计，REMB 作为备用或兜底。

    10. Transport-CC v2 扩展

    Google 在 webrtc.org 中进一步演进了该协议，形成了实验性的 transport-wide-cc-02 版本。

    官方 URI：http://www.webrtc.org/experiments/rtp-hdrext/transport-wide-cc-02

    相比 v1，主要有两处改动：

    10.1 按需反馈（On-demand Feedback）

    v1 的接收端以固定周期（通常每帧或每 RTT）发送反馈。v2 改为由发送端主动请求：在 RTP 包的头扩展中附加一个可选的 Feedback Request 字段：

    • 当扩展长度为 L=3（即 4 字节数据）时，后两个字节为 Feedback Request：
      • T（1 bit）：若置 1，要求反馈报文包含时间信息（timing information）。
      • seq count（15 bits）：要求接收端在反馈中包含最近 seq count 个包的历史状态。若 seq count = 0，等效于不发 Feedback Request（仅保持固定头长度）。

    10.2 可选时间信息

    v2 允许发送端决定是否要求接收端在反馈中包含时间戳。对于仅需丢包信息（不需要延迟估计）的场景，可节省部分带宽开销。

    10.3 与 CCFB（RFC 8888）的关系

    RFC 8888（Congestion Control Feedback in RTCP）是 IETF 正式标准化的另一套拥塞控制反馈机制。它与 Transport-CC 在设计目标上有重叠，两者不能在同一 SDP 媒体描述中同时使用。在 SDP Answer 中，只能选择其中之一。

    11.3 完整的 SDP 片段示例

    11.4 跨媒体 Section 的统一性

    由于 Transport-CC 作用于传输层，设计上应在所有 media section 中指定相同的扩展 ID，以保证序列号计数器的统一性。不过历史上也存在仅对视频配置该扩展、对音频忽略的实现——在这种情况下，拥塞控制将忽略音频流的贡献。

    12. 开销分析

    12.1 RTP 头开销

    每个 RTP 包额外携带 2 字节（序列号）+ 扩展头定长部分，合计约 4 字节。对于典型的 1200 字节视频包，这是约 0.33% 的额外开销，可忽略不计。

    12.2 RTCP 反馈开销

    RTCP Transport Feedback 的大小取决于包的数量和状态分布：

    • 固定头：20 字节（RTCP 公共头 4B + SSRC×2 共 8B + base seq + pkt status count + ref time + fb pkt count 共 8B）
    • 每 16 个包（最坏情况，全部有 delta）：约 2B（状态 vector chunk）+ 7×2B（large delta）= 16B
    • 典型场景（全部小 delta）：约 2B（RLE chunk）+ N×1B

    官方草案建议每帧发送一次反馈，在 30fps 视频下约 30 次/秒。假设每次反馈 40 字节，带宽消耗约 9.6 kbps，与音视频主流量相比微乎其微。

    在上行带宽受限的场景（如移动网络），可降低反馈频率至每 RTT 一次，以节约上行资源。

    13. 实现细节与注意事项

    13.1 反馈报文丢失的处理

    RTCP 反馈报文本身在网络中可能丢失。处理原则：

    • 对于在丢失的反馈报文中应当被报告为”已收到”或”丢失”的包，发送端直接忽略这批包（不纳入延迟估计计算）。
    • 通过 reference time 和 fb pkt count，发送端可感知反馈丢失并做相应补偿。
    • 此行为与 RTCP Receiver Report 丢失时的处理方式一致。

    13.2 序列号回绕

    16-bit 序列号在约 65535 个包后回绕。在 1 Gbps 链路上每秒约发 83000 个 1500 字节的包，回绕周期约 0.8 秒。实现时必须正确处理回绕，通常采用 uint16_t 加法加上有符号差值比较来判断先后顺序。

    13.3 接收端时钟精度

    接收端记录到达时间时，时钟精度直接影响带宽估计质量。建议使用系统的高精度单调时钟（如 Linux 的 CLOCK_MONOTONIC），精度应达到 1ms 或更好。

    13.4 乱序包的处理

    Transport Feedback 的 base sequence number 可以比上一次反馈的末尾序列号更小（即向前”退缩”），以覆盖乱序到达的旧包。但对于此前已经报告为”已收到”的包，不应再重复报告，应标记为”未收到”以避免歧义。

    13.5 与 Pacer 的协作

    Transport-CC 的带宽估计输出（目标码率）通常不直接驱动编码器，而是先经过 Pacer（包速率平滑器） 平滑后再发送。Pacer 将突发帧拆散成均匀的包流，避免瞬时突发导致估计器误判过载。这一点在阅读 WebRTC 源码时尤为关键。

    14. 与其他模块的协作关系

    Transport-CC 在整个 WebRTC 拥塞控制体系中处于”感知神经”的角色：它是 BWE 的数据来源，BWE 的输出再反馈给编码器、Simulcast/SVC 选层逻辑以及 Pacer，形成完整的自适应码率闭环。
