    1. 概述与背景

    H.264，正式名称为 ITU-T Recommendation H.264 | ISO/IEC 14496-10，又称 Advanced Video Coding（AVC） 或 MPEG-4 Part 10，是当今世界上部署最广泛的视频压缩标准。

    历史沿革

    时间 事件
    2003 年 5 月 由 ITU-T 视频编码专家组（VCEG）与 ISO/IEC 运动图像专家组（MPEG）联合成立的”联合视频团队”（JVT）发布第一版
    2004 年 引入 FRExt（Fidelity Range Extensions），支持更高保真度视频，添加 High Profile
    2007 年 引入可伸缩视频编码扩展（SVC）
    2009 年 引入多视角视频编码扩展（MVC）
    2016 年 IETF RFC 7742 将其列为 WebRTC 强制实现（MTI）编解码器之一

    三个名字指向同一个技术规范：

    • H.264：ITU-T 的命名惯例
    • AVC：全称 Advanced Video Coding，ISO/IEC 侧的名称
    • MPEG-4 Part 10：ISO/IEC 14496-10 的简称

    市场地位

    H.264 是有史以来商业化最成功的视频编码标准之一：

    • 全球超过 90% 的视频流内容使用 H.264 编码
    • 在 2024 年，H.264 在桌面和移动平台拥有约 98.23% 的浏览器兼容性
    • 在移动无线标准中被强制要求用于多媒体电话和分组流媒体
    • 是 HLS、RTMP、WebRTC（RFC 7742）等主流协议的核心或默认编码

    尽管 H.265/HEVC、VP9、AV1 等更新一代编码在压缩效率上有 35%65% 的提升，H.264 凭借三大优势持续主导市场：**2010 年以来制造的设备普遍内置硬件解码器、实时编码速度领先后继编码 340 倍、以及主流协议的强制支持**。

    2. 标准体系与规范文档

    理解 H.264 在 WebRTC 中的应用，需要掌握以下核心规范文档：

    文档 内容
    ITU-T H.264 / ISO/IEC 14496-10 H.264 核心技术规范，定义码流语法、解码过程
    RFC 6184 H.264 视频的 RTP Payload 格式（obsoletes RFC 3984）
    RFC 7742 WebRTC 视频处理与编解码器要求，确立 H.264 为 MTI
    RFC 3984 RFC 6184 的前身，历史参考

    这三层规范形成清晰的分工：H.264 标准本身定义”如何压缩”，RFC 6184 定义”如何封包进 RTP”，RFC 7742 定义”WebRTC 场景下必须支持哪些配置”。

    3. 编码核心原理

    H.264 是一种基于宏块（Macroblock）的混合编码器，其核心思路是利用视频中的时间冗余（相邻帧内容相似）和空间冗余（同一帧内相邻区域相似）来实现高效压缩。

    3.1 两层架构：VCL 与 NAL

    H.264 在架构上将编解码职责切分为两层，这一设计使其能够在不同网络协议上灵活运作：

    • VCL（Video Coding Layer）：执行帧内/帧间预测、变换、量化、熵编码等核心压缩操作，其输出为”切片（Slice）”，包含若干宏块数据的比特串。
    • NAL（Network Abstraction Layer）：将 VCL 输出的切片封装为 NAL 单元（NALU），并附加元数据（参数集等），使其适合在 RTP 包、字节流等不同传输环境中使用。

    3.2 帧类型与 GOP 结构

    H.264 定义了三种基本帧（更准确地说是”切片类型”）：

    帧类型 全称 说明
    I 帧 Intra-coded Frame 帧内编码，不依赖任何其他帧，可独立解码。也称关键帧。特殊的 I 帧叫 IDR（Instantaneous Decoding Refresh），解码器收到后会立即清空已缓存的参考帧
    P 帧 Predictive Frame 仅参考前面已解码的帧进行预测，压缩率比 I 帧高
    B 帧 Bi-directional Frame 同时参考前后帧进行双向预测，压缩率最高，但引入额外延迟，WebRTC 实时通话通常不启用 B 帧

    多帧组成一个 GOP（Group of Pictures）。在实时通信场景下，IDR 帧通常由对端通过 RTCP PLI（Picture Loss Indication） 或 FIR（Full Intra Request） 触发，以从丢包中快速恢复。

    3.3 宏块与预测编码

    宏块（Macroblock，MB） 是 H.264 编码的基本处理单位，每个宏块固定为 16×16 亮度（luma）样本，加上对应的色度（chroma）样本（在 4:2:0 格式下为 2 个 8×8 的色度块）。

    帧内预测（Intra Prediction）：利用已编码的相邻宏块像素值预测当前宏块，I 帧中每个宏块均使用帧内预测。H.264 支持 4×4、8×8 和 16×16 三种帧内预测块大小，4×4 模式有 9 种预测方向，能精细描述纹理细节。

    帧间预测（Inter Prediction）：通过运动补偿实现。编码器在参考帧中搜索与当前块最相似的区域（运动估计），记录其偏移量（运动矢量），仅编码残差。H.264 支持 1/4 像素精度的亚像素插值运动补偿，并引入了多参考帧机制（最多 16 个参考帧），显著提升了对复杂运动场景的预测精度。

    H.264 还支持灵活的可变块大小分割（Variable Block-size Motion Compensation，VBMC），宏块可被进一步划分为 16×8、8×16、8×8 及更小的子块，每个子块独立进行运动估计，大幅提升了运动复杂场景的压缩效率。

    3.4 变换、量化与熵编码

    变换（Transform）：对帧内/帧间预测的残差信号施加整数 DCT 变换，将空间域能量集中到少数低频系数。H.264 使用的是精确的 4×4 整数 DCT，避免了浮点运算，保证了编解码器间的一致性。

    量化（Quantization）：通过量化参数 QP（Quantization Parameter） 控制压缩损失。QP 范围 0~51，QP 增大 6 约等于码率减半（质量下降）。编码器根据带宽估计（BWE）动态调整 QP，这是 WebRTC 码率自适应的核心机制之一。

    熵编码（Entropy Coding）：H.264 支持两种熵编码方式：

    • CAVLC（Context-Adaptive Variable Length Coding）：计算复杂度较低，用于 Baseline/Main Profile
    • CABAC（Context-Adaptive Binary Arithmetic Coding）：压缩效率约高出 10%~15%，计算复杂度稍高，用于 Main/High Profile。Constrained Baseline Profile 不支持 CABAC

    去块滤波（In-loop Deblocking Filter）：编码环路内的自适应去块滤波器，平滑宏块边界的方块效应（Blocking Artifact），同时作用于参考帧以提升后续帧的预测质量。这是 H.264 相对早期标准的重要改进之一。

    4. NAL 单元详解

    4.1 NAL 头结构

    每个 NAL 单元以 1 字节的头部开始，其位域定义如下（RFC 6184 §5.3）：

    字段 位数 含义
    F（forbidden_zero_bit） 1 bit 规范要求必须为 0；置 1 表示语法错误
    NRI（nal_ref_idc） 2 bit 重要性指示，00=非参考帧数据，11=最重要（如 IDR、SPS、PPS）
    Type（nal_unit_type） 5 bit NAL 单元类型，123 由 H.264 规范定义，2431 由 RFC 6184 扩展用于 RTP 封装

    4.2 关键 NAL 类型

    Type 值 名称 说明
    1 Coded slice of a non-IDR picture 普通 P/B 帧的切片
    5 Coded slice of an IDR picture IDR 关键帧，解码器收到后清空 DPB（Decoded Picture Buffer）
    6 SEI（Supplemental Enhancement Information） 辅助增强信息，携带时间戳、显示方向、画面冻结等元数据
    7 SPS（Sequence Parameter Set） 序列参数集，包含 Profile、Level、分辨率、帧率等解码所需的全局配置
    8 PPS（Picture Parameter Set） 图像参数集，包含熵编码模式、切片组、运动预测方式、QP、去块滤波等参数
    9 AUD（Access Unit Delimiter） 访问单元分隔符，用于定位帧边界
    24 STAP-A（Single-Time Aggregation Packet type A） RTP 扩展：将多个同时间戳的 NALU 合并入一个 RTP 包
    28 FU-A（Fragmentation Unit type A） RTP 扩展：将一个大 NALU 拆分为多个 RTP 包

    SPS 和 PPS 的重要性：SPS 和 PPS 是解码一切后续 NAL 单元的前提。在 WebRTC 中，每个 IDR 帧之前通常会前置发送 SPS 和 PPS，确保接收端在任何时刻加入会话都能正确初始化解码器。

    4.3 字节流格式与 NALU 流格式

    H.264 码流有两种封装形式：

    Annex B 字节流格式：NALU 之间以起始码（Start Code）0x000001 或 0x00000001 分隔，常见于 MPEG-TS、文件存储和广播场景。

    AVCC/MP4 格式（长度前缀格式）：每个 NALU 前有 4 字节（或 2/1 字节）的长度字段，常见于 MP4 容器和 iOS/Android 平台的 MediaCodec API 输出。

    WebRTC 开发注意：iOS VideoToolbox 输出 AVCC 格式，而 RTP 封装（RFC 6184）要求 NALU 流格式。开发者在对接硬件编码器时，需要在两种格式之间进行转换，通常称为”起始码替换”（Start Code Substitution）。

    5. Profile 与 Level

    H.264 定义了大量可选工具，通过 Profile（配置文件） 和 Level（级别） 两个维度来约束实现复杂度与硬件资源需求。

    5.1 常见 Profile

    Profile profile_idc 主要特性 典型用途
    Baseline Profile (BP) 66 (0x42) 无 B 帧、无 CABAC、无 FMO 低功耗设备、早期移动视频
    Constrained Baseline Profile (CBP) 66 + constraint_set1_flag BP 子集，额外禁止 ASO/FMO WebRTC MTI 要求，兼容性最强
    Main Profile (MP) 77 (0x4D) 支持 B 帧、CABAC DVD、广播
    High Profile (HiP) 100 (0x64) 支持 8x8 帧内预测、自定义量化矩阵 蓝光、高清广播
    Constrained High Profile 100 + constraint_set4/5 High 子集，无 B 帧 WebRTC SHOULD 支持，兼顾质量与兼容性

    Constrained Baseline Profile（CBP）是所有 WebRTC 实现的最低强制要求。值得注意的是，CBP 是 2009 年后 BP 的细化子集——所有 CBP 码流同时也是合法的 BP 码流，因此两者共用 profile_idc = 66，靠 constraint 标志位区分。

    5.2 Level 定义

    Level 定义了编解码器的计算复杂度上限，主要约束参数包括：最大宏块处理速率（max-mbps）、最大帧大小（max-fs，单位为宏块数）、最大码率（max-br）和解码图像缓冲区大小（max-dpb）。

    Level 最大分辨率（示例） 最大帧率 典型应用
    1.2 320×240 15 fps WebRTC CBP 最低要求
    3.1 1280×720 30 fps WebRTC 推荐支持
    4.0 1920×1080 30 fps 高清视频
    4.1 1920×1080 60 fps 蓝光
    5.1 4096×2304 30 fps 超高清

    RFC 7742 要求 WebRTC 实现至少支持 Constrained Baseline Profile Level 1.2，并推荐支持 Constrained High Profile Level 3.1（扩展至 720p@30fps）。

    5.3 profile-level-id 编码规则

    profile-level-id 是 SDP 中用于协商 H.264 能力的关键参数，其值为 6 位十六进制字符串，由三个字节组成：

    字节 含义
    profile_idc Profile 标识符的十进制值（十六进制表示）
    profile_iop Profile 约束标志（constraint flags），各 bit 对应 constraint_set0~5_flag
    level_idc Level 值 × 10（Level 3.1 → 0x1F = 31）

    6. H.264 在 WebRTC 中的应用

    6.1 RFC 7742 强制要求

    RFC 7742 第 6.2 节对 WebRTC 端点提出了以下强制性要求（MUST/SHOULD 语义参照 RFC 2119）：

    1. MUST 支持 RFC 6184 定义的 H.264 RTP payload 格式
    2. MUST 支持 Constrained Baseline Profile Level 1.2
    3. SHOULD 支持 Constrained High Profile Level 3.1（扩展至 720p@30fps）
    4. SDP 中 MUST 包含 profile-level-id 参数，并 MUST 正确解读对端的该参数
    5. MUST 支持 packetization-mode=1（Non-Interleaved Mode）

    此外，RFC 7742 还规定：

    • 默认色彩空间为 sRGB，扫描格式为 Y’CbCr 4:2:0
    • 默认像素宽高比为 1:1（方形像素），除非 SDP 另有指定

    6.2 RTP 封装：RFC 6184

    RFC 6184 定义了三种 NALU 的 RTP 封装模式，由 SDP 参数 packetization-mode 控制：

    模式 packetization-mode 值 支持的 RTP 包类型 说明
    Single NAL Unit Mode 0 仅单 NALU 包 最简单，每包一个 NALU，适合低延迟受控网络
    Non-Interleaved Mode 1 单 NALU、STAP-A、FU-A WebRTC 必须支持，解码顺序等于传输顺序
    Interleaved Mode 2 STAP-B、MTAP16、MTAP24、FU-B 允许乱序传输，增加复杂度，不常用

    三种核心封装包类型详解：

    ① Single NAL Unit Packet（NALU type 1~23）：
    当 NALU 大小不超过 MTU（通常 1200~1400 字节）时，直接将一个完整 NALU 放入一个 RTP 包。RTP payload 即为完整的 NALU（含 NAL 头）。

    ② STAP-A（Single-Time Aggregation Packet type A，NALU type 24）：
    将多个时间戳相同的小 NALU（如 SPS + PPS）合并进一个 RTP 包，节省 RTP 头开销。

    ③ FU-A（Fragmentation Unit type A，NALU type 28）：
    当单个 NALU 超过 MTU 时，将其拆分为多个 FU-A 包传输。每个 FU-A 包含一个 FU indicator 和一个 FU header：

    6.3 SDP 协商示例

    下面是一段典型的 WebRTC Offer SDP 中关于 H.264 的部分：

    关键 SDP 参数说明：

    参数 值示例 含义
    profile-level-id 42e01f CBP Level 3.1
    packetization-mode 1 Non-Interleaved Mode，WebRTC 必须支持
    level-asymmetry-allowed 1 允许发送端和接收端使用不同的 Level，实践中几乎总是设为 1
    max-fs 3600 最大帧大小（以宏块为单位，3600MB ≈ 1080p）
    max-mbps 108000 最大宏块处理速率，超过 Level 默认值时使用
    sprop-parameter-sets base64 字符串 带外（out-of-band）传递的 SPS/PPS，可选

    level-asymmetry-allowed 是一个重要的实践参数。当设为 1 时，Offer 方声明的 Level 仅代表其发送能力，而不约束其接收能力。这意味着一个移动端（发送 Level 3.1）可以接收来自高端服务器（发送 Level 5.1）的视频流，在异构设备互联时非常关键。

    6.4 SEI 消息与 WebRTC 的特殊要求

    RFC 7742 对 SEI（Supplemental Enhancement Information）消息提出了特定要求：

    • Filler payload SEI 和 Full frame freeze SEI：H.264 编码端 MAY 发送，接收端 MUST 能正确处理。Full frame freeze 在视频切换 MCU 中用于保持稳定的显示画面，避免切换时的花屏。
    • Display Orientation SEI（显示方向）：若未使用 CVO RTP 头扩展进行方向信令，实现端 MAY 发送，SHOULD 能正确解读此 SEI 消息，以支持横竖屏旋转。
    • User data registered/unregistered SEI：实现端 MAY 发送和处理，即便不处理，也 MUST 在收到时不产生错误行为。

    7. 硬件加速

    H.264 是目前硬件加速最成熟的视频编解码标准，几乎所有 2010 年后生产的消费级设备都内置了 H.264 硬件解码器。

    7.1 各平台硬件编解码器

    平台 硬件 API / 框架 说明
    macOS / iOS VideoToolbox Apple 提供的原生硬件编解码框架，WebRTC native SDK 已集成，iOS Safari 的 WebRTC 视频强制使用 H.264 + VideoToolbox
    Android MediaCodec Android 4.1+ 提供的硬件编解码 API，WebRTC Android SDK 通过 HardwareVideoEncoderFactory 调用
    Windows Media Foundation / DXVA2 DirectX Video Acceleration，DirectX 集成的视频加速 API
    NVIDIA GPU NVENC / NVDEC 适用于服务器端 SFU/MCU 的高性能 GPU 加速，GTX 600 系列及以上支持
    Intel CPU Intel Quick Sync Sandy Bridge（2011）起，CPU 集成显卡支持 H.264 硬件编解码
    AMD GPU VCE / VCN AMD Video Codec Engine，驱动通过 VAAPI（Linux）或 AMF（Windows）暴露
    Linux VAAPI / V4L2 M2M Video Acceleration API，ARM 设备广泛使用 V4L2 M2M 接口

    软件编码器：开源软件编码器 x264 是 H.264 软件编码的事实标准，广泛用于 FFmpeg、OBS Studio 等工具中，提供极高的压缩效率但 CPU 开销较大。

    7.2 硬件加速的优势与陷阱

    优势：

    • 显著降低 CPU 占用，释放算力给业务逻辑
    • 在低功耗移动设备上实现高分辨率实时编码
    • 硬件解码延迟通常远低于软件解码

    常见陷阱：

    1. 格式转换：iOS VideoToolbox 输出 AVCC（MP4）格式，必须转换为 Annex B 字节流或直接封装为 RTP NALU 流，否则接收端无法解析。

    2. Profile 兼容性：某些硬件编码器默认使用 High Profile 而非 Constrained Baseline，若 SDP 中协商的是 CBP（42e01f）但实际发送 High Profile 码流，接收端（特别是 Firefox）会拒绝解码。

    3. **并非”保证有效”**：即便设备在硬件上支持 H.264 加速，操作系统、驱动、WebRTC 库版本等中间层任一环节缺失，实际调用都可能回退到软件实现或完全失败。Android 碎片化尤为严重，部分设备虽有硬件编码芯片，但 MediaCodec 接口暴露不正确。

    4. NVENC 并发限制：消费级 NVIDIA GPU（GeForce 系列）对 NVENC 并发编码会话数有限制（通常为 2~3 个），服务器端 SFU 需使用 Tesla/A 系列 GPU 或通过软件 fallback 处理超额请求。

    8. 与其他编码的横向对比

    实践结论：H.264 在 WebRTC 实时通信中仍是综合考量最优的选择。若业务场景允许（如 SFU 控制编码参数、接收端可控），VP9 或 AV1 可带来更低带宽成本，但需承担更高的编码延迟和兼容性风险。

    9. 专利与许可

    H.264 是一个专利密集型标准，实施者需注意以下授权池：

    • MPEG LA AVC Patent Pool：由 Sisvel 管理（2024 年起从 MPEG LA 接手），汇集了全球数十家专利持有人的 H.264 相关必要专利。对最终产品的免费分发（如浏览器向用户免费提供 H.264 解码）通常有豁免条款，但具体条款随时间变化，使用前应查阅最新授权协议。
    • Via Licensing：另一个 H.264 专利池，部分厂商通过此渠道获得授权。
    • 软件实现的授权：使用 OpenH264（由思科开发并捐赠专利授权）作为 H.264 软件编解码库，是 Firefox 的解决方案，可在一定范围内绕过专利授权问题。

    相比之下，VP8、VP9 和 AV1 均为开源免专利授权，这是部分大型互联网公司倾向于推广后三者的重要动因。

    10. 实践建议

    WebRTC 应用开发

    1. SDP 协商层：始终在 SDP 中包含 profile-level-id 和 packetization-mode=1。Offer 中可同时提供 CBP（42e01f）和 Constrained High Profile（640c1f）两个 payload type，由对端在 Answer 中选择兼容的最高质量选项。

    2. 关键帧策略：合理响应 RTCP PLI/FIR，及时发送 IDR 帧。IDR 前必须发送 SPS 和 PPS（可合并为一个 STAP-A 包以减少包数）。关键帧间隔（IDR interval）在实时通话中通常设为 2~5 秒；在网络质量差时应缩短间隔以加速恢复。

    3. 移动端硬件加速：

      • iOS：通过 WebRTC SDK 的 RTCVideoEncoderFactoryH264 使用 VideoToolbox
      • Android：使用 HardwareVideoEncoderFactory，并在初始化时启用 Constrained High Profile（需确认设备支持）
      • 始终提供软件编码（LibvpxVp8Encoder 或 OpenH264）作为硬件加速不可用时的 fallback

    4. NALU 格式转换：处理 iOS/Android 硬件编码器输出时，务必将 AVCC 格式转换为 Annex B 或直接构建 RTP NALU，避免将长度前缀误作 Annex B 起始码发送。

    5. 服务端 SFU：使用 NVENC（h264_nvenc）或 Intel Quick Sync（h264_qsv）进行转码，但注意控制并发 session 数量并配置合适的 Rate Control 模式（推荐 CBR 或 ABR 配合 zerolatency tuning）。

    排查常见问题

    现象 可能原因 排查方向
    花屏/绿屏后恢复 IDR 帧丢失 检查 RTCP PLI 响应逻辑、关键帧发送频率
    单向无视频 profile-level-id 不匹配 检查 SDP 的 profile-level-id，确认双端使用兼容的值
    iOS Safari 无法接收视频 Profile/格式不兼容 Safari 对 H.264 profile 严格，确保使用 CBP 42e01f
    编码 CPU 占用过高 未启用硬件加速 确认 VideoToolbox/MediaCodec 已正确初始化
    第一帧延迟过高 SPS/PPS 在 IDR 帧之前未发送 确认 IDR 前的参数集发送逻辑
