前言：为什么需要 PLC

互联网是一个”尽力而为”的网络，IP 协议从来没有承诺每一个数据包都能按时、完整地送达目的地。对于文件下载、网页浏览这类应用来说，TCP 的重传机制可以透明地补偿丢包，用户大不了等几毫秒而已。但对于实时音频通信，哪怕一帧 20ms 的数据延迟超过 150ms，就已经无法通过重传来挽救了。

这就是 PLC（Packet Loss Concealment，丢包隐藏） 存在的理由。

PLC 是一种纯粹的接收端技术。当某个音频包在网络中丢失，解码器不会输出刺耳的爆破音或一段难堪的静默，而是根据已收到的历史音频合成一段”听起来合理”的替代波形，从而维持声音的连续性。这段合成的音频并不能完全还原丢失的内容，但它能让通话双方感知到的音质损伤降到最低。

丢包的成因与影响

在讨论 PLC 之前，我们需要理解丢包是如何发生的。

主要成因：

  • 网络拥塞（Congestion）：路由器或交换机在数据包积压时会主动丢弃报文。
  • 无线链路干扰（Wireless Interference）：Wi-Fi 与蜂窝网络中的信号衰落、同频干扰都可能导致帧误码，进而造成 IP 包丢失。
  • 缓冲区溢出（Buffer Overflow）：接收端抖动缓冲区（Jitter Buffer）满载时，迟到的包会被主动丢弃。这类丢包在 WebRTC 中称为”延迟丢包”（late loss），本质上和网络丢包对 PLC 的挑战是一样的。
  • 路由切换（Route Change）：网络路径变化导致短暂的包序混乱或丢失。

对音频质量的影响：

丢包率（Packet Loss Rate）对主观音质的影响是非线性的。根据 ITU-T E-Model（G.107）的研究：

  • **0%–1%**：用户几乎察觉不到质量下降。
  • **1%–5%**：偶尔听到”断音”，但不影响理解。
  • **5%–10%**：通话质量明显下降，理解困难度上升。
  • **>10%**：音质严重劣化，通话体验极差。

而丢包发生的模式同样关键：随机散点丢包比突发性连续丢包（Burst Loss）容易隐藏得多。连续丢失 5 个包（约 100ms 的音频）对任何 PLC 算法都是严峻考验。

WebRTC 中的丢包应对体系

WebRTC 并不依赖单一技术来对抗丢包，而是构建了一套分层防御体系，PLC 是其中的最后一道防线：

阶段 技术 机制 额外延迟
发送前 FEC（前向纠错） 发送端附带冗余数据 略有增加
发送前 RED（冗余音频数据） 每包携带前帧副本 略有增加
发送前 DRED（深度冗余） 神经网络编码历史帧特征 约 1s 覆盖
传输中 NACK / RTX（重传） 检测丢包后请求重发 至少 1 个 RTT
接收后 PLC（丢包隐藏） 接收端合成替代音频 零额外延迟

PLC 的核心优势在于零额外延迟：无论丢包多么频繁，PLC 都不需要等待重传，可以立即生成输出。这使它成为实时通信中不可替代的最终保障。

PLC 的基本原理

PLC 的工作机制可以用一个简单的流程描述：

PLC 的核心假设来自语音信号的短时平稳性（Short-time Stationarity）：在几十毫秒的窗口内，语音的音调、共振峰、能量包络变化缓慢。这意味着用刚刚接收到的历史帧来预测或外推下一帧，在短时间内是相当合理的。

一个完整的 PLC 系统通常需要处理三个阶段：

  1. 丢包检测：通过 RTP 序列号缺失判断哪一帧丢失了。
  2. 合成生成：根据历史音频生成替代波形。
  3. 平滑过渡：当后续包重新到来时，通过 Overlap-Add（OLA）技术将合成音频与真实解码音频无缝拼接，避免相位跳变产生的”咔哒”声。

经典 PLC 算法详解

静音替换（Silence Substitution）

最简单也是最差的策略，用静音填充丢失帧。在低丢包率下，偶尔出现的短暂静音虽然不自然，但不影响理解。但稍高的丢包率下，这种”打嗝”效果会严重破坏听感。现代系统中，静音替换仅用于连续长时间丢包后的”淡出”阶段，而非首选策略。

波形替换（Waveform Substitution）

最早被实用化的 PLC 技术，原理是直接复制最近一个收到的帧来填补空缺。

对于非周期性噪声信号，单纯重复上一帧会产生明显的机器音。但对于短时平稳的语音信号，1–2 帧的重复在人耳听感上往往可以接受。

改进版本会使用时域交叠相加（OLA, Overlap-Add） 来避免帧边界处的波形不连续：

基于音调的重复（Pitch-based Repetition）

对于有声语音（voiced speech），信号具有明显的基频周期（Pitch Period），通常在 5ms 到 20ms 之间（对应 50Hz–200Hz 的基频）。

基于音调的 PLC 首先从历史帧中估计基频，然后以基频周期为单位循环复制波形，而不是复制整帧。这样生成的替代音频音调更连贯，听感更自然。

与此同时，为了模拟真实语音随时间衰减的特性，合成波形的能量会随着丢包时长线性或指数衰减，并最终淡出为背景噪声或静音。

ITU-T G.711 Appendix I 算法

这是 ITU-T 于 1999 年在 G.711 Appendix I 中正式标准化的高质量低复杂度 PLC 算法，也是历史上最具影响力的 PLC 规范之一。

设计目标：

  • 针对 G.711（PCM，64kbps，8kHz 采样）设计
  • 10ms 帧长（80 个样本）
  • 在 30% 丢包率下保持语音可懂度

算法核心流程（基于 ITU-T STL 文档描述）：

这套算法的关键在于自相关基频检测。记历史缓冲区为 h[n]（n = 0..N-1），则基频周期 T₀ 的估计为：

$$T_0 = \arg\max_{T_{min} \le T \le T_{max}} \sum_{n=0}^{M-1} h[n] \cdot h[n+T]$$

其中 M 为相关窗长度（通常约 20ms）。找到 T₀ 后，生成合成帧 s[n] = h[n - T₀]，并与历史帧末尾进行 OLA 混合以消除接缝。

WSOLA 时域算法

WSOLA（Waveform Similarity Overlap-Add）是一种更精确的时域信号外推方法，广泛用于时间拉伸（Time-Scale Modification）场景，也被引入 PLC。

与简单的基于音调重复不同，WSOLA 在每次外推时会在一定的搜索容差（tolerance） 范围内寻找与当前输出帧末尾波形相似度最高的历史片段，然后通过 OLA 拼接：

这种方法对非整数基频、混合语音、过渡段的处理效果优于简单的基于音调重复，生成的合成音频听感更平滑。WebRTC 的 NetEQ 内部在 expand 模式下使用的就是这类基于相关性的波形外推方法。

基于模型的 PLC（Model-based PLC）

语音编解码器（如 G.729、G.723.1）中的 CELP（Code-Excited Linear Prediction）模型本身包含声道滤波器状态。当发生丢包时，解码器可以利用上一帧已解码的LP（Linear Prediction）滤波器系数和激励信号来外推下一帧，无需额外的 PLC 模块。

这类”内置 PLC”的优势在于：

  • 充分利用了编解码模型的内部状态，合成音频与原始声道模型高度吻合
  • 常见于 ITU-T 标准化的 CELP 编解码器（G.729、G.728 等）

Opus 编解码器内置 PLC

Opus（RFC 6716）是 WebRTC 的标准音频编解码器，由 IETF 于 2012 年正式标准化，融合了 Skype 的 SILK（语音层）与 Xiph.Org 的 CELT（音乐层）。

Opus 在所有模式下都原生支持 PLC，这是其设计目标之一（RFC 6716 §1）。不同模式下的 PLC 策略有所差异：

SILK 层（语音模式）PLC：

SILK 是基于 LP（线性预测）的语音编码层。当发生丢包时，解码器会：

  1. 保持上一帧解码的 LPC（Linear Predictive Coding）滤波器系数
  2. 用噪声激励替代丢失的激励信号，同时渐进缩小激励幅度
  3. 对 LTP（Long-Term Prediction，即基频）参数进行适度衰减
  4. 通过平滑的能量衰减曲线逐渐将合成信号引向背景噪声

CELT 层（音频/音乐模式）PLC：

CELT 基于 MDCT（Modified Discrete Cosine Transform）变换域编码。PLC 时：

  1. 在频域对上一帧的 MDCT 系数施加随机相位扰动（噪声化）
  2. 对整体能量进行指数衰减
  3. 结合时域 OLA 实现平滑过渡

Opus 的 in-band FEC（带内前向纠错）：

值得特别提及的是，Opus SILK 模式还支持带内 FEC。当应用程序通过 OPUS_SET_PACKET_LOSS_PERC 接口向编码器传入预期丢包率时，编码器会将前一帧的低码率副本（称为 LBRR，Low Bit Rate Redundancy）嵌入当前包中。接收端收到后续包时可以从中提取前一帧的冗余数据，实现近乎完美的恢复——这在本质上是一种”一跳延迟的 RED”。

WebRTC NetEQ：PLC 的宿主引擎

在 WebRTC 的 libwebrtc 实现中，PLC 并不是孤立运作的，它深度集成在 NetEQ（Network Equalizer）这个自适应抖动缓冲器中。NetEQ 是 WebRTC 音频通路的核心组件，同时承担抖动吸收、时序控制与丢包隐藏三大职责。

NetEQ 整体架构

根据 Chromium 源码文档，NetEQ 对外暴露两个核心接口：

  • **InsertPacket(rtp_packet)**：网络层每收到一个 RTP 包时调用，将包存入 PacketBuffer
  • **GetAudio()**：音频播放层每 10ms 调用一次，拉取一帧用于播放的 PCM 数据

NetEQ 的决策状态机

NetEQ 的核心是一个决策控制器（DecisionLogic），它根据当前缓冲区水位（Buffer Level）与目标延迟（Target Delay）的差值，决定每次 GetAudio() 应执行哪种操作：

操作模式 触发条件 行为
Normal（正常解码） 缓冲水位 ≈ 目标延迟 正常解码下一个可用包
Expand（扩展/PLC） 缓冲区无可用包（丢包/迟到） 触发 PLC，生成合成帧
Accelerate（加速） 缓冲水位 > 目标延迟（缓冲堆积） 缩短当前帧（时间压缩），加速消耗
Preemptive Expand（预防性扩展） 缓冲水位 < 目标延迟（将要不足） 拉伸当前帧（时间扩展），避免未来丢帧
Merge（合并过渡） 丢包后第一个真实包到来 将 PLC 合成尾部与真实包头部 OLA 融合
CNG（舒适噪声） 收到 DTX 标记帧（说话停止） 生成舒适噪声，模拟背景环境

机器学习驱动的 PLC：WaveNetEQ

传统 PLC 算法基于信号处理假设（短时平稳、基频周期性），对连续多帧丢包的效果有限。2020 年，Google 发布了 WaveNetEQ，将深度学习引入 PLC 领域。

背景：NetEQ 的局限

传统 NetEQ 的 PLC 在以下场景会产生明显的合成失真：

  • 连续丢包超过 3–4 帧（60–80ms）
  • 语音与静音的过渡段（onset/offset）
  • 音调复杂的音乐内容

WaveNetEQ 架构

WaveNetEQ 基于 Google DeepMind 的 WaveRNN 语音合成架构，由两个子网络组成：

技术要点：

  • 条件化网络（Conditioning Network）：以过去音频的 log-Mel 频谱为输入，提取慢变特征（音色、基频趋势、韵律节奏），作为条件向量。
  • 自回归解码器（WaveRNN）：逐样本生成，每个新样本依赖所有已生成的历史样本，可以捕获精细的波形细节。
  • 稀疏化优化：为使模型能在手机端实时运行，Google 对模型进行了激进的权重稀疏化（weight sparsification），在保持音质的前提下大幅降低计算量。

效果与局限：

WaveNetEQ 对中长时连续丢包（40ms–200ms）的主观音质提升显著，在 MUSHRA 和 MOS 主观评测中表现优于传统 NetEQ PLC。然而：

  • 它不是 WebRTC 开源 libwebrtc 的默认实现，目前仅用于 Google 自有客户端（原 Google Duo，现 Google Meet）的某些版本。
  • 对于 10–20ms 的短丢包，传统信号处理方法已经足够，深度学习模型的优势并不明显。
  • 模型推理需要专用的优化和硬件加速，普通第三方应用难以直接移植。

PLC 与其他抗丢包机制的协同

与 FEC 的关系

Opus 的 in-band FEC（LBRR）是 PLC 的”上游补充”：

  • 若 FEC 数据可用（即下一个包如期到来），则优先用 FEC 数据恢复，质量远优于 PLC 合成。
  • 若连续多包丢失（burst loss）导致携带冗余数据的包也丢失，则退化为 PLC。

WebRTC 的 NetEQ 会优先尝试解码 FEC 数据，只有在 FEC 不可用时才触发 PLC。

与 NACK / RTX 的关系

NACK 重传机制与 PLC 在时间尺度上有本质区别：

  • NACK 重传最少需要 1 个 RTT（典型值 50–200ms）才能收到重传包。
  • PLC 生成合成音频是即时的（< 1ms）。

因此，在 WebRTC 音频通路中，通常不对音频使用 NACK（视频才大量使用 NACK），而是完全依赖 FEC + PLC 的组合。

与 RED 的关系

RED（RTP Header Extension Redundancy，RFC 2198）将前一帧的编码数据完整附在当前包中。收到当前包时，若发现前一帧丢失，可立即用附带的冗余数据恢复，质量等同于正常解码。

RED 与 PLC 的配合逻辑：

与 DRED 的关系

DRED（Deep Audio Redundancy，IETF Internet-Draft）是 Opus 的扩展，使用神经网络编码器将最近约 1 秒的音频特征以极低码率嵌入每个数据包中。当发生较长时间的突发丢包后，第一个恢复到来的包中携带的 DRED 数据可以驱动解码器进行语音合成，大幅优于纯粹的 PLC。

从系统层面看，DRED 是 PLC 的强力升级版：PLC 只能基于接收端的本地历史推测，而 DRED 携带了发送端侧的实际音频特征。

PLC 的局限性与质量衡量

局限性

场景 问题
连续丢包 > 60ms 任何 PLC 算法都难以维持可懂度，必须淡出为静音
音调突变（爆破音、清辅音） 基于历史外推的 PLC 会”错过”这些不可预测的瞬态
音乐内容 语音模型假设不成立，合成音频失真明显
高丢包率下的突发 多帧连续丢失时，每帧的错误会累积放大

质量评估方法

客观指标：

  • PESQ（Perceptual Evaluation of Speech Quality，ITU-T P.862）：对比参考语音与失真语音，输出 MOS-LQO 分数（1–4.5）。
  • POLQA（Perceptual Objective Listening Quality Analysis，ITU-T P.863）：PESQ 的继任标准，更好地支持超宽带音频。
  • PLCMOS：专为 PLC 场景设计的神经网络评估模型（Microsoft 提出），可在无参考信号的条件下评估 PLC 后的主观质量。

主观指标：

  • MOS（Mean Opinion Score）：让真人对听感打分（1–5），是最权威的评估方法，但成本高。
  • MUSHRA（Multiple Stimuli with Hidden Reference and Anchor）：多刺激隐藏参考对比测试，常用于比较多种 PLC 算法的相对优劣。

小结

PLC 是 WebRTC 音频质量保障体系中最后也是最重要的一道防线。它的核心价值在于：零额外延迟、纯接收端实现、对所有丢包场景兜底。

从技术演进的脉络来看，PLC 经历了从简单静音填充 → 波形替换 → 基于音调的外推 → 基于语音模型的合成，直到今天基于深度学习的神经网络合成（WaveNetEQ）。每一代技术都在改善”欺骗人耳”的效果，同时保持实时运行的计算约束。

在系统层面，PLC 并非孤立运作，它与 FEC、RED、DRED、NACK 构成了一个完整的抗丢包防御纵深。理解每种机制的触发时机和覆盖场景，是构建高质量实时通信系统的关键所在。

对于大多数 WebRTC 应用开发者而言，NetEQ 中的 PLC 是自动运行的透明模块，无需手动干预。但通过 getStats() 监控 concealedSamples 和 concealmentEvents，并结合 Opus FEC 参数的动态调整，可以有效诊断和缓解网络质量问题对音频体验的影响。
