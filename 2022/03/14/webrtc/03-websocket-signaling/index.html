    1. 为什么 WebRTC 需要信令

    WebRTC 的核心目标是实现浏览器之间的直接点对点媒体传输，但两个陌生的对等端在建立直连之前，必须先通过某种带外（out-of-band）机制完成三件事：

    1. 对等端发现：A 需要找到 B，B 需要知道 A 想要连接它。
    2. 媒体能力协商：双方需要就编解码器、分辨率、码率等参数达成一致，这通过 SDP（Session Description Protocol）完成。
    3. 网络路径协商：双方需要交换 ICE Candidate，确定如何穿越 NAT 或防火墙建立连接。

    这三件事统称为信令（Signaling）。RFC 8825（Overview: Real-Time Protocols for Browser-Based Applications）对 WebRTC 的架构给出了如下描述：

    媒体路径（低路径）直接在浏览器间传输，必须符合 WebRTC 协议规范；信令路径（高路径）经由服务器，格式由应用程序自定义。

    WebRTC 规范有意不强制规定信令协议，这给了开发者极大的灵活性。可以使用 WebSocket、HTTP 长轮询、SIP、XMPP，或任何能可靠传递文本消息的机制。但在实践中，WebSocket 已成为绝对主流，原因如下：

    • 全双工，服务端可主动推送消息（这是 HTTP 轮询无法高效实现的）
    • 持久连接，避免重复建立 TCP 握手的开销
    • 原生支持文本帧，直接传输 JSON 格式的信令消息
    • 所有现代浏览器均原生支持（基于 W3C WebSocket API）

    2. 信令协议的自由选择

    如前所述，WebRTC 规范不定义信令传输协议。这一设计决策来自 JSEP（JavaScript Session Establishment Protocol，RFC 8829）的核心哲学：

    “WebRTC call setup has been designed to focus on controlling the media plane, leaving signaling-plane behavior up to the application as much as possible. The rationale is that different applications may prefer to use different protocols, such as the existing SIP call signaling protocol, or something custom to the particular application.”
    —— RFC 8829, Section 1

    在实际工程中，常见的信令传输方案有：

    方案 优点 缺点
    WebSocket 全双工，低延迟，服务端推送，原生浏览器支持 需要维护长连接；部分反向代理需特殊配置
    HTTP 长轮询 兼容性极强，部署简单 延迟高，服务器资源消耗大
    HTTP SSE（Server-Sent Events） 服务端推送简单 单向（服务端→客户端），客户端仍需 HTTP POST 发送
    SIP over WebSocket（RFC 7118） 与传统电话网互通 复杂度高，适合 VoIP 场景
    XMPP over WebSocket 天然支持在线状态与消息路由 协议较重，使用场景特定

    本文重点讲解 WebSocket 方案。

    3. WebSocket 协议基础（RFC 6455）

    3.1 设计背景与动机

    在 WebSocket 出现之前，浏览器实现服务端推送只能靠”变通”手段：

    • 长轮询（Long Polling）：客户端发起 HTTP 请求，服务端挂起到有数据才响应，客户端收到后立刻再次请求。
    • HTTP Streaming：服务端保持响应流不关闭，持续写入数据。
    • iframe trick：在隐藏 iframe 里加载持续响应的页面。

    这些方案都在滥用 HTTP，HTTP 的设计本质是请求-响应模型，不适合双向实时通信（参见 RFC 6202）。

    RFC 6455 于 2011 年 12 月发布，由 IETF HyBi 工作组主导，定义了 WebSocket 协议。其核心目标：

    “The goal of this technology is to provide a mechanism for browser-based applications that need two-way communication with servers that does not rely on opening multiple HTTP connections.”
    —— RFC 6455, Abstract

    WebSocket 协议的关键特性：

    • 基于 TCP，以 HTTP Upgrade 握手作为协议切换入口
    • 全双工：客户端与服务端可同时独立地发送数据
    • 持久连接：握手后连接保持，直到一方主动关闭
    • 帧（Frame）：数据以帧为单位传输，帧格式极为精简，头部最小仅 2 字节
    • 原生支持文本与二进制：两种数据类型通过 opcode 区分

    3.2 URI 方案

    RFC 6455 Section 3 定义了 WebSocket 的 URI 方案：

    • ws://：明文 WebSocket，默认端口 80
    • wss://：WebSocket over TLS（即 WebSocket Secure），默认端口 443

    在 WebRTC 信令场景中，必须使用 wss:// （理由见第 9 节）。

    3.3 开放握手（Opening Handshake）

    WebSocket 握手复用 HTTP/1.1 的 Upgrade 机制，这使得 WebSocket 能天然穿越现有的 HTTP 基础设施（代理、负载均衡等）。

    客户端请求示例（RFC 6455 Section 4.1）：

    关键字段说明：

    字段 说明
    Upgrade: websocket 声明请求切换到 WebSocket 协议
    Connection: Upgrade 必须包含 Upgrade token
    Sec-WebSocket-Key 客户端随机生成的 Base64 编码的 16 字节随机数，用于握手验证
    Sec-WebSocket-Version: 13 RFC 6455 对应版本号为 13
    Sec-WebSocket-Protocol 可选，声明期望使用的子协议（应用层协议）

    3.4 数据帧格式

    RFC 6455 Section 5.2 定义了 WebSocket 的基础帧格式：

    各字段详解：

    字段 位宽 说明
    FIN 1 bit 是否为消息的最后一帧。1 表示是最终帧，0 表示消息还未结束（分片传输）
    RSV1/RSV2/RSV3 各 1 bit 保留位，必须为 0，除非通过扩展协商使用（如 permessage-deflate 压缩扩展使用 RSV1）
    Opcode 4 bits 定义帧的类型（见下表）
    MASK 1 bit 是否对 Payload 进行掩码处理。客户端发往服务端的所有帧必须设为 1
    Payload len 7 bits Payload 长度：0–125 直接表示；126 表示后跟 16 位无符号整数；127 表示后跟 64 位无符号整数
    Masking-key 0 或 4 bytes 当 MASK=1 时存在，32 位随机掩码密钥
    Payload Data n bytes 实际数据，若 MASK=1 则是掩码后的数据

    Opcode 定义（RFC 6455 Section 5.2）：

    Opcode 十六进制 含义
    0 0x0 延续帧（Continuation Frame）
    1 0x1 文本帧（UTF-8 编码）
    2 0x2 二进制帧
    3–7 0x3–0x7 保留，非控制帧
    8 0x8 连接关闭（Close）
    9 0x9 Ping
    10 0xA Pong
    11–15 0xB–0xF 保留，控制帧

    在 WebRTC 信令场景中，信令消息通常以 JSON 格式发送，使用 opcode 0x1（文本帧）。

    3.5 客户端到服务端的掩码机制

    RFC 6455 Section 5.3 强制要求：所有从客户端发往服务端的帧必须进行掩码处理（MASK=1），服务端发往客户端的帧必须不掩码（MASK=0）。

    掩码算法：

    解掩码使用相同算法（XOR 的对称性）。

    为什么需要掩码？（RFC 6455 Section 10.3）

    这是为了防范缓存污染攻击（Cache Poisoning Attack）。某些 HTTP 代理服务器不理解 WebSocket 协议，可能会错误地将 WebSocket 帧内容解释为 HTTP 响应并加以缓存。通过对 Payload 进行随机掩码，确保帧内容不会被误识别为有意义的 HTTP 数据。

    注意：即使在 TLS（wss://）下，掩码机制仍然生效，这是 RFC 6455 的强制要求，与是否加密无关。

    3.6 消息分片（Fragmentation）

    RFC 6455 Section 5.4 定义了消息分片机制。一条逻辑消息可以分为多个帧传输：

    规则：

    • 第一帧设置真实 opcode（0x1 或 0x2），后续帧 opcode 为 0x0（延续）
    • 最后一帧 FIN=1
    • 控制帧（Close/Ping/Pong）不允许分片，且可以插入到分片数据帧序列之间

    在信令场景中，SDP 字符串可能较长（通常几百到几千字节），分片机制保证了大消息也能高效传输。

    3.7 控制帧：Ping / Pong / Close

    控制帧用于协议层的连接管理，不携带应用数据。

    Ping（0x9）/ Pong（0xA）：

    • 任意一端可发送 Ping 帧；收到 Ping 的一端必须尽快回复 Pong 帧
    • Pong 帧的 Payload 必须与对应 Ping 帧的 Payload 相同
    • 可用于保活（Keep-Alive）或测量往返延迟
    • 控制帧的 Payload 长度不得超过 125 字节

    在 WebRTC 信令服务器中，通常在应用层也实现心跳机制，以检测信令通道的存活状态。

    3.8 关闭握手（Closing Handshake）

    RFC 6455 Section 7 定义了优雅关闭流程：

    1. 发起方发送 Close 帧（opcode=0x8），可携带状态码和原因
    2. 接收方收到 Close 帧后，回复一个 Close 帧
    3. 发起方收到回复的 Close 帧后，关闭底层 TCP 连接

    常用状态码：

    状态码 含义
    1000 正常关闭（Normal Closure）
    1001 端点离开（Going Away），如服务器关闭或浏览器跳转
    1002 协议错误（Protocol Error）
    1003 不支持的数据类型
    1008 违反策略
    1011 服务端意外情况

    在 WebRTC 信令场景中，当一个 WebRTC 会话结束时（挂断电话），信令通道既可以保留（用于后续会话），也可以随之关闭。

    3.9 子协议协商（Subprotocol）

    通过 Sec-WebSocket-Protocol 头，客户端可以声明期望的应用层子协议，服务端从中选择一个。这对信令服务器很有用，可区分不同版本或类型的信令协议。

    ONVIF WebRTC 规范要求子协议设置为 webrtc.onvif.org；自定义系统可以自行定义子协议名称。

    4. WebSocket 在 WebRTC 信令中的角色

    4.1 信令服务器的职责

    信令服务器是一个中间人，其核心职责非常明确且有限：

    1. 管理客户端连接：每个浏览器通过 WebSocket 与信令服务器建立持久连接
    2. 房间/会话管理：维护”谁在哪个房间”的状态
    3. 消息路由：将 A 发给 B 的信令消息转发给 B，反之亦然
    4. 转发 SDP Offer/Answer：协商媒体能力
    5. 转发 ICE Candidate：协商网络连接路径

    信令服务器从不触碰媒体数据。一旦 WebRTC P2P 连接建立成功，所有音视频流直接在两个浏览器间传输，信令服务器不参与其中。

    “The signaling server never handles audio or video data. Its only job is connection setup. Once the two peers are connected, all media flows directly between them, and the signaling server plays no further role in the call.”

    4.3 SDP 交换流程

    SDP（Session Description Protocol）是 WebRTC 媒体协商的核心内容（后续会出章节进行详细介绍）。在信令层面，SDP 被当作不透明的字符串，由信令服务器原样转发，无需解析其内容。

    Offer/Answer 模型（RFC 3264）：

    1. Alice 调用 pc.createOffer() 生成 SDP Offer
    2. Alice 调用 pc.setLocalDescription(offer) 设置本地描述（此时 ICE 候选收集开始）
    3. Alice 通过 WebSocket 将 Offer 发送给信令服务器
    4. 信令服务器将 Offer 转发给 Bob
    5. Bob 调用 pc.setRemoteDescription(offer) 设置远端描述
    6. Bob 调用 pc.createAnswer() 生成 SDP Answer
    7. Bob 调用 pc.setLocalDescription(answer) 设置本地描述
    8. Bob 通过 WebSocket 将 Answer 发送给信令服务器
    9. 信令服务器将 Answer 转发给 Alice
    10. Alice 调用 pc.setRemoteDescription(answer) 完成协商

    RTCPeerConnection 在 setLocalDescription 之后会异步触发 onicecandidate 事件，应用需要监听此事件并将候选通过 WebSocket 发送给对端。

    MDN 文档中对此的说明：

    “It’s important to note that the server doesn’t need to understand or interpret the signaling data content. Although it’s SDP, even this doesn’t matter so much: the content of the message going through the signaling server is, in effect, a black box.”

    4.5 Trickle ICE

    Trickle ICE（RFC 8838）是一项重要优化：候选无需全部收集完毕再发送，而是边收集边通过信令通道发送给对端，对端可以立即开始尝试连接。

    根据 RFC 8829（JSEP）的描述：

    “Candidate trickling is a technique through which a caller may incrementally provide candidates to the callee after the initial offer has been dispatched… This process allows the callee to begin acting upon the call and setting up the ICE connections immediately, without having to wait for the caller to gather all possible candidates.”

    Trickle ICE 显著减少了连接建立时延，是现代 WebRTC 实现的默认行为。支持 Trickle ICE 的 SDP 中会包含：

    全量 Trickle（Full Trickle）vs 半量 Trickle（Half Trickle）：

    • Full Trickle：Offer 发出时可以不含任何候选，候选随后单独发送
    • Half Trickle：Offer 含所有候选，Answer 端使用 Trickle

    WebRTC 浏览器默认支持 Full Trickle，canTrickleIceCandidates 属性可用于查询对端的支持情况。

    5. JSEP 与信令的关系（RFC 8829）

    JSEP（JavaScript Session Establishment Protocol，RFC 8829）是 WebRTC 信令架构的核心规范，它定义了 JavaScript 应用如何通过 RTCPeerConnection API 控制信令状态机。

    JSEP 的关键设计原则：

    1. 信令传输不在规范范围之内：JSEP 定义信令的内容（SDP），不定义传输机制
    2. ICE 状态机与信令状态机解耦：ICE 的收集与连接检查在浏览器内部独立进行
    3. 应用程序驱动 Offer/Answer 交换：通过 createOffer()、createAnswer()、setLocalDescription()、setRemoteDescription() 四个 API 完成

    JSEP 定义的信令状态机：

    完整状态包括：stable、have-local-offer、have-remote-offer、have-local-pranswer、have-remote-pranswer、closed。

    7. Perfect Negotiation 模式

    当双方同时发起 Offer 时（Offer 碰撞，Offer Collision），早期 WebRTC 实现容易陷入竞争条件（Race Condition）。Perfect Negotiation 模式（由 MDN 和 W3C WebRTC 规范推荐）通过引入”礼貌方（polite peer）”和”非礼貌方（impolite peer）”的角色来解决这一问题。

    核心规则：

    • 礼貌方：在 Offer 碰撞时，丢弃自己的 Offer，接受对方的 Offer 并回复 Answer
    • 非礼貌方：在 Offer 碰撞时，忽略对方的 Offer，坚持自己的 Offer

    Perfect Negotiation（完美协商）模式并不是让“双方都礼貌”，而是人为地在两个对等端之间约定一个角色区分：
    一方设为 polite（礼貌方）,另一方设为 impolite（非礼貌方 / 主动方）。

    这个角色通常通过应用层逻辑预先确定，比如：

    • 根据用户 ID 的字典序（如 peerIdA < peerIdB → A 是 impolite，B 是 polite）
    • 根据谁先加入房间（发起者为 impolite，后来者为 polite）
    • 通过信令服务器分配角色

    注意：pc.setLocalDescription() 不传参数时，浏览器会根据当前状态自动决定生成 Offer 还是 Answer（需要浏览器支持 Implicit rollback，Chrome 80+/Firefox 75+ 支持）。

    9. 安全注意事项

    9.1 使用 WSS（WebSocket over TLS）

    生产环境必须使用 wss:// ，而非明文的 ws://。原因：

    1. SDP 含 DTLS 指纹：SDP 中包含 a=fingerprint 字段，是 DTLS-SRTP 密钥协商的基础。若 SDP 被中间人篡改（将攻击者的指纹替换进去），攻击者可以发动中间人攻击，劫持媒体流。
    2. ICE 候选泄露：ICE Candidate 包含 IP 地址信息，明文传输可能导致网络拓扑暴露。
    3. 浏览器策略：现代浏览器在 HTTPS 页面中禁止使用 ws://（Mixed Content Policy）。

    “Run the server over WSS (WebSocket Secure) using TLS — never plain WebSocket.”

    9.2 身份认证

    信令服务器应在客户端加入房间前完成身份验证，常见方案：

    • JWT（JSON Web Token）：在 WebSocket 升级请求时通过 URL 参数或 Authorization 头携带
    • Cookie 认证：依赖已建立的 HTTP Session
    • HMAC 签名的 Room Ticket：短时效的加密凭证

    9.4 速率限制与房间隔离

    • 速率限制：每个 WebSocket 连接限制每秒消息数量，防止消息洪水攻击
    • 房间隔离：严格确保客户端只能收到同一房间内的消息，不能越权访问其他房间
    • 房间规模限制：对房间最大成员数设置上限，防止资源耗尽

    10. 生产环境的工程实践

    10.1 水平扩展

    单个 Node.js 进程通常可以维护数万个并发 WebSocket 连接，信令服务器本身是轻量的。但在多实例部署时，需要解决跨实例的消息路由问题：

    • 使用 Redis Pub/Sub 在多个信令服务器实例间共享房间状态和路由消息
    • 使用粘性会话（Sticky Session）确保同一房间的用户连接到同一实例（简单但限制扩展）

    11. HTTP/2 与 HTTP/3 上的 WebSocket

    RFC 6455 基于 HTTP/1.1。随着 HTTP/2 的普及，IETF 发布了扩展规范：

    • RFC 8441（2018）：Bootstrapping WebSockets with HTTP/2，使用 HTTP/2 的 Extended CONNECT 方法，允许多个 WebSocket 连接共享一条 TCP 连接，避免 HTTP/1.1 多连接的队头阻塞问题。
    • RFC 9220（2022）：将 RFC 8441 扩展到 HTTP/3 和 QUIC，带来零队头阻塞和 0-RTT 连接恢复等优势。

    对于大多数 WebRTC 信令场景，HTTP/1.1 上的 WebSocket（RFC 6455）已足够使用，HTTP/2 WebSocket 在需要多路复用大量信令连接时有更大价值。

    12. 其他信令方案对比

    方案 协议栈 优点 适用场景
    WebSocket（推荐） TCP + HTTP Upgrade 全双工、低延迟、浏览器原生支持 绝大多数 WebRTC 应用
    HTTP 长轮询 HTTP/1.1 兼容性极强、穿透能力强 防火墙严格环境的降级方案
    SIP over WebSocket（RFC 7118） WebSocket + SIP 与传统电话网互通 企业电话、呼叫中心
    XMPP over WebSocket（RFC 7395） WebSocket + XMPP 在线状态、即时消息集成 有 XMPP 基础设施的场景
    WHIP/WHEP HTTP RESTful、简单 单向直播推流/拉流

    WHIP（WebRTC-HTTP Ingestion Protocol） 和 WHEP（WebRTC-HTTP Egress Protocol） 是近年兴起的基于 HTTP 的单次 Offer/Answer 信令方案，适合媒体服务器（SFU）的推流场景，但不适合需要双向实时信令的多方通话。
