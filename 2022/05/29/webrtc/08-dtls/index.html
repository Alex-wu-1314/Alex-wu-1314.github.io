    1. 为什么 WebRTC 需要 DTLS

    WebRTC 的核心使命是实现浏览器间的实时音视频通信，而实时通信天然运行在 UDP 之上，原因在于 UDP 的低延迟特性，它不像 TCP 那样因重传机制而引入不可预测的延迟抖动。

    然而，UDP 本身没有任何安全机制：数据是明文的，没有完整性校验，也没有身份认证。若不加保护，任何处于网络路径上的攻击者都可以：

    • 窃听（Eavesdropping）：截获并读取音视频内容；
    • 篡改（Tampering）：修改媒体数据流；
    • 伪造（Forgery）：冒充通信一方注入恶意数据。

    IETF RFC 8827（WebRTC Security Architecture） 对此给出了明确且强制性的要求：

    WebRTC 的所有媒体流和数据通道必须使用 DTLS-SRTP 加密，这是唯一被允许的加密机制，没有可选项，没有降级路径。

    具体而言，WebRTC 的加密策略覆盖三层传输：

    传输类型 使用协议 相关 RFC
    音视频媒体（RTP/RTCP） DTLS 握手 → SRTP 加密 RFC 5764
    数据通道（DataChannel/SCTP） DTLS 直接封装 RFC 8261
    STUN（部分场景） DTLS RFC 7350

    关键理解：DTLS 在 WebRTC 中扮演的是密钥协商者而非数据搬运工。握手完成后，实际的媒体数据由 SRTP 加密传输，DTLS 负责在握手阶段安全地派生出 SRTP 所需的对称密钥。

    2. DTLS 协议概览

    DTLS（Datagram Transport Layer Security） 是 TLS 协议在 UDP 数据报传输环境下的适配版本，其设计哲学来自 RFC 9147 中的一句话：

    “The basic design philosophy of DTLS is to construct ‘TLS over datagram transport’.”

    TLS 原本依赖 TCP 提供的有序、可靠传输。UDP 不具备这些特性，因此 TLS 无法直接运行于 UDP 之上，主要有四个障碍：

    1. 序号隐式性：TLS 依赖 TCP 的隐式序号来保证记录的顺序解密，UDP 不提供这种保证；
    2. 握手的顺序依赖性：TLS 握手是严格顺序的状态机，任何乱序都会导致失败；
    3. 数据报大小限制：UDP 单个数据报有大小上限（通常受 MTU 限制，约 1500 字节），TLS 握手消息可能远超此限制；
    4. 重放攻击：UDP 不保证消息仅被接收一次，攻击者可以重放旧数据包。

    DTLS 通过若干扩展机制解决了上述所有问题，同时尽可能保持与 TLS 的代码和结构复用——这也是 DTLS “deliberately designed to be as similar to TLS as possible”（RFC 6347）的核心原则。

    3. DTLS 与 TLS 的关键差异

    3.1 显式序号（Explicit Sequence Numbers）

    DTLS 在 Record Layer 中引入了显式的 epoch 和 sequence_number 字段：

    • epoch：初始值为 0，每次密钥变更（ChangeCipherSpec）后加 1，用于区分不同加密阶段的记录；
    • sequence_number：每条记录单独编号，允许接收方检测乱序、重复或丢失的数据报。

    3.2 重传机制（Retransmission）

    UDP 本身不重传丢失的数据包，因此 DTLS 在握手层面实现了自己的重传逻辑：

    • 每条握手消息发送后，启动一个重传定时器；
    • 若定时器超时前未收到响应，则重发上一个”flight”（消息组）；
    • 定时器采用指数退避策略，防止网络拥塞。

    3.3 消息分片（Message Fragmentation）

    握手消息（如证书）可能超过 UDP MTU。DTLS 在握手消息头中添加了分片字段：

    接收方根据 message_seq、fragment_offset、fragment_length 重组完整的握手消息，避免 IP 层分片（IP 分片更容易导致丢包）。

    3.4 Cookie 机制（Anti-DoS）

    TLS 运行在 TCP 之上，TCP 三次握手天然抵御了一定程度的 DoS 攻击。UDP 是无连接的，恶意客户端可以伪造大量 IP 源地址，触发服务器分配资源，造成放大攻击。

    DTLS 引入了 HelloVerifyRequest + Cookie 机制（RFC 6347 §4.2.1）：

    1. 服务器收到 ClientHello 后，先不分配任何状态，直接返回包含 Cookie 的 HelloVerifyRequest；
    2. Cookie 由服务器用密钥对客户端 IP/端口等信息做 HMAC 计算得出；
    3. 客户端必须在第二个 ClientHello 中携带这个 Cookie；
    4. 服务器验证 Cookie 合法后，才真正开始握手并分配状态。

    这确保了：只有真实可达的客户端才能推进握手，防止 IP 欺骗。

    3.5 重放保护（Replay Protection）

    DTLS 使用滑动窗口（Sliding Window）机制检测重放攻击：接收方维护一个已接收序号的位图，序号落在窗口之外或已在窗口内被标记的数据报将被丢弃。

    4. DTLS Record Layer（记录层）

    Record Layer 是 DTLS 的数据封装格式，所有上层数据（握手消息、应用数据、Alert）都被封装成 DTLS Record 传输。

    4.1 Record 头部结构

    Content Type 的合法值：

    值 类型 说明
    20 change_cipher_spec 通知对端切换加密状态
    21 alert 错误或关闭通知
    22 handshake 握手消息
    23 application_data 加密后的应用数据

    Version 字段的编码（注意：DTLS 版本号与 TLS 相反，数值越小版本越新）：

    字节值 DTLS 版本 对应 TLS 版本
    0xFEFF DTLS 1.0 TLS 1.1
    0xFEFD DTLS 1.2 TLS 1.2
    0xFEFC DTLS 1.3 TLS 1.3

    为什么没有 DTLS 1.1？ 为了与 TLS 版本号对齐，IETF 跳过了 DTLS 1.1，直接从 1.0 跳到了 1.2（RFC 6347 明确说明了这一点）。

    5. DTLS 握手协议详解

    DTLS 握手消息被组织成若干 “Flight”（消息组） ，每个 Flight 包含一条或多条消息，作为一个整体发送和确认。

    5.2 各消息详解

    Flight 1 — ClientHello

    客户端发起握手，携带：

    • client_random：32 字节随机数（4 字节时间戳 + 28 字节随机），用于后续密钥派生；
    • session_id：会话 ID，用于会话恢复（WebRTC 中通常为空）；
    • cookie：第一次发送时为空，收到 HelloVerifyRequest 后填入；
    • cipher_suites：客户端支持的加密套件列表，如 TLS_ECDHE_ECDSA_WITH_AES_128_GCM_SHA256；
    • extensions：扩展列表，WebRTC 场景中关键扩展包括：
      • use_srtp：告知服务器本次握手完成后将用于 SRTP 密钥派生，并列出支持的 SRTP 保护配置（Protection Profile），如 SRTP_AES128_CM_HMAC_SHA1_80；
      • supported_groups：支持的椭圆曲线，通常为 secp256r1；
      • signature_algorithms：支持的签名算法；
      • extended_master_secret（RFC 7627）：增强 Master Secret 安全性。

    Flight 4 — Server 应答消息组

    ServerHello：从客户端提供的列表中选择一个加密套件和压缩方法，返回 server_random。

    Certificate：服务器的 X.509 证书（WebRTC 中通常是自签名证书，见第 7 节）。

    ServerKeyExchange（当使用 ECDHE 时）：包含服务器的椭圆曲线公钥（ephemeral key），以及用服务器证书私钥对其签名的签名值，客户端以此验证公钥未被篡改。

    CertificateRequest（WebRTC 中通常包含）：请求客户端也提供证书，实现双向认证（Mutual Authentication）。这是 WebRTC 与普通 HTTPS 的重要区别，通信双方互相认证。

    ServerHelloDone：标志服务器 Flight 4 结束，等待客户端响应。

    Flight 5 — Client 应答消息组

    Certificate：客户端的自签名证书。

    ClientKeyExchange：客户端的 ECDHE 公钥（ephemeral key）。

    至此，双方均拥有对方的公钥，可以通过 ECDH 计算出Pre-Master Secret：

    Master Secret 的派生（DTLS 1.2 使用 PRF）：

    CertificateVerify：客户端用自己的私钥对之前所有握手消息的哈希进行签名，服务器用客户端证书的公钥验证，确认客户端确实持有证书对应的私钥。

    ChangeCipherSpec：通知服务器，此后客户端发送的消息将使用协商好的密钥加密。epoch 从 0 变为 1。

    Finished：第一条加密消息，包含所有握手消息的 MAC，双方通过验证 Finished 消息确认握手过程未被篡改。

    Flight 6 — Server Finished

    服务器发送自己的 ChangeCipherSpec 和 Finished，握手完成。

    6. DTLS 在 WebRTC 中的角色协商

    DTLS 是客户端/服务器模型，握手由 Client 发起。WebRTC 中两端对等，因此必须通过信令（SDP）协商谁扮演 Client、谁扮演 Server。

    6.1 SDP 中的 a=setup 属性

    角色通过 SDP 的 a=setup 属性协商（RFC 4145，在 DTLS-SRTP 场景中由 RFC 5763 §5 进一步规范）：

    值 含义
    active 该端作为 DTLS Client，主动发起 ClientHello
    passive 该端作为 DTLS Server，等待 ClientHello
    actpass 该端可以扮演任意角色（由对端决定）

    JSEP（RFC 8829）的强制规定：

    • Offer 中的 a=setup 必须为 actpass；
    • Answer 中的 a=setup 必须为 active 或 passive（通常为 active）。

    这样，Offer 方（通常是呼叫发起方）将最终由对方来决定角色，避免双方同时发起 ClientHello 的竞争。

    7. 证书指纹与身份验证

    7.1 自签名证书的挑战

    在普通的 HTTPS 场景中，服务器证书由受信任的 CA 签发，浏览器通过验证 CA 链来认证服务器身份。但在 WebRTC 的 P2P 场景中：

    • 没有中心化的服务器，通信双方都是普通浏览器；
    • 为每个连接都申请 CA 签发的证书不现实；
    • 因此 WebRTC 使用自签名证书（Self-Signed Certificate）。

    自签名证书无法通过 CA 链验证，那么如何防止中间人攻击（MITM）？

    7.2 a=fingerprint：将证书绑定到信令

    答案是 RFC 4572 定义的证书指纹机制，在 WebRTC 中由 RFC 5763 使用：

    1. 每端在连接建立前生成自己的 DTLS 证书（RFC 8827 要求每次新通话生成新的密钥对）；
    2. 用指定的哈希算法（如 SHA-256）计算证书的指纹（Fingerprint）；
    3. 将指纹通过 SDP 的 a=fingerprint 属性，经由信令通道（WebSocket + HTTPS/WSS）传递给对端；
    4. DTLS 握手完成后，双方将对端在握手中出示的证书与从 SDP 中获取的指纹进行比对；
    5. 若指纹不匹配，则立即终止连接。

    7.3 安全前提：信令通道的安全性

    整个机制的安全性依赖于信令通道的完整性。如 RFC 8827 明确指出：若信令通过不安全的 HTTP/WS 传输，攻击者可以替换 SDP 中的指纹，实施中间人攻击。

    因此，WebRTC 应用必须使用 HTTPS + WSS（Secure WebSocket）来传递 SDP，这是整个安全架构的基础假设。

    7.4 WebRTC Identity（可选增强）

    对于需要更强身份绑定的场景，WebRTC 支持通过 Identity Provider（IdP） 将 DTLS 证书与用户身份（如 OAuth 令牌）绑定（RFC 8827 §6），但这属于可选机制，超出本文范围。

    8. DTLS-SRTP：密钥导出机制

    WebRTC 中 DTLS 最核心的功能是为 SRTP 提供密钥材料，这一过程由 RFC 5764（DTLS Extension to Establish Keys for SRTP） 定义。

    8.1 use_srtp 扩展

    在 ClientHello 中，客户端通过 use_srtp 扩展声明它希望将本次 DTLS 会话用于 SRTP 密钥派生，并列出支持的 SRTP Protection Profile：

    常见的 Protection Profile（RFC 8827 强制要求支持）：

    Profile 算法 说明
    SRTP_AES128_CM_HMAC_SHA1_80 AES-128-CM + HMAC-SHA1-80 WebRTC 必须支持
    SRTP_AES128_CM_HMAC_SHA1_32 AES-128-CM + HMAC-SHA1-32 可选，较短的认证标签
    SRTP_AEAD_AES_128_GCM AES-128-GCM 现代算法，提供 AEAD
    SRTP_AEAD_AES_256_GCM AES-256-GCM 更高强度

    8.2 密钥材料导出（Key Material Exporter）

    DTLS 握手完成后，双方使用 TLS Exporter（RFC 5705）从 Master Secret 中派生 SRTP 密钥材料：

    导出的密钥材料按固定格式拆分为四份（以 SRTP_AES128_CM_HMAC_SHA1_80 为例）：

    • client_write_SRTP_master_key：客户端发送 SRTP 包时使用的加密密钥；
    • server_write_SRTP_master_key：服务端发送 SRTP 包时使用的加密密钥；
    • master_salt：与密钥结合，用于派生每个 RTP 包的 IV（初始向量）。

    8.3 DTLS 与 SRTP 的关系示意

    DTLS 握手完成后，不再持续封装媒体数据。这是 DTLS-SRTP 与”直接用 DTLS 传输数据”的本质区别：DTLS 负责一次性的密钥协商，之后媒体数据由轻量级的 SRTP 直接加密，避免 DTLS Record Layer 带来的额外开销。

    9. DTLS 版本演进

    9.1 DTLS 1.0（RFC 4347，2006）

    基于 TLS 1.1，引入了 DTLS 的核心机制（Cookie、序号、重传）。目前已被视为过时版本，存在一些安全问题（如不支持 AEAD 加密套件）。

    9.2 DTLS 1.2（RFC 6347，2012）

    基于 TLS 1.2，这是目前 WebRTC 部署最广泛的版本，引入：

    • 支持 AEAD 加密套件（AES-GCM、ChaCha20-Poly1305）；
    • 改进的 Cookie 机制；
    • 更灵活的 PRF 算法。

    RFC 8827 明确要求 WebRTC 实现必须支持 DTLS 1.2，并指出早期要求 DTLS 1.0 的草案已过时。

    9.3 DTLS 1.3（RFC 9147，2022）

    基于 TLS 1.3，RFC 9147 已于 2022 年 4 月发布，主要改进：

    • 更短的握手：1-RTT 握手（1.2 通常需要 1.5-2 RTT）；
    • 0-RTT 支持：会话恢复场景下可以在第一条消息中发送数据；
    • 更简洁的记录格式：去除了 epoch 字段的冗余编码；
    • 精简密码套件：强制 AEAD，移除所有不安全的旧套件；
    • 基于 TLS 1.3 的 ACK 机制：替代了部分重传逻辑。

    注意：截至本文写作时（2025 年），主流浏览器对 DTLS 1.3 的支持仍在推进中，WebRTC 生产环境中 DTLS 1.2 仍是主流。

    版本对比总结

    特性 DTLS 1.0 DTLS 1.2 DTLS 1.3
    基础 TLS TLS 1.1 TLS 1.2 TLS 1.3
    RFC RFC 4347 RFC 6347 RFC 9147
    发布年份 2006 2012 2022
    AEAD 支持 ✗ ✓ ✓（强制）
    握手 RTT ~2 RTT ~1.5 RTT ~1 RTT
    0-RTT ✗ ✗ ✓
    WebRTC 状态 已淘汰 主流 推进中

    10. 安全考量

    10.1 前向保密（Perfect Forward Secrecy）

    WebRTC 要求优先选择支持 PFS 的加密套件（基于 ECDHE 或 DHE 密钥交换）。原因是：

    • 非 PFS 的套件（如 RSA 密钥交换）中，若服务器长期私钥被泄露，攻击者可以解密所有过去录制的会话；
    • ECDHE 套件每次握手生成临时密钥对，即使长期密钥泄露，历史会话依然安全。

    RFC 8827 明确要求：

    Implementations MUST prefer cipher suites that support Perfect Forward Secrecy.

    10.2 推荐加密套件

    套件 密钥交换 加密 MAC 备注
    TLS_ECDHE_ECDSA_WITH_AES_128_GCM_SHA256 ECDHE AES-128-GCM - (AEAD) 推荐
    TLS_ECDHE_RSA_WITH_AES_128_GCM_SHA256 ECDHE AES-128-GCM - (AEAD) 推荐
    TLS_ECDHE_ECDSA_WITH_CHACHA20_POLY1305_SHA256 ECDHE ChaCha20 Poly1305 移动端友好
    TLS_ECDHE_RSA_WITH_AES_128_CBC_SHA ECDHE AES-128-CBC SHA1 兼容性用途

    10.3 中间人攻击防御

    DTLS-SRTP 通过以下链条防止 MITM：

    关键漏洞点：若攻击者能修改 SDP（即信令通道被攻陷），则可以将对端的 a=fingerprint 替换为自己证书的指纹，从而充当中间人，分别与两端建立 DTLS 会话。这也是为什么 WebRTC 应用必须且只能通过 HTTPS/WSS 传递 SDP。

    10.4 每次通话生成新密钥对

    RFC 8827 要求 WebRTC 实现默认为每次新通话生成新的 DTLS 密钥对：

    The API MUST generate a new DTLS key pair for every new call by default.

    这防止了跨会话的关联攻击，即使攻击者同时捕获了多个会话的流量，也无法通过密钥关联它们。

    11. 完整连接建立时序

    将 DTLS 放入 WebRTC 整体连接流程的上下文中：

    时序约束：DTLS 握手必须在 ICE 连通性检查成功之后才能开始。ICE 建立了底层的 UDP 连接路径，DTLS 在此路径上运行。
