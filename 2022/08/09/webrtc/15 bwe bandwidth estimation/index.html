    1. 为什么需要带宽估计

    互联网是一个”尽力而为”（best-effort）的网络，IP 协议本身不对可用带宽作任何保证，且网络状况会在通话过程中动态变化，用户可能从 Wi-Fi 切换到 LTE，或者与他人共享一条家庭宽带。

    实时媒体传输面临一个根本性矛盾：

    • 发送太多：超过网络容量，导致路由器队列溢出、丢包、延迟飙升，用户体验急剧恶化；
    • 发送太少：画质不必要地降低，浪费了可用带宽，同样影响体验。

    带宽估计（BWE, Bandwidth Estimation）正是为了解决这一矛盾而生的模块。它持续测量当前网络的可用容量，并将估计结果反馈给编码器和发送引擎，实现自适应码率（ABR, Adaptive Bitrate） 调整，在任意时刻以最接近网络容量的码率传输，既不拥塞网络，也不浪费资源。

    与 TCP 的区别：TCP 有内置的拥塞控制机制（如慢启动、AIMD），但 WebRTC 使用 UDP，因为实时媒体无法接受重传带来的延迟。这意味着 WebRTC 必须自己实现拥塞控制，BWE 就是这套机制的核心。

    2. 基本概念与术语

    术语 含义
    BWE Bandwidth Estimation，带宽估计，WebRTC 中指整个带宽估计与拥塞控制子系统
    GCC Google Congestion Control，Google 拥塞控制算法，BWE 的主要实现，详见后续章节
    REMB Receiver Estimated Maximum Bitrate，接收端估计最大码率，一种旧的反馈机制
    TWCC / Transport-CC Transport-Wide Congestion Control，传输层拥塞控制，现代 BWE 使用的反馈机制
    RTCP RTP 控制协议，承载 BWE 所需的各类反馈报文
    RTT Round-Trip Time，往返延迟，带宽估计的重要参考指标
    OWD One-Way Delay，单向延迟，延迟梯度分析的基础
    Inter-group delay variation 包组间延迟变化量，GCC 延迟控制器的核心指标，通常记为 d(i)
    RMCAT RTP Media Congestion Avoidance Techniques，IETF 工作组，负责标准化实时媒体拥塞控制

    3. BWE 的整体架构

    WebRTC 的 BWE 模块并非单一算法，而是由多个子模块组成的流水线：

    整个流程可以概括为：

    1. 发送端为每个 RTP 包打上传输层序列号（TWCC header extension）；
    2. 接收端记录每包的精确到达时间，定期通过 RTCP 反馈给发送端；
    3. 发送端 GCC 根据反馈信息，通过延迟梯度分析和丢包分析，估算当前可用带宽；
    4. BWE 结果驱动 Pacer 调整发送节奏，并通知编码器调整目标码率。

    4. REMB：接收端带宽估计（第一代）

    4.1 工作原理

    REMB（Receiver Estimated Maximum Bitrate）是 WebRTC 早期使用的带宽反馈机制，由接收端负责估算带宽并告知发送端。

    基本流程：

    1. 发送端为每个视频 RTP 包添加 abs-send-time 头部扩展（RFC 5450 定义绝对发送时间格式），携带 24-bit 精度的绝对发送时间；
    2. 接收端收到包后，记录本地到达时间；
    3. 接收端对比发送时间与到达时间，计算包组间延迟变化量 d(i)；
    4. 接收端运行 GCC 延迟控制器，得出估算带宽；
    5. 接收端通过 RTCP REMB 报文将估算结果回传给发送端；
    6. 发送端结合 REMB 值与 RTCP 丢包报告，调整最终目标码率。

    4.3 局限性

    REMB 存在以下不足，促使社区转向 Transport-CC：

    • 估算逻辑在接收端：SFU 等中间节点难以介入或优化算法；
    • 每路流独立反馈：无法跨流统一视角，多路 Simulcast 时估计不准确；
    • 反馈粒度粗：仅返回估算码率，发送端无法自行做更精细分析；
    • 未正式标准化：仍为过期 Internet-Draft，但所有主流浏览器均实现了它。

    5. Transport-CC：发送端带宽估计（第二代）

    5.1 设计动机

    Transport-CC（也称 TWCC，Transport-Wide Congestion Control）将带宽估算的逻辑从接收端移到发送端。接收端只负责如实上报每个包的到达时间，发送端拥有完整的发送时间信息，因此可以更准确地分析延迟梯度，同时也更便于 SFU 实现和算法迭代。

    这一机制定义在 draft-holmer-rmcat-transport-wide-cc-extensions-01，同样未正式标准化，但已是 libwebrtc（Chrome / Firefox）和主流媒体服务器的默认实现。

    6. GCC 算法：BWE 的核心实现

    GCC（Google Congestion Control）是 WebRTC BWE 的实际实现算法，规范见 IETF Internet-Draft draft-ietf-rmcat-gcc-02（Holmer, Lundin et al., Google / Politecnico di Bari, 2016）。

    6.1 系统模型

    GCC 将网络路径抽象为一个队列系统，用包组（group of packets） 而非单包为单位进行分析。

    一个包组定义为：在 burst_time（推荐 5ms）窗口内发出的一批 RTP 包。对每个包组，定义：

    • T(i)：第 i 组最后一包的发送时间（sender side）；

    直觉上，d(i) > 0 意味着这组包比上一组多花了时间，队列在积累；d(i) < 0 意味着队列在排空；d(i) ≈ 0 意味着网络稳定。

    6.2 延迟梯度控制器（Delay-based Controller）

    延迟控制器分为四个阶段：预过滤 → 到达时间滤波 → 过载检测 → 速率控制。

    GCC 将 d(i) 建模为：

    • m(i)：队列积累/排空趋势的均值，是我们真正想估计的量；
    • v(i)：零均值高斯白噪声，代表网络抖动和测量误差。

    当 m(i) 持续增大时，说明发送速率超过了网络容量（over-use）；当 m(i) 持续减小时，说明网络有富余（under-use）；当 m(i) ≈ 0 时，说明正好（normal）。

    6.2.2 预过滤（Pre-filtering）

    网络中断（outage）恢复时，积压的包会突发到达，造成虚假的大 d(i)。预过滤将以下情况的包合并为同一组，以消除此类噪声：

    • 连续包的发送时间差 < burst_time；
    • 包的 d(i) < 0（队列在排空，属于正常赶上，不是新的拥塞）。

    6.3 丢包控制器（Loss-based Controller）

    丢包控制器运行在发送端，基于 RTCP Receiver Report（或 Transport-CC 反馈中的序号缺失）计算丢包率，并据此调整最终目标码率 A_s：

    关键特性：

    • 丢包 < 2%：每个反馈周期乘以 1.08，缓慢爬升；
    • 丢包 2%~10%：维持当前码率，观望；
    • 丢包 > 10%：快速下调，降幅随丢包率线性增加；
    • 丢包控制器的输出不能超过延迟控制器给出的上限 A_r，两者取最小值。

    6.4 两个控制器的协同

    两个控制器各司其职：

    • 延迟控制器预判拥塞，在队列溢出（丢包）发生之前就开始降码，更加主动；
    • 丢包控制器在丢包已经发生时兜底，确保码率不超过网络实际承载能力。

    7. 带宽探测（Bandwidth Probing）

    7.1 为什么需要探测

    BWE 的估计本质上是被动的，它依赖已发送的流量来推断网络容量。但有一种场景无法被动感知：当前发送码率远低于实际网络容量时（例如屏幕共享中内容静止，几乎没有流量），延迟控制器无法收到足够的数据来判断是否可以提高码率。

    如果不主动探测，BWE 估计值会长期保持在较低水平，无法利用富余带宽。

    7.2 探测机制

    libwebrtc 通过发送探测包（probe cluster） 来主动试探网络容量：

    1. 将目标探测码率设为当前 BWE 估计值的某个倍数（如 1.5×、2×）；
    2. 在很短的时间窗口内（约 15ms）集中发送足够多的包，使得发送速率等于探测目标码率；
    3. 接收端通过 Transport-CC 反馈这些探测包的到达时间；
    4. 发送端分析探测包的到达情况，判断网络是否能承受探测码率。

    探测包可以是：

    • 真实的媒体包（最常见）；
    • 填充包（padding）：当媒体流量不足时，用空包凑足探测码率。

    7.3 初始探测与周期探测

    初始探测（Connection startup）：通话建立时，BWE 初始值硬编码为约 300 kbps。为了快速收敛到合理估计，libwebrtc 在通话开始时会执行一系列探测，逐步提高目标码率（指数增长），直到检测到拥塞迹象。

    周期探测（Periodic probing）：当媒体流量长期低于 BWE 估计值时（如静态屏幕共享），libwebrtc 每 5 秒发送一次探测，以维持对网络容量的感知，防止估计值不必要地衰减。

    BWE 的输出最终要作用于编码器，改变实际视频/音频的编码码率。这个闭环是 ABR 的核心：

    联动中的注意事项：

    • 编码器调整有滞后：编码器通常以帧为单位工作，码率变化不能在单帧内立即生效；
    • 分辨率/帧率 vs 质量：在码率降低时，可以选择降分辨率（减少像素）或降质量（增加 QP），WebRTC 默认优先保帧率、其次降分辨率、最后降质量；
    • Simulcast 集成：在 SFU 场景下，BWE 还会触发 Simulcast 层切换，详见第 29 篇。

    9. RMCAT 工作组与其他算法

    IETF 专门成立了 RMCAT（RTP Media Congestion Avoidance Techniques）工作组 来标准化实时媒体拥塞控制。该工作组已结束，产出了几个重要文档：

    9.1 SCReAM

    Self-Clocked Rate Adaptation for Multimedia，RFC 8298。

    SCReAM 借鉴了 LEDBAT 的思路，使用单向延迟（OWD）作为主要信号，目标是最小化队列占用。它已成为正式 RFC，但在实际 WebRTC 部署中使用较少。

    9.2 NADA

    Network-Assisted Dynamic Adaptation，RFC 8698。

    NADA 结合了延迟和丢包信号，并支持显式拥塞通知（ECN）辅助。同样已成为正式 RFC，有在 Firefox 实验性版本中尝试集成的记录，但需要依赖 RFC 8888 定义的 RTCP 反馈格式，而现有 WebRTC 实现均不支持该格式，导致实用性有限。

    9.3 GCC 的标准化状态

    GCC（draft-ietf-rmcat-gcc-02）仅是一个已过期的 Internet-Draft，从未成为正式 RFC。然而它是 libwebrtc 的唯一实现，因此也是几乎所有 WebRTC 端点实际使用的算法。标准化状态与实际部署之间的落差，在 WebRTC 生态中是一个公开的讨论话题。

    10. BWE 在 SFU 场景下的挑战

    在点对点（P2P）通话中，BWE 相对简单：发送端直接感知到接收端的反馈。但在 SFU（Selective Forwarding Unit）场景下，情况更为复杂：

    • Publisher 的 BWE 感知的是 Publisher → SFU 的上行链路；
    • 每个 Subscriber 各自有独立的 SFU → Subscriber 下行链路，带宽状况完全独立；
    • SFU 需要分别为每个下行链路运行 BWE，并根据估计结果决定向该 Subscriber 转发哪一层 Simulcast 流。

    SFU 侧实现 BWE 的挑战：

    1. SFU 是转发者，不是编码者：它不能直接调整编码码率，只能在转发的 Simulcast 层之间切换；
    2. 需要实现完整的 GCC 逻辑：序号管理、Transport-CC 生成、延迟分析等，实现复杂度较高；
    3. 探测需要主动注入 padding 包：SFU 需在媒体流量不足时发送填充包以触发探测，会增加带宽开销。

    11. 调试与可观测性

    11.1 chrome://webrtc-internals

    Chrome 内置的 chrome://webrtc-internals 提供了丰富的 BWE 实时图表，包括：

    • googAvailableSendBandwidth：当前 BWE 估计值；
    • googTransmitBitrate：实际发送码率；
    • googActualEncBitrate：编码器实际码率；
    • googTargetEncBitrate：目标编码码率；
    • googBucketDelay：Pacer 队列延迟。

    11.2 RTC Event Log

    libwebrtc 支持将所有 BWE 相关事件记录到 RTC Event Log（chrome://flags 中启用），包括：

    • 每次 BWE 更新；
    • 探测请求和结果；
    • 目标码率变化。

    可使用 WebRTC Internals Exporter 或 Google RTC Event Log Visualizer 进行可视化分析。

    11.3 关键指标监控

    在生产系统中，应重点监控以下指标：

    指标 健康范围 说明
    BWE 估计值稳定性 波动 < 10% 剧烈波动说明网络不稳定或算法调参问题
    实际发送码率 / BWE 80%–100% 显著低于 100% 说明编码能力或 Pacer 有瓶颈
    目标码率与实际码率的差 < 5% 过大的差距说明编码器响应慢
    探测成功率 > 80% 频繁失败的探测说明网络容量接近上限
