    1. 背景与动机

    实时音视频通信（RTC）的拥塞控制是一个极具挑战性的工程问题，其难点在于：

    • 编码特性限制：媒体流的码率调整不像文件传输那样随意，通常只能按帧或关键帧边界进行调整，且调整粒度较大
    • 丢包敏感性：视频帧对丢包极为敏感，而实时性要求又禁止使用 TCP 式的重传来修复丢包
    • 低延迟要求：对话类场景要求端到端延迟在 150ms 以内，任何排队延迟都会直接损害用户体验
    • 公平性要求：实时流量需要与同路径上的 TCP 流量和平共处，既不能被饿死，也不能无限占用带宽

    传统 TCP 的拥塞控制（CUBIC、Reno 等）基于丢包作为拥塞信号，本质上是”填满队列 → 丢包 → 降速”的被动响应模式，引入了大量排队延迟，不适合实时场景。

    GCC 的核心思路是主动探测：通过观察数据包的单向延迟变化趋势（delay gradient）来提前感知拥塞的苗头，在队列真正满溢（导致丢包）之前就开始降速，从而兼顾低延迟与高吞吐。

    2. 标准化历程

    时间 里程碑
    2012 Chrome M23 首次内置 GCC，用于 Google Hangouts
    2013 Alvestrand 在 IETF 提交首个草案 draft-alvestrand-rtcweb-congestion-00
    2015 草案转移至 RMCAT WG，发布 draft-ietf-rmcat-gcc-00
    2016 最终草案 draft-ietf-rmcat-gcc-02 发布（Stefan Holmer、Henrik Lundin 等，Google & Politecnico di Bari）
    至今 草案未推进为正式 RFC，但已成为 WebRTC 事实标准，Chrome/Firefox/Safari 均有实现

    注意：draft-ietf-rmcat-gcc-02 是目前最权威的公开规范文档，本文所有算法描述均以此为准。该草案虽已过期（Expired），但其算法实现已在各主流浏览器中稳定运行多年。

    3. 系统模型与整体架构

    GCC 定义的系统中包含以下核心组件：

    整体流程：

    1. 发送端将媒体数据交给 Pacer 平滑发送
    2. 接收端记录每个 RTP 包的到达时间，通过 RTCP 反馈（REMB 或 Transport-CC）发回发送端
    3. Delay-based Controller 分析包间延迟变化，估算可用带宽上限 A_r
    4. Loss-based Controller 综合丢包率与 RTT，计算目标发送码率 A_s
    5. 最终目标码率 = min(A_r, A_s)，传递给 Pacer 和编码器

    4. 反馈机制与信令扩展

    GCC 依赖以下 RTP/RTCP 扩展来获取网络测量数据：

    4.1 Absolute Send Time（abs-send-time）

    • 定义：http://www.webrtc.org/experiments/rtp-hdrext/abs-send-time
    • 作用：在每个 RTP 包头部携带发送端的绝对时间戳（24-bit，精度约 3.8μs）
    • 用途：接收端据此计算包间发送间隔 T(i) - T(i-1)，进而得出单向延迟变化量

    4.2 REMB（Receiver Estimated Maximum Bitrate）

    • 定义：draft-alvestrand-rmcat-remb（未标准化，但所有主流浏览器支持）
    • 作用：接收端将 Delay-based Controller 估算出的可用带宽上限，通过 RTCP 消息反馈给发送端
    • 格式：RTCP PSFB（Payload-Specific Feedback），包含估计码率（mantissa + exponent 格式）
    • 推荐发送频率：检测到拥塞时立即发送，否则至少每秒一次

    4.3 Transport-wide Congestion Control（Transport-CC / TWCC）

    • 定义：draft-holmer-rmcat-transport-wide-cc-extensions
    • 作用：接收端为每个 RTP 包（跨所有媒体流）打上传输层序号，并将到达时间戳批量反馈给发送端
    • 优势：所有计算在发送端完成，延迟更低，且对多路复用场景更友好
    • 推荐反馈间隔：每收到一个视频帧反馈一次，或至少每 30ms 一次（仅音频或多路流场景）；带宽受限时可放宽至每 100ms 一次

    5. 发送引擎（Pacer）

    Pacer 负责将编码器产生的数据”平滑”地发送到网络，避免突发流量引发人工拥塞：

    关键参数：

    • burst_time：推荐值 5 ms，即每 5ms 发送一批数据包
    • 每批发送量：目标码率 × burst_time

    Pacer 与拥塞控制器紧密协作：控制器计算出目标码率后，Pacer 按此节奏匀速释放数据包。这样既能避免短时突发撑满缓冲区，又能保证测量到的包间延迟真实反映网络状态，而非发送端自身造成的抖动。

    当编码器输出码率高于目标时，Pacer 允许以 1.5 × 目标码率 的速率排空队列，避免因队列积压引入额外延迟。

    6. 基于延迟的控制器（Delay-based Controller）

    这是 GCC 的核心创新，分为四个串联模块：

    6.1 到达时间模型（Arrival-time Model）

    定义以下核心变量：

    符号 含义
    T(i) 第 i 组包的最后一个包的发送时间戳
    t(i) 第 i 组包的最后一个包的到达时间戳
    T(i) - T(i-1) 包组间发送时间间隔（inter-departure time）
    t(i) - t(i-1) 包组间到达时间间隔（inter-arrival time）
    d(i) 包组间单向延迟变化量（inter-group delay variation）

    直觉理解：

    • d(i) > 0：包到达间隔比发送间隔更长 → 路径上有新的排队延迟产生 → 可能过载
    • d(i) < 0：包到达比预期更快 → 之前的队列正在被清空 → 带宽有余量
    • d(i) ≈ 0：网络状态稳定

    将 d(i) 建模为随机过程：

    • m(i)：延迟梯度的均值（congestion trend），是真正想估计的量
    • v(i)：网络抖动等噪声，建模为零均值高斯白噪声

    当网络过载时，m(i) 会持续增大；当队列排空时，m(i) 减小；无拥塞时，m(i) 接近零。

    6.2 预过滤（Pre-filtering）

    预过滤的目的是处理网络中断后突发恢复等异常场景，避免误判。

    将以下情况的包合并为同一个”包组”：

    1. 在 burst_time 时间窗口内发送的连续包
    2. 到达时间间隔小于 burst_time 且 d(i) < 0 的包（判断为同批到达的突发包）

    这样处理后，突发恢复不会产生大量连续的负 d(i)，避免错误触发”欠载”状态。

    6.4 过载检测器（Over-use Detector）

    将 m_hat(i) 与自适应阈值 del_var_th(i) 比较，得出三种状态：

    条件 状态
    m_hat(i) > del_var_th(i) 且持续超过 overuse_time_th Over-use（过载）
    m_hat(i) < -del_var_th(i) Under-use（欠载）
    其他 Normal（正常）

    重要细节：即使满足过载条件，若 m_hat(i) < m_hat(i-1)（延迟梯度在下降），也不触发过载信号，避免在恢复阶段误降码率。

    6.5 速率控制状态机（Rate Control）

    Delay-based Controller 的速率控制模块是一个三状态机：

    状态转移表（来自 draft-ietf-rmcat-gcc-02 第 5.5 节）：

    当前状态 \ 检测信号 Over-use Normal Under-use
    Hold Decrease Increase Hold
    Increase Decrease Increase Hold
    Decrease Decrease Hold Hold

    各状态行为：

    Increase（增加）：码率向上探测，分两种模式：

    • 乘性增加（Multiplicative Increase）：远离收敛点时使用
    • 加性增加（Additive Increase）：接近收敛点时使用，更保守 其中 kbps_per_second 推荐值为 0.5 * response_time * expected_packet_size_bits / response_time²

    判断是否接近收敛点：若当前入站码率 R_hat(i) 落在”历史最大码率均值 ± 3个标准差”范围内，则认为接近收敛，切换到加性增加。均值和标准差用平滑因子 0.95 的指数移动平均估计。

    Hold（保持）：维持当前码率，等待队列排空后再进入 Increase 状态。

    7. 基于丢包的控制器（Loss-based Controller）

    Loss-based Controller 在发送端运行，基于 RTCP Receiver Report 中的丢包率 p 调整发送码率估计 A_s_hat：

    设计逻辑：

    • p < 2%：认为丢包是随机噪声（非拥塞），以 5%/次 的速率缓慢增加码率
    • 2% ≤ p < 10%：认为网络处于轻微拥塞边缘，维持当前码率，等待更多信息
    • p ≥ 10%：明确拥塞，以 0.5p 的比例乘性降低码率（例如 10% 丢包 → 降至原来的 95%；20% 丢包 → 降至原来的 90%）

    为什么 2%-10% 区间不响应？论文作者指出：若传输信道因过载产生少量丢包，这个比例很快就会超过 10%。如果丢包率长时间维持在 2%-10% 而不上升，说明丢包更可能来自随机信道误差（如无线信道），而非拥塞，因此不应贸然降速。

    8. 两个控制器的协同工作

    两个控制器分别产生各自的码率估计，最终目标码率取较小值：

    职责分工：

    • Delay-based Controller：提前感知拥塞趋势（低延迟但信噪比较低），给出带宽上限
    • Loss-based Controller：在真实丢包发生时作为保底兜底（信号滞后但确定），确保不会因盲目增加码率而导致大量丢包

    两者形成互补：延迟控制器主导”无拥塞 → 探测 → 轻微拥塞”阶段，丢包控制器在”严重拥塞”时强制介入。

    9. GCC 的两代演进：REMB-GCC 与 TFB-GCC

    第一代：REMB-GCC（基于接收端的延迟估计）

    • Delay-based Controller 运行在接收端
    • 计算结果通过 REMB 消息反馈给发送端
    • 发送端的 Loss-based Controller 以 REMB 值作为上限

    缺点：RTT 附加延迟导致反馈不及时；REMB 消息粒度较粗；多流场景下效率低。

    第二代：TFB-GCC（基于 Transport-CC 的发送端估计）

    • 所有控制逻辑移到发送端
    • 接收端仅负责记录到达时间戳并通过 Transport-CC 反馈
    • 发送端实现趋势线滤波器（Trendline Filter）取代 Kalman 滤波，通过线性回归估计延迟趋势
    • 支持多路流共享同一个拥塞控制器

    Trendline Filter（TFB-GCC 的延迟估计核心）：

    对最近 N 个包的单向延迟变化量 d(i) 进行线性回归，斜率即为拥塞趋势：

    斜率持续为正 → 排队延迟在增长 → 过载信号。

    这种方式对噪声更鲁棒，也更容易跨多路流汇总计算。

    目前（2024年）Chrome 默认使用 TFB-GCC（Transport-CC + 发送端估计）。旧的 REMB-GCC 路径已被标记为 deprecated，但因兼容性需要仍然保留。

    11. 性能特性与局限性

    优点

    收敛性：GCC 能在均匀网络条件下（无并发竞争流）收敛到接近链路容量的稳定工作点。

    公平性：多路 GCC 流共享同一瓶颈链路时，能获得较公平的带宽分配（intra-protocol fairness）。

    低延迟：相比 TCP 类算法，GCC 保持的队列深度更小，端到端延迟更低。

    鲁棒性：对随机丢包（如无线链路）有一定抵抗力，不会因偶发丢包大幅降速。

    与 TCP 共存：自适应阈值机制使 GCC 能在 TCP 流量压力下保持基本的带宽份额，避免饿死。

    局限性

    收敛速度慢：当链路带宽突然增大时，GCC 需要约 20-30 秒才能重新探测到新的可用带宽，期间视频质量无法充分利用新增带宽。

    锯齿抖动：由于 AIMD（Additive Increase / Multiplicative Decrease）的固有特性，在动态链路上码率呈锯齿状波动，视频质量不够稳定。

    与 TCP 竞争处于劣势：虽然自适应阈值有所缓解，但在与激进的 TCP CUBIC 流竞争时，GCC 流仍然倾向于获得更少的带宽份额，表现出不完全的 TCP 友好性。

    反向路径干扰：反向路径（RTCP 反馈路径）的拥塞会影响到 REMB 消息的及时性，间接影响拥塞控制的响应速度。

    控制器位置 发送端（TFB）/ 接收端（REMB） 发送端 发送端 发送端
    与 TCP 竞争 中等 较弱 中等 激进
    收敛速度 慢（~25s） 中等 中等 快
    带宽利用率 高（~82% @ 5% 丢包） 较低 中等 高
