    1. 为什么需要 ICE

    现代互联网中，绝大多数设备并不拥有全球唯一的公网 IP 地址。NAT（Network Address Translation，网络地址转换）技术的广泛部署，从家用路由器到企业防火墙，使得直接在两个端点之间建立 P2P 连接变得困难重重。

    具体挑战包括：

    • 私网地址不可路由：10.x.x.x、192.168.x.x、172.16-31.x.x 等私网地址无法在公网上直接寻址。
    • NAT 类型多样：Full Cone、Restricted Cone、Port-Restricted Cone、Symmetric NAT 对入站连接的限制程度各不相同，其中 Symmetric NAT 对于 P2P 穿透最为棘手。
    • 防火墙策略各异：企业环境往往只允许特定端口（如 80/443）的出站连接，甚至做深度包检测（DPI）。
    • 多层 NAT 叠加：设备可能处于多级 NAT 之后，路径更加复杂。

    在 ICE 出现之前，应用层通常需要为每种 NAT 类型手动设计穿透方案，既不通用，也难以维护。ICE 的出现统一了这一过程：系统地枚举所有可能的连接路径（候选地址），按优先级逐一验证，选出最优路径。

    2. ICE 协议规范概览

    ICE 由 IETF MMUSIC 工作组制定，核心规范演进如下：

    RFC 时间 说明
    RFC 4091 / 4092 2005 早期草案（已废弃）
    RFC 5245 2010 ICE 第一个正式标准，Offer/Answer 绑定版本
    RFC 8445 2018 当前主流标准，解耦了 ICE 与 SDP Offer/Answer，增加 IPv6、Consent Freshness 等支持，废弃 Aggressive Nomination
    RFC 8839 2021 ICE 的 SDP Offer/Answer 使用规范（从 RFC 8445 中分离）
    RFC 8838 2021 Trickle ICE，候选地址增量交换扩展
    RFC 6544 2012 TCP 候选地址扩展
    RFC 7675 2015 STUN Consent Freshness，连接持续授权机制

    WebRTC 的实现（Chrome、Firefox、Safari）均以 RFC 8445 + RFC 8839 + RFC 8838 为基础。

    3. 核心概念与术语

    在深入流程之前，先搞清楚 RFC 8445 中的关键术语：

    3.1 ICE Agent（ICE 代理）

    参与 ICE 候选交换的协议实体，即通信的每一端。一次会话中有且仅有两个 Agent：

    • Initiating Agent（发起方）：发起 ICE 候选交换流程的一方，通常是 Offer 的发送者。
    • Responding Agent（响应方）：接收并响应发起方的一方，通常是 Answer 的发送者。

    3.2 控制角色：Controlling / Controlled

    ICE 中还存在一对逻辑角色，与 Initiating/Responding 有所不同：

    • Controlling Agent（控制方）：负责最终提名（Nominate） 使用哪个候选对进行通信。两个 Full Implementation 时，发起方担任 Controlling 角色；一端为 Lite Implementation 时，Full 实现的一方担任 Controlling 角色。
    • Controlled Agent（受控方）：等待 Controlling Agent 提名，并遵照执行。

    若两端角色冲突（均声称自己是 Controlling），通过 STUN 请求中的 tiebreaker 值解决。

    3.3 Transport Address（传输地址）

    IP 地址 + 端口号 + 协议（UDP 或 TCP）的三元组，是候选地址的物理载体。

    3.4 Candidate（候选地址）

    对一个传输地址的描述，包含类型、优先级、Foundation 等元数据。每个候选地址与一个特定的媒体组件（Component） 关联：

    • Component ID = 1：RTP 流量
    • Component ID = 2：RTCP 流量（若未使用 RTCP-MUX，则需要单独候选）

    3.5 Foundation（基础标识符）

    Foundation 是一个字符串，用于标识”具有相同网络特征”的候选地址集合。当两个候选地址的类型、Base IP、传输协议、STUN/TURN 服务器地址完全相同时，它们共享同一个 Foundation。

    Foundation 的主要用途是冻结算法（Frozen Algorithm）：避免对具有相同 Foundation 的候选对进行重复检查，提高效率。

    3.6 Base（基础地址）

    候选地址的”原始来源”地址：

    • Host 候选的 Base 是其自身。
    • Server-Reflexive 候选的 Base 是发出 STUN 请求的本地 Host 候选。
    • Relayed 候选的 Base 是 TURN 服务器分配的中继地址本身。

    3.7 Full vs Lite Implementation

    • Full Implementation：执行完整的 ICE 流程，包括候选收集、连接性检查、状态机维护，支持所有 NAT 穿透场景。浏览器、移动端均使用 Full 实现。
    • Lite Implementation：简化版，省略状态机和主动连接性检查，仅响应对端发来的检查请求。适用于拥有公网地址、无需 NAT 穿透的服务器端。

    4. 候选地址的类型

    ICE 定义了四种候选类型，按 NAT 穿透能力和优先级从高到低排列：

    4.1 Host Candidate（主机候选）

    直接从本机网络接口获取的 IP 地址和端口。

    • 优先级最高，延迟最低。
    • 在同一局域网内的两个 Agent 之间，Host 候选往往可以直接建立连接。
    • 隐私风险：Host 候选会暴露用户的真实私网 IP，WebRTC 在浏览器侧对此有 mDNS 匿名化机制（RFC 8828）。

    4.2 Server-Reflexive Candidate（服务器反射候选，srflx）

    通过向 STUN 服务器发送 Binding Request，由 STUN 服务器在响应中告知 Agent 其公网侧的 IP 地址和端口（即 NAT 的映射地址）。

    • raddr / rport：对应的 Base 地址（私网侧）。
    • 适用于 Full Cone / Restricted Cone / Port-Restricted Cone NAT。
    • 对 Symmetric NAT 无效（每次连接目的地不同，端口映射也不同）。

    4.3 Peer-Reflexive Candidate（对等反射候选，prflx）

    在连接性检查过程中动态发现的候选。当 Agent A 发送 STUN Binding Request 给 Agent B 时，B 可能观察到一个此前未在 SDP 中宣告的源地址，这个地址会被 B 记录为一个 Peer-Reflexive 候选。

    Peer-Reflexive 候选不在 SDP 中预先声明，而是在 STUN 连接检查的响应过程中实时发现并加入候选对列表。这在网络路径发生变化、或 Trickle ICE 场景下尤为重要。

    RFC 8445 附录 B.7 特别说明：应优先选择 Peer-Reflexive 候选而非 Server-Reflexive 候选，因为前者反映了连接检查时刻真实的 NAT 映射，更加准确。

    4.4 Relayed Candidate（中继候选，relay）

    当 Agent 无法通过 STUN 建立直接连接时，向 TURN 服务器申请一个中继地址（Relayed Transport Address）。所有流量经由 TURN 服务器转发。

    • raddr / rport：Agent 的 Server-Reflexive 地址（相对于 TURN 服务器观察到的地址）。
    • 优先级最低，因为引入了额外的网络跳数和延迟。
    • 是连接成功率的”兜底”方案，尤其对 Symmetric NAT 和严格防火墙场景至关重要。
    • 典型部署中，TURN 中继连接率可以作为网络质量的负向指标。

    5. 候选地址的收集（Gathering）

    候选收集由 ICE Agent 在本地启动，通常在 setLocalDescription() 调用后触发。流程如下：

    5.1 Host 候选收集

    Agent 枚举本机所有可用的网络接口（包括 IPv4 和 IPv6），在每个接口上分配一个或多个端口，形成 Host 候选。

    对于多宿主主机（多网卡），每个接口都会产生独立的候选地址。

    5.2 Server-Reflexive 候选收集

    Agent 向配置的 STUN 服务器发送 UDP STUN Binding Request，STUN 服务器将其观察到的 Agent 公网地址填入 XOR-MAPPED-ADDRESS 属性，返回给 Agent。这就是 Agent 的 Server-Reflexive 地址。

    注意：一个 STUN 请求（由某个 Host 候选发出）可以同时为该 Host 候选生成对应的 srflx 候选。如果多个 Host 候选共享同一个 NAT 出口（例如同一路由器），它们可能映射到相同的公网地址，此时 ICE 会进行冗余候选消除（Redundant Candidate Pruning），避免重复检查。

    5.3 Relayed 候选收集

    Agent 向 TURN 服务器发送 TURN Allocate Request，TURN 服务器在其公网 IP 上分配一个端口，作为该 Agent 的中继地址。TURN 服务器在 Allocate Response 中返回：

    • RELAYED-ADDRESS：中继候选地址
    • XOR-MAPPED-ADDRESS：Agent 相对于 TURN 服务器的 srflx 地址

    5.4 收集状态与 iceGatheringState

    WebRTC API 通过 RTCPeerConnection.iceGatheringState 暴露收集状态：

    状态 含义
    new 尚未开始收集
    gathering 正在收集候选地址
    complete 收集完成（所有候选已生成）

    收集完成后，若使用传统 ICE（非 Trickle），Agent 会将全部候选放入 SDP 一并发送；若使用 Trickle ICE，则每发现一个候选就立即通过信令通道推送。

    除候选属性外，SDP 中还包含以下 ICE 相关的会话级属性：

    • **ice-ufrag**：ICE 用户名片段（Username Fragment），用于 STUN 请求的认证。
    • **ice-pwd**：ICE 密码，用于 STUN 消息完整性验证（HMAC-SHA1）。
    • **ice-options: trickle**：声明支持 Trickle ICE。
    • **ice-lite**：声明本端为 Lite Implementation。

    8. 候选对（Candidate Pair）的形成与排序

    双方通过信令通道交换候选列表后，每个 ICE Agent 将本地候选与远端候选两两配对，形成候选对（Candidate Pair）。

    8.1 配对规则

    • 本地候选与远端候选的传输协议必须相同（均为 UDP 或均为 TCP）。
    • 本地候选与远端候选的地址族（IPv4/IPv6）应一致（即不跨族配对）。
    • 每个数据流的每个组件独立配对。

    一个数据流包含 RTP（Component 1）和 RTCP（Component 2，若不使用 RTCP-MUX），因此每个流的候选对数量 = 本地候选数 × 远端候选数 × 组件数。

    8.3 Checklist（检查列表）

    每个数据流对应一个 Checklist，其中包含排序后的候选对列表。候选对的状态机如下：

    • Frozen：默认初始状态，每个 Foundation 只有一个候选对被”解冻”（Waiting），其余保持 Frozen，直到同 Foundation 的其他流检查完成。
    • Waiting：可以开始发起 STUN Binding Request。
    • In-Progress：已发出请求，等待响应。
    • Succeeded：收到成功响应，该对进入 Valid List（有效列表）。
    • Failed：超时或收到错误响应。

    9. 连接性检查（Connectivity Checks）

    连接性检查是 ICE 的核心机制，通过 STUN Binding Request/Response 实现对称的双向验证。

    9.1 基本流程：四次握手

    双方同时对同一候选对进行检查，这种对称机制可以”打洞”，两端几乎同时向对方发送包，NAT 设备会将对端的包识别为已建立映射的回程流量，从而放行。

    9.2 STUN 请求中的关键属性

    • **USERNAME**：格式为 远端ufrag:本端ufrag，用于 Agent 识别这是哪个 ICE 会话的检查请求。
    • **MESSAGE-INTEGRITY**：基于 ice-pwd 的 HMAC-SHA1，防止篡改。
    • **PRIORITY**：Peer-Reflexive 候选的优先级（若发现新候选时使用）。
    • **ICE-CONTROLLING / ICE-CONTROLLED**：标识发送方的角色，附带一个 64 位随机 tiebreaker 值。
    • **USE-CANDIDATE**（仅 Controlling Agent 在提名时使用）：通知对端提名该候选对。

    9.3 Triggered Checks（触发检查）

    当 Agent 收到对端发来的 STUN Binding Request 时，如果对应的候选对状态为 Frozen 或 Waiting，则会将其移至队列前部立即进行检查（Triggered Check）。这确保对端已发现某条路径时，本端也能快速验证对称性。

    9.4 Peer-Reflexive Candidate 的发现

    在连接性检查过程中，若 Agent B 收到来自 Agent A 的 STUN 请求，其源地址不在 B 已知的候选列表中，则 B 将该地址记录为 Peer-Reflexive 候选，并为其创建新的候选对加入 Checklist（RFC 8445 §7.3.1.3）。这在以下场景中常见：

    • STUN 服务器探测到的 srflx 地址与实际连接检查时观察到的地址不同（Symmetric NAT）。
    • Trickle ICE 场景下，远端尚未发来所有候选时，连接检查先于候选信息到达。

    10. 候选提名（Nomination）与 ICE 结束

    10.1 提名过程（RFC 8445 中的 Regular Nomination）

    RFC 8445 废弃了 RFC 5245 中的 Aggressive Nomination，仅保留 Regular Nomination：

    1. Controlling Agent 等待至少有一个有效对出现在 Valid List 中。
    2. Controlling Agent 选定一个（或多个，取最高优先级）有效对，向其发送带有 USE-CANDIDATE 属性的 STUN Binding Request。
    3. 若 Controlled Agent 接受该提名请求并成功响应，该候选对被标记为 Nominated。
    4. 两端均将该对设为 Selected Pair，用于实际媒体传输。
    5. 对应数据流的 Checklist 状态变为 Completed，ICE 对该流结束。

    10.2 ICE 整体状态

    整个 ICE 会话的状态通过 RTCPeerConnection.iceConnectionState 反映：

    状态 含义
    new ICE 代理等待远端候选
    checking 正在进行连接性检查
    connected 至少找到一个有效路径
    completed 所有数据流均已完成 ICE 并提名
    failed 所有候选对检查均失败
    disconnected 连接中断（keepalive 超时）
    closed ICE 会话已关闭

    11. Trickle ICE：增量式候选交换

    传统 ICE 要求 Agent 等待所有候选收集完成，再将完整列表放入 SDP 发送。这在慢速网络或 TURN 分配延迟时会造成显著的连接建立延迟。

    Trickle ICE（RFC 8838，2021）允许 Agent 在收集到候选的同时，通过信令通道逐个推送候选给对端，并立即开始连接性检查。

    11.2 End-of-Candidates 指示

    当 Agent 完成所有候选收集后，须通过信令通道发送 end-of-candidates 指示，让对端知道不再会有新候选到来。这使得对端可以：

    • 更新 Checklist 状态。
    • 在所有组件都未找到有效对时，判定 ICE 失败。
    • 在已有有效对（即使优先级较低，如 relay）时加速 ICE 结束。

    11.3 Trickle ICE 对延迟的改善

    在典型场景下，Trickle ICE 可将连接建立时间从数秒降低到数百毫秒，显著提升用户体验。在 WebRTC 实现中，Trickle ICE 已成为事实上的默认行为，通过 SDP a=ice-options:trickle 协商开启。

    12. ICE Restart

    当网络路径发生变化时（如 Wi-Fi 切换、IP 地址变更），可以触发 ICE Restart 重新协商连接路径，而无需重新创建 PeerConnection。

    13. Keepalive 与 Consent Freshness

    13.1 NAT Keepalive

    NAT 设备维护的地址映射表有超时时间（通常 30 秒到几分钟不等）。若长时间无数据包流过，NAT 映射会被释放，导致连接中断。ICE 通过定期发送 STUN Binding Indication（无需响应的单向 STUN 消息）来维持 NAT 映射，默认间隔为 15 秒（RFC 8445 §11）。

    13.2 Consent Freshness（RFC 7675）

    ICE 连接建立后，如何确认远端仍在线且同意继续接收流量？RFC 7675 定义了持续授权检查（Consent Freshness） 机制：

    • Agent 每 5 秒（随机化为 4–6 秒）发送一次 STUN Binding Request 给对端。
    • 若 30 秒内未收到任何有效响应，则视为授权过期，停止发送数据。
    • 每次 Consent Check 使用新的 STUN Transaction ID，且请求发送一次（不重传）。

    这一机制解决了两类安全问题：

    1. 放大攻击：防止恶意 JavaScript 利用浏览器向不知情的第三方发送大量流量。
    2. 会话劫持检测：若对端下线或网络中断，发送方可以及时感知并停止发送。

    14. ICE 与 WebRTC 安全：DTLS/SRTP 绑定

    ICE 负责找到一条可用的网络路径，但它本身不加密媒体数据。WebRTC 强制要求在 ICE 建立的路径上运行 DTLS（Datagram Transport Layer Security），完成密钥协商后再通过 SRTP 加密实际的媒体数据。

    ICE 与 DTLS 的绑定关系：

    1. ICE 通过连接性检查验证路径的物理可达性，同时借助 ice-ufrag/ice-pwd 的 HMAC-SHA1 验证实现信令绑定（防止中间人插入伪造候选）。
    2. 在 ICE 完成、Selected Pair 确定后，DTLS 握手在该路径上进行，交换证书并协商 SRTP 密钥材料。
    3. SDP 中的 a=fingerprint 属性携带 DTLS 证书的指纹，通过信令通道传递，确保 DTLS 握手不被中间人替换证书。

    这种设计实现了信令层（SDP）→ 传输层（ICE）→ 加密层（DTLS）→ 媒体层（SRTP） 的层层绑定，完整防护了 WebRTC 的安全威胁模型。

    • STUN 候选未能收集（STUN 服务器配置错误）。
    • Symmetric NAT 环境，srflx 候选无法穿透。
