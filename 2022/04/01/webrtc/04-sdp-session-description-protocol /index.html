    1. SDP 是什么，不是什么

    SDP（Session Description Protocol，会话描述协议）是 WebRTC 中媒体能力协商的核心载体。在理解 SDP 之前，首先要明确它的边界：

    SDP 是：

    • 一种纯文本格式，用于描述多媒体通信会话的参数
    • 告知对方”我支持哪些编解码器”、”我的 ICE 凭证是什么”、”我的 DTLS 指纹是什么”
    • WebRTC Offer/Answer 协商的核心内容

    SDP 不是：

    • 一个传输协议（它本身不负责传输自己，需要借助信令通道）
    • 一个媒体传输协议（它只描述参数，实际媒体由 RTP/SRTP 传输）
    • 一个加密协议（DTLS-SRTP 负责加密，SDP 只携带 DTLS 指纹用于验证）

    “SDP does not deliver any media streams itself but is used between endpoints for negotiation of network metrics, media types, and other associated properties.”

    2. 规范演进

    年份 规范 说明
    1998 RFC 2327 SDP 初版，随 SAP（Session Announcement Protocol）发布
    2006 RFC 4566 重要更新，澄清了多处歧义，废弃了 k= 加密字段
    2021 RFC 8866 当前有效规范，废弃 RFC 4566，进一步澄清语义

    WebRTC 对 SDP 的使用受以下规范约束：

    • RFC 8866：SDP 基础格式定义
    • RFC 3264：Offer/Answer 模型（SDP 如何用于协商）
    • RFC 8829（JSEP）：JavaScript 环境下 SDP 的生成与处理规则
    • RFC 8843：BUNDLE（多媒体流在单一传输通道复用）
    • RFC 8853：Simulcast SDP 扩展
    • RFC 8830：a=msid MediaStream 标识扩展

    4. 会话级字段详解

    4.1 v= 版本

    当前 SDP 版本固定为 0，这是 RFC 4566 以来唯一合法的值。

    4.2 o= 发起者与会话标识

    示例：

    各子字段说明：

    子字段 示例值 说明
    <username> - 发起方用户名，WebRTC 中通常用 - 占位
    <sess-id> 4567890123456789 会话 ID，随机数字，在会话生命周期内不变
    <sess-version> 2 会话版本号，每次重新协商时递增
    <nettype> IN 网络类型，固定为 IN（Internet）
    <addrtype> IP4 地址类型，IP4 或 IP6
    <unicast-address> 127.0.0.1 发起方地址，WebRTC 中通常为 127.0.0.1 或 0.0.0.0（实际地址通过 ICE 协商）

    WebRTC 实践：<sess-id> 和 <sess-version> 共同唯一标识一个会话，<sess-version> 递增表示会话发生了重新协商（Re-negotiation）。

    4.3 s= 会话名称

    RFC 8866 要求此字段必须存在且非空，WebRTC 中通常用 - 作为占位符。

    4.4 t= 时间

    格式：t=<start-time> <stop-time>，时间以 NTP 秒数表示（距 1900-01-01 的秒数）。

    WebRTC 中固定使用 t=0 0，表示没有固定的开始和结束时间（永久会话）。

    4.5 c= 连接信息

    格式：c=<nettype> <addrtype> <connection-address>

    在 WebRTC 中，实际连接地址通过 ICE Candidate 确定，c= 行通常写 0.0.0.0 作为占位。

    4.6 b= 带宽

    类型 说明
    AS Application Specific，最大带宽（kbps），对应 RTP 会话带宽
    TIAS Transport Independent Application Specific（RFC 3890），以 bps 为单位，更精确
    CT Conference Total，会议总带宽上限

    WebRTC 中常见的 b=AS 用于限制单条媒体流的最大发送码率，也是 SFU 限速的常用手段。

    协议栈 含义
    UDP/TLS/RTP/SAVPF UDP + DTLS + SRTP + RTCP 反馈（RFC 4585）

    5.4 a=sendrecv / sendonly / recvonly / inactive — 媒体方向

    属性 含义
    a=sendrecv 双向（默认）
    a=sendonly 只发送（对端应设 recvonly）
    a=recvonly 只接收
    a=inactive 暂停媒体

    在 SFU 场景中，sendonly/recvonly 的配合尤为常见。

    6. WebRTC 专属 SDP 扩展属性

    标准 SDP（RFC 8866）不包含 WebRTC 所需的全部信息，以下属性来自 WebRTC 相关 RFC 扩展。

    6.1 ICE 相关属性

    a=ice-ufrag 和 a=ice-pwd（RFC 8839）：

    ICE 用户名片段和密码，用于 STUN 绑定请求认证。密码长度至少 22 个 Base64 字符。在 ICE 重启时会更换（新 Offer 中提供新的 ufrag/pwd）。

    a=ice-options:trickle（RFC 8838）：

    声明支持 Trickle ICE，候选可以在 Offer/Answer 之后增量发送，是现代 WebRTC 的默认行为。

    a=candidate（RFC 8839）格式：

    a=end-of-candidates（RFC 8840）：

    声明本端候选收集完毕，对端无需再等待。在 Trickle ICE 中，event.candidate === null 时发送此行。

    6.2 DTLS 相关属性

    a=fingerprint（RFC 8122）：

    DTLS 证书的哈希摘要，对端在 DTLS 握手时据此验证身份，防止中间人攻击。格式：a=fingerprint:<hash-function> <fingerprint>，哈希算法通常为 sha-256。

    安全关键：若攻击者篡改信令中的 SDP，替换 a=fingerprint 为自己证书的指纹，即可实施 DTLS 中间人攻击并解密媒体流。这正是信令必须使用 TLS（WSS）的根本原因。

    a=setup（RFC 5763）：

    定义 DTLS 握手角色：

    值 含义
    active 主动发起握手（DTLS Client）
    passive 被动等待（DTLS Server）
    actpass 两者皆可（Offer 方常用）

    典型流程：Offer 写 actpass，Answer 写 active（Answer 方主动发起 DTLS 握手）。

    6.3 BUNDLE — 多路复用（RFC 8843）

    BUNDLE 允许多个 m= 块共享同一个底层传输连接（同一个 ICE 候选对 / 同一个 DTLS 会话 / 同一个 UDP 端口）。

    未使用 BUNDLE 时，每个 m= 块需要独立的 ICE 协商和 DTLS 握手，耗费大量端口资源和时间。

    使用 BUNDLE 时，每个捆绑的 m= 块必须包含：

    接收端通过 RTP 包头中的 MID 扩展字段区分不同媒体流。WebRTC 浏览器默认启用并要求 BUNDLE 协商成功。

    6.4 a=rtcp-mux — RTP/RTCP 端口复用（RFC 5761）

    允许 RTP 和 RTCP 使用同一个 UDP 端口，而非传统的两个端口。配合 BUNDLE，现代 WebRTC 通话通常只需要 1 个 UDP 端口承载所有音视频的 RTP 和 RTCP。

    6.5 a=mid — 媒体标识符

    为每个 m= 块分配字符串标识符（Media ID），在 BUNDLE 中用于标识归属，也是 RTCRtpTransceiver.mid 属性的值。

    6.6 a=msid — MediaStream 标识（RFC 8830）

    将 RTP 流与 W3C API 中的 MediaStream 和 MediaStreamTrack 对象关联：

    • <stream-id>：对应 MediaStream.id
    • <track-id>：对应 MediaStreamTrack.id

    接收端通过 a=msid 信息构建 RTCTrackEvent.streams 数组。

    6.7 a=ssrc — 同步源标识

    SSRC（Synchronization Source） 是 RTP 包头中的 32 位随机标识符。cname（Canonical Name）用于跨流同步（音视频唇形同步）。

    6.8 a=extmap — RTP 头部扩展映射（RFC 8285）

    在 SDP 中声明 RTP 头部扩展 ID 与其语义 URI 的映射：

    扩展 URI 用途 RFC
    urn:ietf:params:rtp-hdrext:ssrc-audio-level 携带音频音量级别，VAD 检测 RFC 6464
    urn:ietf:params:rtp-hdrext:sdes:mid BUNDLE MID 标识 RFC 8843
    urn:ietf:params:rtp-hdrext:sdes:rtp-stream-id Simulcast RID 标识 RFC 8852

    urn:ietf:params:rtp-hdrext:toffset 传输时间偏移（唇形同步） RFC 5450

    6.9 a=rtcp-fb — RTCP 反馈能力（RFC 4585）

    属性 含义

    7. 一个完整的 WebRTC SDP Offer 逐行注解

    以下是典型浏览器生成的 WebRTC SDP Offer，含音频、视频和 DataChannel：

    会话级头部：版本 0，会话 ID 随机，时间无限制。

    BUNDLE 组（RFC 8843）：mid=0、1、2 的媒体描述共用同一个传输通道。

    允许在同一 RTP 包中混合使用单字节和双字节头部扩展格式（RFC 8285）。

    媒体流语义声明，WMS（WebRTC Media Streams）是 WebRTC 的 msid 语义标识符。

    音频 m= 块：

    音频媒体描述，端口 9 占位。PT 按优先级：111（Opus）> 103（ISAC/16k）> 0（PCMU）> 8（PCMA）> 126（telephone-event）。

    ICE 认证凭证（RFC 8839）+ Trickle ICE 支持声明。

    DTLS 证书指纹（RFC 8122），对端据此验证 DTLS 握手身份。

    DTLS 角色（RFC 5763）：Offer 方声明 actpass，由 Answer 方决定谁主动握手。

    此媒体描述在 BUNDLE 组中的标识符。

    RTP 头部扩展映射：音量检测、绝对发送时间（BWE）、Transport-CC（拥塞控制）、BUNDLE MID。

    双向传输；MediaStream/Track 关联；RTP 与 RTCP 共用端口。

    Opus 编解码器定义：48kHz 双声道，支持 Transport-CC 反馈，最小打包 10ms，启用内置 FEC。

    备选编解码器 G.711（PCMU/PCMA）及 DTMF 电话事件（支持事件 0–95）。

    声明音频 SSRC 及其 CNAME（音视频同步用）和 msid 关联。

    视频 m= 块：

    视频媒体描述，注意 ICE 凭证和 DTLS 指纹与音频相同（BUNDLE 共用传输通道）。

    视频新增：传输时间偏移（唇形同步）、Simulcast RTP-Stream-ID 和 repaired-RTP-Stream-ID（RTX 重传流标识）。

    a=rtcp-rsize（RFC 5506）：支持简化 RTCP 反馈包，减少控制流开销。

    VP8 及其全套 RTCP 反馈机制（REMB 带宽估计、Transport-CC、FIR/NACK/PLI 关键帧控制）。

    PT 97 是 PT 96（VP8）的 RTX 重传流（RFC 4588）。apt=96 表示此 RTX 流为 VP8 服务。

    VP9 和 H.264（Constrained Baseline Level 3.1）及各自的 RTX 重传流。

    FID 组关联主视频流 SSRC（3849985795）与 RTX 重传流 SSRC（1920781173）。

    DataChannel m= 块：

    DataChannel 媒体描述。协议 UDP/DTLS/SCTP；SCTP 端口固定 5000；最大消息 256KB。

    8.2 Answer 对 Offer 的响应规则

    根据 RFC 3264：

    • 媒体描述数量：Answer 中的 m= 块数量不得超过 Offer
    • 媒体描述顺序：Answer 中的 m= 块必须与 Offer 中的顺序对应

    • 编解码器交集：Answer 只能选择 Offer 中提供的编解码器（取子集）
    • 方向协商：若 Offer 为 sendrecv，Answer 可选 sendrecv/sendonly/recvonly/inactive

    9. JSEP 与 SDP 的关系（RFC 8829）

    9.1 Unified Plan vs Plan B

    Plan B（已废弃）：

    • 每种媒体类型只有一个 m= 块
    • 多个同类轨道通过在同一 m= 块中添加多个 a=ssrc 区分
    • Chrome 旧版本使用此方案

    Unified Plan（当前标准，RFC 8829）：

    • 每个媒体轨道（RTCRtpTransceiver）对应一个独立的 m= 块
    • 每个 m= 块通过 a=msid 与 MediaStreamTrack 关联
    • 配合 BUNDLE，所有 m= 块共用同一个传输通道
    • Chrome 72+、Firefox、Safari 均默认使用 Unified Plan

    Unified Plan 使每个轨道的参数可以独立控制，是 RTCRtpTransceiver API 的设计基础。
