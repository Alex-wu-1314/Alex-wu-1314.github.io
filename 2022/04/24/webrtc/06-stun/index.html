    1. 为什么需要 STUN

    互联网中绝大多数终端设备（手机、PC、浏览器）都处于 NAT（Network Address Translation）设备后方，持有的是私有 IP 地址（如 192.168.x.x、10.x.x.x）。当两个位于不同私有网络的端点想建立 P2P 通信时，它们各自无法直接知道对方（乃至自身）在公网上的地址与端口。

    STUN 解决的核心问题：让处于 NAT 后方的端点，通过询问公网上的 STUN 服务器，发现自己被 NAT 分配的公网 IP 地址和端口，即 reflexive transport address（反射传输地址）。

    这一地址随后可以作为 ICE 候选（candidate）提交给对端，尝试建立直连通道。

    2. NAT 基础与分类

    理解 STUN 之前，必须先理解 NAT 的工作方式与分类，因为 NAT 的类型直接决定了 STUN 能否成功辅助打洞。

    2.1 NAT 的基本原理

    NAT 设备维护一张 映射表（binding table），记录内网的 (内部IP, 内部Port) 与公网的 (外部IP, 外部Port) 之间的对应关系。当内网主机发出 UDP 包时，NAT 用公网地址替换源地址，并在映射表中记录这条规则；当公网侧的响应包到达时，NAT 按映射表将目标地址替换回内网地址并转发。

    2.2 NAT 的分类（RFC 3489 经典分类）

    RFC 3489 定义了四种 NAT 类型，按穿透难度从易到难排列：

    Full Cone NAT（全锥型）

    一旦映射建立，任何外部主机都可以通过 203.0.113.1:10000 向内网主机发送数据。这是最容易穿透的类型，STUN 完全有效。

    Restricted Cone NAT（受限锥型）

    映射规则与 Full Cone 相同（同一内部地址始终映射到同一外部地址），但过滤规则限制：只有内网主机曾主动发送过数据包的外部 IP，才能向该映射端口发送数据（端口不限）。

    STUN 有效，但双方需要先互相发一个探测包以”打孔”。

    Port Restricted Cone NAT（端口受限锥型）

    比 Restricted Cone 更严格：不仅 IP 需要匹配，外部来源的端口也必须匹配之前通信过的端口。

    STUN 有效，但对双方同步打洞的时序要求更高。

    Symmetric NAT（对称型）

    这是最难穿透的类型。NAT 会为每一个不同的目标 (IP, Port) 分配不同的外部端口：

    由于 STUN 服务器观察到的地址（10001）与实际和 Peer 通信时 NAT 分配的地址（10002）不同，STUN 返回的 reflexive address 对 P2P 直连无效。此场景必须使用 TURN 中继。

    2.3 RFC 4787 现代分类（Endpoint-Independent vs. Dependent）

    RFC 3489 的四分法在实践中被发现与真实 NAT 行为不完全对应。RFC 4787 引入了更精确的术语：

    维度 描述
    Endpoint-Independent Mapping (EIM) 无论向谁发送，相同的内部地址映射到相同的外部地址。对应 Full/Restricted/Port-Restricted Cone
    Address/Port-Dependent Mapping (APDM) 每个不同的目标都有不同的映射。对应 Symmetric NAT
    Endpoint-Independent Filtering (EIF) 任何外部主机都能发包进来。对应 Full Cone
    Address-Dependent Filtering (ADF) 只有之前通信过的 IP 可以发包进来 。对应 Restricted Cone
    Address and Port-Dependent Filtering (APDF) 只有之前通信过的 IP+Port 可以发包进来。对应 Port-Restricted/Symmetric NAT

    关键结论：STUN 只对 EIM（Endpoint-Independent Mapping）有效。具有 APDM 的 Symmetric NAT 必须回退到 TURN 中继。

    3. STUN 的演进

    RFC 3489 — Classic STUN（2003）

    最初版本将 STUN 定位为完整的 NAT 穿透解决方案。客户端通过一系列 STUN 查询来判断自身的 NAT 类型，并据此决定向对方声明哪个地址。

    然而实践证明这种方法存在根本性缺陷：

    • 现实中 NAT 的行为远比四种类型复杂，大量 NAT 无法被干净地归类；
    • 该方法无法验证学到的地址是否真正对 P2P 通信有效；
    • 存在安全漏洞：攻击者可以向客户端返回错误的 Mapped Address。

    RFC 5389 — Session Traversal Utilities for NAT（2008）

    将 STUN 重新定位为工具性协议（tool），而非完整解决方案。主要变化：

    • 支持 TCP（原来只支持 UDP）；
    • 新增 Magic Cookie 字段（固定值 0x2112A442），用于与其他协议的多路复用解复用，同时保持对旧版的向后兼容；
    • 将 Mapped Address 改为 XOR-MAPPED-ADDRESS 编码（XOR 混淆），防止中间 ALG 设备篡改；
    • 重命名：STUN 全称从 “Simple Traversal of UDP through NAT” 更改为 “Session Traversal Utilities for NAT”；
    • 改进了可扩展性框架（新 Method、新 Attribute、新错误码）。

    RFC 8489 — 现行标准（2020）

    RFC 5389 的更新版本，主要增强了安全性：

    • 认证中引入 SHA-256 替代 MD5（MESSAGE-INTEGRITY-SHA256）；
    • 新增 USERHASH 属性，支持匿名用户名；
    • 增加密码算法协商机制（PASSWORD-ALGORITHMS / PASSWORD-ALGORITHM 属性）；
    • 新增 ALTERNATE-DOMAIN 属性。

    4. STUN 协议概述

    根据 RFC 8489，STUN 是一个客户端-服务器协议，支持两类事务（Transaction）：

    Request/Response 事务

    客户端发送 Request，服务器返回 Success Response 或 Error Response。双方通过 96 位的 Transaction ID 关联请求与响应。这是最常见的模式，Binding Request/Response 就属于此类。

    Indication 事务

    任意一方（客户端或服务器）可以发送 Indication，不产生响应。Indication 不可靠，也不可认证。在 TURN 中，Send Indication 和 Data Indication 就使用此机制。

    STUN Usage

    STUN 规范本身只定义了协议框架；具体使用方式由上层的 STUN Usage 定义。Usage 规定了：

    • 何时发送 STUN 消息；
    • 使用哪些可选属性；
    • 使用什么 STUN 服务器；
    • 采用何种认证机制。

    常见的 STUN Usage 包括：

    Usage RFC 说明
    ICE RFC 8445 WebRTC 连接建立的核心，使用 STUN 做候选收集和连接检查
    SIP Outbound RFC 5626 SIP 信令的 NAT 穿透
    NAT Behavior Discovery RFC 5780 探测 NAT 映射和过滤行为
    TURN RFC 8656 以 STUN 为基础的中继扩展

    5. 报文结构详解

    5.1 STUN 消息头（20 字节固定头）

    所有 STUN 消息都以一个 20 字节的固定头部开始，格式如下（RFC 8489 §5）：

    各字段说明：

    最高 2 位（固定为 00）

    所有 STUN 消息的最高 2 位必须为零。这使得 STUN 可以与其他协议复用同一个端口，通过判断首字节高 2 位来区分协议类型（例如 RTP/RTCP 首字节高 2 位为 10）。

    STUN Message Type（14 位）

    消息类型由 Method（12 位） 和 Class（2 位） 共同编码，两部分交织在这 14 位中：

    Class 取值（2 位）：

    Class 值 含义
    0b00 Request
    0b01 Indication
    0b10 Success Response
    0b11 Error Response

    目前 RFC 8489 只定义了一种 Method：Binding（值为 0x001）。

    典型消息类型编码举例：

    消息 编码
    Binding Request 0x0001
    Binding Success Response 0x0101
    Binding Error Response 0x0111
    Binding Indication 0x0011

    Message Length（16 位）

    消息体（attributes 部分）的字节长度，不含 20 字节头部，且必须为 4 的倍数。

    Magic Cookie（32 位，固定值）

    固定值 0x2112A442（网络字节序）。这是 RFC 5389 引入的关键字段，有两个作用：

    1. 区分新版 STUN 与 RFC 3489 旧版 STUN；
    2. 辅助在同一端口上多路复用时识别 STUN 报文；
    3. 作为 XOR-MAPPED-ADDRESS 计算中的 XOR 掩码。

    Transaction ID（96 位）

    由发起方随机生成，用于将 Request 与 Response 配对。服务器在响应中原样回显此字段。

    5.2 STUN 属性（TLV 格式）

    头部之后跟随零个或多个属性，每个属性采用 TLV（Type-Length-Value） 编码：

    • Type：16 位，属性类型；
    • Length：16 位，Value 字段的字节数；
    • Value：变长，必须以 4 字节对齐（不足补零填充，填充字节不计入 Length）。

    属性类型根据最高位的取值分为两类：

    类型范围 语义
    0x0000 – 0x7FFF Comprehension-Required：若接收方不理解此属性，必须返回错误
    0x8000 – 0xFFFF Comprehension-Optional：若不理解，可安全忽略

    RFC 8489 定义的核心属性：

    属性 类型值 说明
    MAPPED-ADDRESS 0x0001 旧版反射地址（兼容 RFC 3489）
    USERNAME 0x0006 认证用用户名
    MESSAGE-INTEGRITY 0x0008 HMAC-SHA1 完整性校验
    ERROR-CODE 0x0009 错误码及原因短语
    UNKNOWN-ATTRIBUTES 0x000A 未知属性列表
    REALM 0x0014 长期凭证的 realm
    NONCE 0x0015 防重放攻击的 nonce
    XOR-MAPPED-ADDRESS 0x0020 XOR 编码的反射地址（推荐使用）
    SOFTWARE 0x8022 软件版本信息（可选）
    ALTERNATE-SERVER 0x8023 备用服务器地址
    FINGERPRINT 0x8028 CRC-32 校验，辅助多路复用识别
    MESSAGE-INTEGRITY-SHA256 0x001C HMAC-SHA256（RFC 8489 新增）
    PASSWORD-ALGORITHM 0x001D 密码算法指定（RFC 8489 新增）
    USERHASH 0x001E 哈希化用户名（RFC 8489 新增）
    PASSWORD-ALGORITHMS 0x8002 服务器支持的密码算法列表（RFC 8489 新增）

    5.3 XOR-MAPPED-ADDRESS 的编码方式

    这是 RFC 5389 引入的重要改进。反射地址并非直接存储，而是与 Magic Cookie 进行 XOR 运算后存储：

    引入 XOR 编码的原因：部分网络中间设备（ALG，Application Layer Gateway）会扫描 UDP 负载中的 IP 地址并进行修改，从而破坏 MAPPED-ADDRESS 中的地址信息。XOR 混淆使地址不直接暴露，规避了这类设备的干扰。

    6. Binding 方法：获取公网地址

    Binding 是 STUN 目前定义的唯一 Method，也是 WebRTC 中最核心的 STUN 操作。

    6.1 工作流程

    步骤说明：

    1. 客户端（192.168.1.10:5000）向公网 STUN 服务器发送 Binding Request（通常不含任何属性，无需认证）；
    2. 报文经过 NAT 时，NAT 将源地址替换为公网地址 203.0.113.1:10000，并在映射表中建立记录；
    3. STUN 服务器收到请求后，观察到源地址为 203.0.113.1:10000，将其封装在 XOR-MAPPED-ADDRESS 属性中，发回 Binding Success Response；
    4. 响应经 NAT 转发回客户端；
    5. 客户端解析响应，获得自己的 reflexive transport address：203.0.113.1:10000。

    注意：如果客户端经过多层 NAT，STUN 返回的是最靠近服务器一侧的 NAT 分配的地址（即最外层 NAT 的映射），而非内层 NAT 的地址。

    6.2 Binding 用于 Keep-Alive

    NAT 映射有生命周期（通常 UDP 绑定为 30 秒至数分钟不等）。若长时间没有数据包经过，NAT 可能删除映射，导致通道中断。

    STUN Binding Request/Indication 可被用作 keep-alive 心跳包，定期发送以维持 NAT 映射的活跃状态。在 WebRTC 的 ICE 实现中，ICE keep-alive 消息本质上就是周期性的 STUN Binding Indication。

    6.3 Binding 用于连接检查（ICE 专用）

    在 ICE 连接检查阶段，STUN Binding Request 还承担连接可达性验证的职责：

    • 消息中携带 PRIORITY、USE-CANDIDATE、ICE-CONTROLLED/ICE-CONTROLLING 等 ICE 专用属性（定义于 RFC 8445）；
    • 使用短期凭证（Short-Term Credential） 进行认证（凭证来自 SDP offer/answer 中的 ice-ufrag/ice-pwd）；
    • 双向的 Binding Request/Response 交换证明两端可以互相通信。

    7. 认证机制

    RFC 8489 定义了两种认证机制：

    7.1 短期凭证机制（Short-Term Credential Mechanism）

    用途：ICE 连接检查。

    凭证在会话期间生成，随 SDP offer/answer 通过信令通道交换（ice-ufrag 和 ice-pwd）。

    • HMAC Key：密钥直接为 ice-pwd 的 UTF-8 编码字节；
    • 完整性保护：消息中附带 MESSAGE-INTEGRITY（HMAC-SHA1）或 MESSAGE-INTEGRITY-SHA256（HMAC-SHA256）属性；
    • 凭证随 ICE 会话刷新而更新（ICE restart 时）。

    7.2 长期凭证机制（Long-Term Credential Mechanism）

    用途：TURN 服务器认证。

    采用挑战-响应模式，防止重放攻击：

    1. 客户端首次发送不带认证的请求；
    2. 服务器返回 401 Unauthorized，携带 REALM 和 NONCE；
    3. 客户端重发请求，携带 USERNAME（或 USERHASH）、REALM、NONCE 和 MESSAGE-INTEGRITY；
    4. HMAC Key = MD5("<username>:<realm>:<password>") （RFC 5389 及之前）或 SHA-256 派生（RFC 8489）。

    RFC 8489 的改进：NONCE 中嵌入 nonce cookie（固定前缀 obMatJos2），通过其后的比特位表达服务器支持的安全特性（如密码算法），以抵御降级攻击（Bid-Down Attack）。

    8. FINGERPRINT 机制

    为了在同一端口上同时运行 STUN 和其他协议（如 RTP），RFC 5389 引入了 FINGERPRINT 属性：

    • 计算方法：对 STUN 消息（含 FINGERPRINT 属性头，但 Value 填零）做 CRC-32，然后与 0x5354554E 进行 XOR；
    • 将结果填入 FINGERPRINT 的 Value，附加在消息末尾；
    • 接收方验证此值，通过则可高置信度确认这是 STUN 报文。

    与首字节高 2 位为零的机制相比，FINGERPRINT 提供更高的识别可靠性，但计算略有开销。WebRTC 实现通常同时使用两种机制。

    9. 传输协议支持

    传输层协议 支持情况 说明
    UDP 必须支持 默认协议，最常用；RFC 8489 规定 STUN 服务器默认端口为 3478
    TCP 必须支持 用于无法使用 UDP 的场景
    TLS over TCP 可选 提供加密通道，端口 5349
    DTLS over UDP 可选 UDP 上的 TLS，端口 5349

    UDP 重传机制

    由于 UDP 不提供可靠传输，STUN 自身实现了应用层重传：

    • 初始重传超时（RTO）：RFC 建议初始值为 500ms；
    • 采用指数退避（Binary Exponential Backoff）：RTO × 1、× 2、× 4、× 8……；
    • 共尝试 Rc 次（RFC 建议默认 Rc = 7），总超时约 39.5 秒；
    • 若在此期间收到响应，则取消后续重传。

    10. STUN 在 WebRTC/ICE 中的角色

    在 WebRTC 的 ICE 框架（RFC 8445）中，STUN 承担以下具体职责：

    10.1 候选地址收集（Candidate Gathering）

    ICE Agent 在本地收集三类候选：

    候选类型 地址来源 STUN 角色
    host candidate 本机网卡 IP 无需 STUN
    server reflexive candidate（srflx） STUN 服务器返回的 reflexive address 核心用途
    relayed candidate（relay） TURN 服务器分配的中继地址 TURN 基于 STUN 扩展实现

    收集 srflx 候选的过程就是标准的 Binding Request/Response 交互。

    10.2 连接检查（Connectivity Checks）

    ICE 将两端收集的候选进行配对（candidate pairs），然后逐对执行 STUN Binding Request 检查：

    • 请求中携带 PRIORITY（候选优先级）、ICE-CONTROLLING/ICE-CONTROLLED（角色）等属性；
    • 使用 USERNAME（<remote-ufrag>:<local-ufrag>）和短期凭证（MESSAGE-INTEGRITY）进行认证；
    • 双向检查通过则该 candidate pair 标记为 Succeeded；
    • Controlling Agent 选定最佳 pair 后，在请求中附加 USE-CANDIDATE 属性提名。

    10.3 ICE Keep-Alive

    连接建立后，ICE 周期性发送 STUN Binding Indication（或无需响应的 Binding Request）以维持 NAT 绑定。WebRTC 默认 keep-alive 间隔为 15 秒（RFC 8445 §11）。

    10.4 ICE Restart

    当网络切换（如 Wi-Fi 切换到 4G）时，ICE 可以通过更新 ice-ufrag/ice-pwd 触发 ICE restart，重新收集候选并重新执行连接检查，而无需重建 DTLS/SRTP 通道。

    11. STUN 的局限性

    STUN 不是万能的 NAT 穿透方案，存在以下明显局限：

    11.1 无法穿透 Symmetric NAT

    如前所述，当一方或双方位于 Symmetric NAT 后时，STUN 观察到的反射地址对实际 P2P 通信无效。此时必须使用 TURN 中继。

    在实际网络中，企业防火墙、移动运营商 CGNAT（Carrier-Grade NAT）普遍采用 Symmetric NAT 或等效的严格过滤策略。这也是为什么生产级 WebRTC 应用必须部署 TURN 服务器。

    11.2 反射地址的时效性

    NAT 映射会在空闲一段时间后过期，反射地址随之失效。如果 ICE 候选在信令中传递的时间过长（如用户在 SDP offer/answer 之间停留），候选地址可能已失效，需要触发 ICE restart。

    11.3 多层 NAT 下的精度问题

    在多层 NAT 环境下，STUN 只能暴露最外层 NAT 的映射地址，内层 NAT 的信息不可见。

    11.4 无法独立解决问题

    STUN 必须与 ICE 框架配合使用，单独使用 STUN（Classic STUN）已被证明不可靠，RFC 8489 明确不鼓励这种用法。

    12. 安全考量

    12.1 外部攻击

    DoS 放大攻击：攻击者以受害者 IP 作为源地址向 STUN 服务器发送大量请求，利用 STUN 响应对受害者发动 DoS 放大攻击。

    缓解措施：现代 STUN 请求与响应大小相近，放大倍数很低；部署速率限制。

    反射地址伪造：攻击者可以向客户端发送伪造的 Binding Response，其中包含错误的 Mapped Address。

    缓解措施：STUN 与 ICE 联合使用时，连接检查会验证地址的真实可用性，单纯的地址伪造无法造成成功的连接劫持。

    12.2 内部攻击

    若攻击者能观察/篡改 STUN 流量（中间人），且没有 MESSAGE-INTEGRITY 保护，则可以篡改反射地址。

    缓解措施：对需要安全性的场景（如 TURN 认证），必须使用长期凭证 + MESSAGE-INTEGRITY。ICE 使用短期凭证保护连接检查。

    12.3 密码降级攻击（Bid-Down Attack，RFC 8489 新增防护）

    攻击者可能将认证请求中的密码算法协商降级到较弱的 MD5。RFC 8489 通过在 NONCE 字段中嵌入安全特性位，让客户端能够检测到此类降级。

    12.4 隐私考量

    STUN 服务器会记录客户端的 IP 地址，公共 STUN 服务器可能收集用户位置信息。RFC 8489 引入的 USERHASH 属性允许客户端使用哈希化用户名，提升匿名性。

    14. 小结

    方面 说明
    协议性质 轻量级客户端-服务器协议，工具性，不独立成为完整解决方案
    核心功能 发现 reflexive transport address（公网 IP:Port）
    报文结构 20 字节固定头 + TLV 属性列表；Magic Cookie 0x2112A442
    核心 Method Binding Request/Response；Binding Indication（keep-alive）
    认证 短期凭证（ICE 使用）；长期凭证（TURN 使用）
    传输层 UDP（主）/ TCP / TLS-over-TCP / DTLS-over-UDP
    NAT 穿透能力 仅对 EIM（非 Symmetric）NAT 有效
    现行规范 RFC 8489（2020），废止 RFC 5389（2008）

    与其他章节的关联：

    • 与ICE：ICE 是 STUN 的上层 Usage，使用 STUN Binding 做候选收集和连接检查，是 STUN 在 WebRTC 中的主要使用场景；
    • 与TURN：当 STUN 无法穿透 Symmetric NAT 时，TURN 作为 STUN 的扩展（Extension）提供中继方案，TURN 报文格式和认证机制直接基于 STUN；
    • 与DTLS：ICE 连接建立后，DTLS 握手才开始，STUN/ICE 先行完成地址协商；
    • 与WebSocket：STUN 候选通过信令通道（WebSocket）以 SDP 或 Trickle ICE 形式交换给对端。
