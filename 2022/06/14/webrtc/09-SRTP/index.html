    1. 为什么需要 SRTP

    RTP（Real-time Transport Protocol）本身是明文协议。一个经典的 RTP 数据包在网络上传输时，任何能抓到数据包的中间人都可以：

    • 窃听（Eavesdropping）：还原出音视频流的原始内容；
    • 篡改（Tampering）：修改音视频数据，注入噪声或伪造内容；
    • 重放攻击（Replay Attack）：将合法的历史数据包重新注入网络。

    RTP 设计之初（RFC 3550，2003年）是面向受信任网络的，安全性留给上层应用解决。随着 VoIP、视频会议在公共互联网上的大规模部署，明文传输的风险变得不可忽视。

    SRTP（Secure Real-time Transport Protocol）就是为此而生的。它在 不改变 RTP 头部结构、不引入显著延迟和带宽开销 的前提下，为 RTP 数据流提供：

    安全特性 说明
    机密性（Confidentiality） 对 RTP/RTCP 载荷加密，防止窃听
    消息认证与完整性（Authentication & Integrity） 检测篡改，确保包来自合法发送方
    防重放（Replay Protection） 拒绝接受已处理过的历史数据包

    SRTP 由 Cisco 与 Ericsson Research 联合设计，2004 年 3 月发布为 RFC 3711，是 IETF AVT 工作组的标准轨道（Standards Track）文档。WebRTC 规范（RFC 8827）强制要求所有媒体流均使用 SRTP，且必须通过 DTLS-SRTP 完成密钥协商，这使 SRTP 成为现代实时音视频安全的基石。

    2. 协议概览与核心 RFC

    SRTP 在协议栈中的位置可以理解为 RTP 的一个 “stack bump（堆栈中间层）“：从应用层看，仍然是发送/接收 RTP 包；SRTP 层在发送时对载荷加密并附加认证标签，在接收时验证并解密后还原为普通 RTP 包交给上层处理。

    SRTP 本身 不处理密钥交换，密钥由外部密钥管理协议提供，在 WebRTC 场景中即 DTLS-SRTP（RFC 5764）。

    核心 RFC 体系

    RFC 发布年份 内容摘要
    RFC 3711 2004 SRTP 核心规范，定义基础框架、AES-CM 加密、HMAC-SHA1 认证、密钥派生
    RFC 5764 2010 DTLS-SRTP：通过 DTLS 握手为 SRTP 建立密钥，WebRTC 强制使用
    RFC 6904 2013 扩展：对 RTP 头部扩展字段加密（此前只加密载荷）
    RFC 7714 2015 引入 AES-GCM AEAD 算法，提供更高效的认证加密
    RFC 8723 2020 Double Encryption：为 SFU 场景提供端到端 + 跳到跳的双层加密
    RFC 9335 2023 完全加密 RTP 头部扩展与 CSRC

    3. SRTP 包格式

    SRTP 包格式与 RTP 高度兼容，头部字段完全保留，加密仅作用于载荷（Payload），并在包尾附加可选的 MKI 和认证标签。RFC 3711 Section 3.1 给出了完整的包结构图：

    关键字段说明：

    • Encrypted Portion（加密部分）：仅包含 RTP 载荷（及 RTP padding）。RTP 头部本身不被加密（默认情况下），这使得中间节点可以在不解密的情况下读取序列号、SSRC 等路由信息。
    • Authenticated Portion（认证部分）：涵盖整个 RTP 头部 + 加密后的载荷，认证标签保护的范围比加密更广，这意味着头部篡改也会导致认证失败。
    • MKI（Master Key Indicator，可选）：长度可配置，用于在有多个 Master Key 的场景中标识当前使用的密钥，便于密钥轮换（re-keying）。MKI 本身不受完整性保护。
    • Authentication Tag（认证标签，强烈推荐）：默认 80 bit（HMAC-SHA1 截断），也支持 32 bit（仅限某些场景）。

    注意：加密在先，认证在后（Encrypt-then-MAC）。发送方先对载荷加密，再对”头部 + 加密后载荷”计算认证标签；接收方先验证标签，通过后再解密。这一顺序对安全性至关重要。

    参数 说明

    s_l（16 bit） 接收方维护，记录当前已收到的最高序列号

    消息认证算法标识 如 HMAC-SHA1

    5. 密钥体系：Master Key 与 Session Key

    SRTP 使用两层密钥结构，这是其设计亮点之一：

    设计目的：

    1. 最小化密钥暴露量：单个 Master Key 派生出多个用途不同的 Session Key，每个密钥只用于特定操作，减少攻击面。
    2. SRTP 与 SRTCP 使用不同 Session Key：即使 Master Key 相同，两者的加密/认证密钥完全隔离。
    3. 周期性密钥刷新：通过 key_derivation_rate 配置，SRTP 可以定期重新派生 Session Key，限制攻击者可收集的同一密钥下的密文量。

    6. 加密算法

    RFC 3711 预定义了三种加密算法，并允许通过新的配套 RFC 扩展更多算法。

    6.1 AES Counter Mode（AES-CM）

    默认且强制实现的加密算法，将 AES 块密码用作流密码（Stream Cipher）。

    优点：

    • 流密码模式，加密载荷任意长度无需 padding；
    • 支持随机访问和并行加密；
    • 抗丢包：即使前一包丢失，后续包仍可独立解密；
    • AES 硬件加速（AES-NI）成熟，性能极高。

    注意：AES-CM 仅提供机密性，不提供完整性保护。完整性需依赖 HMAC-SHA1 认证标签。

    6.2 AES f8 Mode

    专为移动通信场景（3GPP）设计的变体，基于 AES 的 f8 运行模式，主要应用于特定电信标准，WebRTC 中较少使用。

    其 IV 构造更为复杂，将 RTP 头部字段（包括 PT、序列号、时间戳、SSRC）编码进 IV，提供隐式头部绑定，无需单独的认证即可部分防止头部替换。

    6.3 AES-GCM（RFC 7714）

    2015 年 RFC 7714 引入了 AES-GCM（Galois/Counter Mode），这是一种 AEAD（Authenticated Encryption with Associated Data） 算法，将加密和认证合并为单一操作。

    关键特性：

    • 同时提供机密性 + 完整性，不需要单独的 HMAC；
    • 认证标签长度为 128 bit（比 HMAC-SHA1 的 80 bit 更长，更安全）；
    • GCM 的认证操作（GHASH）天然适合硬件并行实现，性能优越；
    • 不允许截断认证标签（RFC 7714 明确禁止），避免弱认证风险。

    套件标识 说明
    SRTP_AEAD_AES_128_GCM AES-128-GCM，密钥 16B，盐 12B，认证标签 16B
    SRTP_AEAD_AES_256_GCM AES-256-GCM，密钥 32B，盐 12B，认证标签 16B

    由于 AES-GCM 将认证嵌入密文中，SRTP 包结构发生变化：认证标签不再单独附加，而是包含在”加密部分”末尾。

    7. 消息认证：HMAC-SHA1

    针对 AES-CM（非 AES-GCM）加密，SRTP 使用 HMAC-SHA1（RFC 2104）提供消息认证。

    RFC 3711 Section 4.2.1 规定：

    1. HMAC 的计算范围 M = RTP 头部 || 加密后的载荷（即 Authenticated Portion）；
    2. 使用会话认证密钥 k_a 计算：HMAC-SHA1(k_a, M)；
    3. HMAC-SHA1 输出 160 bit，截断为 80 bit（默认，即 n_tag = 80）附加到包尾；
    4. 也支持 32 bit 截断（SRTP_AES128_CM_HMAC_SHA1_32），但安全性较低，WebRTC 中不推荐用于视频流。

    重要说明：SRTP 的消息认证在严格意义上是 完整性保护（Integrity Protection），而非数据源认证（Data Origin Authentication）。在组播场景中，多个参与者共享同一密钥，收到认证通过的包只能证明包来自”知道密钥的某方”，不能确认具体发送者。

    8. 防重放保护

    重放攻击（Replay Attack）是将合法的历史数据包重新注入网络，可导致语音/视频抖动、回声，甚至欺骗应用逻辑。

    SRTP 通过滑动窗口（Sliding Window）+ 重放列表（Replay List） 防御重放攻击：

    处理逻辑（RFC 3711 Section 3.3）：

    1. 根据当前 ROC 和接收到的 SEQ 重建 48 bit 包索引 i；
    2. 若 i 早于窗口左边界 → 直接丢弃（过旧）；
    3. 若 i 在窗口内且对应位已置 1 → 丢弃（已接收过）；
    4. 通过认证标签验证（必须通过认证才能更新状态）；
    5. 认证通过后，更新重放列表位图，必要时滑动窗口。

    关键原则：只有通过认证的包才会更新重放列表，防止攻击者通过注入伪造包污染重放状态。

    9. SRTCP

    RTCP 承载统计信息、成员控制、反馈信令等，若被篡改可导致码率控制被破坏、成员信息被伪造，危害不亚于媒体数据的泄露。因此 RFC 3711 同步定义了 SRTCP（Secure RTCP）。

    SRTCP 包格式

    SRTCP 在 RTCP 包尾追加三个强制字段和一个可选字段：

    • SRTCP Index（31 bit）：显式携带序列号（不像 SRTP 那样需要从 ROC+SEQ 推算），接收方直接读取，无需估算。
    • E（Encrypt Flag，1 bit）：标识 RTCP 载荷是否被加密，SRTCP 支持选择性不加密（使用 NULL cipher）。
    • Authentication Tag：SRTCP 的消息认证是 强制的（MANDATORY），即使不加密也必须认证。

    SRTCP 的重要差异

    对比项 SRTP SRTCP
    序列号 隐式（从 RTP SEQ + ROC 推算） 显式（SRTCP Index 字段）
    认证 强烈推荐（RECOMMENDED） 强制（MANDATORY）
    加密 默认开启 可选（E flag 控制）
    密钥 独立 Session Key 集合 独立 Session Key 集合（可共享 Master Key）

    10. 密钥协商：DTLS-SRTP（RFC 5764）

    SRTP 本身不包含密钥交换机制，这是其设计哲学之一：将安全协议与密钥管理解耦。在 WebRTC 中，密钥交换强制使用 DTLS-SRTP（RFC 5764）。

    DTLS-SRTP 工作原理

    DTLS-SRTP 的关键机制：

    1. use_srtp DTLS 扩展：在 ClientHello/ServerHello 中协商 SRTP 密码套件（Protection Profile），如 SRTP_AES128_CM_HMAC_SHA1_80；
    2. Keying Material 导出：DTLS 握手完成后，双方通过 TLS PRF（伪随机函数）导出密钥材料，分别生成发送方和接收方的 Master Key 及 Master Salt；
    3. 指纹验证（Fingerprint）：SDP 中携带对端证书的指纹（如 a=fingerprint:sha-256 <hash>），接收方在 DTLS 握手时验证对端证书是否与 SDP 中的指纹匹配，防止中间人攻击。

    包多路复用（Demultiplexing）

    DTLS 握手包与 SRTP 媒体包复用同一 UDP 端口，依靠首字节区分协议类型（RFC 5764 Section 5.1.2）：

    首字节范围 协议
    [0, 3] STUN
    [16, 19] ZRTP
    [20, 63] DTLS
    [64, 79] TURN
    [128, 191] RTP/RTCP

    11. WebRTC 中的 SRTP 实践

    强制要求

    根据 RFC 8827（WebRTC Security Architecture）：

    • WebRTC 实现必须支持 SRTP，且必须使用 DTLS-SRTP 进行密钥协商；
    • 禁止以不加密的 RTP 传输真实媒体（调试模式需显式配置）；
    • 必须支持 SRTP_AES128_CM_HMAC_SHA1_80 密码套件。

    WebRTC Native（libwebrtc）中的实现

    WebRTC 官方实现（Chromium / libwebrtc）的 SRTP 层：

    • 底层加密库：libsrtp（Cisco 开源，https://github.com/cisco/libsrtp）；
    • 核心类：webrtc::SrtpSession，封装 libsrtp 的加解密接口；
    • 上层类：webrtc::DtlsSrtpTransport，在 DTLS 握手完成时提取密钥材料，调用 SrtpSession::SetSend / SetRecv 配置密钥；
    • 加解密均为原地操作（in-place），避免额外内存拷贝：ProtectRtp / UnprotectRtp。

    支持的密码套件

    套件名 密钥长度 认证 默认启用
    SRTP_AES128_CM_HMAC_SHA1_80 AES-128 HMAC-SHA1（80 bit tag） 是
    SRTP_AEAD_AES_128_GCM AES-128 GCM（128 bit tag） 是
    SRTP_AEAD_AES_256_GCM AES-256 GCM（128 bit tag） 是
    SRTP_AES128_CM_HMAC_SHA1_32 AES-128 HMAC-SHA1（32 bit tag） 否

    两个 WebRTC 端点互通时默认协商 SRTP_AES128_CM_HMAC_SHA1_80；GCM 套件可被非 WebRTC 对端优先选择。

    SDP 中的 SRTP 相关属性

    • UDP/TLS/RTP/SAVPF：表示使用 DTLS-SRTP（TLS over UDP）；
    • a=fingerprint：对端 DTLS 证书的指纹，用于防中间人；
    • a=rtcp-mux：RTCP 与 RTP 复用同一端口，SRTCP 也通过同一 DTLS 关联保护；
    • 无需 a=crypto（SDES 方式，已不推荐）。

    13. 安全性分析与注意事项

    信令层是最大攻击面

    DTLS-SRTP 的媒体加密本身很难被直接攻破，现实中的攻击更多针对信令层：

    • 若 SDP 通过明文 WebSocket（ws://）传输，攻击者可替换 SDP 中的 DTLS 指纹，实施中间人攻击（MITM）；
    • 防御：信令必须使用 wss://（WebSocket over TLS）或等效加密通道，并严格校验 DTLS 指纹与 SDP 中的一致性。

    SSRC 碰撞（Two-Time Pad 风险）

    AES-CM 是流密码，若两个不同的 RTP 包使用相同的 IV（即相同的 SSRC + 相同的包索引），则会产生”密钥流重用”（Two-Time Pad）：

    缓解措施（RFC 3711 Section 9.1）：

    • 同一会话中不同 SSRC 的流必须使用不同的 Session Key，或确保 SSRC 不碰撞；
    • WebRTC 中，不同 MediaStream 使用不同的加密上下文，规避此风险。

    Master Key 使用量限制

    每个 Master Key 保护的包数量存在上限（与 AES-CM 的密钥流长度相关），超出限制后必须触发 re-keying，否则安全性下降。RFC 3711 建议实现方监控 Master Key 的包计数器。

    Header Extension 加密（RFC 6904）

    标准 SRTP 默认不加密 RTP 头部扩展（如 audio level、video orientation），这些字段在明文状态下可被 SFU 读取用于路由，但也可能泄露元数据。RFC 6904 扩展了 SRTP 以支持对特定头部扩展的选择性加密。

    SFU 场景下的端到端加密

    在 SFU（Selective Forwarding Unit）架构中，SRTP 只保护客户端到 SFU 这一跳，SFU 需要解密后才能转发。若需要真正的端到端加密，可使用 RFC 8723 Double Encryption：

    SFU 只能处理外层（hop-by-hop）加密，无法访问内层（end-to-end）媒体内容。

    15. 小结

    SRTP 是 WebRTC 媒体安全的核心执行层，其设计理念值得细细品味：

    • 最小开销原则：仅加密载荷，保留头部路由信息；认证标签尽量短（80 bit）；流密码避免 padding；整体 per-packet 开销仅增加约 14 字节。
    • 解耦设计：将加密框架与密钥管理彻底分离，SRTP 只管”用密钥做什么”，密钥来源（DTLS-SRTP / SDES / MIKEY）可替换。
    • 可扩展性：通过配套 RFC 机制（Section 6）不断引入新算法（AES-GCM、双层加密等），保持协议的长期生命力。

    在 WebRTC 的完整安全链路中，DTLS 负责握手与密钥协商，SRTP 负责每一个媒体包的实际加解密。理解 SRTP 的包格式、密钥派生、IV 构造以及与 DTLS 的配合关系，是排查 WebRTC 媒体加密问题、优化加密性能的必要基础。
