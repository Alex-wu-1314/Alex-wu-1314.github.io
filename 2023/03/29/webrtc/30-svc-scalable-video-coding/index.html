    1. 什么是 SVC

    Scalable Video Coding（SVC，可伸缩视频编码） 是一种将同一视频源编码为分层码流（Layered Bitstream）的技术。编码器产出的码流由一个基础层（Base Layer） 和若干增强层（Enhancement Layer） 叠加而成；解码器可以只解码基础层，也可以选择性地叠加增强层，从而在分辨率、帧率、画质等维度上灵活适配不同接收端的带宽与算力。

    “SVC encodes a video source in a single stream, with multiple layers that can be selectively decoded to obtain video with particular resolutions, bitrate, or quality.”
    — MDN Web Docs, Introduction to WebRTC protocols

    与传统的单一码率编码不同，SVC 的本质在于层间复用：高层增强包的编码依赖低层信息，因此编码一次即可服务多种质量需求，而无需像 Simulcast 那样对同一画面进行多次独立编码。

    在 WebRTC 场景中，SVC 与 SFU（Selective Forwarding Unit）协同工作的价值尤为显著：SFU 无需转码，只需按照每个接收端的网络条件，选择性地转发对应的层包，便可实现细粒度的自适应传输。

    2. 历史背景与标准化进程

    时间 事件
    2005 年 H.264/AVC 的 Annex G 正式纳入 SVC 扩展，由 ITU-T 与 ISO/IEC MPEG 联合标准化，编号为 ITU-T H.264 Amendment 3 / ISO/IEC 14496-10:2005/Amd.3
    2011 年 IETF 发布 RFC 6190，定义 H.264 SVC 的 RTP Payload Format，支持单/多 RTP 会话传输
    2013 年 Vidyo 与 Google 宣布合作，为 VP9 引入可伸缩扩展
    2019 年 W3C Web Real-Time Communications 工作组发布 Scalable Video Coding (SVC) Extension for WebRTC 首个公开工作草案
    2024 年 8 月 W3C 发布最新工作草案版本（Working Draft），持续迭代

    SVC 并非一种独立的编码格式，而是一种编码特性，可嵌入到 VP9、AV1、H.264、H.265 等主流视频编解码器中。

    3. SVC 的三种可伸缩性维度

    3.1 时间可伸缩性（Temporal Scalability）

    时间可伸缩性通过控制帧率（Frame Rate） 实现伸缩，是 WebRTC 中最早且最广泛被支持的 SVC 形态。

    编码器将帧序列划分为多个时间层（Temporal Layer）：

    • T0（基础时间层）：仅包含关键帧与参考帧，可独立解码，帧率最低（如 7.5 fps）
    • T1：依赖 T0，叠加后帧率翻倍（如 15 fps）
    • T2：依赖 T0 和 T1，叠加后再次翻倍（如 30 fps）

    SFU 可以在不触碰数据包内容的情况下，通过丢弃高层时间层的包来降低帧率，将 30 fps 流降为 15 fps 或 7.5 fps 转发给带宽受限的接收端。

    “The real advantage of temporal scalability is the flexibility it provides. For instance, if we want to reduce the frame rate from 30 to 15 frames per second, we can simply drop all L1 frames, maintaining the dependency chain between the L0 frames. This can be done by the SFU.”
    — BlogGeek.me, Temporal Scalability

    重要约束：在时间层切换时，SFU 需要等待带有切换点标记（Switching Point / U-bit） 的帧，才能安全地切入高层，避免解码器引用丢失的参考帧而花屏。

    3.2 空间可伸缩性（Spatial Scalability）

    空间可伸缩性通过不同的分辨率（Resolution） 实现伸缩。高层空间层的编码依赖低层空间层的重建图像（上采样作为预测参考）：

    层间的分辨率比例通常为 2:1（即标准 SVC）或 1.5:1（即带 h 后缀的模式，如 L2T2h）。

    3.3 质量可伸缩性（Quality / SNR Scalability）

    质量可伸缩性在相同分辨率和帧率下，通过增加增强层来提升画面的 SNR（信噪比）和视觉质量，本质上是对残差（Residual）信息的分层精细化编码。

    这种伸缩性在 WebRTC 实践中的使用相对较少，多见于广播和流媒体存储场景。

    4. 层依赖结构与关键帧

    SVC 的核心难点之一是层间依赖关系（Inter-Layer Dependency） 的管理。根据依赖强度的不同，业界形成了两种主要模式：

    4.1 Full SVC

    在完整 SVC（Full SVC / Continuous SVC）模式下，高层空间层的每一帧都参考了低层空间层的对应帧。这意味着：

    • 编码效率最高：最大化利用层间预测，整体码率最省
    • SFU 层切换困难：只有在关键帧处才能安全切换到更低的空间层，且 SFU 需要深入理解码流结构

    对应的 scalabilityMode 命名格式为 LxTy（如 L3T3）。

    4.2 K-SVC（Key-frame SVC）

    K-SVC（Keyframe SVC）是一种折中方案，高层空间层仅在关键帧（Key Frame）处依赖低层，其余 P 帧在空间维度上独立编码：

    优势：

    • SFU 可在任意 P 帧处切换空间层，无需等待下一个关键帧
    • 带宽利用更平滑：避免关键帧后的码率突刺

    劣势：

    • 相较 Full SVC，非关键帧压缩效率稍低

    对应 scalabilityMode 命名格式为 LxTy_KEY（如 L3T3_KEY）。

    K-SVC 带时间位移（KEY_SHIFT）：进一步优化时间层的预测结构，降低码率突刺，通过将后续帧的时间标识符上移（Shift up）来实现更平滑的带宽使用曲线。格式为 LxTy_KEY_SHIFT。

    5. SVC 与 Simulcast 的对比

    理解 SVC 需要将其与 Simulcast 进行对比，因为二者解决的是同一个问题，如何让 SFU 服务不同能力的接收端，但方法论截然不同。

    维度 SVC Simulcast
    编码方式 单次编码，产出分层码流 多次独立编码，产出多路完整流
    上行带宽 较低（层间共享预测信息） 较高（多路冗余）
    上行 CPU 较高（层间依赖计算复杂） 较低（各路独立编码，可复用硬件编码器）
    SFU 复杂度 高（需理解层依赖，精确控制转发切换点） 低（直接选流转发，无需解析码流内部）
    切换粒度 细（时间层可逐帧切换，空间层视模式而定） 粗（整流切换，存在切换延迟）
    硬件加速 支持较差（SVC 层间依赖难以硬件化） 支持好（各路流可独立走硬件编码器）
    编解码器要求 VP9、AV1 原生支持；H.264/SVC 通过 Annex G VP8、H.264、VP9 均支持
    行业成熟度 仍在发展中（尤其全空间 SVC） 成熟，广泛部署

    “While SVC is technically superior and more bandwidth-efficient, Simulcast remains the industry workhorse because it works reliably across almost all browsers and devices.”
    — BlogGeek.me, WebRTC Simulcast

    实践层面：目前业界最常见的是Simulcast + 时间可伸缩性（Temporal SVC） 的组合：每一路 Simulcast 流内部使用时间分层（L1T2 或 L1T3），SFU 在流粒度上切换分辨率，在帧粒度上按时间层控制帧率与码率，兼顾了两种技术的优点。

    6. RTP 传输机制（RFC 6190 & RFC 7656）

    6.1 单会话传输（SST）与多会话传输（MST）

    RFC 6190（RTP Payload Format for Scalable Video Coding，2011 年 5 月）定义了 H.264 SVC 的 RTP 封装格式，并引入了两种基本传输模式：

    单会话传输（Single-Session Transmission, SST）：将所有可伸缩层的 NAL 单元（Network Abstraction Layer Unit）封装在单个 RTP 会话中传输。简单直接，是 WebRTC 规范（W3C SVC Extension）唯一支持的模式。

    多会话传输（Multi-Session Transmission, MST）：将不同可伸缩层分散到多个 RTP 会话中独立传输，可灵活利用差分网络路径，但复杂度高，W3C SVC Extension 明确不支持此模式。

    6.2 SRST 与 MRST

    在 SST 框架内，RFC 7656（A Taxonomy of Semantics and Mechanisms for Real-Time Transport Protocol (RTP) Sources）进一步区分了：

    SRST（Single RTP stream Single Transport）：所有层的数据通过单个 RTP 流（单个 SSRC）发送。W3C SVC Extension for WebRTC 仅支持 SRST。

    支持 SRST 的 RTP Payload 规范包括：

    • VP8：RFC 7741
    • VP9：VP9-PAYLOAD
    • AV1：AV1-RTP-SPEC
    • H.264：RFC 6184
    • H.265：RFC 7798

    MRST（Multiple RTP stream Single Transport）：多层通过多个 RTP 流（每层独立 SSRC）在同一传输通道中传输。W3C SVC Extension 同样不支持此模式。

    7. SVC in WebRTC：W3C 规范

    W3C Scalable Video Coding (SVC) Extension for WebRTC（最新工作草案：2024 年 8 月 17 日）是在 WebRTC 标准基础上的扩展，不改变现有 WebRTC 对象和方法的行为，而是通过扩展 RTCRtpEncodingParameters 字典，新增一个 scalabilityMode 字段来声明期望的 SVC 模式。

    规范地址：https://www.w3.org/TR/webrtc-svc/
    编辑者草稿：https://w3c.github.io/webrtc-svc/

    7.3 WebIDL API

    规范通过扩展 RTCRtpEncodingParameters 字典新增字段：

    涉及的主要 API 操作：

    **addTransceiver()**：在建立 Transceiver 时通过 sendEncodings 声明 SVC 模式。若所请求的 scalabilityMode 值不被任何已实现的编码器支持，则抛出 OperationError。

    **setParameters()**：在协商完成后动态修改 SVC 模式，无需重新触发 SDP Offer/Answer 协商。若当前协商的编解码器不支持所请求的模式，则拒绝并返回 InvalidModificationError。

    **getParameters()**：获取当前实际生效的 scalabilityMode 值。实际生效值可能与请求值不同（如协商的编解码器不支持时，用户代理可回退到其他值）。

    8. 各编解码器的 SVC 支持情况

    8.1 H.264 SVC（Annex G）

    规范：ITU-T H.264 Annex G（SVC 扩展）、RFC 6190（RTP 封装）

    H.264 SVC 是最早成熟的 SVC 实现，由 Fraunhofer HHI、Vidyo 等主导开发，在 2005 年纳入 H.264 标准。WebRTC 中通过 H264-SVC 媒体子类型描述，但在 WebRTC 实践中支持有限，主要因为：

    • 专利负担重（ MPEG-LA H.264 SVC 专利池）
    • WebRTC 主流实现（libwebrtc / Chrome）未实现 H.264 空间 SVC
    • W3C SVC Extension 对 H.264 仅支持时间可伸缩性（即 L1T2、L1T3）

    8.2 VP8

    VP8 通过 RTP 的时间层扩展（TL0PICIDX、TID 字段）支持时间可伸缩性，最多 3 个时间层。但 VP8 不支持空间可伸缩性，因此无法实现真正意义上的完整 SVC。

    在 WebRTC 中，VP8 的时间可伸缩性长期是事实上的标配，Chrome Simulcast 默认自动附带时间层（通常为 L1T2 或 L1T3）。

    “Temporal scalability is available in VP8, but only in simulcast.”
    — BlogGeek.me

    W3C SVC Extension 对 VP8 仅支持时间层模式（L1T1、L1T2、L1T3）。

    8.3 VP9

    VP9 是 WebRTC 生态中 SVC 支持最为完善的编解码器，同时支持：

    • 时间可伸缩性（最多 3 层）
    • 空间可伸缩性（Full SVC 和 K-SVC）
    • S 模式（单 SSRC Simulcast + 时间层）

    VP9 Payload 规范（草案：draft-ietf-payload-vp9）在 RTP 包头扩展中携带 VP9 Payload Descriptor，包含层 ID（空间层 S、时间层 T）、关键帧标记、切换点标记（U-bit）等信息，使 SFU 无需解析 VP9 码流内部即可做出层转发决策。

    Chrome 对 VP9 SVC 的支持历程：

    • 早期：仅在实验性 flag 后支持时间层（L1T2、L1T3）
    • 后期：逐步开放空间层支持（L3T3_KEY 等 K-SVC 模式）

    8.4 AV1

    AV1 是目前 SVC 支持最完善、最现代的开源视频编解码器：

    • 原生支持时间层、空间层、质量层
    • AV1 RTP Payload 规范（AV1-RTP-SPEC v1.0）已正式发布
    • 支持 SRST 单 SSRC 传输
    • W3C SVC Extension 支持 AV1 的 L、S 全系列模式

    AV1 SVC 的 Chrome 支持（包括 Full SVC）是近年来 WebRTC 媒体能力的重要里程碑，由 CoSMo / Millicast 团队联合推动。

    8.5 H.265（HEVC）

    H.265 HEVC 标准内含 SHVC（Scalable HEVC）扩展，支持多层编码。W3C SVC Extension 对 H.265 仅支持时间可伸缩性，空间 SVC 在 WebRTC 中目前不被规范覆盖。

    SFU 进行层选择的关键信息来源于 RTP 包头扩展（不同编解码器各异）：

    • VP9：VP9 Payload Descriptor 中的 S（空间层 ID）、T（时间层 ID）、U（切换点）、B（层起始包）等字段
    • AV1：AV1 RTP 头扩展中的 Dependency Descriptor（DD）扩展，包含完整的层依赖图（Dependency Structure）
    • VP8：RTP 扩展头中的时间层 TID 字段

    层切换的安全时机：

    1. 时间层降级：可在任意非参考帧时刻丢弃高时间层包，无需等待关键帧
    2. 时间层升级：需等待下一个切换点（U-bit = 1 的帧），确保不引用已丢弃的帧
    3. 空间层降级（Full SVC）：必须等待下一个关键帧（IDR）时刻
    4. 空间层降级（K-SVC）：可在任意 P 帧时刻直接降级，因 P 帧无层间依赖

    SFU 通常通过向发送端发送 RTCP REMB 或 Transport-CC 反馈，配合 PLI（Picture Loss Indication）请求关键帧来触发层切换流程。

    11. 实际部署注意事项

    SDP 协商与 scalabilityMode 的关系

    setParameters() 不触发 SDP 重协商，这是 WebRTC 规范的刻意设计。这意味着：

    • SVC 模式在 Offer/Answer 阶段不会通过 SDP 明确协商
    • 协商的是编解码器（如 VP9），而 SVC 层数在协商完成后才通过 setParameters() 配置
    • 若协商后的编解码器不支持所配置的 scalabilityMode，用户代理可能回退到默认值

    因此，若应用强依赖特定 SVC 能力，应先通过 setCodecPreferences() 限制编解码器列表，确保协商结果满足要求。

    SFU 能力协商

    SFU 的 SVC 层选择能力通过 RTCRtpReceiver.getCapabilities() 暴露（作为 SFM Capabilities），应用可据此判断 SFU 支持哪些 scalabilityMode，避免配置超出 SFU 能力的模式。

    硬件编码器限制

    空间 SVC（多空间层）通常无法利用硬件编码加速，会回退到软件编码，对移动端 CPU 资源有较大压力。如果设备无法支持请求的 SVC 配置，会抛出 RTCError（hardware-encoder-error）。

    丢包与 SVC

    SVC 层间依赖在丢包场景下存在级联损坏风险：基础层包的丢失会导致所有依赖该包的增强层无法解码。因此：

    • SVC 场景下 NACK 重传更为重要
    • 可考虑对基础层包叠加 FEC（冗余纠错）保护
    • K-SVC 模式因 P 帧无层间依赖，对随机丢包的容错性优于 Full SVC

    12. 浏览器支持现状

    浏览器 时间 SVC（L1Tx） 空间 SVC（LxTy） S 模式 主要编解码器
    Chrome ✅ 稳定支持 ✅ 支持（VP9 K-SVC、AV1） ✅ VP9、AV1、VP8
    Firefox ✅ 时间层（VP8） ⚠️ 有限支持 ⚠️ 部分 VP8、VP9
    Safari ✅ H.264 时间层 ❌ 基本不支持 ❌ H.264、H.265
    Edge（Chromium） ✅ 同 Chrome ✅ 同 Chrome ✅ 同 Chrome

    以上为截至 2024 年底的一般状况，具体支持情况随浏览器版本持续更新，建议通过 navigator.mediaCapabilities.encodingInfo() 运行时查询。

    目前 VP9 + K-SVC（L3T3_KEY） 是生产环境中兼具可用性与效率的最优选择；AV1 SVC 则代表未来方向，随着硬件编解码支持的普及，有望成为下一个行业标准。
