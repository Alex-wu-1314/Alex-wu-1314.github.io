    1. 什么是 Simulcast

    Simulcast（联播，源自 “simultaneous broadcast”）是指发送端将同一个媒体源同时编码为多路独立的 RTP 流，每路流使用不同的分辨率、帧率或码率，以满足不同接收端的能力需求。

    按照 RFC 8853 的正式定义：

    Simulcast is defined in this memo as the act of simultaneously sending multiple different encoded streams of the same media source — e.g., the same video source encoded with different video-encoder types or image resolutions.

    2. 为什么需要 Simulcast

    2.1 多方视频会议的核心矛盾

    在多方视频会议中，参与者的网络条件、设备性能和屏幕分辨率往往存在巨大差异。假设有如下场景：

    参与者 下行带宽 屏幕大小 期望画质
    Alice 50 Mbps 4K 显示器 1080p
    Bob 2 Mbps 手机屏幕 360p
    Carol 10 Mbps 笔记本 720p

    发送端（如 Dave）面临三种选择：

    1. 单流发送：发送一路固定质量的流。质量高了 Bob 无法接收，质量低了 Alice 体验差。
    2. 服务端转码（MCU/Transcoding RTP Mixer）：服务器接收一路高质量流，再分别转码后分发。服务端 CPU 消耗极高，引入额外的编解码延迟，且需要访问媒体内容（对端到端加密不友好）。
    3. Simulcast + SFU：发送端发出多路流，SFU 直接转发最合适的一路。服务端无需解码，CPU 消耗极低，延迟最小。

    Simulcast + SFU 是目前工业界应对异构网络环境的主流方案。

    2.2 “慢用户”级联问题

    在没有 Simulcast 的纯 SFU 架构中，存在一个经典的”慢用户问题”：

    1. 某个带宽受限的订阅方（Bob）因频繁丢包无法正常解码，持续向发送端发送 PLI（Picture Loss Indication） 请求关键帧。
    2. 发送端响应 PLI，产生新的关键帧。关键帧体积远大于普通帧，会短暂拉高发送带宽。
    3. 其他订阅方因带宽增加也受到影响，触发更多 PLI，形成级联恶化效应。

    引入 Simulcast 后，SFU 可以向 Bob 转发低质量层，彻底解耦慢用户与其他用户之间的相互影响。

    2.3 RFC 8853 归纳的主要用例

    RFC 8853 第 3 节定义了三类核心用例：

    • REQ-3.1 适配异构接收端：同一媒体源需要服务于具有不同带宽、分辨率、编解码能力的接收方。
    • REQ-3.2 应用特定的媒体源处理：例如会议系统将活跃发言人以高分辨率展示，其他参与者以缩略图（thumbnail）展示，需要同时提供高低两路流。
    • REQ-3.3 接收方的媒体源偏好：接收方可能因为布局变化（如从全屏切换到小窗口）而主动请求不同质量层。

    3. 核心标准与规范

    Simulcast 在 WebRTC 中的标准化工作涉及多个 IETF RFC，它们共同构成完整的协议栈：

    RFC 标题 核心内容
    RFC 8853 Using Simulcast in SDP and RTP Sessions Simulcast 的 SDP 信令扩展，定义 a=simulcast 属性，由 Ericsson 和 Cisco 主导，2021 年 1 月成为 Proposed Standard
    RFC 8851 RTP Payload Format Restrictions 定义 a=rid（Restriction Identifier）SDP 属性，用于标识和约束各路 RTP 流的格式参数
    RFC 8852 RTP Stream Identifier Source Description (SDES) 定义 RtpStreamId 和 RepairedRtpStreamId 两个 RTCP SDES 条目，作为 RTP 头部扩展携带 RID 值
    RFC 8829 JavaScript Session Establishment Protocol (JSEP) 定义浏览器端 SDP offer/answer 处理逻辑，包含 Simulcast 的 sendEncodings 处理规则
    RFC 8834 Media Transport and Use of RTP in WebRTC 描述 WebRTC 中使用多路 RTP 会话实现 Simulcast 的方式
    RFC 8843 Negotiating Media Multiplexing Using SDP (BUNDLE) Simulcast 多路流在 BUNDLE 复用时的处理
    W3C WebRTC WebRTC 1.0: Real-Time Communication in Browsers 浏览器 JavaScript API，定义 addTransceiver + sendEncodings 接口

    关键点：RFC 8853 是 Simulcast 信令的核心规范。RFC 8851 和 RFC 8852 分别定义了流标识符的 SDP 属性和 RTP 头部扩展机制，三者相互依赖，缺一不可。

    4. RTP 层的 Simulcast 实现

    4.1 流标识机制：从 SSRC 到 RID

    在 WebRTC 的早期实现中，Simulcast 的各路流通过不同的 SSRC（Synchronization Source） 来区分，并在 SDP 中通过 a=ssrc-group:SIM 分组。但这种方式存在问题：SSRC 在 RTP 会话生命周期内可能因冲突而发生变化（RFC 3550 允许此行为），导致流的追踪不稳定。

    现代 WebRTC 标准化方案采用 RID（RTP Stream Identifier） 机制：

    • RID 是一个不超过 255 字节的 ASCII 字符串，作为 RTP 头部扩展（Header Extension）随每个 RTP 包携带。
    • RFC 8852 定义了两个 RTCP SDES 条目：
      • RtpStreamId：标识当前流（即发送时的 RID 值）。
      • RepairedRtpStreamId：在 RTX（重传流）中标识被修复的原始流。
    • RID 值通过 RFC 7941 定义的 SDES Item Header Extension 机制（urn:ietf:params:rtp-hdrext:sdes:rtp-stream-id）携带在 RTP 包头中。

    4.3 RTP 包序列号重写

    当 SFU 在不同 Simulcast 层之间切换转发时，会遇到一个重要的技术挑战：每路 Simulcast 流有独立的 RTP 序列号空间和时间戳空间。例如，从高质量层切换到中质量层时，序列号可能出现大幅跳变，导致接收端误判为丢包并触发大量 NACK 请求。

    为此，SFU 在切换层时必须执行以下操作：

    1. 序列号连续化：维护每路流的当前序列号，切换时对新层的序列号进行偏移，保持输出流的序列号单调递增。
    2. 时间戳映射：对输出流的 RTP 时间戳进行适当调整，确保接收端 Jitter Buffer 正常工作。
    3. 关键帧等待：在完成层切换的序列号重写后，必须等到新层发来关键帧（IDR 帧）后才能开始转发，否则接收端无法完成解码初始化。必要时通过 RTCP PLI/FIR 向发送端请求关键帧。

    5. SDP 信令协商

    5.1 核心 SDP 属性

    Simulcast 协商涉及以下几个关键 SDP 属性（均为 media-level 属性）：

    SDP 属性 定义来源 作用
    a=rid RFC 8851 声明一路 RTP 流的 RID 值及其格式限制（分辨率、码率、帧率等）
    a=simulcast RFC 8853 将多个 RID 分组，声明发送和/或接收的 Simulcast 配置
    a=extmap RFC 8285 声明 RTP 头部扩展的映射关系，包括 rtp-stream-id 扩展

    5.2 a=rid 属性语法

    • rid-id：RID 标识符，一个字母数字字符串（通常用 “f”、”h”、”q” 或 “high”、”medium”、”low”）
    • direction：send 或 recv
    • pt：可选，限定该流使用的 RTP Payload Type
    • restriction-list：可选的格式限制，如 max-width、max-height、max-fps、max-mbps、max-br（最大码率，单位 bps/1000）等

    5.3 a=simulcast 属性语法

    • 分号（;）分隔多个独立的编码流（同时发送的各路）
    • 逗号（,）分隔同一路流的替代格式（备选）

    6.4 SDP Munging（不推荐）

    在规范化 API 出现之前，开发者通常通过手动修改 SDP 字符串（即”SDP Munging”）来实现 Simulcast。这种方式已被 W3C 和 IETF 明确不推荐，因为：

    • 依赖 SDP 内部格式，脆弱且难以维护
    • 不同浏览器的 SDP 格式存在差异
    • 违反 JSEP 规范的封装原则

    应始终使用 addTransceiver + sendEncodings 的标准 API。

    7. SFU 在 Simulcast 中的角色

    7.1 整体架构

    7.2 SFU 的核心职责

    层订阅管理（Layer Subscription）：

    SFU 为每个订阅者维护一个期望层（desired layer）和一个允许层（allowed layer）。最终实际转发层由两者取较小值决定：

    • desired_layer：由应用层逻辑决定，通常基于订阅者的视频渲染大小。例如，订阅者以 320×180 的小窗口显示，则不必发送高分辨率层。
    • bandwidth_allowed_layer：由 SFU 的带宽估计（BWE）模块决定。

    层切换时序：

    SFU 在执行层切换时必须满足以下时序要求：

    1. 确定需要切换到新层（如从 h 升级到 f）
    2. 向发布者发送 RTCP PLI（Picture Loss Indication） 或 FIR（Full Intra Request），请求关键帧
    3. 等待新层的关键帧（IDR 帧）到达
    4. 关键帧到达后，执行序列号和时间戳的重写
    5. 开始向该订阅者转发新层的 RTP 数据包

    这一机制确保接收端始终从关键帧开始解码，不会出现花屏。

    发布者带宽受限时的降层处理：

    当发布者上行带宽不足时，浏览器会自动停止最高分辨率层的发送（SFU 检测到某个 RID 的包停止到达）。SFU 需要：

    • 检测到层消失后，将正在接收该层的订阅者降级到更低层
    • 向发布者发送 PLI，确保降级后能立即获得关键帧

    7.3 RTCP 反馈路由

    SFU 还需要处理上下行的 RTCP 反馈消息，并进行适当的聚合与屏蔽：

    • REMB / Transport-CC：SFU 汇总来自各订阅者的带宽反馈，向发布者报告的是 SFU → 发布者链路的上行带宽状况，而非任何单一订阅者的下行带宽。
    • NACK：SFU 可以缓存最近发送的 RTP 包，自行响应订阅者的 NACK 请求，无需向发布者请求重传。
    • PLI/FIR：SFU 通常对多个订阅者的 PLI/FIR 进行防抖（debounce）处理，避免向发布者发送过多关键帧请求。

    8. 时域分层（Temporal Scalability）

    Simulcast 通常与时域分层（Temporal Scalability）结合使用，进一步提升带宽适配的精细度。

    8.1 时域层的概念

    时域层（Temporal Layer）基于帧间依赖关系，将视频流中的帧按时间维度分为不同层级：

    • TID=0（基础层）：关键帧和每隔若干帧的参考帧，仅参考基础层中的帧。低带宽订阅者只接收此层，得到较低帧率（如 15fps）。
    • TID=1（增强层）：介于基础层帧之间的帧，接收后可获得更高帧率（如 30fps）。

    通过仅丢弃 TID=1 的帧，SFU 可以在不等待关键帧的情况下快速将某订阅者的帧率减半，带宽立即降低约 50%。

    8.2 VP8 与时域层

    VP8 在其 RTP Payload Descriptor 中携带 TID（Temporal Layer ID）字段，Chrome 在启用 Simulcast 时会自动为每个空间层增加 2 个时间层（TID=0 和 TID=1），有时多达 3 层。

    9. Simulcast 与 SVC 的对比

    Simulcast 和 SVC（Scalable Video Coding，可伸缩视频编码）解决的是同一个问题：如何向带宽各异的接收端分发同一视频源。两者各有优劣：

    对比维度 Simulcast SVC（如 VP9/AV1 的 SVC 模式）
    发送端带宽 较高（发送多路独立流） 较低（单路流但包含所有层）
    发送端 CPU 较高（多个独立编码器） 较低（单编码器，增量编码层）
    SFU 复杂度 较低（仅需转发正确的流） 较高（需解析帧依赖关系）
    层切换延迟 较高（需等待新层的关键帧） 较低（空间层切换需等切换点，时间层可立即切换）
    编解码兼容性 极好（VP8、H.264 均可） 有限（VP9、AV1；H.264 的 SVC 支持有限）
    硬件加速 良好（每路独立流，现有硬件支持） 有限（SVC 模式的硬件编解码器支持不普遍）
    端到端加密友好度 一般（SFU 需识别 RID） 更好（SFU 只需按层过滤，不需解密内容）

    实践结论：

    • 对于需要广泛兼容性（特别是移动端 H.264 硬件加速）的场景，Simulcast 是首选。
    • 对于 VP9/AV1 场景且 SFU 支持足够复杂的情况下，SVC 可以节省发送端带宽。
    • 部分现代 SFU（如 LiveKit、mediasoup）同时支持两种模式，并根据发布者能力自动选择。

    10. 编解码器支持情况

    10.1 VP8

    VP8 是 WebRTC 中最早支持 Simulcast 的编解码器，也是支持最完善的。其 RTP Payload Descriptor（RFC 7741）中包含 TID（Temporal Layer ID）和 KEYIDX 等字段，供 SFU 进行帧级分析。Chrome、Firefox、Safari 对 VP8 Simulcast 均有良好支持。

    10.2 H.264

    H.264 的 Simulcast 支持稍晚于 VP8。由于 H.264 标准中没有原生的时域层元数据，其时域分层能力不如 VP8 灵活，但空间层 Simulcast（多个独立编码流）完全可行。Chrome M79 开始支持 H.264 Simulcast，且利用硬件编码器，CPU 效率优于 VP8 软件编码。

    10.3 VP9

    VP9 原生支持 SVC 模式，但也可以用于多路独立流的 Simulcast。VP9 的 scalabilityMode 参数（W3C SVC 扩展规范）提供了更灵活的空间+时间分层能力（如 L3T3：3 空间层 × 3 时间层）。

    10.4 AV1

    AV1 是新一代开放视频编解码器，原生支持强大的 SVC 功能。W3C SVC 扩展规范已支持 AV1 的多种 scalabilityMode，但因 AV1 的软件编解码 CPU 消耗较高，目前在实时场景的普及还需等待硬件加速的大规模落地。

    10.5 Opus（音频）

    Simulcast 通常只应用于视频流。音频（Opus）一般不使用 Simulcast，而是通过 Opus 自身的 DTX（Discontinuous Transmission）、带宽自适应等机制处理网络变化。RFC 8853 中 REQ-5.3 也提及了与 Comfort Noise 等音频相关机制的兼容性要求。

    12. 常见问题与注意事项

    12.1 最低分辨率要求

    要启用三层 Simulcast（1:2:4 缩放），源视频的分辨率至少需要满足最低层的分辨率要求。以 4 倍缩放的最低层为例，若最低层要求 320×180，则源分辨率至少需要 1280×720。部分 WebRTC 实现（如 ex_webrtc）明确要求最小起始分辨率为 960×540 才能运行三层 Simulcast。

    12.2 关键帧与层切换的延迟

    层切换并非即时发生。SFU 向发送端发送 PLI，发送端响应并编码关键帧，期间可能有数十到数百毫秒的延迟。在此期间，订阅者可能短暂看到画面冻结。

    12.3 发布者上行带宽自动降级

    当发布者上行带宽不足时，浏览器会自动降低或停止某些 Simulcast 层的发送。这是浏览器内部 BWE（带宽估计）和拥塞控制模块的行为。SFU 需要监测各层 RTP 流是否持续到达，一旦检测到层消失，立即将接收该层的订阅者迁移到更低层。

    12.4 SSRC 变化处理

    虽然 RID-based Simulcast 不再依赖 SSRC 进行流标识，但 SSRC 仍可能在会话中发生变化（如冲突检测触发的 SSRC 重新随机化）。SFU 和接收端实现需要能够处理 SSRC 变化的情况，依赖 RID 扩展重新建立映射关系。

    12.5 与 RTX（重传）的交互

    RFC 8852 定义了 RepairedRtpStreamId SDES 条目（RTP 头部扩展 urn:ietf:params:rtp-hdrext:sdes:repaired-rtp-stream-id），用于 RTX 重传流中标识被修复的原始 Simulcast 流。SFU 在处理重传时需要正确解析此扩展，以将重传包与正确的原始流关联。

    12.6 与 BUNDLE 的关系

    WebRTC 通常启用 BUNDLE（将所有媒体复用到单一传输层）。Simulcast 的三路流（RID f、h、q）均在同一个 BUNDLE 传输通道中传输，通过 MID + RID 头部扩展进行解复用。这要求接收端实现正确处理 RTP 头部扩展的优先级顺序。

    12.7 SDP 协商的向后兼容性

    RFC 8853 的设计中包含了对非 Simulcast 传统客户端的兼容性要求（REQ-6.1）。SFU 在向不支持 Simulcast 的遗留客户端转发时，只需选择一路流转发即可，无需对 SDP 进行特殊处理。
