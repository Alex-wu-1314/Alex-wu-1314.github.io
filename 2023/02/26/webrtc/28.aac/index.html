    1. 背景与标准化历程

    AAC（Advanced Audio Coding，高级音频编码）是由 Dolby、AT&T、Fraunhofer、Sony 等多家机构联合研发的有损音频压缩标准，最初作为 MPEG-2 规范的一部分诞生，后在 MPEG-4 框架下进一步完善。

    关键时间节点：

    年份 事件
    1997 AAC 作为 MPEG-2 Part 7 正式发布，标准号 ISO/IEC 13818-7，也称 MPEG-2 NBC（Non-Backward Compatible）
    1999 MPEG-4 Audio 第一版发布，AAC 被纳入 ISO/IEC 14496-3，引入更灵活的 Audio Object 架构
    2003 HE-AAC v1（AAC-LC + SBR）标准化，发布于 ISO/IEC 14496-3:2001/Amd 1:2003
    2006 HE-AAC v2（HE-AAC v1 + Parametric Stereo）标准化，发布于 ISO/IEC 14496-3:2005/Amd 2:2006
    2003 IETF 发布 RFC 3640，定义 MPEG-4 基本流的 RTP 载荷格式，正式规范 AAC-lbr / AAC-hbr 的网络传输

    AAC 被设计为 MP3（MPEG-2 Audio Layer III）的继任者。在相同码率下，AAC 通过更先进的心理声学模型和更灵活的滤波器组，能够提供比 MP3 更高的感知音质。今天，AAC 是 Apple 全生态（iTunes、Apple Music、iPhone）、YouTube、Android、DAB+ 数字广播等平台的默认或主流音频格式。

    2. AAC 家族一览

    AAC 并不是单一编解码器，而是一个由多种配置（Profile）和扩展工具组合构成的家族：

    各配置的适用码率参考：

    配置 典型码率（立体声） 典型场景
    AAC-LC 128–256 kbps 音乐流媒体、录制存储
    HE-AAC v1 48–80 kbps 互联网广播、低带宽直播
    HE-AAC v2 16–40 kbps 超低码率移动流媒体
    AAC-LD / ELD 32–64 kbps 双向实时通话（Apple FaceTime）
    xHE-AAC 12–300 kbps 下一代广播（DRM、DAB+）

    3. 编码核心技术原理

    3.1 感知音频编码基础

    AAC 是一种感知音频编码器（Perceptual Audio Codec）。其核心思想是：人耳并非对所有声音频率和响度同等敏感——某些声音信息在感知上是”不可见的”，可以被丢弃而不被人耳察觉。

    AAC 通过两条主线减少数据量：

    1. 丢弃感知不相关信息：依据心理声学模型（Psychoacoustic Model），确定哪些频率成分在当前时刻被其他声音”掩蔽”，可以用更少比特甚至不编码这些成分。
    2. 消除统计冗余：通过 Huffman 熵编码压缩量化后仍存在的统计冗余。

    3.2 MDCT 滤波器组

    AAC 的时频转换核心是改进离散余弦变换（MDCT，Modified Discrete Cosine Transform）。与 MP3 使用 32 子带多相滤波器组不同，AAC 直接使用 MDCT，省去了一级滤波器组，简化了架构并提升了频率分辨率。

    MDCT 的数学定义：

    $$X[k] = \sum_{n=0}^{N-1} x[n] \cdot \cos\left[\frac{\pi}{N}\left(n + \frac{N/2 + 1}{2}\right)\left(k + \frac{1}{2}\right)\right], \quad 0 \le k < N/2$$

    其中 $N$ 为窗口长度。

    AAC 支持自适应窗口切换：

    • 长窗口（Long Window）：$N = 2048$ 个样本（约 46.4ms @ 44.1kHz），提供高频率分辨率，适合平稳信号（如持续音调）。
    • 短窗口（Short Window）：$N = 256$ 个样本，分 8 段处理相同时长，提供高时间分辨率，适合瞬态信号（如鼓声、打击乐），避免预回声（Pre-echo）失真。
    • Start / Stop 窗口：在长短窗口间平滑切换的过渡窗口。

    MDCT 的关键特性是时域混叠消除（TDAC，Time Domain Aliasing Cancellation）：相邻帧 50% 重叠，解码时叠加即可完美重建，消除混叠失真。

    3.3 心理声学模型

    心理声学模型是 AAC 编码器的”大脑”，它分析输入信号并为量化器提供掩蔽阈值（Masking Threshold）——即在每个频率上，可以注入多大的量化噪声而不被人耳感知。

    主要利用的心理声学现象：

    • 同时掩蔽（Simultaneous Masking）：强音调成分会掩蔽同一时刻相邻频率上的弱信号。
    • 时间掩蔽（Temporal Masking）：包括前掩蔽（Pre-masking，约 2–5ms）和后掩蔽（Post-masking，约 100ms），强声音前后的时间窗口内，弱信号难以被感知。
    • 绝对听觉阈值（Absolute Threshold of Hearing）：人耳在极低声压级（尤其高频和低频端）无法感知声音，这些频率可以用极少比特编码。

    AAC 的心理声学模型使用 1/3 临界带宽（Critical Bandwidth） 作为频率分辨率单位，这是复杂度与性能之间的工程折中——精度高于整带宽分析，但计算量低于更细粒度方法。

    3.4 量化与 Huffman 熵编码

    获得掩蔽阈值后，编码器执行迭代量化-失真检验循环（Rate-Distortion Loop）：

    1. 将 MDCT 系数划分为若干尺度因子带（Scalefactor Band），每个带共享一个量化步长（Scalefactor）。
    2. 对各带的 MDCT 系数按 $\text{int}(|X[k]|^{3/4})$ 的非均匀量化公式量化。
    3. 对量化索引进行 Huffman 编码（AAC 定义了多个码表）。
    4. 检查各带量化噪声是否超过掩蔽阈值，若超过则细化量化步长，迭代直至满足。

    非均匀量化的指数 3/4 是特意设计的：它在高幅值系数上提供更细的量化（保留信号细节），在低幅值系数上允许更粗的量化（这些信号通常被掩蔽）。

    3.5 时域噪声整形（TNS）

    时域噪声整形（TNS，Temporal Noise Shaping） 是 AAC 相比 MP3 的重要创新之一。

    它解决的问题：对于瞬态信号（如人声的辅音爆破），即使使用短窗口，量化噪声也会在时域上均匀分布，在爆破音之前产生可听见的”预回声”。

    TNS 通过在频域施加预测滤波，将量化噪声的时域分布整形为跟随信号包络——噪声集中在信号本身所在的时刻，而非扩散到信号前后，从而使噪声在感知上不可见。

    3.6 联合声道编码（M/S Stereo & Intensity Stereo）

    AAC 提供两种立体声冗余压缩工具：

    • M/S 立体声（Mid/Side Stereo）：将左右声道（L, R）转换为和差信号（M = (L+R)/2，S = (L−R)/2）。当两声道相关性高时，S 分量能量很小，可用很少比特编码，有效减少总码率。
    • 强度立体声（Intensity Stereo）：在高频段（人耳对高频方向感知不灵敏），只编码一个共享的频谱包络，通过比例因子近似重建左右声道，进一步降低高频编码开销。

    3.7 感知噪声替代（PNS）

    对于类似噪声的频谱分量（如混响、环境声），感知噪声替代（PNS，Perceptual Noise Substitution） 不传输实际的量化系数，而是传输一个能量参数。解码端用该能量参数生成同等功率的随机噪声填充对应频率，感知效果几乎一致，但节省了大量比特。

    3.8 长期预测（LTP）

    长期预测（LTP，Long-Term Prediction） 利用音频信号的周期性（尤其是人声基频），用之前已解码的帧来预测当前帧，编码预测残差而非原始信号。这对含有稳定周期结构的音频（语音、弦乐）能有效降低码率。

    4. HE-AAC：高效扩展技术

    4.1 频谱带复制（SBR）

    频谱带复制（SBR，Spectral Band Replication） 是 HE-AAC v1 的核心扩展，由 Coding Technologies（现已被 Dolby 收购）开发，并于 2003 年由 MPEG 标准化。

    核心思想：人耳对高频成分（通常 > 6kHz）的感知主要依赖其频谱包络（Envelope）形状，而非精确的谐波结构。高频谐波通常是低频谐波的泛音，存在强相关性。

    SBR 工作流程：

    SBR 的关键效益是码率减半：核心编码器只需处理半带宽信号，用原来一半的码率即可达到相近的感知质量。例如 HE-AAC v1 @ 48 kbps 能达到接近 AAC-LC @ 96 kbps 的感知质量。

    4.2 参数化立体声（PS）

    参数化立体声（PS，Parametric Stereo） 是 HE-AAC v2 在 HE-AAC v1 基础上叠加的扩展，标准化于 ISO/IEC 14496-3:2005/Amd 2:2006。

    核心思想：在极低码率下，与其完整传输两条独立声道，不如将立体声信号降混为单声道进行核心编码，同时用极少的参数描述空间感信息，由解码端重建立体声。

    PS 的空间参数包括：

    • IID（Inter-channel Intensity Difference）：左右声道强度差，描述声像位置。
    • ICC（Inter-channel Cross-Correlation）：声道间互相关，描述空间宽度/混响感。
    • IPD/OPD（Inter-channel Phase Difference / Overall Phase Difference）：相位差，精细调整空间定位。

    整套参数每帧约需 2–3 kbps，结合 HE-AAC v1 的单声道主流，HE-AAC v2 能在 24 kbps 左右 提供可接受的立体声质量——是超低码率移动流媒体的理想选择。

    5. 码流封装格式

    AAC 音频数据本身由一系列音频帧（Audio Frame）构成，但这些帧需要封装格式才能被解复用器正确识别和解析。

    5.1 ADIF（Audio Data Interchange Format）

    • 特点：仅有一个文件头，后接连续的 AAC 帧。
    • 适用场景：文件存储，不支持随机访问和流式传输。
    • 现状：基本已被弃用，MPEG-4 规范将其视为”informative”，不强制要求解码器支持。

    5.2 ADTS（Audio Data Transport Stream）

    ADTS 是流式传输最常见的封装格式，每帧前附加一个 7 字节（无 CRC）或 9 字节（有 CRC）的同步头：

    ADTS 的优势：每帧都有同步字（0xFFF），支持随机接入和断点续接，是 HLS 分片（.aac 文件）和 RTMP 流中最常见的格式。

    5.3 LATM / LOAS

    • LATM（Low-overhead MPEG-4 Audio Transport Multiplex）：MPEG-4 定义的低开销传输格式，支持多路 AAC 流复用。
    • LOAS（Low Overhead Audio Stream）：LATM 的自同步封装，用于 DVB 传输流（当使用 SBR 或 PS 扩展时）。

    LATM 的头开销比 ADTS 更小（不含 Syncword），但需要带外（out-of-band）的配置信息（AudioSpecificConfig），因此更适合有信令通道的场景，如 RTSP 或 RTP over MPEG-4（见下节）。

    6. RTP 传输：RFC 3640 详解

    在网络传输层面，AAC 通过 RTP 承载，RFC 3640（RTP Payload Format for Transport of MPEG-4 Elementary Streams，2003 年 11 月）是主要规范。对于通过 LATM 封装的 AAC，另有 RFC 3016（及其后续 draft-ietf-payload-rfc3016bis）规定了 MP4A-LATM 载荷类型。

    6.1 RFC 3640 载荷结构

    RFC 3640 将 RTP 载荷划分为三段：

    Access Unit（AU） 在 AAC 语境下即一个 AAC 音频帧（通常含 1024 个 PCM 样本）。

    AU Header 字段（各字段长度由 SDP fmtp 参数指定）：

    字段 含义
    AU-size 本 AU 的字节数（用于多帧拼接时的解析）
    AU-Index 第一帧的序列号
    AU-Index-delta 后续帧相对于前一帧的序列号增量（用于交错解交错）
    CTS-flag / CTS-delta 组合时间戳（可选）
    DTS-flag / DTS-delta 解码时间戳（可选）
    RAP-flag 随机访问点标志

    RFC 3640 定义了多个传输模式（Mode），SDP 中用 mode 参数指定；对 AAC 最重要的两个：

    6.2 AAC-lbr 模式（低码率 AAC）

    用于码率 ≤ 64 kbps 的场景（如 HE-AAC）：

    • sizeLength=6（AU 大小字段 6 bit，最大 63 字节/帧）
    • indexLength=2，indexDeltaLength=2
    • 支持帧交错（Interleaving），增强丢包鲁棒性

    6.3 AAC-hbr 模式（高码率 AAC）

    用于码率 > 64 kbps 的场景（如 AAC-LC 128kbps+）：

    • sizeLength=13（AU 大小字段 13 bit，最大 8191 字节/帧）
    • indexLength=3，indexDeltaLength=3
    • 支持单帧、多帧拼包（Concatenation）、大帧分片（Fragmentation）

    当 AAC 帧较小时（例如 64 kbps 立体声帧约 200 字节），RFC 3640 允许在单个 RTP 包中拼装多个 AU——在以太网 MTU（1500 字节）环境下平均可装约 7 帧，显著提升传输效率。

    7. 在直播与录制场景中的应用

    7.1 直播协议栈中的 AAC

    在当今主流直播生态中，AAC 几乎是不可绕过的基础设施：

    RTMP 中的 AAC 封装：RTMP 音频消息使用 AudioTagHeader，SoundFormat=10 标识 AAC，后跟一个字节的 AACPacketType：0x00 表示 AudioSpecificConfig（在推流开始时发送一次），0x01 表示实际 AAC 音频帧（ADTS 去掉头部后的裸帧）。

    HLS 中的 AAC：HLS 规范（RFC 8216）要求 AAC 音频以 ADTS 封装写入 MPEG-2 TS 分片，或以 ftyp=mp42 封装写入 fMP4 分片。Apple 推荐 HLS 使用 HE-AAC v2 @ 40kbps 用于低带宽回退，AAC-LC @ 160kbps 作为标准质量档。

    7.2 录制场景

    录制场景通常将 AAC 封装到 MP4（ISO Base Media File Format，ISO/IEC 14496-12）或 MOV 容器：

    • stsd box（Sample Description Box） 中包含 mp4a 编解码类型，内嵌 esds（Elementary Stream Descriptor）或 AudioSpecificConfig。
    • AAC-LC @ 256 kbps 是 Apple 设备录制的默认配置，提供接近透明的感知质量。

    7.3 典型平台音频策略

    平台 编码 码率

    Apple Music AAC-LC 256kbps（iTunes Plus）
    Twitch AAC-LC 128–160kbps（推荐）
    DAB+ 数字广播 HE-AAC v2 24–48kbps

    8. AAC 与 Opus 的边界：在 WebRTC 生态中的定位

    理解 AAC 在 WebRTC 生态中的定位，需要先明确 WebRTC 的强制性规范。

    根据 RFC 7874（WebRTC Audio Codec and Processing Requirements），WebRTC 浏览器必须（MUST）支持以下音频编解码器：

    • Opus（6–510 kbps，RFC 6716）—— WebRTC 主力编解码器
    • G.711 PCMU/PCMA（64 kbps）—— 电话网络互通

    AAC 不在 WebRTC 强制要求之列。

    8.1 为什么 WebRTC 默认使用 Opus 而非 AAC？

    维度 Opus AAC-LC
    算法延迟 2.5–26.5ms（可配置） 40–100ms（典型），最低约 20ms（HE-AAC v2）
    WebRTC 强制要求 ✅ RFC 7874 ❌
    专利/授权 完全免专利费（Xiph/IETF） 需授权（Via Licensing Alliance，约 $0.98/decoder）
    低码率质量（< 64kbps） 更优 较弱（HE-AAC 有改善）
    自适应码率 内建支持（6–510 kbps 无缝切换） 需外部控制
    FEC 内建 ✅ In-band FEC ❌
    编解码器统一性 语音/音乐统一处理 多配置，复杂

    实时通话（WebRTC）场景需要的是低延迟和带宽自适应，Opus 在这两方面均有压倒性优势。

    8.2 AAC 进入 WebRTC 生态的场景

    尽管如此，AAC 并非与 WebRTC 完全无关：

    场景一：SFU 转码/转封装

    WebRTC 推流 → SFU（如 Janus, mediasoup）→ RTMP/HLS 分发时，SFU 需要将 Opus 转码为 AAC。这是因为下游的 CDN、播放器、录制系统普遍只支持 AAC，而不支持 Opus。

    场景二：Safari 的 WebRTC 实现

    Safari（基于 WebKit）历史上在 WebRTC 中提供了对 AAC 的可选支持（RTCRtpCodecCapability 中包含 audio/mp4a-latm），主要用于与 Apple 自有平台的互通。但大多数主流 WebRTC 应用仍使用 Opus 作为主选编解码器。

    场景三：低延迟直播（LL-HLS / WebRTC 分发）

    在低延迟直播（Low Latency HLS）场景中，WebRTC 有时被用作”最后一公里”的推流协议（如 WHIP 协议），但到达播放端时仍会转换为 AAC+HLS/DASH。

    8.3 延迟对比

    AAC-LD/ELD 虽然延迟改善明显，但仍高于 Opus，且属于 Apple 主导的私有生态，在开放 WebRTC 场景中几乎不被使用。

    硬件加速

    AAC 编解码的运算量相比视频编解码很小，但硬件加速仍有价值：

    • Apple： AudioToolbox 框架提供 AAC 硬件/系统加速编解码，iOS/macOS 设备均支持。

    • DSP/嵌入式： 大量 IoT 音频芯片内建 AAC 解码硬件，适合低功耗场景。

    10. 常见问题与工程陷阱

    10.1 ADTS 头部重复封装问题

    在将 AAC 写入 MP4 容器时，不应包含 ADTS 头——MP4 的 esds/AudioSpecificConfig 已承担了带外配置传递的职责，再加 ADTS 头会导致解码失败。从 RTMP 流中提取的 AAC 帧通常已去掉 ADTS 头（RTMP FLV 封装的 AudioSpecificConfig 独立发送），直接写入 MP4 即可。

    反之，将 MP4 中的裸 AAC 帧直接写成 .aac 文件时，需要手动添加 ADTS 头，否则播放器无法识别。

    10.2 AudioSpecificConfig 解析

    SDP fmtp 中的 config=XXXX 是 AudioSpecificConfig 的十六进制字符串，必须正确解析才能初始化解码器。常见错误包括：

    • 忽略 SBR 扩展标志（extensionAudioObjectType = 5），导致 HE-AAC 流被按 AAC-LC 解码（音质变差、采样率减半）。
    • 未处理 samplingFrequencyIndex = 0xF（扩展形式），此时后续 24 bit 才是实际采样率值。

    10.3 HE-AAC 的向后兼容解码

    HE-AAC v1/v2 采用向后兼容设计：不支持 SBR/PS 的老旧 AAC-LC 解码器可以只解码核心 AAC 层，输出采样率将是目标采样率的一半（因为核心 AAC 以半速运行）。这会导致音频缺失高频成分并且速度异常。在实践中需确认解码库版本支持 SBR/PS 解码（libfdk_aac 和现代 FFmpeg 均支持）。

    10.4 RFC 3640 中的时间戳

    RFC 3640 规定 AAC 的 RTP 时间戳以采样率为单位计算。对于 AAC-LC（每帧 1024 样本，44100Hz 采样率）：

    SDP 中 a=rtpmap 的时钟频率（/44100）必须与 AudioSpecificConfig 中的 samplingFrequencyIndex 一致，否则会导致音视频同步漂移。

    10.5 RTMP 中 AAC 序列头的时序

    在 RTMP 推流中，AudioSpecificConfig 必须先于任何 AAC 音频帧发送（AACPacketType=0x00），否则拉流端和录制端无法初始化解码器，前几秒音频会静音或解码失败。
