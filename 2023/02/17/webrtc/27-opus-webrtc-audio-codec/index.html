    Opus —— WebRTC 音频的事实标准编码

    Alex Wu
    2023-02-17
    webrtc

    相关规范

    • RFC 6716 — Definition of the Opus Audio Codec（2012）
    • RFC 7587 — RTP Payload Format for the Opus Speech and Audio Codec（2015）
    • RFC 7874 — WebRTC Audio Codec and Processing Requirements（2016）
    • RFC 8251 — Updates to the Opus Audio Codec（2017）

    1. 概述

    Opus 是由 IETF（Internet Engineering Task Force）于 2012 年正式标准化的开放、免版税的音频编解码器，规范编号 RFC 6716。它由 Mozilla Corporation 与 Skype Technologies（现 Microsoft）合作开发，融合了 Skype 的 SILK 语音编码技术与 Xiph.Org 的 CELT 音频编码技术，是第一个被主流标准组织批准的、完全开源的高质量音频编解码器。

    在 WebRTC 的世界里，Opus 的地位不可撼动。RFC 7874（WebRTC Audio Codec and Processing Requirements）明确规定：

    WebRTC endpoints are REQUIRED to implement the following audio codecs: Opus [RFC6716] with the payload format specified in [RFC7587].

    即所有合规的 WebRTC 实现（浏览器、原生客户端、媒体服务器）必须支持 Opus。这一强制要求使 Opus 成为 WebRTC 音频互操作的基础，也是其被称为”事实标准”的法理依据。

    为什么 Opus 能脱颖而出？

    在 Opus 之前，音频编码领域高度碎片化：

    • 语音场景用 SILK（Skype）、Speex、AMR-NB/WB；
    • 音乐/通用场景用 Vorbis、AAC、MP3；
    • 实时通信场景用 G.711、G.722。

    没有任何一个编解码器能同时覆盖从低码率语音到高质量音乐的全部场景，且大多数都受专利约束。Opus 通过将 SILK 与 CELT 深度整合，第一次让一个编解码器覆盖了几乎所有的网络音频应用场景，且完全免费开源。

    2. 历史与标准化进程

    时间 事件
    2007 年 Xiph.Org 开始开发 CELT（Constrained Energy Lapped Transform）
    2009 年 Skype 开发 SILK 编解码器，用于 Skype VoIP 通话
    2010 年 IETF CODEC 工作组成立，开始制定统一的互联网音频编解码器标准
    2011 年 11 月 IETF 工作组发布最后一次码流格式征集意见（Last Call）
    2012 年 1 月 8 日 码流格式冻结，不再接受变更
    2012 年 7 月 2 日 Opus 被 IETF 批准标准化
    2012 年 9 月 10 日 RFC 6716 正式发布，libopus 1.0 同日发布
    2015 年 6 月 RFC 7587 发布，规范 Opus 在 RTP 上的封装格式
    2016 年 5 月 RFC 7874 发布，将 Opus 列为 WebRTC 必须实现的音频编解码器
    2017 年 RFC 8251 发布，对 RFC 6716 进行若干勘误和更新
    2024 年 libopus 1.5 发布，引入基于深度学习的 DRED（Deep Redundancy）机制

    参与 Opus 开发的主要机构包括：Broadcom、Google、Microsoft（通过 Skype）、Mozilla、Octasic、Xiph.Org 以及 IETF 本身。

    3. 核心特性

    RFC 6716 在摘要中精炼地描述了 Opus 的主要特性：

    Opus is designed to handle a wide range of interactive audio applications, including Voice over IP, videoconferencing, in-game chat, and even live, distributed music performances. It scales from low bitrate narrowband speech at 6 kbit/s to very high quality stereo music at 510 kbit/s.

    具体而言：

    3.1 超宽码率范围

    • 最低码率：6 kbit/s（窄带语音，NB）
    • 最高码率：510 kbit/s（全频带立体声音乐，FB Stereo）
    • 码率可在运行时动态调整，无需重新协商信令

    根据 RFC 6716，不同场景的推荐码率（20 ms 帧大小）：

    场景 带宽类型 推荐码率
    窄带语音（NB Speech） 0–4 kHz 8–12 kbit/s
    宽带语音（WB Speech） 0–8 kHz 16–20 kbit/s
    全频带语音（FB Speech） 0–20 kHz 28–40 kbit/s
    全频带单声道音乐 0–20 kHz 48–64 kbit/s
    全频带立体声音乐 0–20 kHz 64–128 kbit/s

    3.2 极低算法延迟

    Opus 支持 2.5 ms 到 60 ms 的帧大小，算法延迟（不含网络延迟）最低约 5 ms（CELT 模式 2.5 ms 帧 + 2.5 ms look-ahead）。这比 AAC-ELD（15 ms）更低，是对延迟要求极高场景（如网络音乐合奏）的理想选择。

    3.3 支持多种采样率和声道

    带宽缩写 全称 音频带宽 采样率
    NB Narrowband 0–4 kHz 8 kHz
    MB Mediumband 0–6 kHz 12 kHz
    WB Wideband 0–8 kHz 16 kHz
    SWB Super-wideband 0–12 kHz 24 kHz
    FB Fullband 0–20 kHz 48 kHz

    声道支持：单声道、立体声，以及多声道扩展（通过 multiopus 格式）。

    3.4 完全开放与免版税

    Opus 以新 BSD 许可证发布，任何人可以免费使用，没有专利授权费，没有 RAND 条件，是迄今为止唯一被主流标准机构认可的完全免版税宽带音频编解码器。

    4. 内部架构：SILK、CELT 与混合模式

    Opus 在架构上并非简单地在两个编解码器之间切换，而是对 SILK 和 CELT 进行了深度修改和整合。RFC 6716 将其描述为一个统一的编解码器，内部包含三种工作模式。

    4.1 SILK 模式（语音优化）

    SILK 是基于线性预测编码（LPC, Linear Predictive Coding）的语音编解码器，最初由 Koen Vos 在 Skype 开发。它针对人类语音的声学特性进行优化：

    • 使用CELP（Code-Excited Linear Prediction）结构，适合建模语音共振峰特性
    • 支持采样率：8 kHz、12 kHz、16 kHz（对应 NB、MB、WB）
    • 帧大小支持：10 ms、20 ms、40 ms、60 ms（SILK 支持将多个 20 ms 帧打包以降低开销）
    • 包含长期预测滤波器（Long-term Prediction Filter），利用语音的周期性（基音）进行预测
    • 支持 LBRR（Low Bitrate Redundancy）帧，即内嵌 FEC（见第 8 节）
    • 内建 VAD（Voice Activity Detection）和 DTX（Discontinuous Transmission）支持

    SILK 在低码率（6–20 kbit/s）语音场景中表现优异，其 LPC 模型对语音频谱有极高的压缩效率。

    4.2 CELT 模式（音乐与高质量音频）

    CELT（Constrained Energy Lapped Transform）是基于 MDCT（Modified Discrete Cosine Transform）的变换域编解码器，由 Xiph.Org 的 Jean-Marc Valin 等人开发：

    • 始终工作在 48 kHz 采样率，覆盖全频带（0–20 kHz）
    • 帧大小支持：2.5 ms、5 ms、10 ms、20 ms（比 SILK 支持更短帧，延迟更低）
    • 基于心理声学模型进行频带分配，类似 AAC 的感知编码
    • 包含谱复制（Spectral Replication）和噪声生成（Noise Generation），类似 AAC 的 SBR 和 PNS
    • 利用 Range Encoder（算术编码变体）进行熵编码
    • 适用于音乐、宽频带语音、以及所有对音质有高要求的场景

    4.3 混合模式（Hybrid Mode）

    混合模式是 Opus 的独特创新。如 Xiph.Org FAQ 所述：

    In addition to a Linear Prediction SILK mode and an MDCT CELT mode it has a hybrid mode, where speech frequencies up to 8 kHz are encoded with LP while those between 8 and 20 kHz are encoded with MDCT.

    具体地说：

    • 0–8 kHz 频率范围：由 SILK（LPC）编码，利用语音共振峰预测
    • 8–20 kHz 频率范围：由 CELT（MDCT）编码，补充高频细节
    • 两个编码器的输出共同打包进同一个 Opus 数据包

    这种设计允许 Opus 在约 32 kbit/s 的码率下实现极高的超宽带（SWB）和全频带（FB）语音质量——在此码率下，单独使用 SILK 或 CELT 都无法达到同等效果。

    4.4 模式选择逻辑

    编码器根据以下因素动态选择工作模式：

    • 输入信号类型（语音 vs 音乐），通过音乐检测器识别
    • 目标码率：低码率倾向 SILK，高码率倾向 CELT
    • 目标带宽：SWB 及以上倾向混合模式或纯 CELT
    • 目标延迟：极低延迟场景强制使用 CELT

    模式切换是无感知的（seamless），无需任何带外信令，解码端可以从 Opus 包头中自动识别当前模式。

    5. 码流格式（Bitstream Format）

    RFC 6716 定义了 Opus 码流的精确格式。每个 Opus 数据包（Packet）由一个或多个帧（Frame）组成。

    5.1 TOC（Table of Contents）字节

    每个 Opus 数据包以一个 TOC 字节开头，用于描述包内帧的配置：

    • config（5 bits）：编码配置索引（0–31），决定了工作模式、带宽和帧大小
    • s（1 bit）：立体声标志（0 = 单声道，1 = 立体声）
    • c（2 bits）：帧打包代码
      • 00：包内包含 1 帧
      • 01：包内包含 2 帧，每帧等长
      • 10：包内包含 2 帧，帧长不等
      • 11：包内包含任意数量的帧（由后续 Frame Count Byte 指定）

    5.2 配置索引（Config）与工作模式映射

    RFC 6716 定义了 32 种配置，核心映射关系如下：

    Config 索引 模式 带宽 帧大小
    0–3 SILK NB 10/20/40/60 ms
    4–7 SILK MB 10/20/40/60 ms
    8–11 SILK WB 10/20/40/60 ms
    12–13 Hybrid SWB 10/20 ms
    14–15 Hybrid FB 10/20 ms
    16–19 CELT NB 2.5/5/10/20 ms
    20–23 CELT WB 2.5/5/10/20 ms
    24–27 CELT SWB 2.5/5/10/20 ms
    28–31 CELT FB 2.5/5/10/20 ms

    5.3 多帧打包

    当 c = 11 时，TOC 字节后跟一个 Frame Count Byte（FCB）：

    • v（1 bit）：VBR 标志（1 = 可变码率）
    • p（1 bit）：是否包含 Opus Padding
    • M（6 bits）：帧数量（1–63）

    单个 Opus 数据包最多可以包含多个帧，总时长不超过 120 ms。在实时通信中，默认通常使用 20 ms 单帧，每秒发送 50 个 RTP 包，这是 IP/UDP/RTP 头部开销与延迟之间的最佳平衡点。

    6. 在 WebRTC 中的 RTP 封装（RFC 7587）

    RFC 7587 定义了 Opus 在 RTP 上的封装格式，是 Opus 与 WebRTC 媒体传输层对接的关键规范。

    6.1 RTP 头部使用规则

    • Payload Type：动态分配（96–127 范围），在 SDP 中通过 rtpmap 协商
    • Timestamp：以 48000 Hz 为时钟频率递增（不论实际采样率如何，RTP 时钟固定为 48000 Hz）
      • 例如，一个 20 ms 的帧包含 960 个采样（48000 × 0.02），时间戳每包增加 960
    • Marker bit（M bit）：Opus 通常不使用 Marker bit，但若接收端需标记说话人轮换或 DTX 结束，可以置位
    • SSRC：标准 RTP SSRC，每路 Opus 流使用唯一值
    • 序列号：标准 16 位 RTP 序列号，用于检测丢包（Opus FEC/PLC 依赖此序列号）

    7. SDP 协商（RFC 7587 § 7）

    在 WebRTC 的 Offer/Answer 流程中，Opus 通过 SDP 进行能力协商。RFC 7587 规定了 Opus 的 SDP 媒体类型注册和参数语义。

    7.1 rtpmap 格式

    无论实际通话采样率如何，Opus 的 SDP rtpmap 中时钟频率固定为 48000，声道数固定为 2：

    这是 RFC 7587 的强制规定，原因是 Opus 可在运行时动态改变采样率，rtpmap 中的值仅用于 RTP 时钟同步，与实际编码无关。

    7.2 fmtp 参数

    通过 a=fmtp 行传递 Opus 的可选参数，格式为分号分隔的键值对：

    参数 方向 含义 默认值
    maxplaybackrate 解码端→编码端 接收端建议的最大播放采样率（Hz） 48000
    sprop-maxcapturerate 编码端→解码端 发送端最大采集采样率（Hz） 48000
    maxaveragebitrate 解码端→编码端 接收端建议的最大平均码率（bps） 无限制
    stereo 解码端→编码端 接收端是否希望接收立体声（1/0） 0（单声道）
    sprop-stereo 编码端→解码端 发送端是否会发送立体声（1/0） 0
    useinbandfec 解码端→编码端 接收端是否支持 Opus 内嵌 FEC（1/0） 0
    usedtx 解码端→编码端 是否使用 DTX（1/0） 0
    cbr 解码端→编码端 是否使用 CBR（1/0） 0（VBR）

    注意：useinbandfec 和 usedtx 是单向接收参数（unidirectional receive-only），即它们描述的是本端解码能力对发送端编码行为的期望，而非对自身发送行为的约束。

    7.4 ptime 与 maxptime

    a=ptime 指定建议的打包时长（毫秒）。WebRTC 默认使用 ptime=20（20 ms，最常见配置）。RFC 7587 规定：

    • 若对端 ptime 与本端不一致，编码端应尽量配合，但无需强制遵守
    • maxptime 限制接收端能接受的最大帧长（超过此值解码端可能无法处理）

    8. 抗丢包机制

    Opus 提供了多层次的丢包应对策略，这对 WebRTC 在不稳定网络中的音质保障至关重要。

    8.1 内嵌 FEC：LBRR（Low Bitrate Redundancy）

    LBRR 是 Opus SILK 模式特有的带内前向纠错机制。其原理是：

    编码端在发送当前帧（帧 N）时，同时在码流中附加一份前一帧（帧 N-1）的低码率冗余编码。

    当帧 N-1 丢失时，接收端可以从帧 N 中提取 LBRR 数据恢复帧 N-1 的音频，而无需重传（zero additional round-trip delay）。

    关键约束：

    • LBRR 仅在 SILK 模式和混合模式下可用，纯 CELT 模式不支持
    • 需要编码端通过 OPUS_SET_INBAND_FEC(1) 启用，并通过 OPUS_SET_PACKET_LOSS_PERC 告知预期丢包率
    • 接收端需在 SDP 中声明 useinbandfec=1 以告知对端自己能处理 LBRR 数据
    • 每个 Opus 包只包含前一帧的冗余，因此 LBRR 只能恢复非连续单帧丢失；连续丢包仍需 PLC

    8.2 深度冗余：DRED（Deep Redundancy）

    DRED 是 Amazon Web Services（Jean-Marc Valin 团队）开发的基于神经网络的深度冗余机制，于 2024 年在 libopus 1.5 中正式发布。

    与 LBRR 的区别：

    • LBRR 仅包含前 1 帧冗余，且冗余码率约为正常包的 2/3
    • DRED 最多可包含 1 秒或更长时间的音频特征冗余，但冗余码率仅约为正常包的 1/50
    • DRED 使用神经编码器提取声学特征，解码端利用这些特征通过神经声码器合成丢失的语音

    DRED 扩展已通过 IETF 草案标准化流程，并在 WebRTC 实现中逐步引入。

    8.3 内置 PLC（Packet Loss Concealment）

    Opus 解码器内置 PLC 能力，当检测到 RTP 序列号不连续时，调用 opus_decode(NULL, 0, ...) 触发 PLC：

    • CELT 模式 PLC：基于频谱外推，对音乐内容处理较好
    • SILK 模式 PLC：基于语音合成模型，对语音内容处理较好

    libopus 从 1.3 版本开始引入了神经网络辅助的 PLC 算法（NNLPC），在 1.5 版本中进一步完善。

    8.4 DTX（Discontinuous Transmission）

    DTX 是在静音期间停止发送 RTP 包以节省带宽的机制。Opus 内建 VAD，当检测到静音时，编码器可以大幅降低发送频率。

    • 通过 SDP usedtx=1 协商启用
    • DTX 在 WebRTC 中仅在 SILK/Hybrid 模式下工作（纯 CELT 流通常不开启 DTX）
    • 静音期间，Opus 生成低码率的舒适噪声（Comfort Noise），避免接收端感知到突然静音

    RFC 7874 明确指出：Since Opus provides its own CN mechanism, the use of [RFC 3389] CN with Opus is NOT RECOMMENDED.

    9. 变码率（VBR）与固定码率（CBR）

    Opus 默认使用 VBR（Variable Bitrate），根据信号复杂度动态分配码率：

    • 静音或简单音频：降低码率，减少带宽消耗
    • 复杂音频：增加码率，保证音质

    WebRTC 场景中，某些安全性要求高的应用（如端对端加密语音通话）可能使用 CBR 以防止通过码率变化泄露语音内容信息（防流量分析攻击）。RFC 6562（Guidelines for the Use of Variable Bit Rate Audio with Secure RTP）对此有详细讨论。通过 SDP cbr=1 可以请求对端使用 CBR。

    15. 小结

    Opus 不是一个普通的音频编解码器，它是互联网实时音频通信的基础设施：

    • 标准强制：RFC 7874 将其列为 WebRTC 必须实现的编解码器，所有浏览器原生支持
    • 技术先进：SILK + CELT 混合架构，覆盖 6–510 kbps 全码率范围，算法延迟最低 5 ms
    • 网络友好：内置 LBRR FEC、PLC、DTX，专为不可靠网络设计
    • 完全开放：免版税、开源（BSD 许可），无专利风险
    • 持续演进：从 libopus 1.0 到 1.5，不断引入神经网络 PLC（NNLPC）和深度冗余（DRED）等前沿技术

    理解 Opus 的内部机制，是深入掌握 WebRTC 音频栈的必要前提。在后续章节中，AEC 回声消除（第 22 章）、FEC 前向纠错（第 20 章）和 PLC 丢包隐藏（第 21 章）都与 Opus 的工作原理密切相关。
