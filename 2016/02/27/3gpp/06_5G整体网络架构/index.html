    1. 5G 网络分层总览与 CUPS 设计理念

    1.1 四大功能层次

    5G 系统（5GS，5G System）在逻辑上被划分为四个主要功能层次，它们协同工作，完整覆盖从终端接入到数据网络之间的全部网络功能。这种清晰的分层不仅便于工程实现，也是 5G 在灵活性、可扩展性上超越前几代系统的根本结构基础。

    （1）用户终端层（UE，User Equipment）

    UE 是 5G 系统的端点，它向网络发起接入请求、承载业务数据。UE 由两个逻辑组件构成：

    • ME（Mobile Equipment）：即手机或终端硬件本体，内嵌 NR 基带芯片，负责处理空口协议栈。
    • USIM（Universal Subscriber Identity Module）：存储用户身份（SUPI）、鉴权密钥（K）及相关安全参数，是运营商识别和鉴权用户的根信任锚。

    UE 通过空中接口 NR-Uu 与 NG-RAN 建立无线连接，并在其上运行 NAS（非接入层）协议与 5GC 进行信令交互。

    （2）接入网层（NG-RAN，Next Generation Radio Access Network）

    NG-RAN 是 5G 无线接入网，其核心网元是 gNB（gNodeB），另外还支持 ng-eNB（将 LTE eNodeB 升级后接入 5GC 的网元，用于 NSA 向 SA 过渡场景）。

    gNB 在内部实现了 CU/DU/RU 三层功能拆分：

    子单元 全称 主要功能
    CU-CP Central Unit – Control Plane RRC、PDCP-C，负责无线资源控制和切换决策
    CU-UP Central Unit – User Plane PDCP-U、SDAP，处理用户面数据包的安全与 QoS
    DU Distributed Unit RLC、MAC、低层 PHY，靠近天线部署
    RU/AAU Radio Unit / Active Antenna Unit 射频及物理层最低层处理，与天线阵列一体化

    NG-RAN 通过 N2 接口（控制面）连接 5GC 的 AMF，通过 N3 接口（用户面）连接 5GC 的 UPF。gNB 之间通过 Xn 接口互联，用于切换信令和数据转发。

    （3）5G 核心网层（5GC，5G Core Network）

    5GC 是整个系统的”大脑”，采用服务化架构（SBA），将传统的单体网元拆解为一组以微服务形式运行的网络功能（NF，Network Function），各 NF 通过统一的服务化接口（SBI）进行交互。

    5GC 在逻辑上严格区分控制面和用户面：

    • 控制面 NF：AMF、SMF、PCF、AUSF、UDM、UDR、NRF、NSSF、NEF、NWDAF 等——负责注册、鉴权、策略、会话管理等所有信令处理。
    • 用户面 NF：仅有 UPF（User Plane Function）——负责所有用户数据包的转发、QoS 执行、计费数据采集等。

    （4）传输网（Transport Network）

    传输网是连接上述各层的物理承载网络，包括前传（Fronthaul，DU↔RU）、中传（Midhaul，DU↔CU）和回传（Backhaul，CU↔5GC）三个段落。传输网对协议栈透明，但其时延、带宽和抖动指标直接影响 5G 空口性能（尤其 URLLC 场景），因此 3GPP 对不同功能拆分点的传输需求也有明确的定义（见 TS 38.801）。

    1.2 控制面与用户面分离（CUPS）

    CUPS（Control and User Plane Separation） 是 5G 架构最重要的设计原则之一。其思想起源于 3GPP Release 14 对 LTE EPC 的改进（将 SGW 和 PGW 各自的控制面与用户面分离），在 Release 15 的 5GC 中被彻底实施和标准化。

    CUPS 的核心逻辑

    在传统网络中，控制逻辑（如何转发）和数据转发（实际转发）耦合在同一网元中，扩容时必须整体扩容。CUPS 将这两者剥离：

    • SMF 是控制面的会话管理者：它不接触任何用户数据，但通过 N4 接口（使用 PFCP 协议——Packet Forwarding Control Protocol）向 UPF 下发包检测规则（PDR）、转发动作规则（FAR）、QoS 执行规则（QER）、使用报告规则（URR） 等指令，完全指挥 UPF 的行为。

    • UPF 是用户面的执行者：它只负责按照 SMF 下发的规则转发数据包，本身不做任何策略判断。UPF 可以部署在网络的任意位置，从集中式数据中心到靠近基站的边缘节点，而不需要改变控制面的逻辑。

    CUPS 带来的核心价值

    能力 说明
    弹性扩缩容 用户面流量增大时，仅水平扩展 UPF 即可，无需触碰 SMF
    MEC 下沉 UPF 可独立部署在靠近 gNB 的边缘站点，实现极低时延的本地分流
    UL-CL / Branching Point SMF 可插入上行分类器（UL Classifier）UPF，将不同流量分叉路由到不同数据网络
    多 UPF 链式部署 支持 I-UPF（中间 UPF）+ PSA-UPF（锚点 UPF）级联，N9 接口连接两者
    切片隔离 不同网络切片可使用独立的 UPF 实例，实现资源和流量的物理隔离

    用户面数据路径

    完整的用户面路径为：

    • N3：gNB 与第一个 UPF 之间，使用 GTP-U（GPRS Tunnelling Protocol for User Plane）隧道封装，每条 PDU Session 对应一条 GTP-U 隧道，由 TEID（Tunnel Endpoint Identifier）区分。
    • N9：UPF 与 UPF 之间，同样使用 GTP-U，用于 I-UPF 与 PSA-UPF 之间的数据传递，也用于 UE 移动过程中的路径切换。
    • N6：PSA-UPF 与外部数据网络之间，使用标准 IP 路由，可接入 Internet、运营商内网或 MEC 平台。
    • N4：SMF 与 UPF 之间的控制接口，使用 PFCP（TS 29.244 定义），不承载用户数据。

    1.3 5G 与 4G LTE 架构对比

    理解 5G 架构最直观的方式是与 LTE EPC 对比：

    维度 4G LTE EPC 5G 5GC
    架构风格 传统网元（MME / SGW / PGW / PCRF / HSS） 服务化架构（SBA），NF 微服务
    控制/用户面 软分离（R14 引入 CUPS，但非强制） 彻底硬分离，SMF vs UPF
    接口协议 GTPv2-C（控制面）/ GTP-U（用户面） HTTP/2 + JSON（SBI 控制面）/ GTP-U（用户面）
    NF 发现 静态配置或 DNS NRF 动态服务发现
    网络切片 不支持（DCN 有限支持） 原生支持，S-NSSAI / NSSF
    移动性 MME 集中管理 AMF 分布式，支持 RRC_INACTIVE
    会话管理 MME + SGW-C + PGW-C 协同 SMF 统一管理 PDU Session
    数据面锚点 PGW-U UPF（可多级链式）
    QoS 粒度 EPS Bearer（GBR / non-GBR） QoS Flow（5QI 参数化）

    2. 服务化架构（SBA）与接口命名全景

    2.1 SBA 核心设计哲学

    服务化架构（SBA，Service-Based Architecture） 是 5GC 控制面的根本设计范式，由 TS 23.501 第 4 章正式定义，并在 TS 29.500 中规定了其技术实现。

    SBA 借鉴了互联网微服务架构的思想：

    • 网络功能即服务提供者：每个 NF 对外暴露一组标准化的”NF 服务（NF Service）”，其他 NF 可以消费这些服务，无需点对点的专有接口。
    • REST over HTTP/2：所有 SBI（Service-Based Interface）均基于 HTTP/2 协议，消息体使用 JSON（OpenAPI 3.0 定义接口规范），完全采用 RESTful 风格，GET / POST / PUT / PATCH / DELETE / SUBSCRIBE / NOTIFY。
    • 去中心化：NF 之间可以直接调用（通过 NRF 发现对方），不再依赖集中式的信令总线。
    • 云原生友好：NF 可以以容器/Kubernetes Pod 形式部署，实现自动伸缩、滚动升级、故障自愈。

    SBA 中的架构可以用两种等价的视图来表达（TS 23.501 图 4.2.3-1 和 4.2.3-2）：

    1. 服务化接口视图（Service-Based Interface View）：所有 NF 围绕一条逻辑”总线”，每个 NF 在总线上注册并发布自己的服务，其他 NF 订阅/请求这些服务。
    2. 参考点视图（Reference Point View）：传统的 N1、N2、N3…参考点表示，更直观地体现点对点的逻辑关系和协议。

    这两种视图描述的是同一套系统，前者强调架构的服务化特性，后者便于具体接口和流程的描述。

    2.2 控制面网络功能（NF）概览

    以下是 5GC 控制面的核心 NF 及其职责（按 TS 23.501 §6 的定义）：

    AMF（Access and Mobility Management Function）——接入与移动性管理功能

    AMF 是 UE 接入 5GC 的第一个入口点，承担几乎所有与 UE 连接相关的控制职责：

    • 终结 NAS 信令（UE↔AMF，即 N1 接口上的 NAS 协议）
    • 处理 UE 注册（Registration）、去注册（Deregistration）
    • 管理 UE 的连接态（Connected）与空闲态（Idle）转换
    • 协调移动性管理（跟踪区更新、切换协调）
    • 将 PDU Session 请求转发给 SMF（N11 接口）
    • 与 AUSF 协作完成 5G-AKA / EAP-AKA’ 鉴权
    • 向 NG-RAN 下发 UE 上下文和 QoS 配置（N2 接口）
    • 承担寻呼（Paging）的触发角色

    一个 AMF 实例可以同时服务大量 UE，但每个 UE 在同一时刻只连接一个 AMF。

    SMF（Session Management Function）——会话管理功能

    SMF 是 5GC 中负责数据会话全生命周期管理的 NF：

    • 建立、修改、释放 PDU Session（PDU Session Establishment / Modification / Release）
    • 为 UE 分配 IP 地址（作为 DHCP 服务器或 IPAM 代理）
    • 选择并控制 UPF（通过 N4/PFCP 接口下发转发规则）
    • 与 PCF 交互（N7 接口）获取会话策略
    • 与 UDM 交互（N10 接口）获取会话管理订阅数据
    • 管理 QoS Flow 到 DRB 的映射参数协商
    • 支持计费数据的触发与上报（N40 接口连接 CHF）

    SMF 不接触任何用户数据，只负责控制面的会话逻辑。

    PCF（Policy Control Function）——策略控制功能

    PCF 是 5GC 的统一策略引擎，替代了 LTE 中的 PCRF：

    • 为 AMF 提供接入与移动性策略（AM Policy，N15 接口）
    • 为 SMF 提供会话级别的 QoS 和计费策略（SM Policy，N7 接口）
    • 与 AF 交互（N5 接口）接受应用层的策略请求（如 IMS 的 QoS 预留）
    • 通过 UDR 读取订阅的策略数据

    AUSF（Authentication Server Function）——鉴权服务功能

    AUSF 是 5G 鉴权的服务端：

    • 接受 AMF 的鉴权请求（N12 接口）
    • 与 UDM 交互（N13 接口）获取鉴权向量（Authentication Vectors）
    • 执行 5G-AKA 或 EAP-AKA’ 鉴权流程
    • 计算 KAUSF，并将 KSEAF 交给 AMF 用于后续密钥派生

    UDM（Unified Data Management）——统一数据管理功能

    UDM 是 5GC 的订阅数据管理中心，对应 LTE 中 HSS 的数据管理部分：

    • 存储和管理用户签约数据（接入权限、QoS 配置、切片权限等）
    • 生成 5G 鉴权向量（SUPI + K → AV）
    • 提供 SUCI 解密（还原 SUPI）
    • 管理 UE 的注册状态（AMF 注册、SMF 会话注册）
    • 通过 UDR 持久化存储数据

    UDM 是逻辑功能层，其数据实际存储在 UDR（Unified Data Repository） 中，UDM 通过 Nudr 接口访问 UDR。

    NRF（Network Repository Function）——网络功能仓库

    NRF 是 SBA 的服务注册与发现中心：

    • 维护所有 NF 实例的注册信息（NF Profile：类型、地址、容量、服务列表、切片支持等）
    • 响应 NF 的发现请求（NF Discovery），返回满足条件的 NF 实例列表
    • 监控 NF 的健康状态（心跳机制）
    • 支持 OAuth 2.0 鉴权（R16 引入）

    NSSF（Network Slice Selection Function）——网络切片选择功能

    NSSF 负责切片相关的选择决策：

    • 根据 UE 请求的 NSSAI 和订阅信息，确定允许的切片（Allowed NSSAI）
    • 为 UE 选择合适的 AMF 集合（基于切片能力）
    • 与 NRF 协作完成切片感知的 NF 发现

    NEF（Network Exposure Function）——网络能力开放功能

    NEF 是 5GC 对外开放能力的统一门户：

    • 将 5GC 内部的 NF 服务安全地暴露给第三方 AF（Application Function）和 MEC 应用
    • 提供北向 API，支持能力开放（如 QoS 预留、位置信息、事件订阅）
    • 负责外部请求的鉴权、限速和翻译

    NWDAF（Network Data Analytics Function）——网络数据分析功能

    NWDAF（R15 引入，R16/R17 持续增强）是 5G 网络内嵌的 AI/ML 分析引擎：

    • 汇聚来自各 NF 的网络数据
    • 输出分析结果（如负载预测、异常检测、UE 行为分析）
    • 供其他 NF（AMF、SMF、PCF 等）消费，驱动智能决策

    2.3 服务化接口（SBI）命名规则

    5GC 中，SBI 的命名遵循统一规则：

    N{NF名称} 表示该 NF 对外暴露的服务化接口，例如：

    NF SBI 名称 代表性服务
    AMF Namf Namf_Communication、Namf_EventExposure、Namf_Location
    SMF Nsmf Nsmf_PDUSession、Nsmf_EventExposure
    PCF Npcf Npcf_AMPolicyControl、Npcf_SMPolicyControl、Npcf_PolicyAuthorization
    AUSF Nausf Nausf_UEAuthentication、Nausf_SoRProtection
    UDM Nudm Nudm_SubscriberDataManagement、Nudm_UEAuthentication
    UDR Nudr Nudr_DataRepository
    NRF Nnrf Nnrf_NFManagement、Nnrf_NFDiscovery
    NSSF Nnssf Nnssf_NSSelection、Nnssf_NSSAIAvailability
    NEF Nnef Nnef_EventExposure、Nnef_PFDManagement
    PCF Npcf Npcf_UEPolicyControl
    NWDAF Nnwdaf Nnwdaf_AnalyticsInfo、Nnwdaf_EventsSubscription
    AF Naf Naf_EventExposure
    BSF Nbsf Nbsf_Management
    CHF Nchf Nchf_SpendingLimitControl、Nchf_ConvergedCharging

    注意：SBI 仅存在于 5GC 控制面 NF 之间。UPF 的 N4 接口、RAN 侧的 N1/N2/N3 接口不属于 SBI 范畴，因为它们不涉及 NF 间的服务调用关系。

    2.4 参考点接口全景（N1～N15 / Xn / F1 / E1 / NG）

    除了服务化接口视图，5G 系统同样可以用传统的参考点（Reference Point）视图描述，每个 Nx 参考点定义了两个节点之间的接口协议和消息规范。

    核心网 N 系列接口

    接口 连接双方 面 协议 主要功能
    N1 UE ↔ AMF（经 NG-RAN 透传） 控制 NAS（5GMM / 5GSM） UE 注册、鉴权、会话管理的 NAS 信令终结点
    N2 NG-RAN ↔ AMF 控制 NGAP（NG Application Protocol） RAN 与 5GC 的控制面接口，传递 UE 上下文、切换命令、寻呼等
    N3 NG-RAN ↔ UPF 用户 GTP-U over UDP/IP 无线接入网到核心网的用户面隧道
    N4 SMF ↔ UPF 控制（CUPS） PFCP（Packet Forwarding Control Protocol） SMF 向 UPF 下发/更新/删除转发规则（PDR/FAR/QER/URR）
    N5 PCF ↔ AF 控制 HTTP/2 + JSON（Npcf_PolicyAuthorization） AF 向 PCF 请求 QoS 策略（如 IMS 媒体流带宽预留）
    N6 UPF ↔ DN 用户 IP（标准路由） UPF 到外部数据网络（Internet / MEC / 企业内网）的接口
    N7 SMF ↔ PCF 控制 HTTP/2 + JSON（Npcf_SMPolicyControl） SMF 向 PCF 请求/更新 PDU Session 策略
    N8 AMF ↔ UDM 控制 HTTP/2 + JSON（Nudm_UECM / Nudm_SDM） AMF 获取接入与移动性相关的订阅数据
    N9 UPF ↔ UPF 用户 GTP-U over UDP/IP 中间 UPF（I-UPF）与锚点 UPF（PSA-UPF）之间的用户面隧道；也用于漫游 HR 场景（V-UPF ↔ H-UPF）
    N10 SMF ↔ UDM 控制 HTTP/2 + JSON（Nudm_SDM / Nudm_UECM） SMF 获取会话管理相关的订阅数据，并注册会话信息
    N11 AMF ↔ SMF 控制 HTTP/2 + JSON（Namf_Communication / Nsmf_PDUSession） AMF 触发 SMF 的 PDU Session 操作（建立/修改/释放）
    N12 AMF ↔ AUSF 控制 HTTP/2 + JSON（Nausf_UEAuthentication） AMF 向 AUSF 发起鉴权请求
    N13 AUSF ↔ UDM 控制 HTTP/2 + JSON（Nudm_UEAuthentication） AUSF 从 UDM 获取鉴权向量
    N14 AMF ↔ AMF 控制 HTTP/2 + JSON（Namf_Communication） AMF 间的 UE 上下文转移（切换、AMF 重选场景）
    N15 AMF ↔ PCF 控制 HTTP/2 + JSON（Npcf_AMPolicyControl） AMF 向 PCF 请求接入与移动性策略（如 UE 路由选择策略 URSP）

    N5～N15 中的 N5、N7、N8、N10、N11、N12、N13、N14、N15 在服务化视图中均对应 SBI，而 N1、N2、N3、N4、N6、N9 是传统参考点（无对应 SBI），因为它们涉及 UE / RAN / UPF，不属于纯 5GC NF 间的服务调用。

    NG-RAN 侧接口

    接口 连接双方 说明
    NG gNB / ng-eNB ↔ 5GC NG 是 RAN 与核心网之间的接口族，逻辑上包含 N2（控制面，终结于 AMF）和 N3（用户面，终结于 UPF）。NG 接口在 TS 38.413（NGAP）和 TS 29.281（GTP-U）中定义。
    Xn gNB ↔ gNB / ng-eNB 基站间接口，用于切换信令（XnAP 协议，TS 38.423）和直接数据转发（XnU，GTP-U）。支持 Xn 切换可避免数据经过核心网，降低切换时延。
    F1 CU ↔ DU gNB 内部接口，CU-CP 与 DU 之间的控制面（F1-C，F1AP 协议，TS 38.473），以及 CU-UP 与 DU 之间的用户面（F1-U，GTP-U，TS 38.474）。
    E1 CU-CP ↔ CU-UP gNB 内部接口，控制面单元与用户面单元之间的控制信令（E1AP，TS 38.463），用于 CU-CP 管理 CU-UP 上的 PDCP 承载配置。

    2.5 NRF 服务发现机制

    NRF 是 SBA 正常运转的基础设施，其工作流程如下：

    ① NF 注册（NF Registration）

    NF 启动后，通过调用 Nnrf_NFManagement_NFRegister 向 NRF 注册自己的 NF Profile，内容包括：

    • NF 类型（AMF / SMF / PCF …）
    • NF 实例 ID（UUID）
    • FQDN / IP 地址 + 端口
    • 支持的 NF 服务列表及版本
    • 承载的 S-NSSAI（切片标识）
    • PLMN ID
    • 服务区域（TAI List / 地理范围）
    • 容量 / 优先级

    NF 还需定期发送心跳（Nnrf_NFManagement_NFUpdate），若心跳超时则 NRF 标记该实例为不可用。

    ② NF 发现（NF Discovery）

    当 NF-A 需要调用 NF-B 的服务时：

    1. NF-A 向 NRF 发送 Nnrf_NFDiscovery_Request，携带目标 NF 类型、所需服务名称、S-NSSAI、PLMN ID 等过滤条件。
    2. NRF 返回匹配的 NF 实例列表（包含地址和服务端点）。
    3. NF-A 根据返回的列表（考虑优先级、容量、就近原则）选择一个实例，直接调用其 SBI。

    ③ OAuth 2.0 授权（R16 引入）

    R16 起，NF 间的 SBI 调用需携带 NRF 签发的 Access Token（JWT 格式），NRF 充当 Authorization Server 角色，确保只有授权的 NF 才能消费特定服务，防止未授权访问。

    3. 漫游架构与网络切片

    3.1 漫游基础：HPLMN vs VPLMN

    当 UE 离开其签约运营商（HPLMN，Home Public Land Mobile Network）的覆盖区域，进入另一个运营商（VPLMN，Visited Public Land Mobile Network）的网络时，就会触发漫游场景。

    5G 漫游建立在运营商间签订漫游协议的基础上，技术实现遵循 TS 23.501 §4.2 中的漫游架构定义。漫游时的主要问题是：

    • 注册：UE 通过 VPLMN 的接入网注册，但身份鉴权必须依赖 HPLMN 的 UDM / AUSF。
    • 数据会话：用户数据流量的锚定网络是 HPLMN 还是 VPLMN？这产生了两种主要的漫游架构。
    • 切片映射：HPLMN 的 S-NSSAI 与 VPLMN 的 S-NSSAI 可能不同，需要映射。

    在任何漫游场景中，以下规则始终成立：

    • V-AMF 处于 VPLMN，服务 UE 的接入和移动性管理。
    • UDM / AUSF 始终位于 HPLMN（因为用户身份和密钥存储在归属网络）。
    • 具体的 SMF 和 UPF 位置取决于漫游模式（HR 或 LBO）。

    3.2 Home-Routed（HR）漫游

    在 Home-Routed（归属路由）漫游 中，用户数据最终通过 HPLMN 的用户面网关出口，即：

    控制面：

    关键特征：

    • 有两个 SMF：V-SMF（VPLMN）作为会话锚点并管理本地 RAN/UPF，H-SMF（HPLMN）负责最终的策略决定和数据网络接入，两者通过 N16 接口（Nsmf_PDUSession 的跨 PLMN 变体）协调。
    • 有两个 UPF：V-UPF 终结 N3（RAN↔UPF），H-UPF 是 PDU Session 锚点，负责 N6 出口。两个 UPF 通过 N9 接口（跨 PLMN 的 GTP-U 隧道）连接。
    • 计费：V-SMF 连接 V-CHF（Visited Charging Function）生成访问方 CDR；H-SMF 连接 H-CHF 生成归属方 CDR。
    • IMS 语音（VoNR）在漫游场景下强制使用 HR 模式（N9 Home Routing，N9HR），因为 IMS 基础设施部署在 HPLMN。

    适用场景： 运营商希望保持对用户数据的完全控制（计费完整性、合规性），以及 IMS 语音业务。

    3.3 Local Breakout（LBO）漫游

    在 Local Breakout（本地分流）漫游 中，用户数据直接通过 VPLMN 的本地 UPF 出口，不回传至 HPLMN：

    控制面：

    关键特征：

    • 只有一个 SMF：V-SMF，位于 VPLMN，完全管理 PDU Session。
    • 只有一个 UPF：V-UPF，位于 VPLMN，直接通过 N6 出口到数据网络（如 Internet）。
    • V-SMF 仍需通过 N10 接口访问 HPLMN 的 UDM 获取用户订阅数据。
    • V-PCF 可与 H-PCF（HPLMN）通过 N24 接口交换策略信息（可选）。
    • 用户数据不经过 HPLMN，路径更短，时延更低。
    • HPLMN 对用户数据流量的可见性较低。

    适用场景： 对延迟敏感的业务（游戏、视频通话、流媒体）；运营商协议中允许 VPLMN 本地处理数据流量时。

    SMF 位置 V-SMF + H-SMF（跨 PLMN N16） V-SMF（仅 VPLMN）

    3.5 网络切片基础概念

    网络切片（Network Slicing） 是 5G 的标志性特性，允许在同一套物理基础设施上构建多个逻辑上相互独立的虚拟网络，每个虚拟网络（即一个切片）具有定制化的网络能力和性能特征，服务于特定的业务场景或客户群体。

    网络切片的概念贯穿三个层次（TS 28.530 定义）：

    3GPP 定义了三类标准化的切片/服务类型（SST，Slice/Service Type）：

    SST 值 名称 对应场景
    1 eMBB 增强移动宽带（高带宽，如视频流、AR/VR）
    2 URLLC 超可靠低时延通信（工业控制、远程手术）
    3 MIoT 大规模物联网（海量低功耗设备连接）
    4 V2X 车联网（车辆间通信）

    运营商也可以定义私有的 SST 值（128~255），配合 SD（Slice Differentiator）组合成完整的 S-NSSAI。

    3.6 S-NSSAI 与 NSSAI 详解

    S-NSSAI（Single Network Slice Selection Assistance Information） 是标识一个具体网络切片的信息元素，由两部分组成（TS 23.501 §5.15.2）：

    • SST（Slice/Service Type）：标识切片的服务类型（1=eMBB，2=URLLC，3=MIoT 等）。
    • SD（Slice Differentiator）：在同一 SST 下进一步区分不同切片实例（如区分企业专线切片 vs 公共 eMBB 切片）。

    S-NSSAI 的取值与 PLMN 绑定（HPLMN S-NSSAI vs VPLMN S-NSSAI），漫游时可能需要映射。

    NSSAI（Network Slice Selection Assistance Information） 是 S-NSSAI 的集合，有以下几种：

    NSSAI 类型 说明
    Requested NSSAI UE 在注册时向网络请求的切片集合（由 UE 根据 URSP 策略构建）
    Allowed NSSAI 网络（AMF，经 NSSF 决策）批准 UE 在当前 PLMN 中可以使用的切片集合
    Configured NSSAI 运营商预先为 UE 配置的切片集合（存储在 UE 中，按 PLMN 分组）
    Rejected NSSAI 网络拒绝的切片集合及原因（如不支持、未签约、无资源等）
    Pending NSSAI 需要额外注册（Network Slice-Specific Authentication）才能使用的切片

    UE 最多可同时注册在 8 个切片（Allowed NSSAI 中最多 8 个 S-NSSAI），但实际业务数量受切片资源和签约限制。

    3.8 网络切片实例（NSI）端到端架构

    一个完整的 NSI 在端到端架构上包含以下组件：

    NF 共享 vs 专属的设计原则（TS 23.501 §5.15.4）：

    • AMF：原则上可被多个切片共享（一个 AMF 实例服务多个 S-NSSAI 上的 UE），但 AMF 需支持所有相关切片的 NAS 处理。
    • SMF、PCF、UPF：通常为每个切片部署专属实例，以实现资源隔离和独立的策略控制——这是切片 SLA 保障的核心。
    • UDM、NRF、AUSF：通常在全网共享（跨切片统一管理订阅数据和服务发现）。

    切片隔离级别：

    隔离级别 描述
    软隔离 NF 实例共享，但逻辑上通过参数区分（最低隔离，资源效率最高）
    NF 实例隔离 SMF/UPF 等关键 NF 独立部署，控制面与用户面隔离（推荐实践）
    硬件隔离 专用物理服务器/网卡，用于政企专线、URLLC 等严格 SLA 场景

    3.9 漫游场景下的网络切片

    漫游与切片结合时，S-NSSAI 的处理变得更为复杂（TS 23.501 §5.15.4.1）：

    S-NSSAI 映射机制：

    HPLMN 为 UE 分配了 HPLMN S-NSSAI = {SST=1, SD=AAAAAA}（eMBB 切片），而 VPLMN 有自己的切片标识体系 VPLMN S-NSSAI = {SST=1, SD=111111}。

    漫游时，网络需要在 HPLMN S-NSSAI 和 VPLMN S-NSSAI 之间建立映射关系（Mapping of NSSAI）：

    1. UE 在注册时的 Requested NSSAI 中携带 HPLMN S-NSSAI，同时在 mapping 字段中提供对应的 VPLMN S-NSSAI（若 UE 已知该映射）。
    2. V-AMF 向 V-NSSF 查询，V-NSSF 返回 VPLMN 侧的 Allowed NSSAI 及对应的 Mapping of Allowed NSSAI（告知每个 VPLMN S-NSSAI 对应哪个 HPLMN S-NSSAI）。
    3. 在 HR 漫游中，V-SMF 使用 VPLMN S-NSSAI，H-SMF 使用 HPLMN S-NSSAI，两者在 N16 信令中协调映射。
    4. 在 LBO 漫游中，V-NSSF 确定 VPLMN 侧的切片选择；V-SMF 通过 N10 接口访问 HPLMN 的 UDM 时，传递 HPLMN S-NSSAI 以便 UDM 查询签约数据。

    漫游切片限制：

    VPLMN 可能不支持 HPLMN 的所有切片类型（如 HPLMN 的 URLLC 切片在某 VPLMN 无对应），此时网络会通过 Rejected NSSAI 告知 UE，UE 只能使用 VPLMN 支持的切片（如回落到 eMBB 切片）。

    4. 总结

    本文从三个维度系统梳理了 5G 整体网络架构：

    网络分层与 CUPS： 5G 系统由 UE、NG-RAN、5GC、传输网四层构成。5GC 内部的 CUPS 设计将 SMF（控制面）和 UPF（用户面）彻底解耦，用户数据路径（UE→gNB→N3→UPF→N6→DN）与控制信令路径（UE→N1→AMF→N11→SMF→N4→UPF）完全独立，为 MEC 下沉、弹性扩容和流量分叉提供了坚实的架构基础。

    SBA 与接口体系： 5GC 控制面采用服务化架构，AMF、SMF、PCF、AUSF、UDM、NRF、NSSF 等 NF 通过基于 HTTP/2 + JSON 的 SBI（Namf / Nsmf / Npcf …）进行解耦调用，NRF 提供动态服务注册与发现。同时，N1～N15 参考点体系为各接口的协议和流程提供了精确的定义，NG / Xn / F1 / E1 则覆盖了 RAN 侧内部与外部的连接需求。

    漫游与网络切片： 漫游架构分为 HR（归属路由，HPLMN 控制数据出口）和 LBO（本地分流，VPLMN 本地出口）两种模式，前者保障运营商管控能力，后者降低用户时延。网络切片通过 S-NSSAI 标识、NSSF 选择、以及端到端专属 NF 实例，实现了在同一物理网络上提供 eMBB / URLLC / MIoT 等差异化服务的能力，是 5G 实现 B2B 业务多样化的核心技术支撑。
