    1. MAC 子层概述

    MAC（Medium Access Control，媒体接入控制）是 5G NR Layer 2 协议栈的最底层子层，位于 RLC 之下、物理层（PHY）之上。其核心职责由 3GPP TS 38.321 规范定义，主要包括：

    功能域 具体功能
    信道映射 逻辑信道 ↔ 传输信道的映射与复用
    调度支撑 BSR（缓冲状态上报）、SR（调度请求）、PHR（功率余量上报）
    随机接入 RACH 过程控制（Msg1Msg4 / msgAmsgB）
    HARQ 停等协议管理、软合并协调、ACK/NACK 反馈
    时间对齐 定时提前（Timing Advance）维护
    SCell 管理 辅载波（SCell）的激活与去激活
    BWP 切换 带宽部分（Bandwidth Part）切换控制

    MAC 子层为上层（RLC）提供逻辑信道服务，为下层（PHY）提供传输信道接口。其核心角色可概括为：**”将来自多个逻辑信道的数据按优先级复用进传输块，并通过 HARQ 保障传输可靠性”**。

    2. 信道体系：逻辑信道与传输信道

    2.1 逻辑信道（Logical Channel）

    逻辑信道是 RLC 与 MAC 之间的服务访问点（SAP），按所承载信息的类型划分：

    控制信道（Control Channels）

    信道名称 英文全称 方向 用途
    BCCH Broadcast Control Channel DL 广播系统信息（MIB / SIB）
    PCCH Paging Control Channel DL 寻呼消息
    CCCH Common Control Channel UL/DL RRC 连接建立前的公共控制信令
    DCCH Dedicated Control Channel UL/DL UE 专用控制信令（RRC 消息）

    业务信道（Traffic Channel）

    信道名称 英文全称 方向 用途
    DTCH Dedicated Traffic Channel UL/DL 用户面数据（SDAP/PDCP SDU）

    注：与 LTE 相比，NR 中取消了 MCCH / MTCH（多播相关信道），在 R17 之前不支持广播多播业务（后续 Release 中作为增强功能引入）。

    2.2 传输信道（Transport Channel）

    传输信道是 MAC 与 PHY 之间的服务访问点（SAP），按传输方式和物理特性划分：

    下行传输信道

    信道名称 英文全称 对应物理信道 用途
    BCH Broadcast Channel PBCH 承载 MIB，固定编码格式，覆盖全小区
    DL-SCH Downlink Shared Channel PDSCH 主要下行数据承载，支持 HARQ、动态调度、波束赋形
    PCH Paging Channel PDSCH 承载寻呼消息，支持 DRX

    上行传输信道

    信道名称 英文全称 对应物理信道 用途
    UL-SCH Uplink Shared Channel PUSCH 主要上行数据承载，支持 HARQ、动态/配置授权
    RACH Random Access Channel PRACH 随机接入前导码

    3. MAC PDU 结构

    MAC Control Element（MAC CE）

    MAC CE 是 MAC 层内嵌的控制信息，通过 LCID 的特殊取值标识。常见 MAC CE 类型：

    MAC CE LCID 值 方向 说明
    C-RNTI 011100 UL 上行竞争解决，携带 UE 的 C-RNTI
    BSR（Short） 111101 UL 短格式缓冲状态上报
    BSR（Long） 111110 UL 长格式缓冲状态上报
    PHR（Single） 111001 UL 单小区功率余量上报
    Timing Advance Command 000001 DL gNB 下发定时提前命令
    DRX Command 000010 DL 激活 DRX
    TCI State Indication 000101 DL 下行 TCI 状态指示（R15+）
    SP ZP CSI-RS RES SET ACT/DEACT 多值 DL 半持久 CSI-RS 资源集激活/去激活

    TS 38.321 V18.5.0 中已定义超过 50 种 MAC CE 类型，覆盖资源分配、同步、安全激活、多 TRP 协作等全方位场景。

    优先级排序：在 MAC PDU 组装时，MAC CE 按 LCID 优先级排序，高优先级 MAC CE 优先组装。

    4. 逻辑信道优先级与复用（LCP）

    LCP（Logical Channel Prioritization）过程是 MAC 上行复用的核心机制，解决以下问题：当 gNB 为 UE 分配了上行资源（UL Grant）时，UE 如何决定将哪些逻辑信道的数据填入这次传输机会？

    规范参考：TS 38.321 Section 5.4.3

    4.1 优先级参数配置

    gNB 通过 RRC 信令（LogicalChannelConfig）为每个逻辑信道配置以下参数：

    参数 含义 取值范围
    priority 逻辑信道优先级（数值越小优先级越高） 1（最高）~ 16（最低）
    prioritisedBitRate（PBR） 优先保障的比特率（令牌桶填充速率） 0, 8, 16, 32, 64, 128, 256, 512 kbps, infinity
    bucketSizeDuration（BSD） 令牌桶持续时长，决定桶的容量上限 5, 10, 20, 50, 100, 150, 300, 500, 1000 ms

    逻辑信道组（LCG，Logical Channel Group）：每个逻辑信道可被 RRC 分配到 LCG 0~7 中的某个组，用于 BSR 上报（8 个 LCG）。

    4.2 令牌桶算法（Token Bucket）

    LCP 采用令牌桶（Token Bucket）机制实现速率保障：

    • B_i > 0：逻辑信道 i 处于”有保障配额”状态
    • B_i ≤ 0：保障配额已耗尽，进入次优先级竞争

    4.3 LCP 算法执行步骤

    当 UE 收到 UL Grant（大小为 TBS 字节），执行如下 LCP 流程：

    第一轮（优先保障）：

    1. 对所有满足映射限制的逻辑信道，按 priority 从高到低排列
    2. 对每个 B_i > 0 的逻辑信道，尽量从其缓冲区取数据填入 MAC SDU，直到：
      • 该信道的 B_i 被耗尽，或
      • 该信道的缓冲区清空，或
      • 传输块空间不足
    3. 填完后扣减对应 B_i

    第二轮（剩余空间竞争）：

    1. 若传输块仍有剩余空间，继续按 priority 从高到低服务所有有数据的逻辑信道（不考虑 B_i 是否为正），直到传输块填满

    核心原则：PBR > 0 保证了”有保障速率”的逻辑信道（如 VoNR 语音 DRB）始终能在令牌范围内得到服务；超出 PBR 的部分则按纯优先级竞争剩余资源。PBR = infinity 意味着该信道可耗尽全部资源。

    4.4 映射限制（Mapping Restriction）

    5G NR 相比 LTE 引入了更细粒度的逻辑信道映射限制，通过 RRC 为每个逻辑信道配置：

    限制参数 说明
    allowedSCS-List 允许使用的子载波间隔（Numerology）
    maxPUSCH-Duration 允许的最大 PUSCH 持续时长
    configuredGrantType1Allowed 是否允许使用 Configured Grant Type 1 调度
    allowedServingCells 允许传输的载波小区集合
    allowedCG-List 允许使用的 Configured Grant 列表
    allowedPHY-PriorityIndex 允许的动态授权 PHY 优先级索引

    这些限制允许网络将不同 SCS（子载波间隔）的 PUSCH 资源与特定业务的逻辑信道严格绑定，例如 URLLC 业务强制使用小 TTI（μ=3，0.125ms slot）。

    4.5 调度请求（SR）与缓冲状态上报（BSR）

    调度请求（Scheduling Request，SR）：

    当 UE 上行缓冲区从空变为非空（有新数据），且当前没有 UL Grant 时，UE 通过 PUCCH 发送 SR，请求 gNB 为其分配上行资源。

    • RRC 为每个 SR 配置 sr-ProhibitTimer（防止 SR 过于频繁）和 sr-TransMax（最大重传次数）
    • 若 SR 发送达到上限仍未获得 Grant，则触发 RLF（Radio Link Failure）处理

    缓冲状态上报（Buffer Status Report，BSR）：

    BSR 告知 gNB 当前 UE 上行各逻辑信道组的缓冲数据量，供 gNB 调度决策使用。TS 38.321 定义了 4 种 BSR 格式：

    BSR 类型 触发条件 格式说明
    Regular BSR 有新数据到达且属于更高优先级 LCG 上报单个或多个 LCG
    Periodic BSR periodicBSR-Timer 超时 周期上报所有 LCG
    Padding BSR MAC PDU 中有填充（Padding）空间 利用填充空间上报
    Short Truncated BSR 多个 LCG 有数据但空间不足 只上报最高优先级 LCG

    BSR 字段结构：

    • Short BSR / Short Truncated BSR：8 bits，其中 3 bits = LCG ID，5 bits = 缓冲大小索引（0~31，对应 0 ~ 150,000 bytes）
    • Long BSR / Long Truncated BSR：包含所有有数据的 LCG（由 LCGi 位图标识），每个 LCG 8 bits 缓冲大小索引（0~255，颗粒度更高）

    5. HARQ 机制

    HARQ（Hybrid Automatic Repeat reQuest，混合自动重传请求）是 5G NR 可靠传输的基石，结合了 前向纠错（FEC） 与 自动重传（ARQ） 的优势。规范参考：TS 38.321（MAC 层 HARQ 过程管理）+ TS 38.214（物理层软合并实现）。

    5.1 ARQ / HARQ / FEC 的关系

    在 5G 协议栈中，重传由两层机制联合保障：

    • HARQ（MAC 层）：快速重传（亚毫秒级时延感知），残留错误率约 0.1%~1%
    • ARQ（RLC 层）：处理 HARQ 无法纠正的残余错误，实现近乎无损传输

    5.2 停等协议（Stop-and-Wait）与多进程并行

    最简单的 HARQ 实现是停等（SAW，Stop-And-Wait）协议：发送端发出一个 TB（Transport Block），等待接收端反馈 ACK/NACK 后再发下一个。

    问题：等待反馈期间发送端完全空闲，信道利用率极低。

    解决方案：多进程并行 HARQ（Multi-Process SAW）：

    5G NR HARQ 进程数（TS 38.321 & TS 38.214）：

    方向 最大 HARQ 进程数 说明
    下行（DL-SCH） 16 由 nrofHARQ-ProcessesForPDSCH（RRC）配置，默认 8
    上行（UL-SCH） 16 由 DCI 中的 HARQ 进程号字段（4 bits）指示，最多 16 个

    NR 将 LTE 的最大 8 个进程扩展到 16 个，以适应更大的 RTT（特别是 FR2 毫米波和 TDD 场景）。

    每个 HARQ 进程独立维护：

    • 进程状态（等待 ACK / 空闲）
    • 软缓冲区（存储已接收但未能解码的软比特）
    • 当前冗余版本（RV）

    5.3 软合并（Soft Combining）原理

    HARQ 的精髓在于软合并：接收端不丢弃错误接收的 TB，而是将其存入软缓冲区（Soft Buffer），与后续重传版本在比特域（LLR 域）进行叠加。

    软合并不要求每次传输完全相同——这正是 Chase Combining 与 Incremental Redundancy 的分叉点。

    5.4 Chase Combining（CC）

    原理：每次重传发送与初传完全相同的比特集合（使用相同的打孔模式），接收端对相同位置的 LLR 直接累加。

    优点：

    • 实现简单，UE 软缓冲区需求固定（等于一个 TB 大小）
    • 接收端处理复杂度低

    缺点：

    • 频谱效率较低：重传没有携带任何新的编码信息
    • 增益仅来自功率累积（分集增益），没有编码增益

    5.5 Incremental Redundancy（IR）

    原理：每次重传采用不同的打孔模式（不同 RV），发送不同的冗余比特，接收端将历次传输的比特拼接后获得更完整的编码字。

    优点：

    • 同时获得分集增益 + 编码增益，综合性能优于 CC
    • 码率随重传次数下降，自适应信道质量

    缺点：

    • 软缓冲区需求更大（需存储多次不同 RV 的 LLR）
    • 接收端复杂度略高

    5G NR 的选择：NR 标准中实际以 IR 为主，CC 可视为 IR 在 RV 固定为 0 时的特殊情形。

    5.6 冗余版本（Redundancy Version，RV）

    5G NR LDPC 编码定义了 4 个冗余版本：RV0、RV1、RV2、RV3，对应母码中不同的起始打孔位置。

    RV 起始位置（环形缓冲区） 特性
    RV0 0 自解码版本，优先包含系统位，初传通常使用
    RV2 ~1/3 处 重传常用
    RV3 ~1/2 处 重传常用
    RV1 ~1/4 处（LDPC 特有） 重传使用

    RV 序列的典型调度顺序：RV0 → RV2 → RV3 → RV1 → RV0（循环）

    RV0 是自解码（self-decodable）版本，意味着即使没有先前传输的软比特，仅凭 RV0 本身也可能成功解码（取决于信道质量）。其他 RV 在与 RV0 合并后才能有效发挥作用。

    5.7 HARQ 反馈：NDI / ACK / NACK

    NDI（New Data Indicator，新数据指示）：

    NDI 是 DCI（下行控制信息）中的 1-bit 字段，用于区分初传与重传：

    • NDI 翻转（Toggle）：表示这是新的传输（新 TB），接收端清空该进程的软缓冲区
    • NDI 不变：表示这是重传，接收端将新收到的 LLR 与缓冲区中的旧 LLR 合并

    HARQ-ACK 反馈：

    接收端通过 PUCCH（或 PUSCH 捎带）向发送端反馈 ACK/NACK：

    • ACK：CRC 校验通过，TB 解码成功，发送端可释放该进程
    • NACK：CRC 校验失败，请求重传

    DTX（Discontinuous Transmission）：gNB 收不到 UE 的 HARQ 反馈时的特殊状态，可能触发重传。

    5.8 下行 HARQ 与上行 HARQ 的异同

    维度 下行 HARQ 上行 HARQ
    定时方式 异步（Asynchronous） 异步（Asynchronous，NR 与 LTE 不同）
    软缓冲区位置 UE 端 gNB 端
    重传调度 gNB 通过 DCI 调度 gNB 通过 DCI（UL grant）调度
    HARQ 反馈 UE → gNB（via PUCCH/PUSCH） gNB → UE（via PDCCH，PHICH 已废弃）
    CBG 支持 支持（CBGTI + CBGFI） 支持（CBGTI，无 CBGFI）
    进程数 最多 16（RRC 配置） 最多 16（DCI 4 bits 指示）

    NR vs LTE 关键差异：LTE 上行 HARQ 是同步的（固定 8ms RTT，进程号隐含在时序中）；NR 上行改为异步，进程号由 DCI 显式携带，使得调度更灵活，更适应 TDD 的多样时隙配置。

    5.9 码块组重传（CBG-based Retransmission）

    当一个 TB 过大时，会被分割为多个码块（Code Block，CB），若干 CB 组成一个码块组（Code Block Group，CBG）。

    CBG 重传的动机：若仅有少数 CB 解码失败，全 TB 重传浪费大量无线资源。CBG 重传只需重发失败的 CBG，大幅提升频谱效率。

    配置：通过 RRC codeBlockGroupTransmission 配置，每个 TB 可包含 2、4、6 或 8 个 CBG。

    下行 CBG 相关 DCI 字段（TS 38.214）：

    字段 说明
    CBGTI（Code Block Group Transmission Information） 位图，指示哪些 CBG 在本次传输中出现（1=传输，0=不传输）
    CBGFI（Code Block Group Flushing Information） 1 bit，指示是否清空软缓冲区（1=清空，0=合并）

    上行 CBG：上行也支持 CBGTI（gNB 告知 UE 重传哪些 CBG），但无需 CBGFI（软缓冲区在 gNB 侧，gNB 自行决定是否清空）。

    5.10 HARQ 定时关系

    HARQ 定时关系在 NR 中由 TS 38.213 定义，通过以下参数控制：

    PDSCH-to-HARQ 反馈时序（PDSCH-to-HARQ_feedback timing indicator）：

    UE 收到 PDSCH 后，在 K1 个 slot 后通过 PUCCH 发送 HARQ-ACK。K1 的取值集合（dl-DataToUL-ACK）由 RRC 配置，典型值为 {1, 2, 3, 4, 5, 6, 7, 8}。

    PUSCH 调度时序（PDCCH 到 PUSCH 的时延 K2）：

    gNB 发送 UL Grant（DCI format 0_x）后，UE 在 K2 个 slot 后在 PUSCH 上传输数据，K2 的最小值由 Numerology 决定。

    NR 灵活定时的优势：

    • NR 的 HARQ 定时完全灵活可配，使得 URLLC 场景可使用极小的 K1（如 1 slot），最小化反馈延迟
    • TDD 场景下，K1/K2 的灵活配置可避免 UL/DL 时隙冲突

    6. MAC 与 RLC 的协作：ARQ + HARQ 双层重传

    5G NR 采用双层重传架构，MAC 层 HARQ 与 RLC 层 ARQ 分工明确：

    维度 HARQ（MAC 层） ARQ（RLC 层）
    目标错误率 0.1%1%（BLER 目标） 接近 0%（近无损）
    重传粒度 Transport Block / CBG RLC SDU / 分段
    重传延迟 亚毫秒~数毫秒 数毫秒~数十毫秒
    重传次数 通常 ≤ 4 次（可配置） 可多次（窗口机制）
    软合并 核心特性 不支持

    7. 总结与关键参数速查

    逻辑信道与传输信道速查

    逻辑信道 类型 方向 → 传输信道 → 物理信道
    BCCH 控制 DL BCH / DL-SCH PBCH / PDSCH
    PCCH 控制 DL PCH PDSCH
    CCCH 控制 UL/DL UL-SCH / DL-SCH PUSCH / PDSCH
    DCCH 控制 UL/DL UL-SCH / DL-SCH PUSCH / PDSCH
    DTCH 业务 UL/DL UL-SCH / DL-SCH PUSCH / PDSCH

    LCP 关键参数

    参数 作用 配置位置
    priority（1~16） 决定信道服务顺序 RRC LogicalChannelConfig
    prioritisedBitRate 令牌桶填充速率，保障最低速率 RRC LogicalChannelConfig
    bucketSizeDuration 令牌桶容量 = PBR × BSD RRC LogicalChannelConfig
    allowedSCS-List 绑定特定 Numerology RRC LogicalChannelConfig
    LCG 0~7 BSR 上报分组 RRC LogicalChannelConfig

    HARQ 关键参数

    参数 说明 典型值
    HARQ 进程数（DL） RRC 配置，最大 16 8（默认）
    HARQ 进程数（UL） DCI 4-bit 指示 最大 16
    RV 序列 初传与重传的冗余版本 0→2→3→1
    K1（PDSCH→HARQ-ACK） UE 发送 ACK 的时延（slot 数） 1~8，RRC 配置
    K2（UL Grant→PUSCH） UE 发送 PUSCH 的时延（slot 数） Numerology 相关
    CBG 数量 每 TB 的码块组数 2/4/6/8，RRC 配置
    最大重传次数 HARQ 最大重传上限 通常 4（可调）

    5G NR HARQ vs LTE HARQ 关键差异

    特性 LTE 5G NR
    DL HARQ 定时 异步 异步
    UL HARQ 定时 同步 异步
    UL HARQ 进程数 8（FDD）/最多 8（TDD） 最多 16
    CBG 重传 不支持 支持（R15+）
    HARQ 反馈信道 PUCCH + PHICH（UL ACK 用 PHICH） PUCCH（PHICH 废弃）
    最小重传时延 8ms（FDD） 灵活，取决于 Numerology 和 K1
