    1. 控制面协议栈总览

    5G 控制面协议栈贯穿 UE、gNB 到核心网，承载着网络中所有的管理信令。从整体视角看，控制面分为两大平面：

    关键原则：

    • NAS（Non-Access Stratum）：终止于核心网，UE ↔ AMF（5GMM）或 UE ↔ SMF（5GSM），对 RAN 透明。
    • RRC（Radio Resource Control）：终止于 gNB，负责 UE 与基站之间的空口信令控制。
    • RRC 通过 DL/UL Information Transfer 消息透传 NAS PDU，实现 NAS 信令的端到端传递。
    • 控制面与用户面分离（CUPS）是 5G 架构的核心设计原则，体现在 SMF 控制 UPF 进行用户面转发。

    2. NAS 层（非接入层）

    2.1 NAS 在 5G 架构中的位置

    根据 3GPP TS 24.501，NAS 是 UE 与 5G 核心网之间的最高控制层，通过 N1 接口（参考点）传递信令。NAS 对接入网（NG-RAN）完全透明，gNB 只负责封装和转发 NAS 消息，不解析其内容。

    NAS 层的主要功能包括：

    功能类别 具体内容
    注册管理（RM） 初始注册、移动性注册更新、周期性注册、注销
    连接管理（CM） N1 NAS 信令连接建立与释放、服务请求
    会话管理（SM） PDU Session 建立 / 修改 / 释放
    安全控制 NAS 加密（NEA）与完整性保护（NIA）算法协商与激活
    标识管理 SUCI 处理、5G-GUTI 分配、身份请求
    鉴权 5G-AKA / EAP-AKA’ 过程

    TS 24.501 第 4 章明确指出，NAS 协议为 UE 和 AMF 之间（5GMM）以及 UE 和 SMF 之间（5GSM）提供过程，支持 3GPP 接入与非 3GPP 接入两种方式。

    2.2 NAS 的两大子层：5GMM 与 5GSM

    5G NAS 协议由两个子层构成，职责明确分离：

    核心区别：

    • 5GMM 工作在 UE 级别（per Access Type），每种接入方式（3GPP / 非 3GPP）独立维护一套 5GMM 状态机。
    • 5GSM 工作在 PDU Session 级别，每个 PDU Session 独立管理。
    • 5GSM 消息通过被 5GMM 的 UL NAS TRANSPORT / DL NAS TRANSPORT 消息携带（Piggyback 机制）传递到 SMF。

    2.3 5GMM：移动性管理

    5GMM 子层负责跟踪 UE 位置、鉴权、安全控制及标识管理。

    5GMM 状态机

    根据 TS 24.501 第 5.1.3.2 节，UE 侧的 5GMM 状态机如下：

    在已注册（5GMM-REGISTERED）状态下，UE 还有连接子状态：

    • 5GMM-IDLE（CM-IDLE）：N1 NAS 信令连接已释放，UE 处于空闲态。
    • 5GMM-CONNECTED（CM-CONNECTED）：N1 NAS 信令连接已建立，UE 可以发送/接收 NAS 消息。
    • 5GMM-CONNECTED with RRC Inactive Indication（R16+）：NAS 连接维持，但 RRC 处于 INACTIVE 态（对应 RRC 三态中的 RRC_INACTIVE）。

    5GMM 主要过程

    注册管理（RM）过程（TS 24.501 第 5.5 节）：

    过程名称 触发条件 主要目的
    初始注册（Initial Registration） UE 开机 / 从 DEREGISTERED 进入 建立 5GMM 上下文，分配 5G-GUTI
    移动性注册更新（Mobility Registration Update） UE 移动出 TA 列表 更新 AMF 中 UE 位置信息
    周期性注册更新（Periodic Registration Update） T3512 超时 保持与网络的联系，防止被注销
    去注册（Deregistration） UE 关机 / 用户手动 / 网络请求 释放 5GMM 上下文
    紧急注册 紧急呼叫需求 有限服务下的紧急接入

    连接管理（CM）过程（TS 24.501 第 5.6 节）：

    • Service Request：UE 在 5GMM-IDLE 下发起，建立 N1 信令连接，恢复用户面资源。
      • 分为：MO（UE 主动发起）和 MT（网络 Paging 触发）两类。
    • Paging：网络通过寻呼机制通知处于 CM-IDLE 态的 UE。
    • Notification：SMF 通过 AMF 通知 UE PDU Session 相关事件。

    5GMM 安全过程

    鉴权（Authentication）（TS 24.501 第 5.4.1 节）：

    1. AMF 收到注册请求后，向 AUSF 发起鉴权数据请求。
    2. AUSF 向 UDM 获取鉴权向量（Authentication Vector）。
    3. AMF 向 UE 发送 Authentication Request（携带 RAND 和 AUTN）。
    4. UE 验证 AUTN、计算 RES*，返回 Authentication Response。
    5. AMF 将 RES* 发给 AUSF 比对验证。

    安全模式控制（SMC）（TS 24.501 第 5.4.2 节）：

    1. 鉴权成功后，AMF 向 UE 发送 Security Mode Command（携带选定的 NAS 算法和密钥集标识符 ngKSI）。
    2. UE 激活 NAS 安全，返回 Security Mode Complete（首个已加密且完整性保护的 NAS 消息）。

    2.4 5GSM：会话管理

    5GSM 子层的核心任务是管理 UE 与 SMF 之间的 PDU Session 生命周期（TS 24.501 第 6 章）。

    5GSM 状态机（UE 侧，per PDU Session）

    PDU Session 类型

    根据 TS 24.501 第 6.2.2 节，PDU Session 支持以下类型：

    类型 说明
    IPv4 纯 IPv4 地址分配
    IPv6 纯 IPv6 前缀分配
    IPv4v6 双栈，同时支持 IPv4 和 IPv6
    Unstructured 非结构化数据，用于物联网点对点传输（如 LADN）
    Ethernet 以太网帧传输，用于企业专线场景

    5GSM 主要过程

    过程 发起方 说明
    PDU Session Establishment UE 发起 请求建立新 PDU Session，协商 QoS
    PDU Session Modification UE 或 网络发起 修改 QoS Flow 参数，增减 QoS Rule
    PDU Session Release UE 或 网络发起 释放 PDU Session 及相关资源

    Piggyback 机制：PDU Session Establishment Request（5GSM 消息）通常被携带在 Registration Request 或 Service Request 的 NAS PDU container 信息元素（IE）中一并发送，减少信令往返次数。

    2.5 NAS 消息结构与安全保护

    NAS 消息格式

    5G NAS 消息的基本格式（TS 24.501 第 9 章）：

    Extended Protocol Discriminator：

    • 0x7E：5GMM 消息
    • 0x2E：5GSM 消息

    Security Header Type 定义消息安全级别：

    值 含义
    0000 未保护（Plain NAS）
    0001 完整性保护，非加密
    0010 完整性保护 + 加密
    0011 完整性保护，非加密（含新安全上下文）
    0100 完整性保护 + 加密（含新安全上下文）
    1100 安全模式完成（Security Mode Complete）特殊类型

    NAS 安全算法

    根据 TS 33.501，5G NAS 定义以下加密/完整性保护算法：

    算法标识 加密（NEA） 完整性（NIA）
    0 NULL（5G-EA0，不加密） NULL（5G-IA0，不保护）
    1 128-NEA1（SNOW 3G） 128-NIA1（SNOW 3G）
    2 128-NEA2（AES-CTR） 128-NIA2（AES-CMAC）
    3 128-NEA3（ZUC） 128-NIA3（ZUC）

    注意：5G 要求对所有 NAS 消息执行完整性保护（除初始几条信令外），加密是可选的（但强烈建议）。这与 LTE 中可以协商不保护的设计有所不同。

    2.6 NAS 传输机制：消息透传与封装

    NAS 消息在传输路径上的封装关系：

    NAS 层对 gNB 完全透明，gNB 不能解析 NAS 消息内容（仅知道这是一个 NAS PDU），这是接入网与核心网解耦的基本保证。

    3. RRC 层（无线资源控制层）

    3.1 RRC 在协议栈中的位置

    RRC（Radio Resource Control）是 3GPP TS 38.331 规定的 Layer 3 空口协议，终止于 gNB（更准确地说是 gNB-CU 的控制面），是空口控制面最核心的协议层。

    RRC 消息通过 信令无线承载（SRB） 承载，经 PDCP → RLC → MAC → PHY 发送至 gNB，再由 gNB 通过 NGAP（N2 接口）将 NAS 消息转发给 AMF。

    3.2 RRC 核心功能全景

    根据 TS 38.331 第 4 章，RRC 层的主要功能如下：

    具体功能列表：

    1. 系统信息的广播：MIB、SIB1 及其他 SIB 的调度与广播
    2. 寻呼（Paging）：CN 发起寻呼和 RAN 发起寻呼的控制
    3. RRC 连接建立、维护与释放：包括 RRC_CONNECTED ↔ RRC_IDLE 转换
    4. RRC_INACTIVE 态管理（R15 引入）：RNA（RAN Notification Area）配置与管理
    5. 安全功能：RRC 层加密与完整性保护算法协商与密钥激活
    6. 无线承载（Radio Bearer）的建立、修改与释放：SRB 和 DRB 的配置
    7. 移动性控制：测量配置、切换命令、RRC 重建立（Re-establishment）
    8. UE 能力信息的协商
    9. NAS 消息透传：在 UE 与 AMF/SMF 之间传递 NAS PDU
    10. QoS 流到 DRB 的映射支持（配合 SDAP 层）
    11. 辅小区组（SCG）配置：用于双连接（DC）场景
    12. 时间戳分发（R16+）：5G 时间同步分发
    13. 小数据传输（SDT）配置（R17+）：在 RRC_INACTIVE 态下的小数据传输

    3.3 系统信息广播（SI）

    系统信息是基站周期性广播的，供所有 UE 获取小区配置。5G NR 的系统信息体系比 LTE 更加灵活，引入了按需（On-demand）获取机制。

    系统信息层次结构

    On-demand SI（按需系统信息）

    从 R15 引入，UE 可以在需要时向 gNB 请求特定 SIB（除 MIB 和 SIB1 外），gNB 在收到请求后临时广播相应 SIB，无需一直周期性广播，节省空口资源。

    流程：

    1. UE 通过 PRACH（Random Access）发送 SI Request（特殊前导码）。
    2. gNB 识别 SI Request 后，在接下来的 SI Window 内广播被请求的 SIB。

    3.4 RRC 状态机

    5G NR 相较于 LTE 引入了第三个 RRC 状态：RRC_INACTIVE，这是 5G 的重要创新之一。

    三种状态特性对比

    特性 RRC_IDLE RRC_INACTIVE RRC_CONNECTED
    RRC 上下文（gNB） ✗ ✓（保留于 Anchor gNB） ✓
    NAS/AS 安全上下文 ✗ ✓ ✓
    用户面配置（PDU Session） ✗ 暂挂（Suspended） ✓（活跃）
    N2/N3 连接 ✗ ✗ ✓
    下行数据接收 ✗（需经 Paging→接入） ✗（需 Resume） ✓
    上行数据发送 ✗ ✗（或 SDT） ✓
    小区监听（Paging） ✓ ✓（RAN Paging） ——
    小区重选 ✓ ✓（RAN 控制） ✗
    节电效果 最佳 中等 最差
    恢复速度 最慢 中等（快于 IDLE） ——

    RRC_INACTIVE 的核心优势：UE 在 gNB 中保留 AS 上下文和 RRC 配置，恢复连接时无需重新建立全套配置，时延显著低于从 RRC_IDLE 建立连接。这对 URLLC 和频繁小数据业务场景（如 IoT）意义重大。

    RNA（RAN Notification Area）：处于 RRC_INACTIVE 的 UE 在 RNA 范围内移动时，不需要通知网络（类似于 NAS 层的 TA）；超出 RNA 时，触发 RNA Update（类似于 TA Update），通过 RRCResumeRequest 携带原因值执行。

    3.5 RRC 连接管理流程

    RRC 建立流程（RRC Setup）

    1. RRCSetupRequest（UE → gNB）：通过 SRB0（CCCH）发送，包含 UE 标识（ng-5G-S-TMSI 或随机值）和建立原因。
    2. RRCSetup（gNB → UE）：包含 SRB1 配置、初始 BWP 配置、Master Cell Group 配置等。
    3. RRCSetupComplete（UE → gNB）：通过 SRB1（DCCH）发送，携带所选 PLMN ID 及 NAS 消息（如 Registration Request）。

    RRC 恢复流程（RRC Resume，从 INACTIVE）

    RRC 重建立流程（RRC Re-establishment）

    当 RRC_CONNECTED 下发生无线链路失败（RLF：Radio Link Failure）时触发：

    1. UE 检测到 T310 超时（N310 次 OOS 指示后未收到 N311 次 IS 指示）。
    2. UE 进入 RRC_CONNECTED 失败状态，启动 T311，尝试小区选择。
    3. 在新小区发送 RRCReestablishmentRequest（包含 UE C-RNTI、物理小区 ID、短 MAC-I 验证）。
    4. 新 gNB 通过 Xn 接口向旧 gNB 获取 UE 上下文。
    5. gNB 回复 RRCReestablishment 消息完成重建立。

    3.6 无线承载与逻辑信道配置

    RRC 负责配置所有无线承载（Radio Bearer）：

    3.7 信令无线承载（SRB）详解

    SRB 是 RRC 层定义的用于承载控制信令的专用无线承载，定义于 TS 38.331 第 4.2.2 节。

    SRB 类型与特性

    SRB 用途 逻辑信道 RLC 模式 安全保护 建立时机
    SRB0 初始 RRC 消息（RRCSetupRequest/Setup） CCCH TM（透明模式） 无 始终存在
    SRB1 RRC 消息（可含 NAS Piggyback），SRB2 建立前也承载 NAS DCCH AM（确认模式） RRC 安全激活后（完整性+加密） RRCSetup 时建立
    SRB2 NAS 消息专用，低优先级（低于 SRB1） DCCH AM（确认模式） 是（完整性+加密） AS 安全激活后
    SRB3 EN-DC / NR-DC 场景中 SgNB 直接信令（RRCReconfiguration / MeasurementReport 等） DCCH AM（确认模式） 是 EN-DC 配置时（可选）

    为何 SRB0 使用 TM-RLC？
    初始接入阶段（RACH 期间），UE 尚未配置 RLC AM 所需的参数，且 RRCSetupRequest 消息极短，不需要重传机制。TM 模式零开销，最适合初始信令传输。

    SRB1 与 SRB2 的优先级差异：
    SRB1 优先级高于 SRB2，保证 RRC 控制消息（如切换命令）能优先于 NAS 消息传输，避免 NAS 消息堵塞关键 RRC 信令。

    SRB3 的使用场景：
    在 EN-DC（E-UTRA 主站 + NR 从站）或 NR-DC 场景中，SRB3 在 NR 侧提供直接的 RRC 信令路径，支持的消息类型有限（仅 RRCReconfiguration、RRCReconfigurationComplete、MeasurementReport），主要用于对时延敏感的移动性流程，如辅节点内切换。如果未配置 SRB3，则相关消息经由主节点（MeNB/MN）的 X2/Xn 接口中转。

    3.8 测量配置与上报

    测量（Measurement）是 RRC 的重要功能，是移动性决策（切换）的基础。

    测量配置结构

    gNB 通过 RRCReconfiguration 消息中的 measConfig IE 下发测量配置，包括：

    事件类型说明

    事件 触发条件 典型用途
    A1 服务小区测量量高于阈值 关闭异频/异系统测量（节电）
    A2 服务小区测量量低于阈值 触发异频/异系统测量
    A3 邻区测量量比服务小区好超过 offset 同频/同系统切换触发
    A4 邻区测量量高于阈值 负载均衡
    A5 服务小区低于阈值 1 且邻区高于阈值 2 异频切换
    A6 SCG 内邻区比主服务小区好超过 offset 辅小区组切换
    B1 异系统邻区高于阈值 异系统切换触发
    B2 服务小区低于阈值 1 且异系统邻区高于阈值 2 EPS Fallback 触发

    UE 测量满足事件条件且触发时间（TTT：Time-to-Trigger）超时后，通过 MeasurementReport 消息（SRB1）上报测量结果，gNB 根据上报结果决策切换。

    3.9 RRC 安全激活

    RRC 安全（加密 + 完整性保护）通过 SecurityModeCommand / SecurityModeComplete 流程激活：

    密钥来源：

    • KgNB：由 KSEAF（核心网侧）派生，或切换时由源 gNB 的 NH（Next Hop）派生。
    • KRRCenc、KRRCint：由 KgNB 和算法标识派生，用于 RRC 层加密和完整性保护。
    • KUPenc、KUPint：由 KgNB 派生，用于用户面加密和完整性保护（用户面完整性保护为 R15 新增特性）。

    RRC 加密算法（NR-EEA）与完整性保护算法（NR-EIA）：

    算法 加密（NR-EEA / NEA） 完整性（NR-EIA / NIA）
    0 NULL（不加密） NULL（不推荐用于 UP）
    1 SNOW 3G SNOW 3G
    2 AES-CTR AES-CMAC
    3 ZUC ZUC

    3.10 RRC 重配置与能力上报

    RRC 重配置（RRCReconfiguration）

    RRCReconfiguration 是 5G RRC 中最通用的消息，可用于：

    • 增/删 SRB 和 DRB
    • 修改 BWP（带宽部分）配置
    • 更新测量配置
    • 配置辅小区（SCell）
    • 执行切换（含 reconfigurationWithSync）
    • 配置双连接（SCG 配置）

    UE 收到后执行配置，成功则返回 RRCReconfigurationComplete；失败（如无法执行）则触发 RRC 重建立。

    UE 能力上报（UE Capability）

    gNB 通过 UECapabilityEnquiry 查询 UE 能力，UE 回复 UECapabilityInformation，包括：

    • 支持的频段（Band）列表
    • 支持的频段组合（Band Combination）和 CA 能力
    • MIMO 层数（Layers）
    • 调制阶数（最高 256QAM 或 1024QAM）
    • PDCP / RLC / MAC / PHY 特性支持情况
    • 双连接能力（EN-DC / NR-DC）

    gNB 根据 UE 能力选择合适的调度策略和配置。

    4. NAS 与 RRC 的协作关系

    NAS 与 RRC 虽然分属不同协议层，但在 5G 接入流程中紧密协作：

    关键协作点：

    1. RRC 为 NAS 提供信令通道：NAS PDU 被封装在 RRC 的 DLInformationTransfer / ULInformationTransfer 消息中传递，RRC 是 NAS 的”载体”。
    2. NAS 触发 RRC 行为：如注册成功后，AMF 通过 NGAP 下发 InitialContextSetupRequest，触发 gNB 向 UE 发送 RRCReconfiguration 建立 SRB2 和 DRB。
    3. 安全流程分层进行：先 NAS 层安全（5GMM SMC），再 RRC/AS 层安全（RRC SecurityModeCommand），密钥层级依次派生。
    4. 状态联动：NAS 的 CM-CONNECTED 状态与 RRC 的 RRC_CONNECTED 状态对应；CM-IDLE 可对应 RRC_IDLE 或 RRC_INACTIVE（后者为 R15 新引入）。

    5. 与 LTE 的主要差异对比

    特性 4G LTE 5G NR 关键改进
    RRC 状态 2 态（IDLE / CONNECTED） 3 态（IDLE / INACTIVE / CONNECTED） RRC_INACTIVE 降低恢复时延，省电
    NAS 协议 EMM / ESM（TS 24.301） 5GMM / 5GSM（TS 24.501） 支持网络切片、SUCI 隐私保护
    NAS 终止点 MME AMF（5GMM）/ SMF（5GSM） SMF 专司会话管理，CUPS 分离
    SRB 类型 SRB0 / SRB1 / SRB2 SRB0 / SRB1 / SRB2 / SRB3 SRB3 支持 EN-DC 直接信令
    SI 获取 固定周期广播 周期广播 + On-demand（按需） 节省空口资源
    用户面完整性保护 无 支持（可选，KUPint） 增强安全性（URLLC 场景）
    测量事件 A1~A5, B1, B2 A1~A6, B1, B2 A6 事件支持 SCG 侧辅小区切换
    UE 标识隐私 IMSI 有时明文发送 SUCI（基于椭圆曲线加密 IMSI） 空口防 IMSI 嗅探
    小数据传输 无（需先建立 RRC） SDT（R17，INACTIVE 态传小数据） 降低 IoT 场景信令开销

    6. 小结

    本文深入解析了 5G NR 控制面协议栈的两大核心层：

    NAS 层（TS 24.501）：

    • 分为 5GMM（移动性管理，终止于 AMF）和 5GSM（会话管理，终止于 SMF）两个子层。
    • 5GMM 负责注册管理、连接管理和 NAS 层安全（鉴权 + SMC）。
    • 5GSM 负责 PDU Session 的建立、修改与释放，支持 IPv4/v6/Ethernet 等多种 PDU 类型。
    • NAS 消息对 gNB 透明，通过 RRC 的 Information Transfer 消息透传。

    RRC 层（TS 38.331）：

    • 终止于 gNB，负责空口侧所有控制信令。
    • 新引入 RRC_INACTIVE 三态模型，在省电和快速恢复之间取得平衡。
    • 通过 SRB0/1/2/3 承载不同类型和优先级的控制信令。
    • 提供系统信息广播（含按需 SI）、连接管理、安全激活、测量配置、移动性控制等完整功能。

    两层的协作体现在：RRC 是 NAS 的传输载体，NAS 的注册/会话管理流程驱动 RRC 的连接建立和无线承载配置，两者在密钥层级上也具有清晰的上下游派生关系。
