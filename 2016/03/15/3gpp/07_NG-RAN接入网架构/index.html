    1. NG-RAN 总体架构概述

    NG-RAN（Next Generation Radio Access Network）是 5G 系统的接入网部分，定义于 3GPP TS 38.300 和 TS 38.401，负责为用户设备（UE）提供无线接入能力。

    NG-RAN 节点类型如下：

    节点类型 全称 说明
    gNB Next Generation NodeB 提供 5G NR（New Radio）无线接入，终结 NR 空口控制面与用户面协议
    ng-eNB Next Generation evolved NodeB 提供 LTE（E-UTRA）无线接入，连接至 5GC（而非 EPC）

    NG-RAN 的整体拓扑如下图所示：

    NG-RAN 的架构设计遵循以下核心原则（来自 TS 38.401）：

    • 逻辑分离：信令传输网络与数据传输网络逻辑独立；
    • 功能完全分离：NG-RAN 功能与传输网络功能完全解耦；
    • 移动性由 NG-RAN 控制：RRC 连接态下的移动性（切换）由 NG-RAN 全权负责，核心网不介入 RRC 层；
    • 接口开放：所有 NG-RAN 接口（F1、E1、Xn、NG）均基于开放标准定义，实现多厂商互操作。

    2. gNB 内部架构：CU / DU / RU 功能拆分

    2.1 单体 gNB 与功能拆分的演进

    在 LTE 时代，eNB 是一个”单体（Monolithic）”节点，所有协议层功能集中在一个物理设备中（尽管实际部署中常用 RRH+BBU 的远端射频方式，但逻辑上仍是一个节点）。

    进入 5G 时代，3GPP 在 Release 15 研究阶段（TR 38.801）就开始讨论将 gNB 拆分的必要性，其动机包括：

    • 降低 TCO（总拥有成本）：CU 云化部署，多个 DU 共享一个 CU，提升硬件利用率；
    • 灵活的前传/中传/回传选择：根据传输网络条件选择不同的拆分点；
    • 支持 NFV/SDN：CU 功能可以跑在通用 x86 服务器上，实现网络功能虚拟化；
    • 边缘计算：DU 靠近天线侧，可承载时延敏感的用户面处理，CU-UP 也可下沉至边缘节点。

    3GPP 最终在 TS 38.401 中标准化了两种拆分架构（在”单体 gNB”之外）：

    1. CU + DU 拆分（高层分离，Option 2 split）；
    2. CU-CP + CU-UP + DU 三层拆分（在 CU 内部进一步分离控制面与用户面）。

    RU（Radio Unit，射频单元）与 DU 之间的拆分（下层拆分，Lower Layer Split）则由 O-RAN Alliance 在 3GPP 框架之外进一步标准化（见第 5 节）。

    2.2 gNB-CU：Central Unit

    gNB-CU（Central Unit）是 gNB 的高层逻辑节点，部署在较集中的位置（如机房、数据中心）。

    根据 TS 38.401，gNB-CU 承载的协议层包括：

    协议层 面向 说明
    RRC（Radio Resource Control） 控制面 负责 UE 的无线资源配置、移动性管理、系统信息广播等
    PDCP（Packet Data Convergence Protocol） 控制面 + 用户面 头压缩（ROHC）、加解密、完整性保护、重排序
    SDAP（Service Data Adaptation Protocol） 用户面 QoS Flow 到 DRB 的映射（仅 CU-UP 侧）

    gNB-CU 通过 F1 接口与一个或多个 gNB-DU 相连；通过 NG 接口连接 5GC（AMF/UPF）；通过 Xn 接口与邻居 gNB 通信；通过 E1 接口（当 CU 进一步拆分时）连接 gNB-CU-UP。

    关键约束（来自 TS 38.401）：对外部（其他 gNB 和 5GC）而言，gNB-CU 与其连接的所有 gNB-DU 作为一个整体 gNB 呈现，外部节点无需感知内部拆分结构。

    2.3 gNB-CU-CP 与 gNB-CU-UP 分离

    为进一步支持控制面与用户面的独立扩缩容，3GPP 允许将 gNB-CU 再拆分为：

    • gNB-CU-CP（Control Plane）：承载 RRC 和 PDCP 控制面部分；
    • gNB-CU-UP（User Plane）：承载 PDCP 用户面和 SDAP。

    拓扑关系（来自 TS 38.401 Figure 6.1.2-1）：

    具体连接规则：

    连接关系 接口 约束
    gNB-CU-CP ↔ gNB-DU F1-C 一个 DU 只能连接一个 CU-CP
    gNB-CU-UP ↔ gNB-DU F1-U 用户面数据流经 DU 到 CU-UP
    gNB-CU-CP ↔ gNB-CU-UP E1 一个 CU-UP 只能连接一个 CU-CP
    gNB-CU-CP ↔ AMF NG-C 控制信令
    gNB-CU-UP ↔ UPF NG-U 用户数据 GTP-U 隧道

    这种三层拆分的优势：

    • CU-CP 云化：控制面可集中部署，统一管理大量 CU-UP 和 DU；
    • CU-UP 灵活部署：用户面可按业务需求下沉至边缘，降低时延；
    • 独立扩缩容：高峰时段可单独扩容 CU-UP，而无需触及控制面。

    2.4 gNB-DU：Distributed Unit

    gNB-DU（Distributed Unit）是 gNB 的低层逻辑节点，通常部署在靠近天线的位置（如机站机房）。

    gNB-DU 承载的协议层包括：

    协议层 说明
    RLC（Radio Link Control） 数据分段/重组、ARQ 重传（AM 模式）
    MAC（Medium Access Control） 调度、HARQ、逻辑信道复用
    PHY（Physical Layer） 编解码、调制解调、资源映射

    gNB-DU 的核心职责是实时调度和物理层处理，对时延要求严格（调度决策需在亚毫秒级完成）。一个 gNB-CU-CP 可连接多个 gNB-DU，实现大规模覆盖。

    2.5 RU（Radio Unit）与前传接口

    虽然 3GPP 将 PHY 层定义在 gNB-DU 之内，但实际部署中，物理层本身还可以进一步拆分：

    • Higher PHY（留在 DU）：信道编码/解码、速率匹配、层映射等；
    • Lower PHY（下移至 RU）：FFT/IFFT、PRACH 检测、数字预失真（DPD）等；
    • RF 射频处理（在 RU 中）：DAC/ADC、功率放大、天线收发。

    DU 与 RU 之间的接口称为前传（Fronthaul） 接口，3GPP 并未标准化这一接口，而是由 O-RAN Alliance 定义了 Open Fronthaul（eCPRI 上层） 接口规范（见第 5 节）。

    3. F1 与 E1 接口详解

    3.1 F1 接口

    F1 接口定义于 3GPP TS 38.470～38.474 系列规范，是 gNB-CU 与 gNB-DU 之间的接口，分为控制面（F1-C）和用户面（F1-U）两个子接口。

    F1-C（控制面）

    • 传输协议栈：F1AP（F1 Application Protocol）/ SCTP / IP
    • 主要功能：
    功能类别 说明
    F1 接口管理 F1 Setup（建立）、Reset（复位）、Error Indication
    UE 上下文管理 UE Context Setup / Modification / Release（用于 RRC 建立、切换等）
    RRC 消息传递 DL/UL RRC Message Transfer（gNB-CU 和 UE 之间的 RRC 消息透传）
    寻呼 Paging（DU 负责将 PCH 调度到空口）
    系统信息管理 Warning Message Transmission（PWS 公共告警）
    小区管理 gNB-DU Configuration Update（DU 上报小区状态变化）

    F1-U（用户面）

    • 传输协议栈：GTP-U / UDP / IP
    • 主要功能：
    功能类别 说明
    用户数据传输 CU-UP（或 CU）与 DU 之间的下行/上行数据包转发
    流量控制 DL 方向的 Flow Control，避免 DU 缓冲区溢出
    PDCP SN 状态传输 切换场景下，传递 PDCP 序列号状态（数据不丢失）

    F1 接口的设计特点

    1. 承载协议分离：控制信令（F1AP/SCTP）与用户数据（GTP-U/UDP）使用独立协议栈，互不干扰；
    2. DU 多归属（Multi-homing）支持：为增强弹性，一个 DU 可（在特殊配置下）连接多个 CU-CP；
    3. 传输网络无关：F1 接口对传输网络（光纤、微波、IP over WDM 等）无特殊要求，只需满足带宽和时延指标（中传 backhaul 要求通常是 < 10 ms 单向时延）。

    3.2 E1 接口

    E1 接口定义于 3GPP TS 38.460～38.463 系列规范，是 gNB-CU-CP 与 gNB-CU-UP 之间的接口，仅涉及控制面信令（用户数据不经过 E1，而是通过 F1-U 和 NG-U 直接流转）。

    • 传输协议栈：E1AP / SCTP / IP

    主要功能：

    功能类别 说明
    E1 接口管理 E1 Setup、Reset、Error Indication
    Bearer 上下文管理 Bearer Context Setup / Modification / Release（CU-CP 指示 CU-UP 建立/释放 PDCP 承载，携带 QoS、安全参数等）
    数据使用上报 CU-UP 向 CU-CP 上报 UE 数据量（用于计费、非活跃监控）

    E1 接口的引入，使得 CU-CP 可以作为”控制器”统一管理多个 CU-UP，这与 SDN 的控制平面与数据平面分离理念完全契合。

    4. NG-RAN 外部接口：Xn 与 NG

    4.1 Xn 接口

    Xn 接口定义于 3GPP TS 38.420～38.424 系列规范，是 NG-RAN 节点之间（gNB-gNB、gNB-ng-eNB、ng-eNB-ng-eNB）的接口，是 LTE X2 接口在 5G 中的演进版本。

    Xn 接口同样分为控制面（Xn-C）和用户面（Xn-U）：

    Xn-C（控制面）

    • 传输协议栈：XnAP（Xn Application Protocol）/ SCTP / IP
    功能类别 具体内容
    接口管理 Xn Setup、Reset、Configuration Update、Error Indication
    移动性管理 Xn 切换（Handover Preparation / Resource Coordination / Path Switch）、SN 状态传输、UE 上下文获取
    RAN 寻呼 RAN-initiated Paging（RRC_INACTIVE 状态下，目标 gNB 通过 Xn 转发寻呼请求）
    双连接（DC） 辅节点（SN）的添加/重配置/修改/释放，SN 状态变更通知
    资源协调 小区间干扰协调（ICIC）、小区激活/去激活的能量节省信令
    辅助 RAT 数据上报 MR-DC 场景下辅 RAT（如 NR）的流量使用量上报
    Trace 针对特定 UE 的 Trace 会话控制

    Xn-U（用户面）

    • 传输协议栈：GTP-U / UDP / IP
    功能 说明
    数据转发（Data Forwarding） Xn 切换期间，源 gNB 将下行缓存数据转发给目标 gNB（PDCP PDU 级别），确保数据不丢失
    流量控制（Flow Control） 控制数据转发速率，避免目标侧缓冲区过载

    Xn 接口 vs X2 接口的主要差异

    特性 LTE X2 5G Xn
    所属规范 TS 36.42x TS 38.42x
    UE 上下文传输 基础支持 增强支持（更多 5G 特有参数）
    双连接支持 基于 X2 的 MR-DC（EN-DC） 原生 NR-DC 支持
    RAN 寻呼 不支持 RRC_INACTIVE 支持 RRC_INACTIVE + RNA 管理
    核心网 连接 EPC 连接 5GC

    4.2 NG 接口

    NG 接口定义于 3GPP TS 38.410～38.414 系列规范，是 NG-RAN 节点（gNB / ng-eNB）与 5GC 之间的接口，是 LTE S1 接口的 5G 演进版本。

    NG 接口分为：

    • NG-C（控制面）：gNB ↔ AMF，承载 NGAP（NG Application Protocol）
    • NG-U（用户面）：gNB ↔ UPF，承载 GTP-U 隧道

    NG-C（控制面 / NGAP）

    • 传输协议栈：NGAP / SCTP / IP
    • 规范：TS 38.413
    功能类别 具体内容
    NG 接口管理 NG Setup（gNB 向 AMF 注册）、Reset、Configuration Update
    UE 上下文管理 Initial UE Message、UE Context Modification / Release
    注册与移动性 UL/DL NAS Transport（NAS PDU 透传）、UE Registration / Deregistration
    PDU Session 管理 PDU Session Resource Setup / Modify / Release（由 SMF 通过 AMF 触发）
    切换（核心网切换） Handover Required / Command / Complete（跨 gNB 且无 Xn 时使用 NG 切换）
    寻呼 Paging（AMF 发起，下发至 gNB，由 gNB 在 PCH 上广播）
    位置服务 Location Report（UE 位置信息上报 AMF）
    跟踪 Trace 会话激活 / 去激活

    NG-U（用户面）

    • 传输协议栈：GTP-U / UDP / IP
    • 规范：TS 38.414

    NG-U 与 LTE S1-U 类似，承载 PDU Session 中的用户数据，每个 PDU Session 由 gNB 与 UPF 之间的一条 GTP-U 隧道承载（通过 TEID 区分不同用户的隧道）。

    NG-C 与 NG-U 的独立性

    NG-C 和 NG-U 可以连接到不同的物理实体：一个 gNB 可以同时连接多个 AMF（实现 AMF 负载分担）和多个 UPF（实现用户面分布式部署）。这是 5GC CUPS（控制面/用户面分离）设计理念在 RAN 侧接口上的体现。

    5. O-RAN 架构与 RIC

    5.1 O-RAN Alliance 与 3GPP 的关系

    O-RAN Alliance（Open RAN Alliance，开放无线接入网联盟）于 2018 年由 AT&T、中国移动、德国电信、NTT DOCOMO 和 Orange 联合创立，目标是推动 RAN 的开放化、智能化和虚拟化。

    O-RAN 与 3GPP 的关系：

    • 3GPP 定义 gNB 的逻辑架构（CU/DU 拆分、F1/E1 接口）——这是规范性（normative） 标准；
    • O-RAN Alliance 在 3GPP 框架之上，进一步定义：
      • DU↔RU 的开放前传接口（Open Fronthaul，基于 eCPRI）；
      • RAN 智能控制器（RIC） 架构及相关接口（A1、E2、O1）；
      • O-RAN 网络功能的云原生部署规范。

    O-RAN 将 3GPP 的 gNB 组件重命名为：gNB-CU-CP → O-CU-CP，gNB-CU-UP → O-CU-UP，gNB-DU → O-DU，RU → O-RU。

    5.2 O-RAN 整体架构

    5.3 Non-RT RIC 与 rApp

    Non-RT RIC（Non-Real-Time RAN Intelligent Controller）是 SMO 框架内的一个逻辑功能，工作在秒级及以上的时间尺度（> 1 秒），负责：

    • AI/ML 模型训练：基于长期历史数据（流量、信道质量、干扰）训练模型，供 Near-RT RIC 使用；
    • 策略管理：制定网络优化策略，通过 A1 接口下发给 Near-RT RIC；
    • 配置管理：通过 O1 接口对 RAN 网元（O-CU、O-DU、Near-RT RIC）进行配置和监控；
    • 故障管理、性能管理：长周期的 KPI 分析和告警处理。

    Non-RT RIC 上运行的应用称为 rApp（r 代表 RAN），通过 R1 接口访问 Non-RT RIC 提供的服务。典型 rApp 用例包括：

    • 网络切片资源的长期规划；
    • 基于地理分布的频谱策略优化；
    • 能耗优化策略（关闭低负载小区）。

    5.4 Near-RT RIC 与 xApp

    Near-RT RIC（Near-Real-Time RAN Intelligent Controller）工作在 10 ms～1 s 的时间尺度，通过 E2 接口连接 O-CU-CP、O-CU-UP、O-DU 等 RAN 节点（称为 E2 Node），实现精细化的 RAN 控制。

    Near-RT RIC 对 E2 节点提供四类服务：

    服务类型 描述
    Report 请求 E2 节点周期性或事件触发地上报指定指标（如 RSRP、PRB 使用率）
    Insert 指示 E2 节点在特定条件下激活某用户面功能
    Control 指示 E2 节点执行特定控制面动作（如触发切换、调整调度参数）
    Policy 在 E2 节点上配置特定策略参数（如负载均衡阈值）

    Near-RT RIC 上运行的应用称为 xApp（x 代表 any，即任意第三方应用），xApp 可由第三方厂商独立开发，不依赖于 Near-RT RIC 平台本身。典型 xApp 用例：

    • 干扰缓解（Interference Mitigation）xApp：实时监测 PRB 级干扰，动态调整频率资源分配；
    • 流量导引（Traffic Steering）xApp：基于实时负载，将 UE 引导至合适小区或频段；
    • 切换优化（Handover Optimization）xApp：基于 ML 模型调整 A3 事件阈值，降低乒乓切换；
    • QoS 控制（QoS Control）xApp：实时保障高优先级业务的资源供给。

    5.5 O-RAN 关键接口汇总

    接口 连接关系 时延要求 主要功能
    A1 Non-RT RIC ↔ Near-RT RIC > 1 s 策略下发、AI/ML 模型传输
    E2 Near-RT RIC ↔ E2 Nodes（O-CU/O-DU） 10 ms～1 s RAN 数据上报与控制命令下发
    O1 SMO ↔ RAN Nodes 分钟级 OAM（操作、管理、维护）：配置、故障、性能管理
    O2 SMO ↔ O-Cloud - 云资源编排（O-RAN 网络功能的云化部署）
    Open Fronthaul O-DU ↔ O-RU < 100 μs～几 ms 物理层数据（IQ 采样）、控制、同步、管理
    R1 rApp ↔ Non-RT RIC - rApp 访问 Non-RT RIC 服务

    6. 无线空口能力：Massive MIMO 与波束管理

    6.1 Massive MIMO 基本原理

    Massive MIMO（大规模多输入多输出）是 5G NR 最重要的无线技术之一，定义于 3GPP TS 38.211、TS 38.214 等规范。

    与 LTE 最多 8 天线端口不同，5G NR 支持最多 256 个逻辑天线端口，物理天线阵元数量通常为 64～192 个（如典型的 64T64R 配置）。Massive MIMO 的核心优势：

    • 波束赋形增益：将能量集中到目标 UE 方向，显著提升 SNR；
    • 空间复用：同频段同时为多个 UE 提供服务（MU-MIMO）；
    • 干扰抑制：通过精确的波束方向控制，降低对非目标 UE 的干扰。

    Massive MIMO 的实现依赖于有源天线单元（AAU，Active Antenna Unit），将射频单元与天线阵列集成在一起。根据实现方式，波束赋形分为：

    类型 实现方式 特点
    模拟波束赋形 移相器调节各天线元件的相位 每次只能形成一个方向的波束，适合 FR2 mmWave
    数字波束赋形 每个天线独立射频链路，数字域加权 可同时形成多个独立波束，适合 FR1
    混合波束赋形 数字+模拟混合 兼顾灵活性与硬件成本，FR2 常用

    6.2 波束管理流程：P1 / P2 / P3

    波束管理（Beam Management，BM） 定义于 3GPP TS 38.213 和 TS 38.214，目标是在 gNB 和 UE 之间建立并维护最优的 TX/RX 波束对链路（Beam Pair Link，BPL）。

    完整的波束管理分为三个子过程：

    P1：初始波束建立（Beam Sweeping 阶段）

    P1 过程在 UE 进行初始接入或波束恢复时触发，基于 SSB（Synchronization Signal Block） 进行粗粒度的波束扫描：

    1. gNB 以不同方向发射多个 SSB 波束（每个 SSB 对应一个波束方向）；
    2. UE 测量每个 SSB 的 L1-RSRP（Layer 1 参考信号接收功率）；
    3. UE 通过 PUCCH 上报最强的 SSB Beam ID 及对应 RSRP 值（即 Beam Report）；
    4. gNB 根据上报结果，确定与该 UE 通信的最优 SSB 波束方向。

    每个 SSB 波束粒度相对较粗（覆盖角度较宽），P1 的目的是快速缩小搜索范围。

    P2：发射端波束细化（TX Beam Refinement）

    P2 过程在完成 P1 后进行，基于 CSI-RS（Channel State Information Reference Signal） 在 P1 确定的角度范围内进行更精细的波束扫描：

    1. gNB 在 P1 确定的方向附近，发射多个更窄的 CSI-RS 波束（波束权重更集中）；
    2. UE 固定接收波束，测量各 CSI-RS 波束的 L1-RSRP；
    3. UE 上报最优 CSI-RS 资源 ID，gNB 确定精细化的下行发射波束；
    4. 后续 PDSCH 数据传输使用此精细波束。

    P3：接收端波束细化（RX Beam Refinement）

    P3 过程用于优化 UE 侧的接收波束（主要在 FR2 mmWave 场景，UE 也有方向性天线）：

    1. gNB 使用 P2 确定的发射波束，重复发送 CSI-RS（固定 TX 波束）；
    2. UE 切换不同的 RX 波束，测量每个 RX 波束下的 CSI-RS RSRP；
    3. UE 选择 RSRP 最高的 RX 波束作为最优接收波束；
    4. 最终形成最优的 TX-RX 波束对。

    波束管理的周期维护

    初始波束建立后，gNB 需要持续追踪 UE 位置变化导致的最优波束变化：

    • 周期性 CSI-RS 测量：gNB 配置 UE 周期上报 CSI-RS 测量，用于波束跟踪（Beam Tracking）；
    • 事件触发波束切换：当当前 BPL 质量下降（L1-RSRP 低于阈值），触发 P2/P3 重新选择波束；
    • 上行波束管理：gNB 通过测量 UE 发送的 SRS（Sounding Reference Signal） 来确定最优上行接收波束（利用信道互易性 Channel Reciprocity，在 TDD 系统中 UL 和 DL 信道互易，可从 SRS 推断 DL 最优波束）。

    6.3 波束失败恢复（BFR）

    当所有配置的波束质量都低于门限（Beam Failure Detection，BFD），UE 判定为波束失败（Beam Failure），触发 Beam Failure Recovery（BFR） 流程（定义于 TS 38.321）：

    1. 候选波束检测：UE 扫描预配置的候选参考信号（SSB 或 CSI-RS），找到满足质量要求的新候选波束；
    2. BFR 请求发送：UE 在新候选波束对应的 PRACH 资源上（或 PUCCH 资源上）发送 BFR 请求；
    3. gNB 响应：gNB 使用新波束方向进行 PDCCH 应答，完成波束恢复。

    BFR 的时延目标通常在几十毫秒以内，确保业务不中断（区别于完整的 RRC 重建立，BFR 在 MAC 层完成，速度更快）。

    7. FR1 vs FR2 频段特性与部署差异

    7.1 频段划分定义

    根据 3GPP TS 38.104，5G NR 频段被划分为两大频率范围（Frequency Range，FR）：

    频率范围 缩写 频率范围 说明
    Frequency Range 1 FR1 410 MHz ～ 7125 MHz Sub-7GHz，又称”Sub-6 GHz”（严格说包含到 7.125 GHz）
    Frequency Range 2-1 FR2-1 24250 MHz ～ 52600 MHz 毫米波（mmWave）主力频段（R15 定义）
    Frequency Range 2-2 FR2-2 52600 MHz ～ 71000 MHz 更高毫米波频段（R17 引入）

    注：FR3（7 GHz ～ 24 GHz 的”上中频”）已在 2023 年世界无线电通信大会（WRC-23）讨论，但尚未纳入正式 3GPP 标准（截至 R18）。

    典型 FR1 频段（部分）：

    NR Band 上行（UL）频率 下行（DL）频率 双工方式 主要地区
    n1 1920～1980 MHz 2110～2170 MHz FDD 全球
    n28 703～748 MHz 758～803 MHz FDD 亚太、欧洲
    n41 2496～2690 MHz 2496～2690 MHz TDD 中国（中国移动主力）
    n77 3300～4200 MHz 3300～4200 MHz TDD 全球（3.5 GHz 主流）
    n78 3300～3800 MHz 3300～3800 MHz TDD 欧洲、亚洲
    n79 4400～5000 MHz 4400～5000 MHz TDD 中国、日本

    典型 FR2 频段（部分）：

    NR Band 频率范围 双工方式 主要地区
    n257 26500～29500 MHz TDD 美国（28 GHz）
    n258 24250～27500 MHz TDD 欧洲（26 GHz）
    n260 37000～40000 MHz TDD 美国（39 GHz）
    n261 27500～28350 MHz TDD 美国

    7.2 FR1 特性与部署

    FR1（Sub-7 GHz） 是全球 5G 部署的主力频段，具有以下特性：

    传播特性

    • 穿透损耗低：低频段信号绕射能力强，穿墙损耗小（3.5 GHz 典型穿墙损耗约 15～20 dB，远低于 28 GHz 的 40+ dB），室内覆盖能力强；
    • 覆盖范围广：单站覆盖半径可达数公里（低频 700 MHz 甚至可达 10+ km），适合广域覆盖；
    • 多径丰富：Sub-6 GHz 环境中多径反射、散射成分丰富，利于传统空间复用（SU/MU-MIMO）。

    天线与 MIMO

    • FR1 天线尺寸相对较大（波长 = 5 cm～73 cm），单站天线规模通常为 32T32R 或 64T64R（Massive MIMO AAU）；
    • 由于多径丰富，FR1 的 MIMO 增益主要来自数字波束赋形 + 空间复用，而非纯模拟波束扫描；
    • UE 侧通常不需要方向性天线（采用全向或半向天线），UE 不参与 P3 RX 波束细化。

    信道带宽与容量

    • FR1 最大单载波信道带宽 100 MHz（TS 38.101-1）；
    • 通过载波聚合（CA），最多可聚合 16 个载波，理论峰值带宽可达 1600 MHz（实际部署中通常 2～4 个载波聚合）；
    • 子载波间隔（SCS）支持 15 kHz、30 kHz、60 kHz（μ = 0/1/2）。

    典型部署场景

    • 宏站覆盖：700 MHz / 800 MHz 频段用于农村、郊区广覆盖；
    • 容量层：3.5 GHz（n77/n78）是全球 5G 容量的主力担当；
    • 室内覆盖：FR1 可穿墙，适合分布式天线系统（DAS）或小基站室内覆盖。

    7.3 FR2 特性与部署

    FR2（mmWave，毫米波） 是 5G NR 的超高速频段，具有与 FR1 截然不同的物理特性：

    传播特性

    • 路径损耗极大：28 GHz 的自由空间路径损耗比 3.5 GHz 高约 18 dB，穿透损耗极高（混凝土墙体穿透损耗可达 40～80 dB）；
    • 覆盖范围极小：典型小区半径仅 100～300 m，适合密集热点覆盖；
    • 雨衰：降雨对毫米波有明显吸收（约 0.01 dB/m @28 GHz），恶劣天气覆盖下降；
    • 几乎无多径：高频信号在障碍物处被快速吸收，多径成分稀疏，依赖直射路径（LoS）。

    天线与波束赋形

    • FR2 天线尺寸极小（28 GHz 波长 ≈ 10 mm），可在小型面板中集成大量天线（如 256 个天线元素）；
    • 强制需要模拟波束赋形：由于路径损耗极大，必须通过高方向性波束（Beam Gain 可达 20～30 dBi）补偿路径损耗，才能建立可靠链路；
    • UE 侧同样配备方向性天线阵列，需要执行 P3 RX 波束细化；
    • 天线面板（Panel）：FR2 UE 通常配置多个天线面板（如手机四个面），通过面板切换应对 UE 旋转导致的遮挡（手持遮挡 Body Blocking）。

    信道带宽与容量

    • FR2 最大单载波信道带宽 400 MHz（FR2-1，TS 38.101-2）；
    • 子载波间隔（SCS）支持 60 kHz 和 120 kHz（μ = 2/3），极短 Slot 时长（120 kHz SCS 下 Slot = 0.125 ms）；
    • 极高带宽带来超高峰值速率（理论可达 20 Gbps+），适合超高清视频、VR/AR 等 eMBB 极限场景。

    典型部署场景

    • 室外热点：体育场、音乐厅、CBD 商业区——密集用户高吞吐量需求；
    • 室内专网：工厂、机场、商场——利用 FR2 的超高带宽提供”光纤级”无线连接；
    • 固定无线接入（FWA）：楼宇之间的点对点高速链接，替代固网宽带。

    7.4 FR1 vs FR2 对比总结

    特性 FR1（Sub-7 GHz） FR2（mmWave）
    频率范围 410 MHz ～ 7125 MHz 24.25 GHz ～ 71 GHz
    规范 TS 38.101-1 TS 38.101-2
    最大单载波带宽 100 MHz 400 MHz（FR2-1）
    子载波间隔 15 / 30 / 60 kHz 60 / 120 kHz
    覆盖半径 100 m ～ 数十 km 50 ～ 300 m
    穿透能力 较强 极弱（需要 LoS）
    峰值速率 百 Mbps ～ 数 Gbps 可达 20 Gbps+

    8. 本章小结

    本文从四个维度深入剖析了 NG-RAN 接入网架构：

    （1）gNB 内部架构：3GPP TS 38.401 定义了 gNB 的 CU/DU 拆分与 CU-CP/CU-UP 进一步分离，实现了控制面与用户面的解耦，为云化部署和边缘计算提供了架构基础。F1-C（NGAP over SCTP）承载 RRC 消息传递和 UE 上下文管理，F1-U（GTP-U over UDP）承载用户数据；E1 接口则是 CU-CP 统一管理多个 CU-UP 的控制通道。

    （2）外部接口：Xn 接口（TS 38.420～38.424）实现 gNB 间协同，支持 Xn 切换、双连接和 RAN 寻呼；NG 接口（TS 38.410～38.414）连接 5GC，NG-C（NGAP）承载 NAS 透传和移动性信令，NG-U（GTP-U）承载用户面 PDU Session 数据。

    （3）O-RAN 与 RIC：O-RAN Alliance 在 3GPP 基础上定义了 Non-RT RIC（> 1 s，rApp）和 Near-RT RIC（10 ms ～ 1 s，xApp）双层智能控制架构，通过 A1、E2、O1、Open Fronthaul 等开放接口，推动 RAN 的 AI 驱动智能化和多厂商开放生态。

    （4）Massive MIMO 与频段：P1/P2/P3 三阶段波束管理流程实现了从粗粒度波束扫描到精细波束对建立的完整过程，BFR 机制保障了波束失败场景下的快速恢复。FR1 和 FR2 频段在传播特性、天线规模、带宽能力上各有侧重，构成了 5G NR 覆盖（FR1）与容量/速率（FR2）的双轮驱动。
