    1. 用户面协议栈总览

    1.1 协议栈分层结构

    5G NR 用户面协议栈位于 UE 与 gNB 之间，负责将应用层的 IP 数据包安全、可靠、高效地在无线空口上传输。根据 3GPP TS 38.300，用户面协议栈由上至下共分为五个子层：

    注意：SDAP 是 5G NR 相比 LTE 新增的子层，仅存在于连接 5G 核心网（5GC）的场景（SA 架构）。在 NSA（Option 3x）架构中，LTE 侧不配置 SDAP。

    1.2 各层核心职责速览

    子层 核心职责 关键规范
    SDAP QoS Flow → DRB 映射；QFI 标记 TS 37.324
    PDCP 头压缩（ROHC）；加密与完整性保护；PDCP SN 维护；重排序 TS 38.323
    RLC 分段与重组；ARQ（AM 模式）；TM / UM / AM 三模式 TS 38.322
    MAC 逻辑信道到传输信道的复用；HARQ；调度请求 TS 38.321
    PHY 编码调制；多天线处理；物理资源映射 TS 38.211/212/213/214

    1.3 SDU 与 PDU 的概念

    在协议栈中，每一层的数据单元有严格定义：

    • SDU（Service Data Unit）：来自上层的数据单元，即本层的”输入”。
    • PDU（Protocol Data Unit）：经本层处理后交给下层的数据单元，即本层的”输出”。

    数据从 SDAP 自顶向下流动时，每层在 SDU 基础上添加头部（Header），形成本层 PDU，再作为下层 SDU；反方向（上行接收）则依次去头还原。

    2. SDAP：QoS Flow 到 DRB 的映射

    2.1 SDAP 存在的背景与意义

    在 LTE/EPC 体系中，QoS 管理以 EPS Bearer 为粒度，承载与 QoS 参数绑定，接入网（eNB）直接感知 Bearer。

    进入 5G/5GC 时代，核心网引入了更细粒度的 QoS Flow 概念（参见 TS 23.501）：

    • 一个 PDU Session 内可包含多个 QoS Flow，每个 Flow 携带 QFI（QoS Flow Identifier）。
    • UPF 与 gNB 之间通过 GTP-U 隧道传输数据，下行方向通过 GTP-U 扩展头中的 QFI 区分 Flow。
    • gNB 侧需要将 QoS Flow 映射到无线空口的 DRB（Data Radio Bearer） 上。

    SDAP（Service Data Adaptation Protocol） 正是为了承担这一映射职责而被引入。SDAP 规范定义在 TS 37.324，之所以使用 TS 37.x 编号（而非 38.x），是因为 SDAP 同样可适用于 LTE UE 连接 5GC 的场景（EPS 与 NR 共用）。

    2.2 SDAP 实体模型

    根据 TS 37.324 第 4 章，每个 PDU Session 对应一个 SDAP 实体（Dual Connectivity 场景下可配置两个实体）。SDAP 实体存在于 gNB 和 UE 两侧，仅位于用户面，控制面无 SDAP。

    2.3 QoS Flow 到 DRB 的映射规则

    2.3.1 映射关系约束

    TS 37.324 规定：

    • 一个 QoS Flow 在上行方向同一时刻只能映射到一个 DRB（1:1 UL 约束）。
    • 一个 DRB 可以承载来自同一 PDU Session 的多个 QoS Flow（多 Flow 聚合到一个 DRB）。
    • 每个 PDU Session 有且只有一个 Default DRB。当没有明确映射规则时，数据包默认路由到 Default DRB。

    2.3.2 映射配置来源

    QoS Flow 到 DRB 的映射规则由 RRC 层通过 TS 38.331 中的 SDAP-Config IE 配置下发，主要参数包括：

    参数 含义
    defaultDRB 是否为该 PDU Session 的默认 DRB
    mappedQoS-FlowsToAdd 需要映射到本 DRB 的 QFI 列表
    mappedQoS-FlowsToRelease 从本 DRB 移除的 QFI 列表
    sdap-HeaderDL 下行是否添加 SDAP 头部
    sdap-HeaderUL 上行是否添加 SDAP 头部

    2.3.3 End-Marker PDU

    当某个 QoS Flow 需要从一个 DRB 重映射（remap）到另一个 DRB 时，发送侧的 SDAP 实体会在原 DRB 上发送一个 End-Marker Control PDU，通知接收端该 DRB 上该 Flow 的数据流已结束，后续数据将从新 DRB 到达，避免乱序。

    2.4 SDAP PDU 格式

    SDAP PDU 分为 Data PDU 和 Control PDU 两类。

    SDAP Data PDU 格式（含 SDAP 头部时）：

    字段 长度 含义
    D/C 1 bit 区分 Data PDU（1）还是 Control PDU（0）
    RQI 1 bit Reflective QoS Indication，置 1 时通知 UE 使用反射 QoS
    QFI 6 bit QoS Flow Identifier，标识本数据包所属的 QoS Flow

    注：当 sdap-HeaderDL / sdap-HeaderUL 未配置时，SDAP 头部可以省略（无头模式）。此时 SDAP 层对数据透明传递，不添加任何开销。

    2.5 反射 QoS（Reflective QoS）

    反射 QoS 是 5G 引入的一种简化 QoS 配置的机制（定义于 TS 23.501）：

    1. 核心网下行发送数据时，在 SDAP 头部将 RQI 置 1，并携带 QFI。
    2. UE 的 SDAP 实体检测到 RQI=1 后，自动推导出：该 QFI 对应的上行数据应映射到与下行相同的 DRB。
    3. UE 无需等待显式的 NAS 或 RRC 映射规则配置，即可建立上行的 QoS Flow → DRB 映射。

    这一机制特别适合 IP 应用的对称流（如视频通话），减少信令开销。

    3. PDCP：头压缩 / 加密 / 完整性保护 / 重排序

    PDCP（Packet Data Convergence Protocol）由 TS 38.323 定义，是用户面和控制面共用的重要子层（控制面 SRB 与用户面 DRB 均使用 PDCP）。

    3.1 PDCP 子层的完整功能列表

    根据 TS 38.323 第 4.4 节，PDCP 提供以下功能：

    1. 数据传输（用户面与控制面）
    2. PDCP 序列号（SN）维护
    3. 头压缩与解压缩（ROHC / EHC / UDC）
    4. 加密与解密（Ciphering）
    5. 完整性保护与校验（Integrity Protection）
    6. 基于定时器的 SDU 丢弃（t-DiscardTimer）
    7. 针对分裂承载（Split Bearer）的路由
    8. PDCP PDU 复制（Duplication，用于可靠性增强）
    9. 重排序与顺序递交（Reordering & In-order Delivery）
    10. 乱序递交（Out-of-order Delivery，可选配置）
    11. 重复丢弃（Duplicate Discarding）

    3.2 PDCP 实体模型

    每个无线承载（SRB 或 DRB）对应一个 PDCP 实体（DAPS 切换场景下可有两套安全配置）。PDCP 实体由 RRC 配置，位于 CU-CP（控制面）或 CU-UP（用户面，F1 分离架构下）中。

    3.3 PDCP 序列号（SN）与 COUNT 值

    PDCP 为每个 PDU 分配一个 PDCP SN，与 HFN（Hyper Frame Number） 合并构成 COUNT 值：

    COUNT 值在加密和完整性保护中充当输入参数（等同于序列计数），保证每个 PDU 使用唯一的加密流，防止重放攻击。

    NR PDCP SN 长度可配置：

    承载类型 SN 长度 最大 SN 值
    SRB（信令承载） 12 bit 4096
    DRB 12 bit 或 18 bit 4096 / 262144

    相比 LTE 最大 15 bit SN，NR 引入 18 bit SN 是为了支持更高吞吐率下的更大窗口，避免在高速传输时 SN 回绕导致的问题。

    3.4 头压缩（ROHC）

    3.4.1 为何需要头压缩

    IP/UDP/RTP 报头典型大小为 40 字节（IPv4）或 60 字节（IPv6），对于 VoIP 等每帧载荷仅 2030 字节的业务而言，头部开销占比极高。头压缩可将其压缩到 13 字节，显著节省无线资源。

    3.4.2 ROHC 框架（RFC 5795）

    NR PDCP 支持 ROHC（Robust Header Compression），定义于 IETF RFC 5795，核心思想是：

    • 初始化（IR 模式）：传输完整未压缩头部，建立上下文（Context）。
    • 一阶（FO 模式）：传输变化字段的增量，固定字段不传。
    • 二阶（SO 模式）：仅传输最小变化量（如 RTP SN、时间戳），压缩比最高。

    当信道质量较差导致上下文同步失败时，ROHC 可回退到更保守的模式重新同步。

    TS 38.323 支持的 ROHC Profile（TS 38.323 Table 5.7.1-1）：

    Profile ID 适用协议
    0x0000 无压缩（ROHC 不压缩模式）
    0x0001 RTP/UDP/IP（IPv4）
    0x0002 UDP/IP（IPv4）
    0x0003 ESP/IP
    0x0004 IP（IPv4 only）
    0x0006 TCP/IP
    0x0101 RTP/UDP/IP（IPv6）
    0x0102 UDP/IP（IPv6）
    0x0104 IP（IPv6）

    每个 PDCP 实体最多配置一个 ROHC 压缩实例和一个解压缩实例，下行与上行各为单向信道。

    R17 起还引入了 EHC（Ethernet Header Compression） 和 UDC（Uplink Data Compression），分别面向 TSN 以太帧承载和上行数据体积压缩场景。

    3.5 加密（Ciphering）

    3.5.1 加密范围

    根据 TS 38.323 第 5.8 节，加密对象为：

    • PDCP Data PDU 的数据部分（含 MAC-I 字段），不包括 SDAP 头部（如存在）。
    • PDCP Control PDU 不加密。

    3.5.2 加密算法（NEA）

    加密算法由 RRC 配置，支持以下三种（TS 33.501）：

    算法标识 算法名称 说明
    NEA0 空算法 不加密（测试/紧急业务）
    128-NEA1 SNOW 3G 基于流加密
    128-NEA2 AES-CTR 128 bit 密钥，计数器模式
    128-NEA3 ZUC 中国祖冲之算法

    • DIRECTION：上行（0）/ 下行（1）

    3.6 完整性保护（Integrity Protection）

    3.6.1 保护范围

    • SRB（信令承载）：所有 PDCP Data PDU 强制进行完整性保护。
    • DRB（数据承载）：由 RRC 配置决定是否启用（R15 默认不启用，R16 后针对用户面完整性保护有增强选项，但考虑到性能开销，通常仅在高安全需求场景配置）。
    • PDCP Control PDU 不进行完整性保护。

    3.6.2 完整性保护算法（NIA）

    算法标识 算法名称
    NIA0 空算法
    128-NIA1 SNOW 3G
    128-NIA2 AES-CMAC
    128-NIA3 ZUC

    3.6.3 MAC-I 字段

    完整性保护的输出为 MAC-I（Message Authentication Code for Integrity），长度 4 字节（32 bit），附在 PDCP PDU 末尾。接收端重新计算 MAC-I 并与收到的 MAC-I 对比，不一致则丢弃 PDU。

    3.7 重排序（Reordering）与顺序递交

    由于无线信道的多路径传输（Split Bearer / CA）以及 RLC AM 层的 ARQ 乱序，PDCP 接收端收到的 PDU 序号可能不连续。PDCP 通过以下机制处理：

    3.7.1 重排序窗口

    PDCP 维护一个接收窗口，以 COUNT 值排序。状态变量：

    • RX_NEXT：期望接收的下一个 PDCP COUNT 值。
    • RX_DELIV：待递交给上层的最小 COUNT 值（已按序排列的指针）。
    • RX_REORD：触发 t-Reordering 的 COUNT 值。

    3.7.2 t-Reordering 定时器

    当 PDCP 检测到 SN 乱序（收到 COUNT 值比 RX_DELIV 更大但中间有空缺），启动 t-Reordering 定时器。定时器超时前，乱序的 PDU 在缓冲区等待；超时后，无论空缺是否填补，强制将所有已收到的 PDU 按序递交给上层，防止无限等待造成延迟。

    3.7.3 切换场景的 PDCP 重建

    切换时，PDCP 实体可能需要重建（Re-establishment）：

    • AM DRB：未成功递交的 SDU 需要重传（PDCP Status Report 机制），以保证数据不丢失。
    • UM DRB：重建后丢弃窗口外数据，直接递交窗口内有序数据。

    PDCP Status Report（状态报告）包含 FMC（First Missing COUNT），告知对端从哪个 COUNT 值开始重传，用于切换场景的数据恢复。

    4. RLC：三种模式与重传机制

    RLC（Radio Link Control）由 TS 38.322 定义，位于 PDCP 之下、MAC 之上，提供逻辑信道上的数据传输服务。

    4.1 RLC 核心功能

    • 上层 PDU 的传输（分段 / 重组）
    • AM 模式下的 ARQ 错误纠正
    • RLC SDU 的分段（Segmentation）与拼接（按 MAC 层指示的 TB 大小切分）
    • SDU 的丢弃（上层触发）

    NR 相比 LTE 的重要变化：NR RLC 取消了拼接（Concatenation）功能，Concatenation 下移到 MAC 层处理；同时也取消了 LTE 中的 RLC 重分段（Re-segmentation 机制简化）。这使得 RLC PDU 格式更简洁，同时降低了时延（不必等待多个 SDU 再拼接）。

    4.2 RLC 三种模式

    TS 38.322 定义了三种 RLC 模式，对应不同的逻辑信道和业务需求。

    4.2.1 TM（Transparent Mode，透明模式）

    应用场景：

    • BCCH（广播控制信道）：SIB 系统信息广播
    • PCCH（寻呼控制信道）：寻呼消息
    • CCCH（公共控制信道）：RRC 连接建立（初始 Msg3）

    工作机制：

    • TM RLC 实体的 TMD PDU 不含任何 RLC 头部，完全透明传输。
    • 不分段、不重传、不排序，保持原始 SDU 的完整性。
    • 一个 TMD PDU 对应一个 RLC SDU（1:1 关系）。

    优点：开销为零，延迟最低。
    缺点：无差错保护，依赖上层（PDCP）或下层（HARQ）提供可靠性。

    4.2.2 UM（Unacknowledged Mode，非确认模式）

    应用场景：

    • DTCH（专用流量信道）：实时业务，如 VoNR（语音），eMBB 视频流
    • MCCH / MTCH（组播控制 / 流量信道）

    工作机制：

    • UM RLC 实体不提供 ARQ 重传，接收方不向发送方发送 ACK/NACK。
    • 支持 SDU 的分段（Segmentation）：当 MAC 层指示的 TB 空间不足时，可将 RLC SDU 分成多个 UMD PDU 传输。
    • 接收端可进行重组（Reassembly）和重排序（维护接收窗口，由 t-Reassembly 定时器驱动）。
    • SN 长度可配置为 6 bit 或 12 bit（由 RRC 配置）。

    UMD PDU 头部字段：

    字段 描述
    SI（Segmentation Info） 2 bit，指示是否为完整 SDU（00），首分段（01），末分段（10），中间分段（11）
    SN 6 bit 或 12 bit，序列号
    SO（Segment Offset） 16 bit，当为非首分段时携带，指示分段在原 SDU 中的字节偏移

    t-Reassembly 定时器：接收端检测到分段空缺时启动；超时后，将已缓存的分段强制上报或丢弃，防止无限等待。

    4.2.3 AM（Acknowledged Mode，确认模式）

    应用场景：

    • DCCH（专用控制信道）：RRC 信令（SRB）
    • DTCH（专用流量信道）：需要可靠传输的数据（大多数 DRB，如 TCP 数据）

    工作机制：

    AM 是最复杂的 RLC 模式，提供基于 ARQ（Automatic Repeat reQuest）的可靠传输。

    AM RLC 发送侧状态机：

    1. 接收上层 PDU，分配 SN，构建 AMD PDU。
    2. 将 AMD PDU 提交 MAC 层发送，同时保留副本在重传缓冲区。
    3. 当收到 STATUS PDU（NACK）时，将对应 AMD PDU 标记为重传。
    4. 当重传计数（RETX_COUNT）达到 maxRetxThreshold（RRC 配置）时，触发 RLC 错误（通知上层链路失败）。

    STATUS PDU（NACK/ACK 机制）：

    • STATUS PDU 是 AM 模式唯一的控制 PDU（D/C bit = 0）。
    • 包含 ACK_SN（当前已成功连续接收的最大 SN + 1）及 NACK_SN 列表（带 SO 范围，支持部分重传）。
    • 接收端可通过 polling 机制触发发送（发送端在 AMD PDU 中置 P bit）。
    • 为防止 STATUS PDU 风暴，接收端受 t-StatusProhibit 定时器限制（超时前不重复发送）。

    AM RLC PDU 格式（AMD PDU）：

    字段 大小 描述
    D/C 1 bit 数据（1）或控制（0）
    P 1 bit Polling bit，请求接收端发送 STATUS PDU
    SI 2 bit 分段信息
    SN 12 bit 或 18 bit 序列号（RRC 配置）
    SO 16 bit 非首分段时携带，字节偏移

    相比 LTE（最大 10 bit SN），NR AM RLC 支持 12 bit 或 18 bit SN，最大窗口大小分别为 2048 和 131072，可满足极高吞吐场景。

    STATUS PDU 格式关键字段：

    字段 描述
    ACK_SN 最大连续接收的下一个 SN
    NACK_SN 丢失或需重传的 SN
    SOstart/SOend NACK 对应的字节范围（支持部分段的重传）
    NACK_range 连续多个 SN 的 NACK（压缩表达）
    E1/E2/E3 扩展位，指示后续是否还有 NACK_SN / SO / NACK_range

    4.2.4 三种模式对比

    特性 TM UM AM
    RLC 头部 无 有（SI/SN/SO） 有（D/C/P/SI/SN/SO）
    分段支持 ✗ ✓ ✓
    ARQ 重传 ✗ ✗ ✓
    顺序递交 ✗ 可选 ✓
    序列号长度 — 6/12 bit 12/18 bit
    典型逻辑信道 BCCH/PCCH/CCCH DTCH（语音/视频） DCCH/DTCH（数据）
    适用业务 广播/寻呼 实时低延迟业务 可靠数据传输

    4.3 RLC 与 HARQ 的关系

    NR 用户面错误纠正采用两层重传架构：

    • HARQ 工作在 PHY/MAC 层，基于软合并（Chase Combining 或 Incremental Redundancy），RTT 约 8ms，通常最多 4 次重传。
    • RLC ARQ 工作在 RLC AM 层，仅在 HARQ 最终失败（所有重传均不成功）后触发，RTT 相对较长（涉及 L2 反馈）。
    • 两层重传形成互补：HARQ 处理大多数错误（低延迟），ARQ 处理极端错误（高可靠）。

    5. MAC 与 PHY 简述

    本节简要描述 MAC 和 PHY 的职责（后续专题文章将详细展开）。

    5.1 MAC 子层（TS 38.321）

    MAC（Medium Access Control）位于 RLC 之下、PHY 之上，主要职责：

    • 逻辑信道（Logical Channel）→ 传输信道（Transport Channel）的复用/解复用：将来自不同 RLC 实体（不同逻辑信道）的数据拼接打包成一个 MAC PDU（TB，Transport Block）。
    • 调度与资源分配：由 gNB 的 MAC 调度器决定每个 TTI 分配哪些 UE、哪些逻辑信道、多大 TB 的传输机会。
    • HARQ 管理：维护多个 HARQ 进程，管理重传逻辑（停等 + 软合并）。
    • 上行 SR / BSR / PHR 上报：UE 通过调度请求（SR）、缓冲区状态报告（BSR）、功率余量报告（PHR）告知 gNB 自己的需求。
    • 随机接入（RACH）管理：支持基于竞争和非竞争的随机接入过程。

    5.2 PHY 物理层（TS 38.211/212/213/214）

    PHY 负责将 MAC 层的 TB（Transport Block）转换为无线信号发送，主要处理：

    • 信道编码：数据信道使用 LDPC，控制信道使用 Polar Code（NR 相比 LTE 的重大改进）。
    • 速率匹配与调制：QPSK / 16QAM / 64QAM / 256QAM，链路自适应（MCS 选择）。
    • OFDM 波形：下行 CP-OFDM，上行 CP-OFDM（也可选 DFT-s-OFDM 降低 PAPR）。
    • 多天线处理：Massive MIMO 波束赋形，空分复用。
    • 物理信道与信号：PDSCH / PDCCH / PBCH（下行）；PUSCH / PUCCH / PRACH / SRS（上行）；DMRS / CSI-RS 等参考信号。

    7. 小结与对比：NR vs LTE 协议栈差异

    特性 LTE（EPS） 5G NR（5GS）
    QoS 管理层 无独立 SDAP（EPS Bearer 直接映射） 新增 SDAP，QoS Flow → DRB 映射
    PDCP SN 最大长度 15 bit（DRB AM） 18 bit（DRB AM），更大窗口
    头压缩 ROHC ROHC + EHC（R17）+ UDC（R17）
    RLC 拼接（Concatenation） 在 RLC 层做 下移到 MAC 层（RLC 不做拼接）
    RLC AM SN 最大 10 bit（窗口 512） 18 bit（窗口 131072）
    信道编码 Turbo Code 数据信道 LDPC + 控制信道 Polar Code
    HARQ 同步 HARQ（FDD/TDD 各异） 异步 HARQ（灵活时隙调度）
    完整性保护 仅 SRB SRB 必选，DRB 可选（R16 增强）
