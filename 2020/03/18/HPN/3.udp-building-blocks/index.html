    一、UDP 简介：被设计的”空协议”

    用户数据报协议（User Datagram Protocol，UDP）由 Jon Postel 于 1980 年 8 月 正式加入核心网络协议栈。其诞生时机颇为特殊，彼时 TCP 与 IP 规范正好被拆分为两份独立的 RFC。这一背景极为关键：UDP 的核心价值不在于它引入了什么，而在于它主动省略了什么。因此，UDP 在业界常被戏称为”空协议（null protocol）“，其完整规范文档 RFC 768 薄到可以写在一张餐巾纸上。

    数据报（Datagram）与数据包（Packet）的区别

    这两个词常被混用，但存在细微差别：

    • Packet（数据包）：泛指任意格式化的数据块。
    • Datagram（数据报）：特指通过不可靠服务传输的包，没有投递保证，没有失败通知，是一个自包含、独立的数据实体，携带足够信息以从源节点路由到目标节点，且不依赖节点间的任何历史通信记录。

    正是这种”不可靠”特性，让 UDP 有时被非正式地称为 **”Unreliable Datagram Protocol（不可靠数据报协议）”**。

    UDP 的典型应用：DNS 与 WebRTC

    DNS（域名系统） 是 UDP 最广为人知的应用场景，几乎每一次浏览器访问网站都依赖它：在任何数据交换发生之前，都需要先将人类可读的主机名解析为 IP 地址，而这个解析过程走的正是 UDP。

    尽管浏览器自身深度依赖 UDP，但历史上 UDP 从未作为一等公民的传输层协议暴露给网页应用，直到 WebRTC 的出现。

    由 IETF 与 W3C 工作组联合制定的 WebRTC（Web Real-Time Communication） 标准，使浏览器原生支持了基于 UDP 的实时通信能力，包括语音/视频通话和 P2P 通信，并提供了客户端 JavaScript API。这是 UDP 在浏览器生态中的一次重要跃升。

    二、Null Protocol：UDP 的极简设计

    IP 层的基础能力

    要理解 UDP 为何被称为”空协议”，需要先了解其下层的 IP 协议（Internet Protocol）。

    IP 层的核心任务是：根据地址将数据报从源主机传送到目标主机。数据会被封装进 IP 数据包中，包含源/目标地址及路由参数。IP 层同样不提供任何投递保证和失败通知，它将底层网络的不可靠性直接暴露给上层协议。

    如果路由节点因拥塞或高负载而丢包，是否检测、恢复、重传，完全取决于 IP 层之上的协议，如果你需要这种能力的话。

    UDP 的报文结构

    UDP 将用户消息封装进自己的包结构，在 IP 之上仅新增了 4 个字段，构成一个 8 字节的报头：

    字段 说明
    Source Port（源端口） 可选字段
    Destination Port（目标端口） 必选字段，用于标识目标应用
    Length（包长度） 整个 UDP 数据报的字节长度
    Checksum（校验和） 可选字段，可委托给上层应用处理

    值得注意的是，源端口和校验和均为可选字段。IP 包头自带校验，应用层可以选择省略 UDP 校验，将错误检测与纠正全权交给应用自身处理。

    UDP 的本质是在 IP 之上提供”应用多路复用（Application Multiplexing）“，通过源/目标端口区分不同的通信进程，仅此而已。

    UDP 刻意省略的特性

    特性 TCP UDP
    投递保证 ✅ ACK / 重传 / 超时 ❌ 无
    顺序保证 ✅ 序列号 / 重排序 / 队头阻塞 ❌ 无
    连接状态跟踪 ✅ 连接建立 / 断开状态机 ❌ 无
    拥塞控制 ✅ 内置客户端与网络反馈机制 ❌ 无

    此外，TCP 是字节流协议，一条应用消息可以跨多个包传输，不含显式消息边界；而 UDP 数据报有明确的边界，每个数据报对应一个 IP 包，每次 read 操作返回完整消息，数据报不可分片。

    “自己动手”的风险

    UDP 是一个简单的无状态协议，几乎把所有协议设计决策都留给了上层应用。这提供了极大的灵活性，但也带来了相应的工程挑战：

    • 需要处理 NAT 穿透问题（middleboxes 的普遍存在）
    • 需要遵循网络协议设计的最佳实践
    • 非常容易一不小心就实现出一个残缺版的 TCP

    TCP 的各种算法与状态机历经数十年的打磨与改进，涵盖了大量不易复现的机制。在决定基于 UDP 自建协议之前，请务必做好充分的调研与规划。

    三、UDP 与网络地址转换（NAT）

    3.1 NAT 的起源与现状

    IPv4 地址仅有 32 位，最多支持约 42.9 亿个唯一地址。随着 90 年代初互联网主机数量的指数级增长，IPv4 地址耗尽问题迫在眉睫。

    1994 年中期，IP 网络地址转换（NAT，RFC 1631） 规范作为临时解决方案被提出：在网络边缘部署 NAT 设备，维护一张”内网 IP:Port ↔ 公网 IP:Port”的映射表，使内网地址空间可以在不同网络中复用。

    然而，正如”最持久的往往是临时方案”所言，NAT 设备不仅解决了眼前的问题，还迅速成为了企业/家庭路由器、安全设备、防火墙等各类硬件和软件的标配组件，从临时方案演变为互联网基础设施中根深蒂固的一部分。

    3.2 私有网络地址段

    IANA（Internet Assigned Numbers Authority）为私有网络保留了三段 IP 地址范围，通常位于 NAT 设备之后：

    地址范围 地址数量
    10.0.0.0 – 10.255.255.255 16,777,216
    172.16.0.0 – 172.31.255.255 1,048,576
    192.168.0.0 – 192.168.255.255 65,536

    你的路由器分配给本机的地址大概率就来自上述范围之一，这就是你在内网的私有 IP，会由 NAT 设备在与外网通信时完成转换。公网上的任何主机都不能被分配这些私有地址段中的 IP。

    3.3 连接状态与超时问题

    NAT 的核心矛盾在于：NAT 中间件依赖连接状态，而 UDP 恰恰是无状态的。

    对于 TCP，NAT 设备可以通过清晰的握手/断开序列来管理路由记录的生命周期。但 UDP 没有握手，也没有连接终止序列，通信双方随时可能停止发送数据报，毫无预兆。

    为解决这个问题，NAT 设备通过定时器来过期 UDP 路由记录。超时时间因设备厂商、型号、版本和配置而异，没有统一标准。

    因此，对于长时间运行的 UDP 会话，业界的通行做法是：定期发送双向 keepalive 包，以防止 NAT 路由记录超时失效。

    TCP 的 NAT 超时问题：理论上 TCP 不需要额外的 NAT keepalive，因为 TCP 有明确的握手与终止序列。但实践中，许多 NAT 设备对 TCP 和 UDP 会话采用相同的超时逻辑，因此在某些情况下 TCP 也需要 keepalive。如果你的 TCP 连接莫名断开，中间 NAT 超时很可能是罪魁祸首。

    3.4 NAT 穿透问题

    对于 P2P 应用（VoIP、游戏、文件共享等），NAT 带来的更大挑战是：根本无法建立 UDP 连接。这类应用需要同时充当客户端和服务端，以实现双向直连通信。

    主要有两个问题：

    问题一：无法感知公网 IP

    NAT 设备在转发 UDP 包时会改写源端口/地址以及 IP 包中的源 IP。内网客户端只知道自己的私有 IP，如果它将私有 IP 作为应用数据传给外网的对等端，连接必然失败。因此，应用必须首先发现自己的公网 IP，才能与外网对等端共享。

    问题二：仅知道公网 IP 仍不够

    当一个数据包到达 NAT 设备的公网 IP 时，还需要在 NAT 表中找到对应的条目，才能路由到内网的目标主机。如果该条目不存在（对于外网主动发起的连接而言，这是常态），数据包会被直接丢弃。NAT 设备在没有显式端口映射配置的情况下，完全充当了一个简单的包过滤器。

    对于从内网发起连接的客户端应用，这不是问题，因为出站流量会自动建立 NAT 映射记录。但对于需要接收入站连接的 P2P 应用（充当服务端时），就会立刻遇到这个障碍。

    为此，需要借助 TURN、STUN、ICE 等穿透技术来建立 UDP 对等端之间的端到端连接。

    3.5 STUN、TURN 与 ICE

    STUN（Session Traversal Utilities for NAT）

    STUN（RFC 5389）是一种允许应用发现 NAT 存在并获取其公网 IP 和端口的协议。其工作原理依赖一台位于公网的第三方 STUN 服务器：

    STUN 解决的问题：

    1. 发现公网 IP:Port：应用可将此信息告知对等端。
    2. 建立 NAT 路由记录：向 STUN 服务器发出的出站请求会在路径上建立 NAT 映射，使入站回包能够正确路由到内网应用。
    3. 维护 NAT 记录：STUN 协议内置了简单的 keepalive 机制，防止 NAT 路由记录超时。

    两个对等端通过各自的 STUN 服务器获得公网 IP:Port 后，即可直接通信。

    然而，STUN 并非万能，它无法处理所有 NAT 拓扑和网络配置。在某些企业网络环境中，UDP 可能被防火墙直接封锁，STUN 完全失效。

    TURN（Traversal Using Relays around NAT）

    当 STUN 失败时，可以使用 TURN（RFC 5766）作为回退方案。TURN 可基于 UDP 运行，若 UDP 也不可用则切换到 TCP。

    TURN 的核心机制是中继（Relay）：

    工作流程：

    1. 两端各自向同一个 TURN 服务器发送 allocate 请求。
    2. 权限协商完成后，双方都将数据发送给 TURN 服务器，由其中继给对方。

    TURN 的显著缺点：这不再是真正的 P2P 连接，所有流量都经过中继服务器，运营成本高，服务器带宽需足以承载所有数据流。因此，TURN 应作为最后手段的回退方案，而非默认选择。

    STUN vs TURN 的真实世界数据

    Google 的开源 P2P 库 libjingle（用于 Google Talk 等应用）的文档给出了一组有参考价值的数据：

    • 92% 的情况下，可以直接建立连接（使用 STUN）。
    • 8% 的情况下，需要通过中继服务器（使用 TURN）。

    这说明即使有了 STUN，仍有相当比例的用户无法直接建立 P2P 通道，因此可靠的服务还必须配备 TURN 中继作为保障。

    四、UDP 的工程优化建议

    UDP 简单灵活，是构建新传输协议的常用基础，但这种灵活性对实现者来说也是一把双刃剑。你的应用可能需要从头实现部分或全部 TCP 已有的特性，而且每一项都必须与网络上的其他协议和对等端良好协作。

    最关键的风险：不同于 TCP 内置了流控、拥塞控制和拥塞避免，UDP 应用必须自行实现这些机制。不考虑拥塞的 UDP 应用很容易压垮网络，严重时可能导致网络拥塞崩溃（network congestion collapse）。

    RFC 5405 单播 UDP 应用设计指南（节选）

    RFC 5405 专门聚焦于单播 UDP 应用的设计规范，以下是核心建议：

    级别 建议
    MUST 能容忍广泛变化的互联网路径状况
    SHOULD 控制传输速率
    SHOULD 对所有流量执行拥塞控制
    SHOULD 使用与 TCP 相当的带宽
    SHOULD 在发生丢包后退避重传计数器
    SHOULD NOT 发送超过路径 MTU 的数据报
    SHOULD 处理数据报的丢失、重复和乱序
    SHOULD 能容忍最长 2 分钟的投递延迟
    SHOULD 启用 IPv4 UDP 校验和；必须启用 IPv6 校验和
    MAY 在需要时使用 keepalive（最短间隔 15 秒）

    最佳实践总结

    1. 充分研究：设计新传输协议需要大量周密的思考、规划与调研。
    2. 复用现有库：尽可能利用已考虑了 NAT 穿透，并能实现网络公平性的现有库或框架。
    3. 避免重复造轮子：TCP 的拥塞控制算法历经数十年演进，自行实现极易出错。

    好消息：WebRTC 就是这样一个已经为你处理好上述种种复杂性的框架，可以直接在浏览器中使用。

    五、总结

    维度 要点
    协议定位 UDP 是”空协议”，以省略特性为设计哲学，为上层应用提供最大灵活性
    核心机制 8 字节报头，仅提供应用多路复用（端口），无连接、无顺序、无重传
    NAT 挑战 NAT 依赖连接状态而 UDP 无状态，造成超时、穿透等一系列问题
    穿透方案 STUN（发现公网地址）→ TURN（中继回退）→ ICE（自动选择最优路径）
    工程要求 必须自行实现拥塞控制、流控、顺序保证等机制，并遵循 RFC 5405 规范
    推荐路径 优先使用 WebRTC 或实现了 ICE/STUN/TURN 的现有库，而非从零构建

    UDP 的价值在于其极简性与灵活性，它提供了最薄的抽象层，让应用开发者可以按需定制传输行为。但这种自由也伴随着责任：网络公平性、可靠性、顺序保证等问题都需要开发者在应用层认真应对。理解 UDP 的边界与局限，是构建高性能网络应用的重要基础。
