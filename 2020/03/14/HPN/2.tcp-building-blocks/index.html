    1. TCP 简介

    互联网的核心由两个协议构成：IP（Internet Protocol） 负责主机间的路由与寻址，TCP（Transmission Control Protocol） 则在不可靠的信道上提供可靠网络传输的抽象层。

    TCP/IP 协议组（也称 Internet Protocol Suite）由 Vint Cerf 和 Bob Kahn 于 1974 年在论文《A Protocol for Packet Network Intercommunication》中首次提出。经多次修订后，1981 年 v4 规范被拆分为两个独立的 RFC 正式发布：

    • RFC 791 — Internet Protocol
    • RFC 793 — Transmission Control Protocol

    小知识： IPv4 中的”4”正是来自 TCP/IP 协议的第 4 版，因为最初两个协议是耦合在一起的，v4 才正式将它们分开。IPv6 之所以跳过了 v5，是因为 v5 已被另一个实验性协议 Internet Stream Protocol（ST）占用，而该协议从未普及。

    TCP 为应用层屏蔽了网络通信的大部分复杂性，包括：

    • 丢包重传
    • 有序交付
    • 拥塞控制与拥塞避免
    • 数据完整性校验

    关键特性： TCP 保证发送端发送的所有字节与接收端收到的字节完全一致，且顺序相同。因此，TCP 是为准确传输而优化的，而非为及时传输。这一特性在优化 Web 性能时会带来一定挑战。

    尽管 HTTP 标准并不强制要求使用 TCP 作为传输协议，但实践中互联网上所有 HTTP 流量都通过 TCP 传输，因为它提供了大量开箱即用的强大特性。

    2. 三次握手（Three-Way Handshake）

    所有 TCP 连接都从三次握手开始。 在客户端或服务端交换任何应用数据之前，双方必须协商起始报文序号及其他连接参数（序号从双方各自随机选取，出于安全考虑）。

    握手过程如下：

    步骤 方向 说明
    SYN 客户端 → 服务端 客户端随机选择序号 x，发送 SYN 包（可附带额外 TCP 标志和选项）
    SYN-ACK 服务端 → 客户端 服务端将 x 加一，随机选择自己的序号 y，附加标志和选项后发送响应
    ACK 客户端 → 服务端 客户端将 x 和 y 各加一，发送最后的 ACK 包，完成握手

    握手完成后，应用数据才可以开始在客户端和服务端之间流动：客户端可以在发送 ACK 包后立即发送数据，而服务端必须等收到 ACK 后才能发送数据。

    性能影响

    每一个新 TCP 连接在传输任何应用数据之前，都要经历一个完整的往返时延（RTT）。

    以纽约客户端连接伦敦服务器为例，通过光纤链路建立新 TCP 连接的三次握手最少需要 56 毫秒（单程 28ms，往返一次）。这里带宽不是决定因素，延迟完全由传播时间决定。

    三次握手带来的延迟使得创建新 TCP 连接代价高昂，这也是连接复用对所有 TCP 应用来说都是关键优化手段的核心原因。

    3. TCP Fast Open（TFO）

    加载一个网页通常需要从数十个不同主机获取数百个资源，这意味着浏览器可能需要建立数十个新 TCP 连接，每个都要承受握手开销，在慢速移动网络上尤为明显。

    TCP Fast Open（TFO） 旨在通过允许在 SYN 包中直接携带数据，消除新 TCP 连接的延迟惩罚。

    TFO 的限制

    • SYN 包中数据负载有最大大小限制
    • 仅支持特定类型的 HTTP 请求
    • 依赖加密 Cookie，仅对重复连接有效

    启用条件

    • 服务端：Linux kernel v4.1+
    • 客户端：Linux，或 iOS 9+ / macOS 10.11+
    • 应用层需显式启用对应 socket 标志

    效果

    Google 的研究表明，TFO 可以：

    • 降低 HTTP 事务网络延迟约 15%
    • 整体页面加载时间平均降低超过 10%
    • 在高延迟场景下最高可降低 40%

    4. 拥塞控制与拥塞避免

    1984 年，John Nagle 在 RFC 896 中记录了一种被称为”拥塞崩溃（congestion collapse）“的现象：当网络节点间带宽不对称时，一旦往返时延超过最大重传间隔，主机就会不断向网络注入同一数据报的副本，最终导致网络陷入严重故障。

    1986 年，随着网络节点数量（5000+）和多样性的增长，一系列拥塞崩溃事件席卷整个 ARPANET，在某些情况下网络容量下降了 1000 倍，变得几乎无法使用。

    为此，TCP 引入了多种机制来控制双向数据发送速率：流量控制、拥塞控制、拥塞避免。

    4.1 流量控制（Flow Control）

    流量控制防止发送方以接收方无法处理的速率淹没接收方，接收方可能正忙、负载过重，或只愿分配固定大小的缓冲区。

    实现机制： TCP 连接的每一方都会通告自己的接收窗口（rwnd），表示当前可用缓冲区大小。

    • 连接建立时，双方使用系统默认值初始化 rwnd
    • 若某一方跟不上数据速率，它会通告更小的窗口；若窗口降至 0，则视为信号，通知发送方暂停发送
    • 每个 ACK 包都携带最新的 rwnd 值，双方据此动态调整数据流速

    窗口扩展（Window Scaling，RFC 1323）

    原始 TCP 规范为接收窗口分配了 16 位，上限为 65,535 字节（约 64 KB），在高带宽时延积网络中往往不够用。

    RFC 1323 引入了 TCP 窗口扩展选项，允许将最大接收窗口提升至 1 GB。该选项在三次握手期间协商，携带一个值表示未来 ACK 中 16 位窗口字段的左移位数。

    目前所有主流平台默认启用窗口扩展，但部分中间节点（路由器、防火墙）可能重写甚至剥离该选项。

    4.2 慢启动（Slow-Start）

    流量控制解决了发送方淹没接收方的问题，但没有解决淹没底层网络的问题。连接建立之初，双方都不知道可用带宽，需要一种机制来估算并适应不断变化的网络状况。

    1988 年，Van Jacobson 和 Michael J. Karels 提出了四个算法：慢启动、拥塞避免、快速重传、快速恢复，这些算法很快成为 TCP 规范的强制组成部分，被普遍认为正是这些更新防止了 80 至 90 年代互联网的崩溃。

    慢启动的工作原理

    TCP 为每个连接维护一个拥塞窗口（cwnd） 变量（私有，不通告给对方），规定了在收到 ACK 之前最多可以在途发送的数据量。

    实际在途数据上限 = min(rwnd, cwnd)

    慢启动的初始 cwnd 值演化历程：

    • 最初：1 个网络段
    • RFC 2581（1999 年）：4 个段
    • RFC 6928（2013 年）：10 个段（当前标准）

    算法逻辑：

    • 每收到一个 ACK，cwnd 增加一个段（即每个 RTT 内窗口翻倍）
    • 这也是为何该阶段被称为”指数增长“阶段
    • 直到超出接收方流控窗口、系统配置的拥塞阈值（ssthresh），或发生丢包

    对于短促的 HTTP 请求，数据传输往往在达到最大窗口之前就结束了，因此大量 Web 应用的性能实际上受限于服务端和客户端之间的 RTT，而非带宽。

    4.3 慢启动重启（Slow-Start Restart，SSR）

    TCP 还实现了慢启动重启机制：连接空闲一段时间后，拥塞窗口会被重置为保守的默认值。原理很简单，网络状况可能在连接空闲期间发生变化。

    然而，SSR 会对存在周期性突发传输的长连接性能产生显著影响。建议在服务器上禁用 SSR 以提升长连接 HTTP 性能。

    4.4 拥塞避免（Congestion Avoidance）

    TCP 将丢包视为网络拥塞的反馈信号，不是”是否会发生”，而是”何时发生”的问题。

    慢启动以保守窗口初始化连接，每个 RTT 翻倍，直到超出 rwnd、ssthresh，或发生丢包。此时拥塞避免算法接管：

    • 原始 AIMD 算法（Multiplicative Decrease and Additive Increase）： 发生丢包时将 cwnd 减半，然后每个 RTT 线性增加固定量
    • TCP 吞吐量曲线呈现锯齿状，正是拥塞控制和拥塞避免算法调整 cwnd 的结果

    目前主流 TCP 拥塞控制实现包括：TCP Tahoe、TCP Reno、TCP Vegas、TCP New Reno、TCP BIC、TCP CUBIC（Linux 默认）、Compound TCP（Windows 默认）等。

    4.5 比例速率降低（Proportional Rate Reduction，PRR）

    原始 AIMD 算法在丢包后过于保守。PRR（RFC 6937） 是 Google 开发的新算法，目标是在丢包后更快恢复：

    • 相比 AIMD，有丢包的连接平均延迟降低 3%–10%
    • 已成为 Linux 3.2+ 内核的默认拥塞避免算法

    5. 带宽时延积（BDP）

    TCP 内置的拥塞控制机制带来了另一个重要性能含义：最优的发送/接收窗口大小必须根据 RTT 和目标数据速率动态调整。

    核心概念

    带宽时延积（Bandwidth-Delay Product，BDP）： 数据链路容量与端到端延迟的乘积，即任意时刻网络中未被确认的最大在途数据量。

    若发送方或接收方频繁等待 ACK，数据流中就会出现间隙，限制连接的最大吞吐量。解决方法是将窗口大小调整到恰好足够大，使任意一方都能持续发送数据，直到更早数据包的 ACK 返回为止。

    常见瓶颈原因

    • 接收方饱和，通告的 rwnd 较小
    • 网络状况差、丢包率高，导致 cwnd 被频繁重置
    • 中间节点的流量整形（traffic shaping）限制了吞吐量

    注意： BDP 限制同样存在于本地局域网（LAN）中。要在 1ms RTT 的环境下实现 1 Gbit/s，同样需要至少 122 KB 的拥塞窗口。

    6. 队头阻塞（Head-of-Line Blocking）

    TCP 为所有报文分配唯一的序列号，并要求按序交付。一旦某个包在传输途中丢失，后续所有包都必须在接收方的 TCP 缓冲区中等待，直到丢失的包被重传并到达。

    应用层对 TCP 的重传和缓冲队列一无所知，只是在读取 socket 数据时感受到延迟，这就是 TCP 队头阻塞（HOL Blocking）。

    影响

    • HOL 阻塞使应用层无需处理报文重排和重组，简化了代码
    • 但代价是引入了不可预测的延迟抖动（jitter），对实时应用影响较大
    • 部分应用不需要可靠传输或有序交付（如音频、视频、游戏状态更新），TCP 无法满足这类需求

    丢包有时是合理的

    对于 TCP 来说，丢包是必要的，它是反馈机制，帮助双方调整发送速率以避免网络过载。

    对于能容忍丢包的应用（音频、视频、实时游戏），更适合使用 UDP 作为传输层（如 WebRTC 即以 UDP 为基础传输协议）：

    • 音频丢一个包，编解码器可以插入短暂停顿继续处理后续包，用户可能察觉不到
    • 游戏状态更新如果已有 T 时刻的包，T-1 时刻的包就可以丢弃，不必等待

    7. TCP 性能优化实践

    7.1 服务器配置调优

    在调整各种缓冲区和超时参数之前，最好的起点是将主机升级到最新系统版本，这是TCP 最佳实践和底层算法持续演进，大多数改进仅在最新内核中可用。

    关键配置项：

    优化项 说明
    增大初始拥塞窗口 设置为 RFC 6928 的 10 段（Linux kernel 2.6.39+ 默认，建议升级至 3.2+）
    禁用空闲后慢启动重启 改善周期性突发传输的长连接性能
    启用窗口扩展（RFC 1323） 提升高延迟连接的吞吐量上限
    启用 TCP Fast Open 在特定场景下允许在 SYN 包中携带应用数据（需客户端和服务端同时支持）

    7.2 应用层行为调优

    即便 TCP 本身已调优至最佳状态，应用对连接的使用方式仍会产生更大影响：

    • 减少传输数据量： 没有任何优化比”不发送”更快；消除不必要的资源传输
    • 缩短物理距离： 比特的传播速度无法加快，但可以通过 CDN、地理分布的服务器让数据更靠近用户
    • 复用 TCP 连接： 避免每次请求都重新经历三次握手和慢启动阶段

    7.3 性能优化清单

    • 将服务器内核升级到最新版本
    • 确认初始拥塞窗口（cwnd）设置为 10 段
    • 确认窗口扩展（Window Scaling）已启用
    • 禁用空闲后慢启动重启（Slow-Start after Idle）
    • 评估是否可以启用 TCP Fast Open
    • 消除冗余数据传输
    • 对传输数据进行压缩
    • 将服务器部署在靠近用户的位置以减少 RTT
    • 尽可能复用 TCP 连接
    • 参考 TCP Tuning for HTTP 文档中的建议

    总结

    TCP 是一个自适应协议，设计上对所有网络对等方公平，并尽可能高效利用底层网络。理解其核心机制是构建高性能 Web 应用的必要前提。

    以下四条核心原理贯穿始终，无论 TCP 算法如何演化，其性能影响始终不变：

    1. TCP 三次握手引入一个完整 RTT 的延迟
    2. 慢启动应用于每一个新连接
    3. TCP 流量控制和拥塞控制调节所有连接的吞吐量
    4. TCP 吞吐量由当前拥塞窗口大小决定

    在现代高速网络中，TCP 连接的数据传输速率往往受限于收发双方之间的 RTT。带宽在持续增长，而延迟受光速限制已接近物理上限。大多数情况下，延迟而非带宽才是 TCP 的瓶颈。
