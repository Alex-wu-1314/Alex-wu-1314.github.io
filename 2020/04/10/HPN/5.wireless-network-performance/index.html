    无线网络性能深度解析

    Alex Wu
    2020-04-10
    高性能网络

    目录

    1. 无线网络概论
    2. WiFi 深度解析
    3. 移动网络（蜂窝网络）

    一、无线网络概论

    1.1 无处不在的连接

    过去十年中最具变革性的技术趋势之一，是无处不在的网络连接的普及。无论是收发邮件、语音通话、网页浏览，还是其他各种应用场景，我们现在已经期望随时随地都能访问在线服务，无论是在路上、排队时、办公室里、地铁上还是飞机上。

    无线网络处于这一趋势的核心。广义上，无线网络泛指任何不依赖电缆连接的网络，正是这一特性赋予了用户所期待的便利性和移动性。

    当前已被广泛使用的无线技术超过十种：WiFi、Bluetooth、ZigBee、NFC、WiMAX、LTE、HSPA、EV-DO 以及各种 3G 标准、卫星服务等等。尽管技术多样，但大多数无线技术都遵循共同的原理，面临相同的权衡，受到相似的性能约束。

    1.2 无线网络的分类

    根据地理覆盖范围，无线网络可以分为以下几类：

    类型 覆盖范围 典型应用 标准
    个人区域网络（PAN） 人体可触及范围 外设电缆替代 Bluetooth、ZigBee、NFC
    局域网（LAN） 建筑物或园区内 有线网络的无线延伸 IEEE 802.11（WiFi）
    城域网（MAN） 城市范围 无线互联网络 IEEE 802.15（WiMAX）
    广域网（WAN） 全球范围 无线网络接入 蜂窝网络（UMTS、LTE 等）

    这种分类并非绝对。例如，蓝牙最初定位于 PAN 应用，但随着版本迭代，现已具备与 802.11（WiFi）的无缝互操作能力。不同类型的网络面临不同的约束：有的设备有持续供电，有的必须极力优化电池寿命；有的需要 Gbit/s 级数据速率，有的（如 NFC）只传输几十或几百字节。

    1.3 无线网络的性能基础

    香农信道容量公式

    所有通信方式都有一个最大信道容量，由香农（Claude E. Shannon）提出的数学模型精确描述：

    $$C = BW \times \log_2(1 + S/N)$$

    其中：

    • C：信道容量，单位 bps（比特/秒）
    • BW：可用带宽，单位 Hz（赫兹）
    • S/N：信号功率与噪声功率之比（SNR）

    无论技术名称、缩写或规范版本号如何，决定无线网络性能的两个根本约束始终是可用带宽和信噪比（SNR）。

    带宽（Bandwidth）

    与有线世界不同，无线通信天然使用共享媒介——无线电波（即电磁辐射）。发送方和接收方必须事先约定通信所用的频率范围，以实现设备间的互操作。

    频率范围的分配由各国政府管理。在美国，由联邦通信委员会（FCC）负责。由于各国政府法规不同，某些无线技术可能在某地区有效，在另一地区则无效。

    频率分配的历史：

    • 1912年：美国《无线电法》颁布，要求无线电频谱的许可使用（部分起因于泰坦尼克号沉船事故调查）
    • 1934年：《通信法》创建 FCC，FCC 自此负责美国境内的频谱分配
    • 1947年：国际电信会议设立”工业、科学和医疗”（ISM）无线电频段 2.4–2.5 GHz（100 MHz）和 5.725–5.875 GHz（150 MHz）被预留并对所有人开放（无需授权频谱）
    • 2008年：FCC 的 700 MHz 频段拍卖，698–806 MHz 频段以 195.92 亿美元成交

    频率范围的大小直接影响性能：根据香农公式，可用频率范围翻倍，数据速率随之翻倍，这正是 802.11n 将带宽从 20 MHz 提升到 40 MHz 以提升性能的原理。

    不同频率特性对比：

    • 低频信号：传播距离远，覆盖面积大（宏蜂窝），但天线尺寸要求更大，竞争接入的客户端更多
    • 高频信号：数据传输量更大，但传播距离短，形成小覆盖区（微蜂窝），需要更多基础设施

    信号功率（Signal Power）与 SNR

    信噪比（SNR）衡量所需信号与背景噪声和干扰的强度对比。背景噪声越大，信号就必须越强才能传递信息。

    所有无线通信都在共享媒介上进行，其他设备可能产生不必要的干扰。例如，工作在 2.5 GHz 的微波炉可能与 WiFi 频段重叠；邻居的 WiFi 接入点，甚至同一 WiFi 网络中同事的笔记本电脑，都会对你的传输造成干扰。

    为实现在存在干扰情况下的目标数据速率，可以：

    • 增加发射功率以提高信号强度
    • 缩短发射机与接收机之间的距离
    • 或两者兼而有之

    两个重要效应：

    • 近远问题（Near-Far Problem）：强信号可能遮蔽弱信号，使接收机无法检测到来自较远处的弱信号
    • 小区呼吸（Cell Breathing）：覆盖区域的大小随着累积噪声和干扰水平的变化而扩大或缩小

    调制（Modulation）

    数字信号（0 和 1）需要转换为模拟信号（无线电波），这个过程称为调制。不同的”调制字母表”可以以不同的效率编码数字信号：

    • 符号速率（baud）× 每个符号比特数 = 信道比特率
    • 高阶调制字母表需要以降低抗噪声和干扰能力为代价，没有免费的午餐

    1.4 真实世界中的无线性能

    所有无线技术都宣称峰值/最大数据速率（例如，802.11g 可达 54 Mbit/s，802.11n 最高可达 600 Mbit/s，部分运营商宣传 LTE 速率超过 100 Mbit/s）。但这些数据的前提是理想条件：最大可用带宽、频谱独占、零背景噪声、最高吞吐量调制字母表，以及多路无线流（MIMO）并行传输。

    影响无线网络性能的因素：

    • 接收方与发送方之间的距离
    • 当前位置的背景噪声量
    • 同一网络内用户造成的干扰（蜂窝内干扰）
    • 邻近网络中用户造成的干扰（蜂窝间干扰）
    • 接收方和发送方的可用发射功率
    • 处理能力与所选调制方案

    延迟性能：无线网络的延迟性能与所用的具体技术直接相关，需要结合具体标准分析。

    二、WiFi 深度解析

    2.1 WiFi 简介

    WiFi 工作在未授权的 ISM 频谱中，任何人都可以轻松部署，所需硬件简单且廉价，已成为部署最广泛的无线标准之一。”WiFi” 是 WiFi 联盟的商标，技术上任何产品都需经 WiFi 联盟认证才能使用该名称，但实际中它泛指所有基于 IEEE 802.11 标准的产品。

    发展历史：

    • 1997年：第一个 802.11 协议，是以太网标准（IEEE 802.3）向无线通信领域的适配版本
    • 1999年：802.11b 标准推出，WiFi 设备市场开始腾飞
    • 今天：几乎所有新出的台式机、笔记本、平板、智能手机都支持 WiFi

    2.2 从以太网到无线局域网

    802.11 无线标准主要被设计为以太网（802.3）标准的适配和扩展，因此 802.11 系列通常被称为无线局域网（WLAN）。

    以太网的历史可追溯到 1971 年夏威夷大学开发的 ALOHAnet（首个公开演示的无线网络），Ethernet 正是受此启发而来，某种程度上，WiFi 是回归原点。

    核心调度机制：以太网和 WiFi 都将共享媒介视为”随机访问信道”，没有中央进程或调度器控制谁在何时发送数据，每个设备自主决定，所有设备协同保证共享信道性能。

    CSMA/CD vs CSMA/CA：

    • 以太网（CSMA/CD）：发送前先侦听，检测到冲突后双方立即停止并随机退避（指数退避）
    • WiFi（CSMA/CA）：由于无线硬件限制，发送时无法检测冲突，因此采用冲突避免策略：仅在信道空闲时发送完整帧，并等待接收方的显式确认（ACK）后再发送下一帧

    负载与性能：概率访问模型在轻载网络中运行良好，理论上要获得良好信道利用率（最小化冲突），信道负载需保持在 10% 以下。负载增加后，冲突数量迅速上升，导致整个网络性能不稳定，这正是在大型公共场所（如大型会议现场）使用 WiFi 时常常遇到性能问题的原因。

    2.3 WiFi 标准演进

    802.11 协议 发布时间 频率 (GHz) 带宽 (MHz) 每流数据速率 (Mbit/s) 最大 MIMO 流数
    b 1999年9月 2.4 20 1, 2, 5.5, 11 1
    g 2003年6月 2.4 20 6, 9, 12, 18, 24, 36, 48, 54 1
    n 2009年10月 2.4 20 7.2–72.2 4
    n 2009年10月 5 40 15–150 4
    ac ~2014年 5 20/40/80/160 最高 866.7 8

    性能提升路径：”n” 和 “ac” 标准将带宽从 20 MHz 增加到 40 MHz，使用更高阶调制，并通过多天线（MIMO）并行传输多路数据流，理想条件下可实现千兆级吞吐量。

    2.4 WiFi 性能测量与干扰

    信道争用问题：

    • 2.4 GHz 频段仅提供 3 个不重叠的 20 MHz 信道（1、6、11），超过两三个相邻 WiFi 网络时，必然有重叠
    • WiFi 标准没有中央调度器，对吞吐量和延迟均无任何保证
    • 当你的邻居在同一 WiFi 信道上开始流媒体 HD 视频时，你的带宽可能直接减半

    5 GHz 频段的优势：5 GHz 频段（802.11n 和 802.11ac）提供更宽的频率范围，且在大多数环境中干扰仍然较少。双频路由器可兼顾旧设备（2.4 GHz）的兼容性和新设备（5 GHz）的高性能。

    第一跳延迟测量示例（对无线网关执行 ping）：

    频率 (GHz) 中位数 (ms) 95th 百分位 (ms) 99th 百分位 (ms)
    2.4 6.22 34.87 58.91
    5 0.90 1.58 7.89

    2.4 GHz 频段中十多个重叠网络导致从笔记本到不足 20 英尺（约 6 米）外的路由器，第一跳延迟达 35 ms（95th 百分位）。

    WiFi 性能总结：

    • WiFi 不对其用户提供带宽或延迟保证
    • WiFi 带宽随信噪比变化而波动
    • WiFi 发射功率限制为 200 mW（各地区可能更低）
    • 2.4 GHz 和 5 GHz 频段可用频谱有限
    • WiFi 接入点在信道分配上天然存在重叠
    • 所有 WiFi 接入点和客户端争用同一无线信道

    2.5 WiFi 中的数据包丢失

    WiFi 发送的概率调度可能导致大量冲突。然而，这不一定会直接导致更高的 TCP 丢包率，原因是所有 WiFi 协议的数据链路层和物理层都有自己的重传和纠错机制，会向上层网络栈隐藏这些无线冲突。

    实际上，你更可能看到的是数据包到达时间的高变异性，而非直接的 TCP 丢包。802.11n 引入了”帧聚合”特性，允许同时发送和确认多个 WiFi 帧，提升了效率。

    2.6 针对 WiFi 网络的优化策略

    利用无限流量带宽

    WiFi 网络通常是有线 LAN 的延伸，而有线 LAN 通过 DSL、电缆或光纤连接到广域网，且通常是无流量计费（或拥有更高流量上限）的连接。大文件下载、更新和流媒体场景最好在 WiFi 下进行。许多移动运营商也明确推荐”Wi-Fi 卸载”策略，在后台同步和大数据传输时优先使用 WiFi。

    自适应可变带宽

    由于 WiFi 无带宽保证，可用带宽分配可能以秒为单位剧烈变化。对于 HD 视频流等需要数 Mbit/s 带宽的场景，应采用自适应码率流（Adaptive Bitrate Streaming，ABR）：

    • 将资源以多种码率编码并分段（如 5–10 秒的切片）
    • 客户端或服务端监控每个分段的下载速率，动态调整下一个分段的码率
    • 实践中，Netflix 为每个流编码超过 120 个不同版本以适应各种屏幕尺寸和可用带宽

    不应在视频开始时仅测一次带宽就固定码率，应持续进行带宽测量与调整。

    自适应可变延迟

    WiFi 第一跳延迟无保证。在干扰最小、网络空闲时，无线跳延迟可不足 1 毫秒；但在高密度环境中，中位数延迟可达 1–10 ms，并有长尾延迟（偶发 10–50 ms，最差情况可达数百毫秒）。对于延迟敏感型应用，可考虑使用 WebRTC 的非可靠 UDP 传输以降低协议层引入的延迟。

    三、移动网络（蜂窝网络）

    3.1 移动网络的规模

    截至 2013 年初，全球蜂窝连接数约 64 亿，2012 年智能互联设备出货量约 11 亿台。各代蜂窝网络的性能参数：

    代际 数据速率 延迟
    2G 100–400 Kbit/s 300–1000 ms
    3G 0.5–5 Mbit/s 100–500 ms
    4G 1–50 Mbit/s < 100 ms

    实际使用时，数据吞吐量应参考下界估算，数据包延迟应参考上界估算。

    3.2 移动网络发展史

    2G：首批数字数据服务

    • 1979年：日本发布全球第一个商业 1G 网络（模拟系统，无数据能力）
    • 1991年：芬兰发布基于 GSM（全球移动通信系统）标准的首个 2G 网络，引入数字信令，实现首批电路交换移动数据服务（短信 SMS）和峰值 9.6 Kbit/s 的数据速率
    • 1990年代中期：GPRS（通用分组无线服务）引入 GSM，使无线互联网接入成为现实，可达 172 Kbit/s，典型往返延迟高达数百毫秒（常被称为 2.5G）
    • 2003年：美国发布首批 EDGE（GSM 演进增强数据速率）网络（2.75G），峰值数据速率提升至 384 Kbit/s

    3GPP 与 3GPP2 标准组织

    移动数据服务的消费需求快速增长后，无线电网络互操作性成为核心议题：

    • GSM（3GPP路线）：欧洲电信标准研究所（ETSI）于 1990 年代初开发，迅速被全球采用，估计覆盖 80%–85% 的市场份额
    • IS-95/cdmaOne（3GPP2路线）：Qualcomm 开发，主要在北美市场，约占 10%–15% 份额

    1998 年，GSM 和 IS-95 标准组织认识到全球标准演进的必要性，成立了两个全球伙伴项目：

    • 3GPP：负责开发 UMTS（GSM 的 3G 升级），后负责 LTE 标准开发
    • 3GPP2：负责基于 CDMA2000 技术的 3G 规范（IS-95 的后继技术）

    两套标准体系不可互操作，为 IS-95 无线网络设计的设备无法在 GSM 网络上运行，反之亦然，这一限制让许多国际旅行者深有体会。

    3.3 4G 与 IMT-Advanced

    IMT-Advanced 4G 要求

    4G 是 ITU 于 2008 年制定并发布的一套要求（IMT-Advanced），满足这些要求的任何技术都可以被称为 4G。核心要求包括：

    • 基于 IP 分组交换网络
    • 与以前的无线标准（3G、2G）互操作
    • 移动客户端 100 Mbit/s 数据速率，静止时 Gbit/s 以上
    • 控制面延迟 < 100 ms，用户面延迟 < 10 ms
    • 网络资源在用户间动态分配和共享
    • 可变带宽分配（5 至 20 MHz）

    LTE vs LTE-Advanced：严格来说，LTE（Rel. 8）只是 3.9G 过渡标准，LTE-Advanced（Rel. 10）才是真正满足 IMT-Advanced 的 4G 标准。然而，由于营销因素，LTE 和大多数 HSPA+ 网络尽管并不完全满足技术上的 4G 要求，却被运营商冠以”4G”之名，”4G” 标签在今天的使用上实际上是模糊的。

    LTE 核心特性

    • 全 IP 核心网
    • 简化网络架构以降低成本
    • 低延迟（用户面 < 10 ms，控制面 < 100 ms）
    • 新无线接口和调制，支持高吞吐量（100 Mbps）
    • 支持更大带宽分配和载波聚合
    • MIMO 作为所有设备的强制要求

    重要约束：由于无线接口和核心网实现差异，LTE 网络不是现有 3G 基础设施的简单升级，必须在独立的频谱上并行部署。每个支持 LTE 的设备还需要为 3G/2G 提供单独的无线接口，一部手机可能需要 3–6 个无线模块！

    HSPA+ 是 4G 全球化的实际领导者

    HSPA+ 首次引入于 3GPP Rel. 7（2007年）。尽管公众关注点迅速转向 LTE，但 HSPA+ 的开发并未停止，且持续协同演进。选择 HSPA+ 而非立即跳跃到 LTE 的核心原因是成本：

    • 全球 3GPP 3G 技术占据已建立无线市场的大多数份额，意味着运营商有巨大的既有基础设施投资
    • 迁移到 LTE 需要新的无线网络，意味着大量资本支出
    • HSPA+ 提供了一条更加资本高效的路径：运营商可以对现有网络进行增量升级，获得相当的性能

    行业预测显示，HSPA+ 将是本十年大多数 4G 升级的来源，LTE 的大规模部署则是更长远的目标（新标准从首次规范到主流部署大约需要十年）。

    3.6 无线资源控制器（RRC）

    RRC（Radio Resource Controller）是 3G 和 4G 网络特有的、WiFi 所没有的功能。RRC 负责调解设备与无线基站之间的所有连接管理，对延迟、吞吐量和设备电池寿命有直接影响。

    RRC 的职责：调度谁在何时通话、分配带宽、控制信号功率、管理每个设备的功耗状态等。想通过无线信道发送数据？必须先请求 RRC 分配无线资源。

    RRC 的位置：

    • 2G/3G 网络：RRC 位于核心运营商网络中
    • 4G LTE 网络：RRC 逻辑已移至服务无线基站（eNodeB），以提升性能并减少协调延迟

    3G/4G 与 WiFi 功耗对比

    • WiFi：每个设备自行设置发射功率，通常在 30–200 mW 范围
    • 3G/4G：发射功率由网络管理，空闲状态低至 15 mW，高功率传输状态可达 1,000–3,500 mW

    结论：传输大量数据时，WiFi 通常比移动网络更节能（信号强时）；设备大部分时间空闲时，3G/4G 无线电更高效。

    LTE RRC 状态机

    LTE 设备的无线状态由当前为其服务的无线基站控制。3GPP 标准定义了一个明确的状态机：

    状态转换 LTE LTE-Advanced
    空闲→连接 < 100 ms < 50 ms
    DRX→连接 < 50 ms < 10 ms
    用户面单向延迟 < 5 ms < 5 ms

    关键机制：由于 IP 流量是突发性的，TCP 连接是长期的，UDP 没有”传输结束”指示，RRC 使用定时器来触发状态转换，空闲一定时间后降低功耗状态。

    HSPA/UMTS RRC 状态机

    UMTS（HSPA、HSPA+）的 RRC 状态机更复杂，延迟也高得多：

    • 空闲（Idle）：低功耗状态，仅侦听控制流量
    • Cell DCH：高功耗状态，分配上下行网络资源（类似 LTE 连续接收）
    • Cell FACH：中间功耗状态（约 DCH 的一半功耗），通过共享低速信道传输少量数据（通常 < 20 Kbps），设计用于非交互式流量，如后台应用的定期轮询

    关键差异：UMTS 从空闲到 DCH 的转换可能需要长达 2 秒并需要数十条控制消息；FACH 到 DCH 也可能需要 1.5 秒。这也是 LTE 性能更好的原因之一，简化了 RRC 状态转换架构。

    EV-DO（CDMA）RRC 状态机

    EV-DO 是最简单的 RRC 状态机：设备要么处于高功耗连接状态（分配网络资源），要么处于空闲状态。所有网络传输都需要转换到连接状态，延迟与 HSPA 网络相似（数百到数千毫秒）。

    3.7 周期性传输的低效性

    超时驱动的无线状态转换有一个重要后果：间歇性网络访问是移动网络上的反模式。

    能量尾巴（Energy Tail）：每次无线传输（无论数据量多小）都会触发切换到高功耗状态，传输完成后无线电会在高功耗状态持续等待，直到惰性计时器超时，数据量的大小不影响计时器。

    极端案例（AT&T 实验室研究）：

    • 分析应用：Pandora
    • 流媒体音乐时，应用会一次性下载整个音乐文件（正确行为）
    • 但每 60 秒会发送间歇性的受众测量 ping
    • 结果：分析 beacon **占总传输字节的 0.2%**，却占 **总功耗的 46%**！
    • 解决方案：将分析数据合并到更少的请求中，或在无线电已激活时发送

    核心规律：任何一次无线传输，无论多小，都会强制转换到高功耗状态，然后无线电一直保持在这个状态，直到所有定时器触发。

    3.8 端到端运营商架构

    无线接入网络（RAN）

    RAN 是每个运营商网络的第一个主要逻辑组件，负责调解对无线信道的访问并传输数据包。在 LTE 中，每个无线基站（eNodeB）承载 RRC。

    切换管理：当用户有更强的邻近小区信号或当前小区过载时，会被切换到邻近基站。LTE 中基站间切换可在数百毫秒内完成（对用户完全透明），而在早期网络中，这一过程可能需要数秒。

    跟踪区域（Tracking Area）：一个或多个无线基站组成一个”跟踪区域”，核心网知道用户所在的跟踪区域，但可能不知道具体服务的基站，设备在同一跟踪区域内切换不会触发通知，节省设备电量。

    核心网络（EPC）

    LTE 核心网络（演进分组核心，EPC）负责数据路由、计费和策略管理：

    • PGW（分组网关）：连接移动运营商和公共互联网的外部网关，终止所有外部连接，分配和维护设备 IP 地址。重要：PGW 常常同时为数十甚至数百个设备共享同一公共 IP 地址（即 NAT）
    • PCRF（策略和计费规则功能）：维护和评估数据包过滤、检查、QoS 分配、DoS 防护等策略规则
    • SGW（服务网关）：作为无线网络内设备的移动锚点
    • MME（移动管理实体）：有效的用户数据库，管理每个用户在网络上的状态：位置、账户类型、计费状态、已启用服务等所有用户元数据

    关键特性——物理层与应用层连接性分离：由于 PGW 终止所有连接，设备无线状态与应用层连接性是解耦的：

    拆除无线上下文会终止设备与无线基站之间的物理无线链路，但不影响任何 TCP 或 UDP 会话的状态。设备无线电可以处于空闲状态（无与本地无线基站的链路），而之前建立的连接在 PGW 处仍保持活跃。

    这意味着应用层 keepalive 并非保持无线电激活所必需。

    回程（Backhaul）容量与延迟

    LTE 无线接口可达设备与基站之间 100 Mbps，但一旦信号被接收，就必须有足够的容量通过运营商网络传输数据。在实践中，单个无线基站可能服务多达三个附近的无线小区，合计数百个活跃用户，每个基站都需要专用光纤连接！

    这些性能瓶颈无法由应用开发者控制，但它们说明了一个重要事实：4G 接口并不保证最大吞吐量。我们的应用必须适应无线信道、运营商网络以及公共互联网上不断变化的网络状况。

    3.9 移动网络中的数据包流向

    发起请求的流程（LTE 为例）

    假设用户已通过 4G 网络认证，设备处于空闲状态：

    1. RRC 协商（< 100 ms）：手机与附近基站同步，请求建立新的无线上下文（需要多次往返）
    2. 数据传输（用户面单向延迟 < 5 ms）：建立无线上下文后，数据包从用户无线电传输到基站
    3. 核心网络路由（30–100 ms）：数据包经过 SGW 到 PGW
    4. 互联网路由（可变）：数据包从 PGW 出发到达公共互联网目标地址

    总结延迟组成：

    延迟类型 说明 量级
    控制面延迟 RRC 协商（一次性成本） 空闲→激活 < 100 ms；休眠→激活 < 50 ms
    用户面延迟 每个应用数据包的固定成本 < 5 ms
    核心网络延迟 运营商相关，基站→PGW 实践中 30–100 ms
    互联网路由延迟 PGW 到目标地址的可变延迟 可变

    AT&T 各代网络延迟参考：

    网络代际 延迟
    LTE 40–50 ms
    HSPA+ 100–200 ms
    HSPA 150–400 ms
    EDGE 600–750 ms
    GPRS 600–750 ms

    有趣的比较：光在赤道绕地球一圈需要 133.7 ms。许多移动请求的延迟相当于至少绕地球一圈的光速行程。

    入站数据流（惰性设备接收数据）

    设备处于空闲状态但需要从 PGW 路由数据时：

    1. PGW 将入站数据包路由到 SGW
    2. SGW 查询 MME，但 MME 可能只知道跟踪区域而非精确基站
    3. MME 向跟踪区域内所有基站发送寻呼消息，所有基站在共享无线信道上广播通知
    4. 设备定期醒来侦听寻呼消息，发现自己在寻呼列表中后与基站协商建立无线上下文
    5. 完成协商的基站向 MME 发送消息报告用户位置
    6. MME 将答案返回 SGW，SGW 将消息路由到基站
    7. 基站将数据投递到设备

    一旦设备进入连接状态，基站与 SGW 之间建立直接隧道，后续入站数据包直接路由到基站，跳过步骤 2–5，无寻呼开销。第一个数据包始终是延迟最高的！

    3.10 异构网络（HetNets）

    现有的 4G 无线调制技术已接近无线信道理论极限，下一量级的无线性能将不来自无线接口的改进，而来自更智能的网络拓扑，即通过多层异构网络（HetNets）的广泛部署。

    核心思想：在宏蜂窝大范围覆盖的基础上，叠加部署许多小小区，以减少路径损耗、降低发射功率需求，为所有用户提供更好的性能。

    小区类型 覆盖范围
    宏蜂窝（Macrocell） 低密度环境数十平方英里，高密度城市仅 50–300 米
    微蜂窝（Microcell） 特定建筑物
    微微蜂窝（Picocell） 一个或多个楼层
    毫微微蜂窝（Femtocell） 小公寓，利用现有宽带服务作为无线回程

    HetNets 不是简单地用小小区替代宏蜂窝，而是在彼此之上分层叠加多个小区。主要挑战：最小化干扰、提供足够的上行链路容量、实现各层网络之间的无缝切换。

    对开发者的影响：预期不同小区之间的切换次数大幅增加，应用程序的延迟和吞吐量性能可能显著波动。

    3.11 真实世界 3G/4G/WiFi 性能对比

    密歇根大学与 AT&T 实验室开展的全美范围联合研究（2011 年末，3,300 名用户，两个月时间，对比 46 个测量节点）结果显示：

    • LTE 展现出出色的网络吞吐量性能
    • 更令人印象深刻的是，LTE 在 RTT 和数据包抖动延迟方面比其他无线标准更加稳定
    • 至少在这项测试中，LTE 提供了与 WiFi 相当乃至更好的性能

    移动网络不必然缓慢。我们有充分理由相信，可以并且将会使其更快。

    四、移动网络性能优化

    4.1 优化的三个约束

    移动应用设计需要仔细权衡三个不可分割的约束：

    1. 内容呈现（响应式设计等）
    2. 网络性能（协议、连接管理等）
    3. 电池寿命（无线电使用效率）

    这三者相互关联，糟糕的网络决策会对电池寿命产生超比例的负面影响。没有通用解决方案，但任何一个都不能被忽视。

    4.2 节省电池电量

    节省电量是移动端所有相关方（设备制造商、运营商、应用开发者、用户）的核心关切。

    移动无线电的功耗特点：

    • 全功率运行可在数小时内耗尽整块电池
    • 无线电功耗要求随每代无线技术的升级而增加
    • 无线电通常是仅次于屏幕的第二耗电组件
    • 无线电使用相对于传输数据量具有非线性能耗曲线

    核心策略：最大化无线电激活时传输的数据量，同时最小化额外的数据传输次数——不是不使用无线电，而是在用时尽量多传，不用时让它休眠。

    AT&T 应用资源优化工具（ARO）

    ARO 包含两个组件：

    • 采集器：Android 应用（可在真机或模拟器上运行），采集传输的数据包、无线活动及其他交互
    • 分析仪：提供无线状态、能耗和流量模式的洞察，并给出常见性能问题的优化建议（缺少压缩、冗余数据传输等）

    ARO 分析器也接受 tcpdump 或兼容工具产生的 pcap 文件（适用于 iOS 用户）。

    4.3 消除周期性和低效的数据传输

    移动无线电切换到全功率状态的固定成本，与传输数据量无关——**移动网络中不存在”小请求”**。

    核心规则：

    • 轮询（Polling）在移动网络上代价极高，应尽量减少
    • 尽可能使用推送（Push）交付和通知
    • 出站和入站请求应被合并和聚合
    • 非关键请求应推迟到无线电已激活时发送

    推送 vs 轮询：推送通常比轮询高效，但高频率推送流可能与轮询一样甚至更耗电。设计实时更新时应考虑：

    • 最佳更新间隔是什么？是否符合用户预期？
    • 能否使用自适应策略（基于电池电量或用户偏好）？
    • 入站/出站请求能否聚合为更少的网络调用？

    Nagle 算法的应用层类比：TCP 的 Nagle 算法尝试将多个小 TCP 消息合并为一个数据包，移动应用可以在应用层复用这一思想——在服务端按时间、数量或大小聚合消息，而不是为每条消息触发独立推送。Google Cloud Messaging (GCM) 等服务提供了消息聚合投递的能力。

    周期性轮询的电量消耗计算示例：

    • 电池容量：5 瓦时 = 18,000 焦耳
    • 无线电从空闲到连接再回到空闲消耗：约 10 焦耳
    • 1 分钟轮询间隔：600 焦耳/小时
    • 占总电池容量约 3%/小时！ 几个使用不同轮询间隔的应用就足以在中午前耗尽电池

    消除不必要的应用层 Keepalive

    连接状态与无线状态是独立的：无线电可以处于低功耗状态，而 PGW 处维护的连接依然有效。**应用无需保持无线电”激活”**来防止连接被断开——这是常见的误解，且会对电池寿命产生极大的负面影响。

    大多数移动运营商设置 5–30 分钟的 NAT 连接超时。因此，周期性 keepalive（每 5 分钟）可能是必要的，但频率不应超过这个。如果你发现需要更频繁的 keepalive，先检查自己的服务器、代理和负载均衡器配置。

    4.4 预估网络延迟开销

    在移动网络上，单个 HTTP 请求可能产生数百到数千毫秒的延迟开销。

    一次”简单” HTTP 请求的延迟分解（假设 4G 往返时延 100 ms，3.5G+ 往返时延 200 ms）：

    延迟组件 3G 4G
    控制面（RRC 协商） 200–2,500 ms 50–100 ms
    DNS 查询 200 ms 100 ms
    TCP 握手 200 ms 100 ms
    TLS 握手 200–400 ms 100–200 ms
    HTTP 请求 200 ms 100 ms
    总延迟开销 200–3,500 ms 100–600 ms

    这还没有计算服务器响应时间或响应大小（可能需要多次往返），我们已经积累了多达 6 次往返，翻译成时间就是 3G 网络数秒、4G 网络约半秒。

    应对 RRC 状态转换

    如果移动设备已空闲超过几秒钟，应假定第一个数据包将产生数百甚至数千毫秒的额外 RRC 延迟。经验法则：

    • 4G ：额外加 100 ms 一次性控制面延迟成本
    • 3.5G+ ：额外加 150–500 ms
    • 3G ：额外加 500–2,500 ms

    RRC 是移动网络的现实——无法绕过。

    RRC 知识要点：

    • 每个无线标准的 RRC 状态机不同
    • RRC 状态机由每个设备的无线网络管理
    • 需要传输数据时，RRC 提升到高功耗状态
    • 网络配置的超时触发 RRC 降低到低功耗状态
    • 4G LTE 状态转换需要 10–100 毫秒
    • HSPA+ 状态转换与 LTE 相当
    • HSPA 和 CDMA 状态转换可能需要数秒
    • 每次网络传输，无论大小，都会产生能量尾巴

    将用户交互与网络通信解耦

    精心设计的应用可以通过即时反馈让用户感觉很快，即使底层连接很慢：

    • 不要将用户交互、用户反馈和网络通信耦合在一起
    • 应在数百毫秒内确认用户输入
    • 如果需要网络请求，在后台发起，并提供即时的 UI 反馈
    • 控制面延迟单独就可能超过提供即时用户反馈的时间预算

    4.5 针对可变网络接口可用性的设计

    移动网络不同于有线世界——一旦连接建立，不能假定其会保持连接。用户可能进入干扰严重、活跃用户众多或信号覆盖不佳的区域。

    即使使用最新设备，用户也会根据无线环境的持续变化，在 4G、3G 甚至 2G 网络之间切换。

    网络状态监听：可以通过 navigator.onLine 订阅连接状态通知。

    带宽和延迟估算的注意事项：

    • 移动网络上的带宽和延迟估算在数十到数百毫秒（最多一秒）范围内是稳定的，但不能更长
    • 不要缓存或使用旧的带宽估算来做决策——在 4G 上可能测到几百 Kbit/s，移动几英寸后就能达到 Mbit/s 以上
    • 对于长时间的流媒体（如视频），自适应码率流仍然有用，但估算不应被缓存

    流媒体策略：

    • 如果需要下载大文件且有信心整个文件都会被使用，一次性下载整个文件，然后让无线电尽可能长时间保持空闲（Pandora 音乐下载的正确做法）
    • 如果不能完整下载（如 HD 视频），使用自适应码率流以持续适应网络吞吐量变化

    失败处理策略：

    • 不要缓存或猜测网络状态
    • 发出请求，侦听失败，诊断原因
    • 瞬态错误会发生，制定好重试策略
    • 监听连接状态以预测最佳请求策略
    • 使用退避算法进行请求重试，不要无限循环
    • 如果离线，记录并在可能时稍后发送请求
    • 利用 HTML5 AppCache 和 localStorage 实现离线模式

    4.6 突发数据传输并返回空闲

    移动无线接口针对突发传输进行了优化——LTE 和 HSPA+ 等最新一代网络以 1 毫秒为间隔动态分配资源，并优先处理突发数据流：

    核心策略：将请求分组，尽可能快地下载尽量多的数据，然后让无线电返回空闲状态。

    这一策略将提供最佳网络吞吐量并最大化设备电池寿命。

    注意：移动网络上的渐进式加载资源可能弊大于利——通过小块下载内容，应用暴露于更高的吞吐量和延迟变异性，以及更高的无线电操作能耗。

    预取（Prefetch）最佳实践：

    • 如果需要下载大型音乐或视频文件，考虑提前下载整个文件而不是分块流式传输
    • 预取应用内容，投入指标和统计模型以确定哪些内容适合提前下载
    • 提前预取第三方内容（如广告），并添加应用逻辑在必要时显示和更新其状态
    • 消除不必要的间歇性传输

    预取策略的三变量平衡：

    • 传输字节数（带宽成本）
    • 对电池的影响
    • 网络吞吐量和延迟的变异性

    这三个变量不是独立的，一次传输更大批次的字节可以带来更好的吞吐量。良好的预取策略需要了解应用的主要使用模式，实施策略，收集指标验证，然后迭代。

    4.7 卸载到 WiFi 网络

    当前行业估计显示，全球近 90% 的无线流量预计来自室内，且通常在 WiFi 可达的区域。4G 网络虽然峰值吞吐量和延迟可以与 WiFi 竞争，但通常仍会施加月度数据上限。此外，对于大数据传输，WiFi 连接更节能，且不需要 RRC。

    建议：尽可能使用 WiFi，尤其是在构建数据密集型应用时，考虑在不可用时提示用户开启 WiFi。

    4.8 应用协议和应用层最佳实践

    在移动端，协议层和应用层的所有优化都同样关键——甚至更为重要，因为往返延迟高且带宽始终珍贵：

    • TCP 优化：复用 keepalive 连接，通过地理定位将服务器和数据靠近客户端
    • TLS 优化：TLS 会话复用，减少 TLS 握手往返次数
    • UDP 优化：对于实时应用考虑 WebRTC 的 UDP 传输
    • HTTP 优化：HTTP/1.1 连接复用，HTTP/2 多路复用
    • 应用层最佳实践：所有通用 Web 性能最佳实践在移动端都同等适用

    总结

    无线网络性能受制于几个根本性约束：

    层面 关键约束
    物理层 带宽（频谱分配）、信噪比、调制方案
    无线接入层 RRC 状态机、功耗管理、信道争用
    核心网络层 回程容量、路由延迟、NAT/IP 管理
    应用层 传输模式、预取策略、自适应行为

    关键设计原则速查：

    1. 突发传输：集中批量传输，而非频繁小量传输
    2. 预取优先：预测用户需求，提前下载
    3. 解耦 UI 与网络：即时反馈，异步网络操作
    4. 适配变化：自适应码率流、监听网络状态变化、优雅降级
    5. 避免轮询：优先使用推送，合并通知
    6. 关注 RRC：理解状态转换成本，避免产生不必要的能量尾巴
    7. WiFi 优先：大数据传输时提示用户切换 WiFi
    8. 不要假设网络稳定：设计健壮的错误处理和重试逻辑

    无线网络的核心挑战在于共享媒介的竞争性访问、不断变化的无线环境，以及电池电量与网络性能之间不可避免的权衡。理解这些原理，针对性地设计应用架构，才能真正构建高性能的移动应用。
