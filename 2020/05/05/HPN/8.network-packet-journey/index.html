    当你键入网址后，数据包经历了什么

    Alex Wu
    2020-05-05
    高性能网络

    一次完整的 HTTP 请求背后，隐藏着一段数据包的”心路历程”，从浏览器解析 URL，到服务器返回响应，涉及 DNS、TCP、IP、MAC、网卡、交换机、路由器等一系列协议与设备的协同配合。

    目录

    2. 查询真实地址 —— DNS

    1. 浏览器解析 URL —— HTTP

    URL 的结构

    浏览器做的第一步工作，是对 URL 进行解析，从而生成发送给 Web 服务器的请求信息。

    一条完整的 URL 包含以下几个部分：

    组成部分 说明
    协议（scheme） 如 http://、https://
    主机名（host） 如 www.server.com
    端口号（port） 如 :80、:443，可省略
    路径（path） 如 /dir/file.html
    查询参数（query） 如 ?key=value

    当 URL 中没有路径名时，默认访问根目录下预设的文件，通常是 /index.html 或 /default.html。

    生成 HTTP 请求信息

    对 URL 解析后，浏览器确定了 Web 服务器地址和目标文件名，接下来根据这些信息生成 HTTP 请求报文，报文结构包括：

    • 请求行：方法（GET/POST 等）+ URL + HTTP 版本
    • 请求头部：Host、User-Agent、Accept 等字段
    • 请求体：POST 请求时携带的数据

    此时，HTTP 数据包已准备好，但它还需要很多帮助才能抵达目的地。

    2. 查询真实地址 —— DNS

    为什么需要 DNS？

    浏览器解析出域名后，需要将域名转换为 IP 地址，才能让操作系统发送数据。就像打电话必须知道对方的号码一样，DNS（域名系统） 就是保存域名与 IP 对应关系的”电话本”。

    域名的层级结构

    DNS 中的域名用 . 分隔，越靠右层级越高，形成一棵树状结构：

    • 根 DNS 服务器：最顶层，所有 DNS 服务器都保存根域信息
    • 顶级域 DNS 服务器：如负责 .com、.cn 等
    • 权威 DNS 服务器：负责具体域名的解析，”我的域名我做主”

    DNS 解析完整流程

    整个过程犹如”只指路，不带路”，每一层 DNS 只告诉你下一步去哪问，最终由权威 DNS 给出答案。

    3. 协议栈：网络通信的指南

    协议栈的层次结构

    获取到 IP 地址后，浏览器将 HTTP 消息的传输工作委托给操作系统中的协议栈。协议栈分层如下：

    各层核心协议说明：

    协议 职责
    TCP 可靠传输，处理连接、分割、确认、重传
    UDP 不可靠但高效的传输
    IP 网络包路由与寻址
    ICMP 传递网络错误和控制信息
    ARP 通过 IP 地址查询 MAC 地址

    4. 可靠传输 —— TCP

    TCP 报文头部格式

    TCP 头部包含以下关键字段：

    字段 作用
    源端口 / 目标端口 标识通信的应用进程
    序列号（Seq） 解决数据包乱序问题
    确认号（Ack） 确认对方是否收到，解决丢包问题
    状态位（SYN/ACK/FIN/RST） 控制连接的建立与断开
    窗口大小 流量控制，避免接收方被”撑死”或”饿死”

    此外，TCP 还具备拥塞控制能力，当网络拥塞时，通过控制自身发送速率来应对。

    三次握手建立连接

    在传输 HTTP 数据之前，TCP 需要先建立连接，即著名的三次握手：

    三次握手的目的：确保双方都具备发送和接收的能力，缺一不可。

    可通过 netstat -napt 命令在 Linux 中查看当前 TCP 连接状态。

    TCP 数据分割

    当 HTTP 请求消息较大时，TCP 会将数据拆分成多个MSS（最大报文段长度）大小的块分别发送：

    • MTU：一个网络包的最大长度，以太网通常为 1500 字节
    • MSS：去掉 IP 头 + TCP 头后，单个包能承载的最大数据量

    每个分块都会加上 TCP 头部，交由 IP 模块发送。

    TCP 报文结构（此阶段）

    5. 远程定位 —— IP

    IP 报文头部格式

    IP 层负责为数据包添加网络寻址信息：

    字段 说明
    源地址 IP 客户端的 IP 地址
    目标地址 IP 通过 DNS 获得的服务器 IP
    协议号 TCP 为 0x06，UDP 为 0x11
    TTL 生存时间，防止包在网络中无限循环

    多网卡时如何选择源 IP？

    当主机有多个网卡（多个 IP 地址）时，需要查询路由表来决定使用哪块网卡作为出口：

    路由匹配规则：将目标 IP 与每条路由的子网掩码做 与运算（&），结果与路由条目的目标地址比较，匹配则使用该条目对应的网卡和网关。

    • 若所有条目都不匹配，则使用默认路由（子网掩码为 0.0.0.0），将包发往默认网关（路由器）。

    IP 报文结构（此阶段）

    6. 两点传输 —— MAC

    MAC 头部格式

    MAC 层（以太网层）负责在同一局域网内的两个设备之间传输数据包：

    字段 说明
    发送方 MAC 地址 从网卡 ROM 中读取
    接收方 MAC 地址 通过 ARP 协议查询
    以太网类型 0x0800 = IP 协议，0x0806 = ARP 协议

    ARP 协议：根据 IP 找 MAC

    当不知道目标设备的 MAC 地址时，使用 ARP（地址解析协议） 进行广播查询：

    为避免每次都广播，操作系统会将查询结果缓存在 ARP 缓存中（有效期几分钟）：

    发包时优先查 ARP 缓存，命中则直接使用，未命中才发送 ARP 广播。

    MAC 报文结构（此阶段）

    至此，数据包已组装完毕，准备出发！

    7. 出口 —— 网卡

    IP 层生成的网络包只是内存中的二进制数据，无法直接在网线上传输。网卡（NIC） 的职责就是将数字信号转换为电信号（或光信号、无线信号）。

    网卡驱动程序从 IP 模块获取数据包后，会进行以下处理：

    1. 将包复制到网卡内部的缓存区
    2. 在包头部加上报头和起始帧分界符（标记包的起始位置）
    3. 在包末尾加上 FCS（帧校验序列），用于检测传输错误
    4. 将数字信号转换为电信号，通过网线发送出去

    完整数据包结构

    8. 送别者 —— 交换机

    交换机工作在第二层（MAC 层），负责在局域网内将数据包转发到正确的端口。

    交换机的工作流程

    1. 接收电信号：将电信号转为数字信号
    2. FCS 校验：检查数据包是否在传输中损坏
    3. 查 MAC 地址表：根据目标 MAC 地址找到对应端口

    与计算机网卡不同，交换机的端口没有 MAC 地址，它不核对接收方地址，而是接收所有包并存入缓冲区。

    MAC 地址表的工作机制

    交换机维护一张 MAC 地址表：

    MAC 地址 端口
    00:02:B3:1C:9C:F9 3 号端口
    … …

    找不到目标 MAC 地址时：交换机将包泛洪（Flood）到除源端口外的所有端口。目标设备收到后会响应，交换机从响应包中学习到其 MAC 地址并记录，下次就无需泛洪了。

    广播地址（会发送到所有端口）：

    • MAC 广播：FF:FF:FF:FF:FF:FF
    • IP 广播：255.255.255.255

    9. 出境大门 —— 路由器

    路由器 vs 交换机

    对比项 路由器 交换机
    工作层次 第三层（IP 层） 第二层（MAC 层）
    端口特征 有 MAC 地址 + IP 地址 无 MAC 地址
    转发依据 IP 地址 + 路由表 MAC 地址表
    俗称 三层网络设备 二层网络设备

    路由器的包接收流程

    1. 电信号 → 数字信号
    2. FCS 校验
    3. 核对 MAC 地址：只接收目标 MAC 为自身的包，其余丢弃
    4. 去掉 MAC 头部（MAC 头部的使命已完成）
    5. 查询路由表，决定转发目标

    路由表匹配规则

    路由器对每条路由表记录，将目标 IP 与子网掩码做 & 运算，结果与目标地址比较：

    • 匹配成功 → 该条目为候选转发路径
    • 所有条目都不匹配 → 使用默认路由（0.0.0.0/0）

    路由器的发送操作

    根据路由表的网关（Gateway）字段判断：

    • 网关有 IP：包还未到终点，将包转发给该 IP（下一跳路由器），通过 ARP 获取其 MAC 地址，重新封装 MAC 头部后发出
    • 网关为空：目标 IP 即为终点，直接 ARP 查询目标 MAC，交付

    关键规律：在整个网络传输过程中，源 IP 和目标 IP 始终不变，而 MAC 地址在每一跳都会更新，因为 MAC 地址只用于当前链路（两点之间）的传输。

    经过多个路由器的层层转发，数据包最终抵达目标服务器所在的局域网。
