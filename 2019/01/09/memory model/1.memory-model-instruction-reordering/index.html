    指令重排序：你的代码并不总按你写的顺序执行

    Alex Wu
    2019-01-09
    内存模型

    并发编程是现代软件工程中最难掌握的领域之一，很多 bug 不是逻辑错误，而是对底层执行机制的误解造成的。指令重排序（Instruction Reordering）通常是一切混乱的根源。

    1. 一个令人困惑的问题

    你写下这样的代码：

    直觉上，如果线程 2 看到 flag == 1，那么 data 一定已经是 42 了，因为线程 1 先写了 data，再写了 flag。

    但现实中，这段代码可能打印出 0。

    这不是编译器的 bug，也不是 CPU 的故障，而是指令重排序（Instruction Reordering）这一核心机制的正常表现。理解它，是迈向正确并发编程的第一步。

    2. 什么是指令重排序？

    指令重排序是指程序中指令的实际执行顺序，与你在源代码中书写的顺序不同。

    这种重排序发生在两个层面：

    层面 发生时机 执行者
    编译器重排序 编译期 编译器（GCC、Clang、MSVC 等）
    CPU 重排序（乱序执行） 运行期 处理器硬件

    两种重排序都是对程序员透明的优化，在单线程程序中，它们不会改变程序的可见行为。然而，一旦涉及多线程共享内存，这些隐形的重排序就会埋下难以察觉的 bug。

    3. 为什么要重排序？性能是根本驱动力

    3.1 充分利用 CPU 执行单元

    现代 CPU 拥有多个独立的执行单元（ALU、FPU、Load/Store 单元等），它们可以并行运作。若两条指令之间没有数据依赖，CPU 会同时发射并执行它们。

    这里 a 和 x 互不相关，CPU 可以在不同执行单元上同时处理两条指令，而不是顺序等待。这种技术叫指令级并行（ILP, Instruction-Level Parallelism），是超标量处理器的核心能力。

    3.2 隐藏内存访问延迟

    访问内存（RAM）比 CPU 执行指令慢几个数量级。当一条指令需要等待内存数据时，CPU 不会空转，它会提前执行后续的独立指令，把等待时间”填满”。

    在等待指令 2 的内存数据返回期间，CPU 会先执行指令 3，避免流水线停顿（Pipeline Stall）。

    3.3 编译器的静态优化

    编译器在生成机器码时，会对指令序列进行静态分析和重新调度，以获得更好的指令级并行度。常见优化包括：

    4. 单线程安全，多线程危险

    4.1 为什么单线程没问题？

    即便执行顺序被重排，CPU 和编译器都会严格保证：对于同一个线程，程序的可见行为与按源码顺序执行完全一致。

    这通过以下机制实现：

    • 数据依赖追踪：若指令 B 依赖指令 A 的结果，A 绝不会被移到 B 之后。
    • 重排序缓冲区（ROB, Reorder Buffer）：CPU 内部用 ROB 追踪指令的原始程序顺序，保证结果按序提交（Commit），对外呈现一致的状态。

    4.2 多线程的”可见性”问题

    然而，在多核场景下，每个核心有自己的寄存器和多级缓存（L1/L2/L3）。一个核心对内存的写操作，不会立刻对其他核心可见。

    这就是开头例子中”打印 0”的根本原因：写操作对其他核心的可见顺序，可能与写操作的实际执行顺序不同。

    5. 四种重排序类型

    从内存操作（Load/Store）的角度，重排序可以分为四种类型：

    类型 含义 是否常见
    LoadLoad 后续读取被移到前一个读取之前 弱序架构常见（ARM）
    StoreStore 后续写入被移到前一个写入之前 弱序架构常见（ARM）
    LoadStore 读取被移到后续写入之后 罕见
    StoreLoad 写入被移到后续读取之后 最常见，x86 也可能发生

    6. 深入硬件：Store Buffer 与 Invalidate Queue

    为什么 CPU 会发生这样的重排序？背后有具体的硬件原因。

    6.1 Store Buffer（写缓冲区）

    在多核处理器中，当一个核心写入某个缓存行（Cache Line），若该缓存行当前处于 Invalid 状态，核心需要先向其他核心发送”无效化请求”（Invalidate），等所有核心确认后，才能完成写入。

    这个等待过程非常耗时。为了避免 CPU 空转，引入了 Store Buffer：

    • 核心把写操作先放入 Store Buffer，立即继续执行后续指令。
    • Store Buffer 中的写操作在收到其他核心的确认后，再异步写入缓存。

    关键点：Store Buffer 只对本核心可见。其他核心看不到 Store Buffer 中尚未刷入缓存的写操作。

    6.2 Invalidate Queue（无效化队列）

    另一端，当 Core 1 收到 Invalidate 消息时，为了快速返回 ACK（让 Core 0 尽早继续执行），Core 1 会将 Invalidate 消息放入 Invalidate Queue，先回复 ACK，再慢慢处理失效操作。

    这意味着：Core 1 可能已经 ACK 了某个缓存行的失效请求，但本地缓存中该缓存行仍然有效（持有过时数据），直到 Invalidate Queue 中的条目被处理。

    6.3 MESI 协议与缓存一致性

    多核 CPU 通过 MESI 协议（Modified / Exclusive / Shared / Invalid）来维护缓存一致性：

    • Modified（M）：该缓存行仅存在于本核心，且已被修改，与主内存不一致。
    • Exclusive（E）：该缓存行仅存在于本核心，未被修改，与主内存一致。
    • Shared（S）：该缓存行存在于多个核心的缓存中，均为只读。
    • Invalid（I）：该缓存行无效，不可使用。

    MESI 保证了缓存一致性（Cache Coherence），对同一内存地址的操作最终是一致的。但缓存一致性 ≠ 内存一致性（Memory Consistency）。MESI 不保证跨地址的操作顺序，而 Store Buffer 和 Invalidate Queue 的存在，使得不同核心观察到的操作顺序可能大相径庭。

    一句话总结：MESI 解决”看到哪个值”的问题，内存模型解决”按什么顺序看到”的问题。

    8. Happens-Before：描述顺序保证的形式语言

    在讨论并发正确性时，计算机科学家用 Happens-Before（先行发生） 关系来精确描述操作的顺序保证。

    8.1 定义

    若操作 A happens-before 操作 B，则 A 的内存效果在 B 执行前对执行 B 的线程完全可见。

    这是一个偏序关系（Partial Order），满足传递性：若 A happens-before B，且 B happens-before C，则 A happens-before C。

    注意：Happens-before 不等于实际的时间先后顺序。它是一个关于可见性的承诺，而非物理时间的约束。

    8.2 哪些操作建立 Happens-Before 关系？

    以 Java 内存模型（JMM）为例，以下操作天然建立 happens-before 关系：

    规则 说明
    程序顺序规则 同一线程内，前面的操作 happens-before 后面的操作
    volatile 变量规则 对 volatile 变量的写 happens-before 后续对同一变量的读
    锁规则 unlock 操作 happens-before 后续对同一锁的 lock 操作
    线程启动规则 Thread.start() happens-before 被启动线程的任何操作
    线程终止规则 线程中的所有操作 happens-before 其他线程检测到该线程终止
    传递性 若 A hb B 且 B hb C，则 A hb C

    8.3 回到最初的例子

    由于 flag 是 volatile：

    • 步骤 1 happens-before 步骤 2（程序顺序规则）
    • 步骤 2 happens-before 步骤 3（volatile 变量规则）
    • 步骤 3 happens-before 步骤 4（程序顺序规则）
    • 由传递性：步骤 1 happens-before 步骤 4

    这保证了线程 2 若看到 flag == 1，必然能看到 data == 42。

    9. 强内存模型 vs 弱内存模型

    不同 CPU 架构对重排序的限制程度不同，形成了强内存模型和弱内存模型之分：

    9.1 强内存模型：x86/x86-64

    x86 架构实现了相对严格的 TSO（Total Store Order） 模型：

    • 允许 StoreLoad 重排序（写操作可以在后续读操作之后对外可见）
    • 禁止 LoadLoad、StoreStore、LoadStore 重排序
    • Store Buffer 的存在是 StoreLoad 重排序的硬件根源

    9.2 弱内存模型：ARM / PowerPC / RISC-V

    ARM 架构采用弱内存模型，允许几乎所有类型的重排序（LoadLoad、StoreStore、LoadStore、StoreLoad），默认只保证对同一地址操作的一致性。

    这意味着在 ARM 上正确运行的并发代码，需要更多的内存屏障指令，开销也更高。但也正因如此，ARM 在不需要同步的路径上可以做更激进的优化。

    9.3 对程序员的影响

    架构 默认保证 需要屏障的场景
    x86-64 较强（TSO） 主要是 StoreLoad 场景
    ARM / RISC-V 较弱 几乎所有跨线程共享内存操作
    Java / C++ 标准 由语言内存模型定义 使用 volatile / atomic 保证

    好消息是：现代编程语言（Java、C++11 起、Go、Rust）都提供了语言级别的内存模型，通过 volatile、atomic、mutex 等抽象，屏蔽了底层架构差异。程序员只需正确使用这些工具，编译器会自动在需要的地方插入屏障。

    10. 如何写出正确的多线程代码？

    了解了重排序的本质后，解决方案就清晰了：需要使用同步工具来建立 happens-before 关系，明确约束操作的可见性与顺序。

    10.1 互斥锁（Mutex / Lock）

    最经典的同步原语。持有锁期间的所有操作，对下一个获取同一把锁的线程完全可见。

    10.2 原子操作（Atomic Operations）

    原子操作保证操作的不可分割性，同时可以指定内存顺序语义：

    Release-Acquire 语义：

    • Release（store）：该操作之前的所有内存写入，对执行对应 Acquire 操作的线程可见。
    • Acquire（load）：该操作之后的所有内存读取，能看到对应 Release 操作之前的写入。

    10.3 内存屏障（Memory Barrier / Fence）

    内存屏障是最底层的同步工具，直接控制 CPU 和编译器的重排序行为：

    在 Linux 内核中，常见的屏障宏包括：

    • smp_rmb()：读屏障，防止 LoadLoad 重排序
    • smp_wmb()：写屏障，防止 StoreStore 重排序
    • smp_mb()：全屏障，防止所有类型重排序

    11. 总结：建立正确的心智模型

    概念 关键认知
    指令重排序 编译器和 CPU 均会重排序，目的是性能优化
    单线程安全 重排序不改变单线程的可见行为（数据依赖保证）
    多线程危险 重排序会破坏线程间的可见性假设
    Store Buffer CPU 写缓冲区，使写操作延迟对其他核可见，是 StoreLoad 重排序的硬件来源
    Invalidate Queue 失效队列，使缓存失效操作被延迟处理，导致读到过时值
    MESI 协议 保证缓存一致性（Coherence），但不保证操作顺序（Consistency）
    Happens-Before 描述操作间可见性保证的形式化工具
    内存模型 语言/硬件对”重排序边界”的规范，程序员与系统的契约

    核心原则：

    任何对共享内存的并发访问（至少一个写操作），都必须通过同步机制来建立明确的 happens-before 关系。否则，程序行为是未定义的。
