    内存模型详解：线程究竟如何看待内存

    Alex Wu
    2019-01-20
    内存模型

    在上一篇文章中，我们深入探讨了指令重排序的根源–编译器优化与 CPU 乱序执行，这些重排序在单线程中是透明的，但在多线程共享内存时会引发难以调试的 bug。本文将在此基础上，系统介绍内存模型（Memory Model） 这一核心抽象，解释不同硬件架构与编程语言如何定义线程对内存操作的可见性规则。

    1. 为什么需要内存模型？

    设想你正在编写多线程代码，你会隐约假设：

    • 我写入内存的数据，其他线程迟早能读到；
    • 我先写 A 再写 B，其他线程如果看到了 B 的新值，也应该能看到 A 的新值；
    • 代码从上到下执行，不会”倒着跑”。

    这些假设在单核、单线程时代是成立的。但在多核处理器 + 编译器激进优化的今天，这些假设全都可能被打破。

    内存模型正是为了重新建立这些规则而存在的。它是程序员、编译器、CPU 三方之间的一份契约，精确定义了：

    • 哪些内存操作（load/store）允许被重排序；
    • 一个线程的写操作，何时、以何种顺序对其他线程可见；
    • 多个线程如何观察彼此的读写操作。

    没有内存模型，多线程程序的行为就无从推理，每个线程可能以任意顺序观察到操作，导致完全不可预测的结果。

    每种模型都是在可编程性与性能之间做出的不同权衡。本文将逐一深入剖析这三种模型。

    3. 顺序一致性（Sequential Consistency，SC）

    3.1 Lamport 的定义

    顺序一致性（SC）由图灵奖得主 Leslie Lamport 于 1979 年提出，是最直观的内存模型。其定义如下：

    任何执行的结果，都与所有处理器的操作按照某种顺序执行的结果相同，且每个处理器内部的操作在这个序列中维持其原有的程序顺序。

    用人话说，SC 就像一个单一的、全局共享的时间轴：

    • 所有线程的所有操作，都被放在这条时间轴上；
    • 每个线程自己的操作，保持在时间轴上的相对顺序；
    • 所有线程看到的，都是同一条时间轴上的事件序列。

    3.2 SC 的直观模型

    可以把 SC 想象成一个单处理器的时间片调度模型：即使有多个线程，任意时刻也只有一个线程在执行一个操作，执行完再轮到下一个，且所有线程立即可见这次操作的结果。

    3.3 SC 的保证：不可能的结果

    考虑以下经典的”消息传递”测试（初始值 x=0, y=0）：

    在 SC 模型下，r1 == 0 && r2 == 0 是不可能出现的。

    推理如下：Thread 1 的 x=1 和 r1=y 按程序顺序执行，Thread 2 的 y=1 和 r2=x 同理。在任意合法的顺序交错中，要么 x=1 在 r2=x 之前（则 r2=1），要么 y=1 在 r1=y 之前（则 r1=1），两者必居其一。所以两个读操作不可能同时读到 0。

    3.4 SC 的代价

    SC 听起来完美，但实现代价极高：

    • 禁止 Store Buffer：每次写操作必须立即全局可见，无法使用写缓冲区延迟提交；
    • 禁止乱序执行：CPU 不能将后续指令提前执行；
    • 禁止编译器重排序：所有跨线程可见的优化均被限制。

    这些限制会导致严重的性能损失，因此现代硬件没有一个架构完整实现 SC。

    4. 全存储顺序（Total Store Order，TSO）

    4.1 x86 的现实选择

    x86/x86-64 架构（Intel 和 AMD）实现了 Total Store Order（TSO）模型。TSO 是比 SC 稍弱的模型，但仍然强到足够大多数程序员直觉式地推理，同时允许一项关键的性能优化：Store Buffer（写缓冲区）。

    4.2 TSO 的核心机制：每核一个 FIFO 写缓冲区

    在 TSO 模型中，每个 CPU 核心都有一个先进先出（FIFO）的写缓冲区：

    写操作先进入 Store Buffer，异步地按 FIFO 顺序刷入共享内存。读操作直接访问共享内存（或先检查自己的 Store Buffer，称为 store-to-load forwarding）。

    需要注意：这里提到的 Store Buffer（写缓冲区）并不是 CPU Cache（缓存），很多人会把两者搞混。Store Buffer是一个位于 CPU 核心与缓存/内存之间的、专门用于暂存待写入数据的硬件结构。它们虽然紧密协作，但功能和目的不同。

    下面详细解释两者的区别与联系：

    Store Buffer 是什么？

    位置：位于每个 CPU 核心的执行单元（Execution Unit）和 L1 缓存之间。

    作用：

    • 允许核心在执行 store 指令时不等待写操作真正完成（比如缓存未命中、需要获取缓存行的独占权限等），而是先把数据放入 Store Buffer，立即继续执行后续指令，提高性能。
    • 在 TSO 模型中，Store Buffer 是 FIFO 队列，保证本核心的写操作按程序顺序对外可见。

    可见性规则（TSO 关键特性）：

    • Load 操作会“窥探”（snoop）自己的 Store Buffer（称为 store-to-load forwarding），所以本核心能立即看到自己刚写的值；
    • 其他核心不能看到 Store Buffer 中的数据，必须等数据刷入缓存/内存后才对其他核心可见。

    ✅ 所以：Store Buffer 是写操作的临时暂存区，用于解耦执行引擎和内存子系统。

    CPU Cache 是什么？

    层级结构：通常包括 L1（私有）、L2（可能私有或共享）、L3（通常共享）。

    作用：

    • 缓存最近访问的内存数据，减少访问主存延迟；
    • 通过缓存一致性协议（如 MESI）保证多核间缓存行的一致性；
    • 是内存的“副本”，对所有核心（通过一致性协议）最终可见。

    写策略：

    • x86 通常采用 Write-Back + Write-Allocate：写操作先写入缓存（标记为 Modified），之后再回写到主存；
    • 但在写入缓存前，如果缓存行不在本地（缓存未命中），需要先从其他核心或内存获取该行，并获得独占权限。

    ✅ 所以：Cache 是多核共享可见的数据存储层（通过一致性协议），而 Store Buffer 是仅本核心可见的写队列。

    两者的关系

    当一个 store 指令执行时：

    • 数据先放入 Store Buffer；
    • 后台逻辑尝试将该写操作应用到 L1 Cache（可能需要等待缓存行就位）；
    • 一旦写入 L1 Cache（状态变为 Modified），并通过缓存一致性协议广播，其他核心才能看到这个写；
    • Store Buffer 中的条目随后被移除。

    Store Buffer 是“待写队列”，Cache 是“已写副本”。只有当写操作从 Store Buffer 刷入 Cache 后，其他核心才能看到它。

    因此，Store Buffer ≠ CPU Cache，它是理解 x86 TSO 内存模型（以及为何需要内存屏障）的核心概念之一。

    4.3 TSO 的三条规则

    规则 1：Store-Store 顺序保持

    若同一线程先写 x=1，后写 y=2，其他线程观察这两次写操作时，必定先看到 x=1 才能看到 y=2。

    规则 2：Load-Load 顺序保持

    若同一线程先读 r1=x，后读 r2=y，读操作按程序顺序执行。

    规则 3：Store-Load 可以重排序（TSO 与 SC 的核心区别）

    后续的 Load 可以在更早的 Store 之前执行，只要它们访问不同的地址。这正是 Store Buffer 带来的直接后果，写操作被缓冲，而读操作直接访问内存，于是”读”可以先于”写”对外可见。

    4.5 TSO 的 Store-to-Load Forwarding

    TSO 有一个重要的特性：一个核心可以从自己的 Store Buffer 中读取尚未提交到共享内存的值。

    这保证了单线程语义的正确性，即便使用了 Store Buffer，本线程对自己写入的数据也能立即看到最新值。其他线程则需要等 Store Buffer 刷新才能看到。

    当需要打破 Store-Load 重排序时，x86 提供了 MFENCE（Memory Fence）指令：

    MFENCE 会阻塞执行，直到 Store Buffer 中所有的写操作都被刷新到共享内存。其代价是数十至数百个时钟周期的延迟。

    此外，x86 还提供 LFENCE（Load Fence，禁止 Load 重排序）和 SFENCE（Store Fence，禁止 Store 重排序），但在 TSO 模型下这两者很少需要单独使用。

    5. 弱/松散内存模型（Relaxed / Weak Memory Model）

    5.1 ARM 和 POWER 的哲学

    ARM（移动端主流）、POWER（IBM 服务器）等架构实现了更弱的内存模型。这些架构的设计哲学是：默认允许几乎所有重排序，需要顺序保证时，程序员显式指定。

    弱内存模型的概念模型可以这样理解：

    每个核心好似有自己的”内存副本”，写操作以任意顺序传播到其他核心，没有全局一致的写入顺序。

    5.2 弱内存模型允许的重排序

    与 TSO 只允许 Store-Load 重排序不同，弱内存模型允许全部四种重排序：

    6.2 性能与可编程性对比

    6.3 一个直观的类比

    可以把三种模型类比为三种不同的”会议记录”方式：

    • SC（顺序一致性）：会议全程实时直播，所有参会者看到的是完全相同的、按时间顺序的发言记录。
    • TSO：会议记录员先把发言写在草稿纸上（Store Buffer），大家看的是草稿纸缓冲后的内容，同一位发言者的发言顺序保持，但不同发言者的发言可能交替显示。
    • 弱模型：每个参会者各自维护一份笔记，消息以不确定的顺序传到每个人手中，不同人的笔记可能完全不同，需要参会者主动”同步”才能对齐信息。

    7. 语言级内存模型与 DRF-SC 保证

    硬件内存模型因架构而异，这给跨平台的并发编程带来了极大困难。为此，现代编程语言提供了语言级内存模型，作为程序员与底层硬件之间的抽象层。

    7.1 DRF-SC：语言内存模型的黄金标准

    几乎所有现代语言的内存模型都基于同一个核心承诺：

    DRF-SC（Data-Race-Free Sequential Consistency）：如果程序是数据竞争自由的（Data-Race-Free），则其所有执行看起来都是顺序一致的。

    这个承诺的含义是：

    • 程序员的责任：通过正确使用同步机制（锁、原子操作等）消除数据竞争；
    • 语言/编译器的责任：在满足上述条件时，保证程序行为等价于 SC 模型；
    • 代价：编译器在同步点附近插入必要的内存屏障指令，屏蔽底层架构差异。

    数据竞争的定义：两个或多个线程并发地访问同一内存地址，且至少有一个是写操作，且这些访问之间没有 happens-before 关系。

    如果程序没有数据竞争，则程序的所有执行都将表现为顺序一致的。（JLS 17.4.3）

    8. 缓存一致性 vs. 内存一致性：一个常见混淆

    很多人会把缓存一致性（Cache Coherence）与内存一致性（Memory Consistency）混为一谈，但这是两个不同层次的概念：

    概念 解决的问题 保证的范围
    缓存一致性 对同一地址的多个缓存副本，最终看到相同的值 单个内存地址的一致性
    内存一致性 多个线程对不同地址的操作，以什么顺序可见 跨地址的操作顺序

    缓存一致性（如 MESI 协议）是内存一致性的必要条件，但不是充分条件。

    举例说明：MESI 协议保证了当 Core 0 写入 x=1 并刷新到缓存后，Core 1 一定能读到 x=1（而非旧值）。但 MESI 不能保证 Core 0 的 x=1 和 y=2 两次写操作，对 Core 1 和 Core 2 的可见顺序相同。

    9. 总结：建立层次化的理解框架

    内存模型是一个层次化的概念体系，理解它需要同时把握多个层面：

    核心心智模型：

    层面 关键问题 答案
    为什么有内存模型？ 硬件和编译器会重排序，需要规范 内存模型是三方契约
    SC 为何不够用？ 实现代价极高，禁止几乎所有优化 没有硬件完整实现 SC
    TSO 允许什么？ Store Buffer 带来 Store-Load 重排 r1=0, r2=0 在 x86 合法
    弱模型更危险？ 允许全部四种重排序 需要显式 fence 保证顺序
    语言如何应对？ 抽象出 DRF-SC 保证 正确同步 → 行为等价 SC

    给程序员的实践建议：

    1. 优先使用高层抽象：mutex、channel、concurrent 容器，它们已经内置了正确的内存语义。
    2. volatile / atomic 不是万能药：了解它们在你使用的语言中的具体语义，不要盲目添加。
    3. 无锁代码极难正确：除非是性能关键路径，否则不要尝试；即便尝试，也要基于经过验证的模式。
    4. 在弱架构上测试：如果你的代码要在 ARM 上运行，必须在 ARM 上测试，x86 上不出现的 bug 在 ARM 上会爆发。
    5. 用工具辅助验证：ThreadSanitizer（TSan）、Helgrind 等工具可以在运行时检测数据竞争。
