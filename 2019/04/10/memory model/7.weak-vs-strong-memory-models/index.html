    Weak vs. Strong Memory Models

    Alex Wu
    2019-04-10
    内存模型

    Memory Model

    内存重排序（Memory Reordering）的种类繁多，并非所有重排序都以同等概率发生。其具体行为取决于你所面向的处理器架构，以及所使用的编译工具链。

    内存模型（Memory Model） 描述的是：对于给定的处理器或工具链，在运行时相对于给定源代码，会发生哪些类型的内存重排序。需要特别注意的是，内存重排序的效果只有在使用无锁编程（lock-free programming）技术时才能被观察到。

    经过系统整理，内存模型可以划分为以下四大类别（从左到右，每种模型都包含其左侧模型的所有保证，并在此基础上增加额外保证）：

    下文中每种具体的物理硬件设备代表一种硬件内存模型。硬件内存模型描述的是，相对于汇编（或机器）代码，在运行时可以期望何种内存排序行为。

    每个处理器家族在内存重排序方面都有其独特的习惯，这些习惯只能在多核或多处理器配置中被观察到。鉴于多核架构已成主流，熟悉这些模型的差异非常有必要。

    此外，也存在软件内存模型。从理论上讲，一旦你用 C11、C++11 或 Java 编写（并调试好）可移植的无锁代码，软件内存模型才是真正需要关心的。但对硬件内存模型有一定了解，在调试时解释意外行为，以及理解”错误代码为何碰巧在某个特定处理器和工具链上运行正确”时，依然大有裨益。

    一、弱内存模型（Weak Memory Models）

    1.1 完全弱序（Fully Weak / Relaxed）

    在最弱的内存模型中，可能会遭遇所有四种类型的内存重排序：

    重排序类型 说明
    LoadLoad 两个读操作之间可以被重排序
    StoreStore 两个写操作之间可以被重排序
    LoadStore 读操作可以被重排序到后面的写操作之后
    StoreLoad 写操作可以被重排序到后面的读操作之后

    只要不改变单一线程隔离执行时的行为，任何 Load 或 Store 操作都可以与其他任何 Load 或 Store 操作发生有效的重排序。在实际中，重排序可能来自两个层面：

    • 编译器层面：编译器在生成指令时对代码进行重排
    • 处理器层面：处理器在执行时对内存访问进行重排

    当处理器具有弱硬件内存模型时，我们通常称其为弱序（weakly-ordered），或称其具有宽松内存模型（relaxed memory model）。

    代表架构：DEC Alpha

    DEC Alpha 是弱序处理器中最经典的例子，没有任何主流处理器的内存排序比它更弱。

    软件层面的影响：C11 / C++11

    C11 和 C++11 公开了一种弱软件内存模型，在许多方面都受到了 Alpha 架构的影响。当在这些语言中使用底层原子操作时，即使你实际上是在针对 x86/64 这样的强序处理器，也必须仍然指定正确的内存排序约束（如 acquire/release 语义），否则无法防止编译器层面的重排序。

    1.2 弱序 + 数据依赖有序（Weak With Data Dependency Ordering）

    尽管 Alpha 的相关性已随着时间推移逐渐降低，但仍有几个现代 CPU 家族继承了弱硬件排序的传统：

    • ARM：目前存在于数亿部智能手机和平板电脑中，并在多核配置中日益普及。
    • PowerPC：Xbox 360 已将其以多核配置的形式带入了 7000 万个家庭。
    • Itanium：微软已不再在 Windows 中支持它，但 Linux 仍然支持，并可在惠普服务器中找到。

    这些架构的内存模型在各方面与 Alpha 几乎同样弱，但有一个对程序员至关重要的共同特性：它们维护数据依赖有序（Data Dependency Ordering）。

    什么是数据依赖有序？

    如果你在 C/C++ 中编写 A->B（即通过指针 A 读取字段 B），你总是能保证加载到的 B 的值，其新鲜程度至少与 A 的值一样新。Alpha 不提供这种保证，而上述 ARM、PowerPC、Itanium 等则提供。

    实际意义：Linux 内核的 RCU（Read-Copy-Update）机制大量依赖数据依赖有序来实现高效的并发读取。

    二、强内存模型（Strong Memory Models）

    2.1 什么是强内存模型？

    关于”强”与”弱”的确切界限，学界存在一些争议。综合来看，可以给出如下定义：

    强硬件内存模型是指：每条机器指令都隐式地附带 acquire 和 release 语义。因此，当某个 CPU 核心执行一系列写操作时，其他每个 CPU 核心看到这些值变化的顺序，与写入顺序完全一致。

    可以这样理解：强内存模型相当于在弱序基础上做了如下限制：

    • ✅ 所有修改按顺序提交到共享内存（无 StoreStore 重排序）
    • ✅ 从共享内存按顺序读取（无 LoadLoad 重排序）
    • ✅ 指令总是顺序执行（无 LoadStore 重排序）
    • ⚠️ StoreLoad 重排序依然可能发生（这是强模型中唯一保留的重排序类型）

    2.2 x86/64

    x86/64 处理器家族在通常情况下是强序的。

    • 在某些特殊情况下，x86/64 的强排序保证会失效，但作为应用程序员，大多数情况下可以忽略这些边界情况。
    • x86/64 处理器可以乱序执行指令（Out-of-Order Execution），但这是硬件实现细节——重要的是，它依然保持内存交互的顺序，因此在多核环境中仍然可以认为是强序的。
    • 历史上，由于规范的演进，曾存在一些混乱，但目前已经明确。

    2.3 SPARC（TSO 模式）

    SPARC 处理器在 TSO（Total Store Order，全局写顺序）模式 下运行时，是强硬件排序的另一个典型例子。

    TSO 的含义与上述定义有一个微妙的区别：它意味着所有核心对共享内存的写操作，始终存在一个单一的全局顺序。x86/64 同样具备这一特性（详见 Intel x86/64 架构规范 Volume 3，§8.2.3.6-8）。

    TSO 属性通常对编写无锁代码的应用程序员没有直接意义，但它是迈向顺序一致性（Sequential Consistency）的重要一步。

    三、顺序一致性（Sequential Consistency）

    顺序一致性是所有内存模型中最强的保证：

    在顺序一致性内存模型中，不存在任何内存重排序。整个程序的执行等价于所有线程指令的某种顺序交错执行（sequential interleaving）。

    例如，在经典的”内存重排序被当场抓获”实验中，r1 = r2 = 0 的结果在顺序一致性模型下是不可能出现的。

    3.1 硬件层面

    如今，几乎找不到在硬件层面保证顺序一致性的现代多核设备。不过，历史上至少存在过一台顺序一致的双处理器机器——1989 年基于 386 架构的 Compaq SystemPro。根据 Intel 的文档，386 处理器还不足以在运行时执行任何内存重排序。

    3.2 软件层面

    顺序一致性作为软件内存模型才真正变得有趣和实用：

    语言/特性 实现方式
    Java 5+ 将共享变量声明为 volatile
    C++11 在原子类型操作上使用默认排序约束 memory_order_seq_cst

    通过上述方式，工具链会：

    1. 限制编译器重排序
    2. 发出 CPU 特定的指令，充当适当类型的内存屏障

    这样，即使在弱序多核设备上，也可以”模拟”出顺序一致的内存模型。

    注意：阅读 Herlihy & Shavit 所著的《多处理器编程的艺术（The Art of Multiprocessor Programming）》时，请注意其中大多数示例都假设了顺序一致的软件内存模型。

    四、综合对比总结

    特性 完全弱序 (Alpha) 弱序+数据依赖 (ARM/PPC/Itanium) 强序/TSO (x86/64, SPARC TSO) 顺序一致 (SC)
    防止 LoadLoad 重排序 ❌ ❌ ✅ ✅
    防止 StoreStore 重排序 ❌ ❌ ✅ ✅
    防止 LoadStore 重排序 ❌ ❌ ✅ ✅
    防止 StoreLoad 重排序 ❌ ❌ ❌ ✅
    数据依赖有序 ❌ ✅ ✅ ✅
    全局写顺序 (TSO) ❌ ❌ ✅ ✅

    五、硬件模型 vs 软件模型

    维度 硬件内存模型 软件内存模型
    定义层次 相对于汇编/机器代码的内存排序行为 相对于高级语言源代码的内存排序保证
    典型代表 Alpha、ARM、x86/64、SPARC TSO C11/C++11 原子操作语义、Java volatile
    观察环境 多核/多处理器配置 多线程程序
    意义 理解底层行为、调试、解释”碰巧正确”的代码 编写可移植、正确的无锁代码

    六、其他细节说明

    内存模型的完整谱系中还有许多其他细节，包括：

    • 控制依赖（Control Dependencies）
    • 因果一致性（Causal Consistency）
    • 不同内存类型（Different Memory Types）

    但就应用级无锁代码编写而言，这些细节的实际意义相对有限，大多数讨论最终都会回归到上述四个主要类别。
