    Acquire 与 Release 语义：多核编程的内存序基础

    Alex Wu
    2019-04-03
    内存模型

    背景：为什么需要内存序？

    在多核或多处理器环境中编写无锁代码时，一个常见的错误假设是：代码按照书写顺序执行，其他核心也会以相同顺序看到内存变化。

    然而现实并非如此。编译器和 CPU 都可能对内存操作进行重排序（reordering），以提升执行效率。在单线程程序中，这种重排序是透明无害的；但在多线程环境下，如果没有显式的内存序约束，就可能导致难以复现的并发 Bug。

    Acquire 语义和 Release 语义，正是解决这一问题的核心工具。

    核心概念定义

    Acquire 语义（读屏障）

    Acquire 语义只能作用于从共享内存读取的操作（包括 load 或 read-modify-write）。
    该操作被称为 read-acquire。

    效果：阻止 read-acquire 操作与其之后的任何读写操作发生重排序。

    换句话说，一旦 read-acquire 完成，它之后的所有内存访问都能”看到”其他线程在对应 write-release 之前写入的数据。

    Release 语义（写屏障）

    Release 语义只能作用于向共享内存写入的操作（包括 store 或 read-modify-write）。
    该操作被称为 write-release。

    效果：阻止 write-release 操作与其之前的任何读写操作发生重排序。

    两者配合使用

    Acquire 和 Release 语义必须成对使用，才能保证共享数据从一个线程完整传递到另一个线程。

    这里的关键变量 Ready 必须是原子操作，且两端必须分别使用 release/acquire 语义，才能形成正确的同步关系（synchronizes-with）。

    用内存屏障类型来理解

    在内存屏障模型中，有四种基本类型：#LoadLoad、#LoadStore、#StoreLoad、#StoreStore。

    • Acquire 语义 = #LoadLoad + #LoadStore（放置在 read-acquire 之后）
    • Release 语义 = #LoadStore + #StoreStore（放置在 write-release 之前）

    注意：两者都不包含 #StoreLoad 屏障。这意味着 write-release 后紧跟一个 read-acquire，这两个操作之间是允许被重排序的。

    C++11 中的实现方式

    方式一：使用 atomic_thread_fence

    方式二：直接在原子操作上指定内存序（推荐）

    方式二将屏障语义”融入”到原子操作本身，更为简洁且语义更精确。

    各平台的实现细节

    不同 CPU 架构对 Acquire/Release 语义的实现成本差异很大：

    平台 Acquire 实现 Release 实现
    x86/64 编译器屏障（几乎免费） 编译器屏障（几乎免费）
    ARM dmb ish 指令 dmb ish 指令
    PowerPC lwsync 指令 lwsync 指令

    x86/64 是强内存序（strongly-ordered）架构：每次 load 隐含 acquire 语义，每次 store 隐含 release 语义。这也是为什么某些在 x86 上运行正常的无锁代码，放到 ARM 或 PowerPC 上就会出问题——正确性不应依赖具体平台的强序保证。

    独立 Fence 与原子操作上的语义：有何不同？

    • store(1, memory_order_release) 仅阻止前面的操作越过该 store 本身，这个约束只针对这个store操作本身。换句话说，编译器/CPU 不能把前面的操作“越过”这个 store，但 如果后面还有其他 store，前面的操作可能被重排到那些后面的 store 之前，只要不越过当前这个 release store。

      举例说明：

      (1) 不能重排到 (2) 之后，但 (1) 可以 重排到 (3) 之前或之后？

      → 实际上，因为 (1) 不能越过 (2)，而 (2) 在 (3) 之前，所以 (1) 也不会出现在 (3) 之后。

      → 但如果 (3) 是另一个线程中的操作，或者编译器/CPU 有更多自由度时，情况会不同。

      关键在于：release store 只“锚定”自己，不保护后续的 relaxed 操作。

      即：(1) 对 Thread 2 是可见的（因为 release-acquire 同步），但 (3) 是 relaxed，不参与同步，所以即使 Thread 2 看到 x==1，也不能保证看到 y==1。同步关系不会延伸到 relaxed 操作。

    • atomic_thread_fence(memory_order_release) 则阻止前面的操作越过其后的所有 store。

      它的语义是（根据 C++ 标准）：所有在栅栏 之前的 内存操作（包括非原子、relaxed 原子等），都不能被重排到 栅栏之后的任何 release 操作（或写操作） 之后。

      但更重要的是 编译器和硬件层面的排序约束：所有在 fence 之前的读写操作，都不能越过 fence 后面的任何写操作（store）。

      示例对比：

      (1) 不能被重排到 (3) 之后！即使 (3) 是 relaxed，fence 也阻止前面的操作越过其后的任何 store。

      这就是你引述的核心：atomic_thread_fence(memory_order_release) 阻止前面的操作越过其后的所有 store，而不仅仅是某一个 store。

    形式 作用范围 限制强度
    x.store(..., release) 仅约束该 store 本身 较弱：只防止前面的操作越过 这个 store
    atomic_thread_fence(release) 约束 fence 之后的所有 store 更强：前面的操作不能越过 任何后续 store

    在实现 DCLP（双重检查锁定）等模式时，这一区别至关重要。独立 fence 语义更强，但也可能是一种性能悲观化（performance pessimization），应根据实际需求选择。

    常见误区

    1. 误以为 x86 上不需要内存序约束：x86 的强序特性是硬件行为，C++11 的内存模型是软件规范。即使在 x86 上省略约束”碰巧正确”，放到 ARM 设备上就可能崩溃。

    2. 混淆 Acquire/Release 与 seq_cst ：memory_order_seq_cst 提供最强保证，但代价是跨核通信开销。Acquire/Release 是在性能和正确性之间取得平衡的选择。

    3. 单独使用 release 或 acquire：两者必须配对使用，才能形成有效的 synchronizes-with 关系，单方面使用没有意义。

    4. 将 Release 语义用于 load，或将 Acquire 语义用于 store：Release 只适用于写操作，Acquire 只适用于读操作，反过来是无效的。

    小结

    概念 作用对象 防止重排方向 典型用途
    Acquire 语义 读操作（load） 之后的读写不可上移 获取锁、读取共享数据
    Release 语义 写操作（store） 之前的读写不可下移 释放锁、发布共享数据

    Acquire 和 Release 语义是无锁编程的基石。理解它们，不仅能帮助你写出跨平台正确的并发代码，也能让你在性能与正确性之间做出合理权衡——而不是盲目地使用 memory_order_seq_cst 了事。
