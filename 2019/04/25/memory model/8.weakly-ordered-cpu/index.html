    弱序 CPU 的内存重排：一次亲手验证的实验

    Alex Wu
    2019-04-25
    内存模型

    原文：This Is Why They Call It a Weakly-Ordered CPU — Jeff Preshing，2012

    背景：内存重排序与弱序 CPU

    在写无锁（lock-free）编程的博客系列中，Jeff Preshing 长期讨论诸如 acquire/release 语义、弱序 CPU 等话题。然而，这些概念讲起来再透彻，也不如一个活生生的实验更有说服力。

    所谓”弱序 CPU”（weakly-ordered CPU），其最核心的特征是：

    一个 CPU 核心对共享内存的写入顺序，在另一个 CPU 核心看来可能是不同的顺序。

    换句话说，如果核心 A 先写了变量 X，再写了变量 Y，核心 B 可能会先看到 Y 的新值，再看到 X 的新值——与写入顺序相反。

    对于我们日常开发中最常用的 x86/64 架构（Intel 和 AMD 的绝大多数桌面和笔记本处理器），在普通应用场景下并不存在这个问题，因为 x86/64 是强序（strongly-ordered）的：一个核心的写入序列，所有其他核心都能以相同的顺序观察到。

    因此，要想亲眼见到内存重排序的效果，我们需要一颗弱序多核设备。作者选择的是他口袋里的 iPhone 4S，它搭载了一颗双核 ARM 处理器，而 ARM 架构正是弱序的。

    实验设计

    实验目标

    实验的核心对象是一个受互斥锁保护的共享整数 sharedValue。实验流程如下：

    1. 启动两个线程；
    2. 每个线程独立运行，直到它将 sharedValue 累加了 10,000,000 次；
    3. 理论上，两个线程各加 1000 万次，最终结果应当是 20,000,000。

    线程行为

    线程并不阻塞等待互斥锁，而是：

    • 做随机量的”忙等”（busy work，即无用的 CPU 空转）；
    • 在随机时刻尝试加锁；
    • 如果加锁成功，则对 sharedValue 加 1，然后解锁，计数器 +1；
    • 如果加锁失败，就继续做忙等，然后再试。

    互斥锁的实现

    互斥锁是一个概念，有无数种实现方式。我们完全可以用语言标准库提供的 std::mutex，一切都会正确运行，但那样就没什么好展示的了。

    这里自己实现一个极简的互斥锁，然后故意破坏它，以展示弱序硬件带来的后果。

    数据结构

    互斥锁由一个整数 flag 表示：

    • flag == 0：锁未被持有（空闲）；
    • flag == 1：锁已被持有（占用）。

    在 C++11 中，将 flag 声明为原子类型 atomic<int>，以保证 CAS（Compare-And-Swap）操作的原子性。

    加锁

    这里使用了 memory_order_acquire 内存序约束，表示获取语义（acquire semantics）：保证在此操作之后的所有读写，不会被重排到此操作之前——从而确保我们能看到上一个持有锁的线程所做的最新修改。

    解锁

    解锁使用 memory_order_release，表示释放语义（release semantics）：保证在此操作之前的所有读写，不会被重排到此操作之后——从而确保我们对 sharedValue 的修改在释放锁之前已经对外可见。

    Acquire 和 Release 必须成对使用，才能保证共享数据从一个线程安全地传递到另一个线程。

    结果始终小于 20,000,000！ 两个线程各自执行了 1000 万次累加，汇编指令顺序与 C++ 代码意图完全吻合——但最终结果仍然是错误的。

    这正是 CPU 的内存重排序造成的。

    错误根因分析

    以一种可能的重排为例：

    CPU 核心可以将 sharedValue 的写入延迟（保留在本地写入缓冲区中），而 flag 的写入已经全局可见。这就意味着：互斥锁被释放的时候，受保护的数据修改还没来得及传播出去。另一个线程此时进入临界区，读取的是旧值，然后写入的是基于旧值的新值，相当于上一次的修改被静默覆盖了，最终计数永远不会到达 20,000,000。

    这就是”弱序”的直观含义：解锁操作可能在数据写入操作之前，就对其他核心可见了。

    编译器此时会插入 dmb ish（Data Memory Barrier）指令：

    注：在 ARMv8（AArch64）上，这一机制得到了进一步优化。casab（Compare And Swap, Acquire, Byte）和 stlrb（Store-Release Register, Byte）等指令直接将 acquire/release 语义编码进指令本身，无需额外的 dmb 指令。

    修复后的实验结果

    每次都是精确的 20,000,000。自制的互斥锁终于能够正确保护 sharedValue，确保每次解锁时，所有修改都被安全地传递给下一个持有锁的线程。

    延伸讨论

    为什么在 x86/64 上看不到这个问题？

    将同一份带有错误内存序（memory_order_relaxed）的程序，在 x86/64 机器上运行（即使是多核），结果仍然是正确的 20,000,000。

    原因在于 x86/64 是强序架构（Total Store Order, TSO）：所有核心观察到的写入顺序与实际写入顺序一致（除了 StoreLoad 重排这一个例外）。在 x86/64 上，store 和 store 之间不会发生重排，因此上面那种”解锁先于数据写入可见”的情况根本不会出现。

    这带来了一个危险的假象：

    你的代码在 x86/64 上运行完全正确，但实际上使用了错误的内存序。一旦移植到 ARM 或 PowerPC 等弱序平台，就会出现难以复现的数据竞争 bug。

    强序与弱序架构的对比

    架构 内存模型 典型平台
    x86 / x86-64 强序（TSO） 桌面PC、服务器
    ARM（AArch32/AArch64） 弱序 手机、平板、Apple Silicon
    PowerPC 弱序 Xbox 360、部分服务器
    Itanium (IA-64) 弱序 HP 服务器
    Alpha 弱序（最极端） 已退出历史舞台，但深刻影响了 C++11 内存模型

    C++11/C11 的原子操作内存模型正是按照”尽可能弱“的原则设计的，原因是这样能在强序平台上以零开销使用，同时在弱序平台上只在必要时插入屏障。

    单核设备上不会出现这个问题

    文中特别指出：将同一个程序（带错误内存序）放到 iPhone 3GS 或第一代 iPad（均为 ARMv7 但只有单核）上运行，结果同样是正确的 20,000,000。

    这是因为内存重排序在单核场景下是不可见的——CPU 乱序执行时遵守”不改变单线程程序行为“这一铁律，两个线程实际上交替运行在同一颗核心上，重排的写入结果对该核心自身永远是一致的。

    acquire/release 的直觉理解：源代码管理类比

    一个理解 acquire/release 的好方法是将其类比为分布式源代码管理（如 Git）：

    • 每个 CPU 核心就像一台有本地工作区的工作站；
    • sharedValue 和 flag 是远端仓库中的文件；
    • 每个核心都有这些文件的本地副本，不一定是最新的；
    • Release（解锁）= git push，将本地所有修改推送到远端；
    • Acquire（加锁）= git pull，从远端拉取所有最新修改。

    如果解锁时不执行”push”（缺少 release 语义），那么 sharedValue 的修改就还留在本地，其他核心”pull”时拿不到——这正是实验中数据丢失的本质原因。

    总结

    关键点 说明
    弱序 CPU 的本质 一个核心的写入顺序，在另一个核心看来可能不同
    典型弱序架构 ARM、PowerPC、Itanium
    典型强序架构 x86/x86-64（仅有 StoreLoad 一种重排例外）
    内存重排的两个层次 编译期重排（编译器优化）+ 运行期重排（CPU 乱序执行）
    解决方案 正确使用 acquire/release 语义；在需要的地方插入内存屏障
    危险陷阱 在 x86 上测试通过 ≠ 在 ARM 上也正确；弱序 bug 在强序平台上天然被掩盖
    C++11 标准的意义 提供了跨平台统一的原子操作接口，屏蔽了不同架构的差异

    内存重排序是多核编程中最难以察觉的陷阱之一。它在单线程程序中永远不可见，在强序平台上经常被掩盖，只有在弱序多核设备上才会赤裸裸地暴露出来。正确理解并使用 acquire/release 语义，是写出正确无锁代码的基础。
