    编译时的内存序：编译器重排与 Compiler Barrier 详解

    Alex Wu
    2019-03-16
    内存模型

    引言

    从你写下 C/C++ 源代码，到它真正在 CPU 上执行，代码中的内存访问顺序可能已经被悄悄改变了。

    这种改变发生在两个阶段：

    • 编译时：编译器对指令进行重排；
    • 运行时：处理器对指令进行乱序执行。

    两者的动机相同——性能优化。本文聚焦于编译时的内存重排，以及如何用 Compiler Barrier 来约束它。

    核心法则：不能改变单线程程序的行为

    编译器开发者和 CPU 厂商都遵守一条不成文的黄金法则：

    Thou shalt not modify the behavior of a single-threaded program.
    （不得修改单线程程序的可观测行为。）

    正因如此，内存重排在单线程程序中几乎感知不到。在多线程编程中，如果使用了 mutex、semaphore、事件等同步原语，它们本身就会在调用点阻止重排，所以同样感知不到。

    只有在 lock-free 编程中，即多个线程在没有互斥保护的情况下共享内存，内存重排的问题才真正暴露出来。

    编译器重排是什么？

    编译器的职责是将人类可读的源代码翻译成 CPU 可执行的机器码。在这个过程中，编译器有权对指令进行重排，只要不影响单线程下的可观测结果。

    这种重排只在开启优化时才会发生（例如 GCC 的 -O2）。

    什么是 Compiler Barrier？

    Compiler Barrier（编译器屏障）是一种指令，告诉编译器：不要跨越此处对内存访问进行重排。

    GCC 中的 Compiler Barrier

    这条”空汇编”指令不会生成任何机器码，但 "memory" clobber 告诉编译器：这条语句可能读写任意内存位置，因此编译器必须：

    1. 在此之前将寄存器中缓存的值写回内存；
    2. 在此之后重新从内存读取值，而不是使用寄存器缓存。

    它的运行时开销为零，但能有效阻止编译时指令重排。

    Compiler Barrier 的局限性

    Compiler Barrier 只能解决单处理器（单核）场景下的内存重排问题。

    在多核/多处理器系统中，即使编译器生成了正确顺序的指令，处理器本身也可能在运行时对内存操作进行乱序执行。这种乱序对其他 CPU 核心是可见的，而 Compiler Barrier 对此无能为力。

    要在多处理器环境下保证内存访问的正确顺序，需要使用 CPU Fence 指令（即运行时内存屏障）。

    CPU Fence 指令：兼顾编译时与运行时

    CPU Fence 指令（如 PowerPC 的 lwsync、x86 的 mfence）不仅会在 CPU 层面阻止运行时重排，同时也隐含了 Compiler Barrier 的语义。

    编译器不得引入对可能被共享的内存位置的赋值，如果这些赋值在抽象机器中不会发生。

    这从语言规范层面禁止了会引入数据竞争的编译器变换。

    各层次解决方案对比

    场景 推荐方案
    单处理器，阻止编译时重排 asm volatile("" ::: "memory")
    多处理器，需要运行时保证 CPU Fence 指令（mfence、lwsync 等）
    跨平台、现代 C++ 代码 std::atomic + memory_order_release/acquire
    Linux 内核代码 smp_mb() / smp_rmb() / smp_wmb()

    总结

    内存重排是编译器和处理器为了性能优化而进行的常规操作，在单线程场景下完全安全，但在 lock-free 多线程编程中会引发难以调试的 bug。

    核心要点：

    1. 编译器重排发生在编译期，对所有平台（包括单核）都有影响；
    2. Compiler Barrier（如 asm volatile("" ::: "memory")）是零开销的编译期约束，但不提供任何运行时保证；
    3. CPU Fence 指令同时提供编译时和运行时两层保证，适用于多核场景；
    4. C++11 std::atomic 是最推荐的现代跨平台写法，语义清晰，可移植性强。

    理解编译时内存序，是掌握 lock-free 编程的第一步。运行时处理器层面的内存重排，则是另一半需要独立理解的拼图。
