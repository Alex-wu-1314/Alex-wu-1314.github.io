    内存屏障就像版本控制操作

    Alex Wu
    2019-03-29
    内存模型

    背景：为什么需要内存屏障？

    在编写无锁（lock-free）代码时，内存排序（memory ordering）是一个绕不开的核心问题。内存排序分为两个层面：

    • 编译期重排序：编译器为优化性能，可能改变指令顺序；
    • 运行时重排序：处理器本身也可能对内存操作进行重排序。

    本文聚焦于运行时重排序，以及如何通过内存屏障（Memory Barrier / Fence）来约束它。

    运行时重排序在单线程程序中完全不可见，只有在多线程、无锁编程场景下才会暴露问题——尤其是在多核/多处理器系统中。

    常见的内存屏障指令包括：

    • GCC 内联汇编，如 PowerPC 上的 asm volatile("lwsync" ::: "memory")
    • C++11 原子类型上的操作，如 load(std::memory_order_acquire)
    • POSIX 互斥锁操作，如 pthread_mutex_lock

    核心类比：多核系统 ≈ 奇葩的版本控制协作

    理解内存屏障的最佳方式，是将多核系统类比为一种极其奇怪的版本控制协作模式。

    设想这样的场景：

    • 中央仓库（Central Repository）= 主内存 + 共享 L2 缓存
    • 每个程序员的本地工作副本（Local Working Copy）= 每个 CPU 核心的私有 L1 缓存
    • 程序员 Larry 和 Sergey = 两个并发执行的线程

    这个版本控制策略有多奇葩？

    Larry 和 Sergey 对本地副本的修改，会随机地、异步地向中央仓库泄漏（leak），没有任何时序保证：

    • Larry 先修改了文件 X，再修改了 Y 和 Z，但 Y、Z 的变更可能比 X 更早同步到仓库，这就是写操作的重排序；
    • Sergey 从中央仓库拉取变更的时机和顺序同样不确定，这就是读操作的重排序。

    如果两人互不干扰、操作完全不同的文件，这一切都无所谓。但一旦他们开始协作操作同一批文件，问题就来了。

    经典示例：r1 = 0 且 r2 = 0

    假设共享变量 X = 0，Y = 0：

    Larry（线程1） Sergey（线程2）
    X = 1 Y = 1
    r1 = Y r2 = X

    在没有任何同步的情况下，最终可能出现 r1 = 0 且 r2 = 0，两人都还没看到对方的写入。这在版本控制类比中显而易见：Larry 写了本地的 X，还没来得及泄漏到仓库；Sergey 读到的 Y 自然还是旧值 0，反之亦然。

    四种内存屏障类型

    幸好，程序员（处理器）不是完全被动的。可以主动发出 fence 指令来控制内存操作的顺序。以下是四种核心内存屏障类型：

    1. #LoadLoad 屏障 — 对应 Pull（拉取）

    防止屏障前的 Load 与屏障后的 Load 发生重排序。

    类比版本控制中的 git pull / svn update：Sergey 强制从中央仓库拉取最新内容，确保之后读到的数据不早于已知的最新状态。

    使用场景：读取共享标志位后，再读取其保护的数据。

    2. #StoreStore 屏障 — 对应 Push（推送）

    防止屏障前的 Store 与屏障后的 Store 发生重排序。

    类比版本控制中的 git push / svn commit：Larry 把已完成的修改推送到中央仓库，确保数据在标志位之前可见。

    注意：推送是异步的，不保证立即完成，但能保证顺序。

    使用场景：先写数据，再设置共享标志位。

    3. #LoadStore 屏障 — 防止特定指令级重排序

    防止屏障前的 Load 与屏障后的 Store 发生重排序。

    这种类型没有直觉直观的版本控制类比，需要从指令重排序角度理解：

    处理器在某些情况下（如 Load 遇到缓存未命中，Store 命中缓存）会把 Store 提前到 Load 之前执行。#LoadStore 屏障阻止这种行为。

    值得注意：即使存在 #LoadLoad 或 #StoreStore 屏障，#LoadStore 重排序仍可能发生，需要单独处理。

    4. #StoreLoad 屏障 — 对应 Push + 等待完成 + Pull 最新版本

    最强的屏障类型，确保屏障前所有 Store 对其他处理器可见，且屏障后的 Load 获取的是最新值。

    这是唯一能阻止上文 r1 = r2 = 0 情况发生的屏障类型。

    类比版本控制：先强制推送所有本地修改，等待推送完成，然后拉取中央仓库的绝对最新版本。

    为什么 #StoreStore + #LoadLoad 不够？

    • #StoreStore 的推送是延迟异步的，不保证立即生效；
    • #LoadLoad 的拉取不一定拉到最新的 head 版本。

    因此两者组合并不等价于 #StoreLoad。这也解释了为什么 PowerPC 的 lwsync 指令（实现了 #LoadLoad + #LoadStore + #StoreStore，但不含 #StoreLoad）无法阻止 r1 = r2 = 0 的出现。

    全内存屏障（Full Memory Fence）= #StoreLoad + #LoadStore，即四种屏障的合集。在当前所有主流处理器上，凡是实现了 #StoreLoad 的指令，同时也实现了完整的全内存屏障。#StoreLoad 是代价最高的屏障类型。

    四种屏障类型汇总

    屏障类型 防止的重排序 版本控制类比 典型代价
    #LoadLoad Load → Load Pull（拉取） 低
    #StoreStore Store → Store Push（推送，异步） 低
    #LoadStore Load → Store 无直接类比，指令级重排序 低
    #StoreLoad Store → Load Push（同步完成）+ Pull 最新 高

    这个类比的适用范围

    覆盖范围

    这个类比对应的是一种弱内存模型（Weak Memory Model），能有效帮助理解：

    • PowerPC、ARM 等弱内存模型架构；
    • x86/64 虽然内存模型较强（天然保证较多顺序），理解这个类比也有助于写出跨平台安全的代码；
    • C++11 / C11 标准抽象机器模型——基于此模型编写的无锁代码，更容易在各种平台上正确运行。

    不在覆盖范围内

    • 数据依赖屏障（Data Dependency Barriers，如 Alpha 架构的特殊行为）；
    • 硬件底层细节：Store Buffer、缓存一致性协议（MESI）等（推荐阅读 Paul McKenney 的相关论文）。

    小结

    概念 一句话总结
    内存屏障的本质 约束处理器对内存读写操作的重排序
    版本控制类比的价值 用直觉熟悉的 push/pull 操作，对应抽象的 fence 语义
    最需要记住的 #StoreLoad 是最强也最昂贵的屏障，其余三种代价较低
    工程实践建议 优先使用 C++11 原子操作和标准内存序，而非手写 fence 指令

    掌握这四种内存屏障的语义，是理解 C++11 memory_order、Java Memory Model、Linux 内核无锁数据结构等高级话题的重要基础。
