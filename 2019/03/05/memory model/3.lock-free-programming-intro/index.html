    什么是无锁编程？

    很多人把”无锁编程”简单理解为”不使用 mutex 的编程”，这个说法只说对了一部分。

    根据学术界的通用定义，无锁（lock-free）是对一段代码的性质描述，而不是对其实现方式的描述。一段代码被称为无锁，是指它满足以下条件：

    在多个线程并发执行这段代码时，无论线程调度器如何调度（甚至是”恶意”调度），整个系统作为一个整体总能在有限步骤内取得进展（make progress）。

    这里的关键是”系统整体始终能够推进”，而不是”某一个特定线程一定能推进”。

    注意，”lock-free”中的 “lock” 并不是专指 mutex，而是泛指任何可能导致整个应用程序卡死的情况，包括死锁（deadlock）、活锁（livelock），甚至是由于线程调度策略导致某些线程永久无法执行的场景。

    与传统互斥锁的对比

    使用 mutex 的代码天然不是无锁的。原因很简单：假设某个线程持有了 mutex，如果调度器永远不再调度该线程，其他所有等待该锁的线程将被永久阻塞。

    相比之下，无锁代码即使某个线程被挂起，其他线程也可以独立推进，不会受到影响。

    核心技术概览

    当你尝试写出满足无锁定义的代码时，一套相关技术会自然而然地浮现出来：原子操作、内存屏障、ABA 问题的处理……这些并非可以随意选用的独立工具，而是相互关联的知识体系。

    下面逐一介绍这些核心概念。

    1. 原子操作（Atomic Operations）

    原子操作是指在内存上执行的、从外部看来不可分割的操作：没有任何线程能观察到该操作”执行了一半”的中间状态。

    在现代处理器上，许多操作天然就是原子的，例如对已对齐的简单类型（int、指针等）进行的读写操作。

    更重要的是 读-改-写（Read-Modify-Write, RMW） 原子操作，它允许以原子方式完成更复杂的事务。当多个线程同时对同一地址执行 RMW 操作时，它们会依次串行执行，互不干扰。常见的 RMW 操作包括：

    • fetch_add（原子加法）
    • exchange（原子交换）
    • compare_exchange（比较并交换，即 CAS）

    在 C++11 中，std::atomic<T> 提供了全套原子操作的标准接口，如下面这个fetch_add 函数就是其中之一。

    使用规则：任何时候，若两个线程并发访问同一共享变量，且其中至少一个线程执行写操作，则两个线程都必须使用原子操作。否则，就会产生 C++11 标准所称的”数据竞争（data race）”，其后果是未定义行为（undefined behavior）——根本原因在于，非原子操作可能导致”撕裂读/写（torn read/write）”。

    3. ABA 问题

    CAS 的一个经典陷阱是 ABA 问题：

    1. 线程 1 读取共享变量值为 A
    2. 线程 2 将其修改为 B，又修改回 A
    3. 线程 1 执行 CAS，发现值仍为 A，以为没有变化，CAS 成功

    但实际上，共享状态已经被中间修改过了，线程 1 的 CAS 是基于过期的假设做出的。在链表等数据结构中，这可能导致严重的逻辑错误。

    典型的无锁链表的ABA删除操作

    假设有一个无锁单向链表，我们要删除头节点：

    • 初始状态：head → A → B → C
    • 线程 1 执行删除：
      • 读取 head = &A
      • 准备执行 CAS：将 head 从 &A 改为 &B

    但在 CAS 执行前：

    • 线程 2 删除了节点 A（head → B → C）
    • 节点 A 被释放回内存池
    • 线程 3 插入一个新节点，恰好分配到原来 A 的内存地址（即新节点也叫 “A”，地址仍为 &A），此时链表变成：head → A（新）→ …

    然后线程 1 执行 CAS：

    • 它看到 head == &A（但这是新的 A！）
    • CAS 成功，将 head 设置为 &B
    • 结果：新插入的节点 A 被跳过，甚至可能造成链表断裂、内存泄漏或访问已释放内存！

    常见解决方案：

    • 使用带版本号的 CAS（tagged pointer / stamped reference）：将指针与一个单调递增的版本号打包，每次修改都增加版本号，即使值回到 A，版本号也不同。
    • 使用支持双倍宽度 CAS（DWCAS）的硬件指令，如 x86 的 cmpxchg16b。

    4. 顺序一致性（Sequential Consistency）

    顺序一致性是一种内存模型，其含义是：所有线程对内存操作的观察顺序，与程序源代码中的顺序完全一致，且所有线程就这个全局顺序达成一致。

    在顺序一致模型下，不会出现令人费解的内存重排序问题，程序的行为就像所有操作都按照代码顺序、在单处理器上串行执行一样。

    5. 内存重排序（Memory Reordering）

    这是理解无锁编程最难、也最容易被忽视的部分。

    内存重排序是指：代码中的内存操作，在实际执行时（或对其他线程可见时）的顺序，可能与源代码中写的顺序不同。这种重排序有两个来源：

    编译器重排序：编译器在优化时，可能调整指令顺序以提高效率，只要在单线程语义下结果等价即可。

    处理器重排序：现代多核 CPU 出于性能原因（如乱序执行、写缓冲区等），可能以不同于程序顺序的方式对内存操作进行调度，使得其他核心以不同的顺序观察到这些操作的结果。

    在单线程代码或使用了 mutex 的多线程代码中，这些重排序通常不可见（mutex 的实现中已包含必要的内存屏障）。但在无锁代码中，重排序的影响会暴露出来，可能导致极难复现的 bug。

    x86/x64 架构拥有较强的内存模型（strong memory model），大多数情况下不会出现处理器重排序。而 ARM、PowerPC 等架构是弱序（weakly-ordered）的，在多核设备（如手机）上可能明显地观察到重排序效果。

    6. 内存屏障与 Acquire/Release 语义

    为了在多处理器环境下防止内存重排序，需要使用内存屏障（Memory Barrier / Memory Fence）。内存屏障分为以下几类：

    类型 含义
    #LoadLoad 屏障前的 load 必须在屏障后的 load 之前完成
    #LoadStore 屏障前的 load 必须在屏障后的 store 之前完成
    #StoreStore 屏障前的 store 必须在屏障后的 store 之前完成
    #StoreLoad 屏障前的 store 必须在屏障后的 load 之前完成（开销最大）

    在实际编程中，更常用的抽象是 Acquire/Release 语义：

    • Acquire 语义（读取时使用）：屏障后的所有读写操作，不能被重排序到该操作之前。
    • Release 语义（写入时使用）：屏障前的所有读写操作，不能被重排序到该操作之后。

    这对语义可以可靠地在线程间传递信息：一个线程通过 release 写入，另一个线程通过 acquire 读取，就能保证写线程在 release 之前的所有修改，对读线程在 acquire 之后的操作都可见。

    错误或缺失的 acquire/release 语义，是无锁编程中最常见的错误类型。

    常见陷阱与注意事项

    1. 无锁 ≠ 无等待（Wait-Free）
    无锁（lock-free）保证系统整体能推进，但某个特定线程可能因 CAS 反复失败而一直重试。”无等待”（wait-free）是更强的保证，要求每个线程都能在有限步骤内完成操作，但实现成本更高。

    2. 谨慎使用 memory_order_relaxed
    relaxed 语义不提供任何顺序保证，仅保证操作本身的原子性。只有在确认操作间没有依赖关系时（如独立的计数器）才应使用。

    3. 性能不一定比锁高
    无锁编程的价值在于消除死锁、提高在高竞争场景下的吞吐量，但在低竞争场景下，mutex 的性能可能更好，而且代码更易于维护。先做性能分析，再考虑无锁化。

    4. 难以验证正确性
    无锁代码中的 bug 往往与时序相关，极难复现和调试。可以使用 Relacy Race Detector 等工具进行形式化验证和压力测试。

    5. 优先使用标准库中已有的无锁结构
    在自己动手实现之前，优先考虑 std::atomic、并发库中已验证的数据结构，以及 Intel TBB、Folly 等成熟的并发库。

    总结

    无锁编程是一个精密而复杂的领域，其核心知识体系可以总结如下：

    • 无锁的本质是一种系统级的进展性保证，而非简单地”不用锁”
    • 原子操作是无锁编程的基础，确保内存操作的不可分割性
    • CAS 循环是支持多写者的核心模式，需警惕 ABA 问题
    • 内存重排序是多核环境下无锁编程最隐蔽的挑战
    • Acquire/Release 语义是在线程间可靠传递信息的关键工具
    • 顺序一致性提供了最强的保证，但有性能代价；acquire/release 是更精细的折中方案

    理解这些概念之间的层次关系，是写出正确、高效的无锁代码的前提。
