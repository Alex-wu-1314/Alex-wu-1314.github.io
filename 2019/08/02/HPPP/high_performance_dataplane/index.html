    1. 最小化缓存缺失

    缓存缺失（Cache Miss）是高性能数据平面最主要的性能瓶颈之一。现代 CPU 主频与 DRAM 访问延迟之间存在数十到数百倍的差距（通常 L1 命中约 4 周期，主存访问约 200+ 周期）。数据平面每秒需处理数百万甚至数十亿个数据包，任何缓存缺失的累积效应都会导致严重的性能退化。因此，从指令缓存和数据缓存两个维度系统性地减少缓存缺失，是数据平面设计的首要任务。

    1.1 指令缓存——批处理

    BATH-1：数据平面必须完全批处理化

    要求： 数据平面必须构建为若干批处理步骤（Batch Processing Step，BPS），每个步骤的输入和输出均为数据包批次（batch of packets）。逐包（packet-by-packet）处理只能发生在单个步骤内部。

    详细说明：

    批处理是数据平面软件中最重要的结构性设计决策之一。其核心思想是：将相同的处理逻辑应用于一组数据包（通常 32～64 个），而非每收到一个包就完整地走一遍处理流程。

    从指令缓存（L1 I-Cache）的视角来看，批处理的价值在于摊销（amortize）指令加载成本：

    • 当处理第一个数据包时，处理该步骤所需的所有指令会被加载进 L1 I-Cache。
    • 后续同批次的数据包继续使用同一批指令，I-Cache 命中率极高。
    • 相比之下，若采用逐包完整流水线处理，不同处理阶段的代码段会频繁替换 I-Cache，导致大量指令缺失。

    此外，批处理还带来以下好处：

    • 数据预取友好：可以在处理当前包时预取下一个包的数据（见 PF-1）。
    • 分支预测改善：循环结构规则，分支预测器更容易训练。
    • SIMD 向量化：批量操作天然适合 SIMD 指令集优化。

    典型实现框架： DPDK 的 rte_mbuf 批量收包（rte_eth_rx_burst）、VPP（Vector Packet Processing）的 node-graph 架构，均是批处理思想的典型工业实现。

    BATH-2：批处理步骤的代码尺寸必须受控，能够装入目标 L1 I-Cache

    要求： 每个 BPS 的代码大小必须在目标处理器 L1 指令缓存容量范围内，并保留足够余量，以容纳与该步骤并行运行的其他常驻代码（如调度器、中断处理等）。

    详细说明：

    当前主流处理器的 L1 I-Cache 通常为 32KB～64KB（部分嵌入式处理器更小）。一个 BPS 若代码过大，在执行过程中自身就会发生 I-Cache 缺失，完全抵消批处理带来的收益。

    设计准则：

    • 量化评估：在目标平台上使用 perf stat -e L1-icache-load-misses 或 PMU 事件对每个 BPS 进行指令缓存缺失率评估。
    • 控制代码膨胀：避免在 BPS 内部调用大型非内联函数；使用 static inline 控制函数内联边界（也与 LU-1 协同）。
    • 步骤分裂（Step Splitting）：若单个逻辑处理步骤代码过大，应将其拆分为两个串联的 BPS，每个 BPS 独立满足 I-Cache 约束（也与 LU-2 呼应）。
    • 保留余量：实际可用的 I-Cache 空间需减去操作系统/运行时等背景代码占用的部分，通常建议单个 BPS 代码尺寸控制在 L1 I-Cache 的 50%～70% 以内。

    1.2 数据缓存

    数据缓存（L1D/L2 Cache）的效率对数据平面性能同样至关重要。数据包元数据、报文头、查找表等数据结构的布局方式，直接决定了 Cache 命中率。

    1.2.1 预取

    PF-1：必须具备至少一个性能评估环境，能够跟踪 L1D 缓存缺失停顿周期并定位到具体的批处理步骤

    要求： 必须建立性能评估基础设施，能够精确测量每个 BPS 中由 L1D 缓存缺失导致的 CPU 停顿周期（stalled cycles）。

    详细说明：

    “无法测量的东西无法优化。”L1D 缓存缺失的影响因访问模式、数据结构布局和工作集大小而差异巨大，必须通过实测数据驱动优化决策，而非凭经验猜测。

    推荐的性能评估工具链：

    • Linux perf：perf stat -e L1-dcache-load-misses,cycle_activity.stalls_l1d_miss
    • Intel VTune Profiler：提供 Memory Access 分析视图，可精确定位到代码行的 Cache Miss 热点。
    • ARM Streamline / DS-5：针对 ARM 平台的 PMU 事件采集与可视化。
    • DPDK Telemetry / PMD Stats：在数据平面框架层面收集每个处理节点的性能指标。
    • 自定义 PMU 采集：在 BPS 入口/出口插入 PMU 读取代码，计算 per-BPS 的 L1D Miss 停顿周期。

    建立性能基线的意义：

    • 在功能开发完成后，优先优化 L1D Miss 停顿周期最高的 BPS，而非随机优化。
    • 为回归测试提供量化标准，确保代码变更不引入性能退化。
    PF-2：基于实测的 L1D 缺失停顿周期，对停顿过多的 BPS 引入预取以降低影响

    建议： 根据 PF-1 的测量结果，对 L1D 缺失停顿周期超出系统 KPI 允许范围的 BPS，引入数据预取（Prefetch）优化。

    详细说明：

    预取的核心思想是：在 CPU 需要某块数据之前，提前将其从内存加载到缓存，从而将内存延迟与计算并行化，隐藏内存访问延迟。

    1.2.2 访问模式与数据布局

    数据布局是数据缓存效率的决定性因素。良好的数据布局能将 L2/L3 工作集压缩到可接受范围，并最大化 Cache 行的有效利用率。

    APDL-1：不得在性能敏感软件中使用链表，除非有充分理由、文档记录并经性能验证

    要求： 禁止在数据平面性能关键路径上使用链表，除非经过严格的性能评估证明其可接受，并在设计文档中明确说明理由。

    详细说明：

    链表在高性能数据平面中是典型的反模式，原因如下：

    1. 指针追踪导致随机内存访问：遍历链表时，每个节点的地址是前一个节点的运行时计算结果，CPU 无法预测，硬件预取器也无能为力，每次访问都可能引发 Cache Miss。

    2. Cache 行利用率极低：一个典型的链表节点包含 prev、next 指针（各 8 字节）和少量数据，一个 64 字节 Cache 行中有效数据占比极低。

    3. 延迟瀑布效应：若链表较长，指针追踪形成严格的串行依赖链，完全无法利用内存级并行（MLP）。

    例外情况（需文档记录）：

    • 低频控制面操作（配置、管理）。
    • 数据结构规模极小（≤ 4 个节点），完全在 Cache 中。
    • 业务语义确实需要 O(1) 插入/删除且无法用其他结构替代。
    APDL-2：使用引用数组（批次）代替链表

    要求： 使用指向对象的指针数组或索引数组代替链表，除非能证明这对性能有害。

    详细说明：

    指针/索引数组相比链表的优势：

    • 数组本身可顺序访问，Cache 友好。
    • 数组索引（操作批次）天然与批处理模型（BATH-1）契合。
    • 可通过 SIMD 并行处理多个引用。
    APDL-3：考虑使用整数索引代替指针，当索引尺寸小于指针尺寸时

    建议： 当对象存储在固定大小的池（pool）中，且池的容量使得索引可以用比指针更小的整数类型表示时，优先使用整数索引而非指针。

    详细说明：

    在 64 位系统中，指针占 8 字节。若对象池不超过 65536 个条目，用 uint16_t 索引（2 字节）即可，相比指针节省 75% 的存储空间，直接体现为：

    • 更多引用能装入同一 Cache 行（同样 64 字节，可存 32 个 16-bit 索引 vs 8 个指针）。
    • 降低 L1D/L2 Cache 压力，减少缓存缺失。
    • 对大批次的随机访问，显著减小工作集大小。

    注意：索引间接访问会引入一次额外的加法运算，这通常远小于 Cache Miss 的代价。

    APDL-4：优先采用 SoA（Structure of Arrays）布局，特别是针对批处理步骤的上下文数据

    建议： 对批处理步骤中的上下文数据，优先考虑使用 SoA（结构体数组的转置形式——数组的结构体）而非传统 AoS（Array of Structures）。

    详细说明：

    SoA 的优势：

    • SIMD 向量化友好：同类型字段连续存储，可直接加载 SIMD 寄存器进行并行计算。
    • Cache 行利用率高：若某个步骤只需访问 src_ip 字段，AoS 会将无关字段也加载进来浪费 Cache 带宽；SoA 只加载所需字段。
    • 硬件预取效率高：字段数组顺序访问，预取器可高效工作。

    适用场景：当批处理步骤中每次迭代只访问结构体的部分字段时，SoA 收益最大。

    APDL-5：性能评估必须在仿真环境中定义安全的 L2 Cache 大小，以满足系统包处理速率的性能需求

    要求： 必须通过仿真/性能评估，确定满足系统 KPI 所需的最小 L2 Cache 容量（”安全 L2 大小”），并以此作为硬件选型和软件设计的约束条件。

    详细说明：

    数据平面的工作集（working set）大小直接决定了 L2 Cache 的命中率。如果工作集超出 L2 容量，大量访问将穿透到 L3 或主存，延迟大幅增加。

    工作集的主要构成：

    • 每包元数据（mbuf header）× 批次大小
    • 当前处理阶段需要访问的查找表（路由表、流表、策略表等）
    • 环形队列（ring buffers）的控制结构与数据缓冲区

    评估方法：

    1. 统计各 BPS 访问的数据结构尺寸之和。
    2. 在目标平台上，用 perf stat -e LLC-load-misses 等工具测量 L2/L3 命中率。
    3. 调整数据结构布局（压缩、SoA 转换、索引替换等），观察 L2 命中率变化。
    4. 确定满足吞吐量 KPI 时所需的最低 L2 Cache 配置。
    APDL-6：L2 大小定义应评估数据表示方式及可能的优化对 L2 大小的影响

    建议： 在确定 L2 大小要求时，应同步评估不同数据表示方式（如字段压缩、索引vs指针、SoA vs AoS）对工作集大小的影响，并将优化潜力纳入 L2 容量需求的计算。

    详细说明：

    同一业务逻辑，不同的数据表示可能使工作集相差数倍。例如：

    • IPv4 路由表用 uint32_t 存储地址 vs 用字符串表示：工作集差距 4x～8x。
    • 流表条目用 128 字节 vs 精心压缩后 48 字节：工作集差距 2.7x，直接影响 L2 容量需求。

    在硬件选型阶段，若软件优化可以将工作集压缩至能装入更小的 L2 Cache，则可以降低 BOM 成本；反之，若工作集不可压缩，则需要选择 L2 容量更大的处理器。

    APDL-7：在优化 L1D 缺失时，应将数据表示效率和可能的压缩优化作为减少缺失的手段之一

    建议： 当优化 L1D 缺存缺失时，除预取外，也应考虑通过改善数据表示效率（压缩字段、减小结构体大小）来降低 Cache 压力。

    详细说明：

    减少 Cache Miss 有两类正交的手段：

    1. 时间换空间：通过预取，在数据被需要之前就将其放入缓存。
    2. 减小工作集：通过数据压缩，让更多有效数据能装入同样大小的 Cache。

    对于 L1D Cache（通常 32KB～48KB），工作集压缩尤为重要：

    • 将结构体中的 uint64_t 字段尽可能替换为 uint32_t 或 uint16_t。
    • 合并布尔标志位到位域（bit field）。
    • 按字段访问频率排列结构体成员（热点字段在前，冷数据在后）。
    • 拆分热/冷结构体（Hot-Cold Splitting）：将频繁访问的字段和不常访问的字段分开存储。
    APDL-8：软件数据布局必须防止伪共享（False Sharing）

    要求： 数据结构布局必须确保不同线程（核心）各自修改的数据不共享同一 Cache 行。

    详细说明：

    伪共享是多核性能杀手。当两个 CPU 核心分别修改不同的变量，但这两个变量恰好位于同一 Cache 行（通常 64 字节）时，Cache 一致性协议（如 MESI）会强制该 Cache 行在核间来回传递，产生”乒乓效应”，即使两个核心从未真正共享数据。

    检测工具：

    • Linux perf c2c：专门用于检测 Cache-to-Cache 伪共享热点。
    • Intel VTune “Memory Access” 分析。
    APDL-9：多核共享数据必须在代码和设计文档中明确标识

    要求： 所有在多个核心之间共享的数据，必须在代码（注释/类型/命名约定）和设计文档中明确标注，包括其访问模式（只读/读写）和同步方式。

    详细说明：

    共享数据的隐式化是多核系统中 Bug 和性能问题的主要来源。明确标识的目的是：

    • 安全性：确保每处共享数据都有明确的同步机制，防止数据竞争。
    • 性能审查：使代码审查者和架构师能够快速识别潜在的 Cache 争用热点。
    • 可维护性：防止后续开发者无意间增加共享数据的访问频率。

    2. 最小化上下文切换

    上下文切换（包括内核态/用户态切换、线程调度切换、中断处理切换）每次需要保存/恢复数百个寄存器，同时导致 TLB 刷新和 I-Cache/D-Cache 污染，对数据平面吞吐量和延迟影响极大。

    2.1 内核上下文切换

    KCS-1：若应用运行于用户态，系统必须支持内核旁路（Kernel Bypass）

    要求： 用户态数据平面实现必须采用内核旁路技术，使数据包在软件应用与硬件之间的交换不经过内核。

    详细说明：

    传统 Linux 网络栈中，每个数据包的收发需要经历：
    硬件 → 内核驱动 → 内核网络栈 → 系统调用（copy_to_user） → 用户态应用

    每次系统调用（send/recv）会引发用户态↔内核态的模式切换，代价约为 1000～5000 个 CPU 周期。对于 10Gbps 以上的数据平面，这一开销是无法接受的。

    内核旁路技术方案：

    技术 代表实现 原理
    DPDK Intel DPDK UIO/VFIO 驱动，PMD 直接轮询 NIC 寄存器
    XDP/eBPF Linux 内核原生 在驱动层或 TC 层挂载 eBPF 程序，绕过完整网络栈
    RDMA Mellanox/InfiniBand 用户态直接操作 NIC 队列，零拷贝
    AF_XDP Linux 5.x 基于 XDP 的零拷贝用户态套接字
    VPP fd.io VPP 基于 DPDK，向量化包处理框架

    内核旁路的核心优势：

    • 零系统调用：数据路径完全在用户态完成。
    • 零拷贝：数据包 DMA 直接写入用户态内存（Hugepage），无需 kernel→user 拷贝。
    • Poll-mode：用轮询（busy-polling）代替中断，彻底消除中断上下文切换（需配合 EVT-1）。

    MMU-1：若应用运行于有 MMU 保护的用户态，必须控制并最小化 MMU 开销，通常使用大页（Huge Pages）

    要求： 用户态数据平面必须使用大页（Huge Pages）来减少 TLB 缺失，控制 MMU 相关开销。

    详细说明：

    TLB（Translation Lookaside Buffer）是虚拟地址→物理地址转换的缓存。TLB 容量有限（通常 L1 TLB 32～64 个条目），若数据平面的工作集跨越大量 4KB 小页，会导致大量 TLB Miss，每次 TLB Miss 需要多次内存访问完成页表遍历（Page Table Walk），延迟约 100～300 周期。

    大页的收益：

    页大小 单个 TLB 条目覆盖范围 覆盖 1GB 需要的 TLB 条目数
    4KB（标准页） 4 KB 262,144 个
    2MB（大页） 2 MB 512 个
    1GB（巨页） 1 GB 1 个

    使用 2MB 大页，TLB 覆盖同等内存所需的条目数减少 512 倍，TLB Miss 率大幅降低。

    2.2 线程上下文切换——协作式调度

    COOP-1：数据平面必须考虑协作式实现——每个硬件线程绑定一个 OS 线程，并在每个 OS 线程内实现显式协作式调度

    要求： 数据平面应采用一个 OS 线程绑定（pin）到一个 CPU 核心的模型，并在线程内部通过显式协作式调度管理不同任务，任何偏离此设计的方案必须有充分理由。

    详细说明：

    为什么避免抢占式多线程？

    操作系统抢占式调度器会在任意时刻打断当前线程，保存并恢复完整的 CPU 状态（包括 SIMD 寄存器），这带来：

    • 不可预测的调度延迟（jitter）。
    • Cache 和 TLB 污染（被调度进来的其他任务会替换数据平面的 Cache 内容）。
    • 核间负载均衡的额外开销。

    协作式调度模型（Run-to-Completion + 轮询）：

    关键设计要点：

    • CPU 亲和性（CPU Affinity）：用 pthread_setaffinity_np 或 DPDK 的 rte_eal_remote_launch 将线程绑定到特定核，并在 Linux 启动参数中用 isolcpus= 将该核从调度器中隔离。
    • 禁用内核抢占：对绑定的核使用 nohz_full= 参数禁用时钟中断（tickless mode）。
    • 轮询代替中断：用 busy-polling 代替 IRQ 驱动的数据包收发（配合 EVT-1）。

    COOP-2：每个 OS 线程的协作式调度器设计必须管理事件优先级并控制最坏情况延迟

    要求： 协作式调度器必须支持事件优先级，并为高优先级事件提供可预期的最坏情况响应延迟保证。

    详细说明：

    在协作式调度模型中，若所有 BPS 以相同优先级轮询，低优先级的大批量任务会阻塞高优先级事件（如 控制面消息、定时器事件）的响应，导致延迟抖动。

    优先级感知调度器设计示例：

    最坏情况延迟（WCET，Worst Case Execution Time）分析：

    • 对每个 BPS 进行最坏情况执行时间分析。
    • 确保高优先级事件的响应延迟 = 低优先级 BPS 的最大单次执行时间（有界）。

    COOP-3：协作式 OS 线程内的任何内部缓冲区，设计必须管理拥塞，或通过约束调度器行为确保拥塞不可能发生

    要求： 协作式线程内所有内部队列/缓冲区必须具有拥塞保护——要么实现拥塞控制/AQM，要么通过调度约束使队列溢出在设计上不可能发生。

    详细说明：

    在协作式调度器中，若某个 BPS 的输出速率持续超过后续 BPS 的处理速率，内部缓冲区将溢出。若不加处理，这会导致：

    • 内存耗尽（无界队列）。
    • 包乱序或数据损坏（有界队列无保护）。
    • 性能雪崩（背压未传递，上游持续发包）。

    两种解决策略：

    1. 设计约束（无拥塞）：通过流量整形和流水线瓶颈分析，确保系统的速率瓶颈在流水线头部（MC-2），下游 BPS 的处理能力始终≥上游输出速率。
    2. 主动拥塞管理（AQM）：在内部缓冲区上部署 AQM 算法（如 CoDel、RED），在队列积压时主动丢包，防止延迟无限增长（见第 4 章拥塞控制）。

    COOP-4：为满足极低延迟保证，可以接受每个硬件线程使用多个 OS 线程，以隔离和保护高优先级流量

    建议： 若极低延迟要求无法通过单线程协作式调度满足，可以考虑每个核使用两个 OS 线程——一个专用于高优先级处理，另一个处理普通数据流量。

    详细说明：

    URLLC（Ultra-Reliable Low-Latency Communication）要求端到端延迟 <1ms，甚至 <0.5ms，这在单线程协作式调度器中极难保证（受最长 BPS 执行时间约束）。

    方案：使用 Linux 实时优先级线程（SCHED_FIFO）处理 URLLC 流量，普通数据平面线程使用较低优先级。URLLC 线程可以抢占普通线程，但两者之间的切换次数有限，Cache 污染可控。

    此方案的代价是引入少量线程上下文切换开销，需要在延迟保证和吞吐效率之间权衡。

    2.3 事件开销——IRQ 上下文切换

    EVT-1：所有高吞吐量接口必须支持中断合并（Interrupt Mitigation）

    要求： 高吞吐量网络/硬件接口必须支持中断合并（Interrupt Coalescing / Mitigation），以减少 IRQ 引起的上下文切换次数。

    详细说明：

    在高吞吐量场景下，若每个数据包都触发一次 IRQ，100Gbps 的流量可能产生每秒数千万次中断。每次 IRQ：

    • 打断当前执行流，保存上下文（约 100～500 周期）。
    • 污染 I-Cache 和 D-Cache（中断处理程序引入新代码和数据）。
    • 恢复上下文后，CPU 可能需要重新预热 Cache（Cache warmup 惩罚）。

    中断合并策略：

    1. IRQ Coalescing（硬件层面）：配置 NIC 在收到 N 个包或等待 T 微秒后才触发一次 IRQ，减少中断频率。

    2. Poll Mode Driver（PMD）：完全用轮询替代中断（DPDK 的默认模式），彻底消除 IRQ 上下文切换，代价是一个 CPU 核心 100% 用于轮询（适合专用数据平面核心）。

    3. NAPI（Linux 网络层）：中断触发首次包接收通知，随后切换为轮询模式批量收包，合并完成后再恢复中断模式。

    对于数据平面专用核心，推荐 PMD 完全轮询模式；对于通用核心，推荐 NAPI + 激进的中断合并参数。

    3. 多核组织

    现代数据平面运行在多核处理器上，如何组织多核之间的工作分配，直接影响系统整体吞吐量、延迟和扩展性。

    MC-1：多核用户面实现必须考虑流水线（Pipeline）组织，跨核分配处理阶段，任何偏离必须有充分理由

    要求： 多核数据平面应优先采用流水线模型，将处理步骤分布到不同核心，任何偏离（如 run-to-completion 模式）必须有充分的性能论证。

    详细说明：

    多核数据平面的两种主要组织模型：

    1. Run-to-Completion（RTC）模型：

    优点：无核间通信，Cache 局部性好；缺点：扩展性受限于单核处理能力，难以处理复杂多阶段处理。

    2. Pipeline 模型：

    优点：每个核专注于特定处理阶段，I-Cache 局部性极好；可利用不同阶段的处理速率差异做负载均衡；缺点：核间队列引入延迟，需要 Cache 一致性协议传递数据。

    选择 Pipeline 的场景：

    • 处理栈深度大（多个复杂处理阶段）。
    • 各阶段处理时间差异显著，需要独立扩展。
    • 总处理量超过单核能力。

    选择 RTC 的场景：

    • 处理栈简单，单核可满足吞吐量需求。
    • 对延迟极其敏感（pipeline 引入排队延迟）。

    MC-2：设计必须使核间队列拥塞不可能发生（瓶颈在流水线头部），或对可能的瓶颈队列实现拥塞控制算法

    要求： 多核流水线中的核间队列必须有拥塞保护，策略为：将系统瓶颈设计在流水线入口（丢包在最前端，减少无效处理），或在瓶颈队列上部署拥塞控制。

    详细说明：

    核间队列无界增长会导致：

    • 内存耗尽。
    • 延迟无限积压（队列时延 = 队列深度 / 出队速率）。
    • 系统崩溃或性能雪崩。

    最优策略——头部丢包（Head Drop / Tail Drop at Ingress）：

    将系统的接收能力（入队速率）限制在整个流水线最慢阶段的处理能力以内，超出部分在 RX 阶段即丢弃。这样：

    • 后续流水线阶段永远不会过载。
    • 无效处理（最终被丢弃的包）最小化，节省 CPU 和 Cache 资源。
    • 延迟保持稳定（队列始终不积压）。

    MC-3：软件应具备跟踪每个批处理步骤处理成本的仪表盘，以支持核间负载均衡

    建议： 为每个 BPS 添加处理时间/成本测量，生成 per-BPS 性能剖析数据，用于指导流水线阶段在多核间的动态负载均衡。

    详细说明：

    流水线中的负载均衡依赖于精确的 per-stage 成本测量。实现方式：

    通过持续监控各 BPS 的 cycles/packet，可以：

    • 识别流水线瓶颈（成本最高的阶段）。
    • 动态调整批次大小或流水线阶段分配。
    • 在多核扩展时，将高成本阶段分配给更多核心。

    4. 拥塞控制

    拥塞控制是数据平面健壮性的核心。设计不良的拥塞处理会导致系统在高负载下崩溃、延迟爆炸或无法恢复。

    CC-1：数据平面设计必须识别所有资源拥塞可能性，并为每种情况提供关联的拥塞管理方案

    要求： 在设计阶段，必须系统性地枚举所有可能发生拥塞的资源（CPU、内存、队列、DMA 描述符等），并为每个潜在拥塞点定义对应的管理策略。

    详细说明：

    数据平面中常见的拥塞点：

    拥塞点 类型 推荐管理方案
    RX 队列满（NIC） 硬件层 IRQ Coalescing + PMD 轮询频率调整
    核间传递队列满 软件层 AQM（CoDel/RED）或背压
    Mempool 耗尽 内存层 限制最大并发包数；头部丢包
    CPU 过载 计算层 背压传递至头部；AQM 丢包
    TX 队列满（NIC） 硬件层 软件 TX 队列排队；背压上游

    每个拥塞点必须在设计文档中明确记录，并说明其检测方法和管理策略。

    CC-2：数据平面必须支持内部拥塞缓解 AQM（主动队列管理）作为默认方案，可选支持背压

    要求： 默认采用 AQM（Active Queue Management）算法管理内部队列拥塞，可选支持背压（Back Pressure）机制。

    详细说明：

    AQM vs 尾丢包（Tail Drop）：

    传统的尾丢包（队列满才丢包）导致”全局同步”现象——大量 TCP 流同时触发拥塞控制，导致吞吐量剧烈波动（锯齿效应）。

    AQM 的核心思想是主动、提前地丢包（在队列开始积压时），而非等到队列满才丢包：

    • RED（Random Early Detection）：根据平均队列长度，按概率丢包，平滑流量。
    • CoDel（Controlled Delay）：基于排队延迟（而非队列长度）触发丢包，更好地控制时延。
    • FQ-CoDel：结合公平队列和 CoDel，对多流场景效果更佳。

    背压（Back Pressure） 适用于有状态连接且对丢包容忍度低的场景，通过向上游发送”慢速”信号让其降低发送速率，而非直接丢包。

    CC-3：数据平面必须对输入饱和和 CPU 饱和具有鲁棒性，并在饱和结束后能优雅恢复

    要求： 系统在遭受突发流量冲击或 CPU 过载时，不得崩溃、死锁或进入不可恢复状态，且在压力消除后能自动恢复到正常工作状态。

    详细说明：

    鲁棒性设计原则：

    • 有界队列：所有内部队列设置最大深度上限，防止内存耗尽。
    • Mempool 保护：当 Mempool 可用缓冲区低于阈值时，在 RX 阶段开始丢弃新入包。
    • 无状态化恢复：数据平面的每个 BPS 应尽量无状态，或状态可安全丢弃，使得丢包后无需复杂的状态清理。
    • 饱和检测：通过队列深度、处理延迟或 Mempool 使用率检测饱和状态，并记录告警。
    • 自动恢复：当压力消除（队列深度降低、CPU 利用率下降），AQM 丢包概率自动降低，系统恢复正常吞吐量。

    CC-4：在软件饱和时，检测到饱和的 BPS 应使用背压通知流水线头部丢弃数据包，直到饱和结束

    建议： 当流水线末端（如 TX 前的排队 BPS）检测到队列积压超过阈值时，应向流水线头部（RX 入口）发送背压信号，使其在接收端直接丢包，减少无效的端到端处理。

    详细说明：

    背压传递的设计价值：将丢包点从流水线末端移至最前端，从而：

    • 节省被丢包花费在流水线中间各 BPS 的 CPU 时间。
    • 减少 Cache 污染（不处理最终会被丢弃的包）。
    • 降低内部队列压力。

    实现示意：

    背压信号可通过共享原子变量、Per-CPU 标志或显式消息队列传递，需注意信号传递本身的开销（见第 5、6 章关于共享数据的讨论）。

    CC-5：AQM 可支持 ECN 作为丢包的替代方案，当丢包概率超过可配置阈值（默认 10%）时，应回退到丢包模式

    建议： AQM 可选支持 ECN（Explicit Congestion Notification），在队列积压早期标记包而非丢包；当拥塞加剧（丢包概率 > 10%）时，自动切换回丢包模式。

    详细说明：

    ECN 是对丢包式拥塞信号的改进：通过在 IP 头和 TCP 头设置 ECN 标志位，向发送端传递拥塞信号，让发送端主动降速，而无需真正丢包。

    优势：

    • 降低重传次数，提高吞吐量效率。
    • 减少因丢包触发 TCP 慢启动导致的吞吐量剧烈波动。

    限制：

    • 仅对支持 ECN 的端点（TCP 连接两端均需支持）有效。
    • 当拥塞极度严重时，ECN 标记可能无法及时通知发送端（RTT 大于拥塞持续时间），此时必须回退到丢包。
    • 默认阈值 10% 是经验值，可根据业务场景配置调整。

    使用条件：必须检查 IP 包头中的 ECN 能力标志（ECT bit），仅对声明支持 ECN 的数据流进行标记。

    CC-6：设计必须基于峰值流量吞吐量、流量类型和 RTT 规划缓冲容量

    要求： 系统缓冲区容量（Buffer Sizing）必须基于以下参数系统性计算：峰值流量速率、流量特性（突发度、包大小分布）和端到端 RTT。

    详细说明：

    经典带宽-时延积（BDP）缓冲规则：

    但现代多流场景下，BDP 规则往往过于保守，实际所需缓冲更小（因为多流之间的统计复用效应）。更精确的规划需要考虑：

    • 峰值突发（Burst）大小：突发流量需要足够的缓冲来平滑，避免瞬时丢包。
    • 流量类型：实时流（视频/语音）对延迟敏感，缓冲应小（低延迟优先）；弹性流（文件传输）对吞吐敏感，可使用更大缓冲。
    • RTT：RTT 越大，需要越大的缓冲来保持链路利用率（飞行中的数据量 = BDP）。
    • AQM 目标延迟：结合 CoDel 等 AQM 的目标队列时延（通常 5ms），反推最大队列深度。

    5. 最小化线程间共享

    SHARE-1：设计必须识别共享数据，评估其访问和同步成本，并将此成本控制在可接受水平

    要求： 必须系统性地识别所有跨线程共享的数据，量化其同步开销，并确保该开销不影响系统 KPI。

    详细说明：

    线程间数据共享的成本来源：

    1. 锁竞争（Lock Contention）：互斥锁在竞争下退化为串行执行，彻底消除多核并行收益。
    2. Cache 一致性协议开销：写操作会使其他核心的 Cache 副本失效（MESI Invalid），强制其重新从 L3 或主存加载。
    3. 原子操作开销：CAS、fetch_add 等原子操作需要独占 Cache 行，在高竞争下延迟可达普通访问的 100 倍。
    4. 内存屏障（Memory Barrier）开销：阻止编译器和 CPU 的乱序优化，降低 IPC。

    最小化共享的设计策略：

    • Per-Core 数据结构：每个核心维护独立的统计计数器、缓冲区、状态机；汇总时异步聚合。
    • 分区（Partitioning）：将数据按核心 ID 或流 ID 哈希分区，每个核心只操作自己的分区。
    • 消息传递代替共享内存：核间通过无锁队列（如 DPDK rte_ring）传递消息，而非直接读写共享变量。
    • 读多写少的数据使用 RCU（见第 6 章）。

    量化评估工具：

    • perf stat -e cache-misses,cache-references 测量 Cache 争用。
    • perf c2c 检测 Cache-to-Cache 伪共享和真共享热点。
    • lock_stat（Linux 内核锁统计）分析锁竞争。

    6. 热点共享数据

    某些共享数据无法避免（如全局路由表、全局配置），需要特殊的访问模式来降低其开销。

    HSD-1：大型频繁访问的读多写少数据，必须使用无锁方式访问和更新，通常采用 RCU 方案

    要求： 大型频繁读取、偶尔更新的共享数据（如路由表、策略表），必须使用 RCU（Read-Copy-Update）或等效的无锁读机制。

    详细说明：

    RCU 的核心思想：读操作完全无锁，写操作通过”拷贝-修改-替换”保证一致性。

    RCU 使读操作的开销接近于裸内存访问，适合：

    • 路由/转发表（读多，路由变更少）。
    • ACL/策略规则表（运行时偶尔更新）。
    • 全局配置参数。

    Linux 内核和 DPDK 均有成熟的 RCU 实现可直接使用。

    HSD-2：热点共享数据，若不可避免，每批最多访问一次

    要求： 不可避免的热点共享数据，在一个批次（batch）的处理过程中，最多读取一次，并将结果缓存在本地变量中供整批包使用。

    详细说明：

    本地缓存的数据存放在寄存器或栈上，完全无 Cache Miss 和同步开销。

    HSD-3：热点共享数据若不可避免，应尽量放入一个机器字，并用 C11 原子操作更新，以最小化同步开销

    建议： 热点共享数据应尽量压缩到 ≤8 字节（一个机器字），使用 C11 原子操作（atomic_load/atomic_store/atomic_compare_exchange）进行访问，避免使用互斥锁。

    详细说明：

    memory_order_relaxed 适用于统计计数器等无需严格顺序保证的场景；对需要可见性保证的数据使用 memory_order_acquire/release。

    7. 填满流水线

    CPU 流水线利用率（IPC，Instructions Per Cycle）是衡量代码执行效率的关键指标。现代超标量 CPU 每周期可执行 3～6 条指令，但数据相关性、控制相关性和内存延迟会导致流水线停顿，使实际 IPC 远低于理论值。

    FP-1：性能测量环境必须测量每个批处理步骤的有效 IPC 和相对流水线停顿量

    要求： 性能测量基础设施必须能够测量并上报每个 BPS 的有效 IPC 和各类流水线停顿（内存停顿、分支停顿等）的占比。

    详细说明：

    关键 PMU 事件：

    解读指标：

    • IPC = instructions / cycles ：理论上限约 4（4-wide 超标量），实际 >2 通常较好，<1 说明严重停顿。
    • stalls_mem_any / cycles ：内存停顿占比，>30% 说明需要预取或数据布局优化。
    • branch-misses / instructions ：分支预测失效率，>1% 说明存在难以预测的条件分支。

    IPC 优化目标：通过消除内存停顿（预取/数据布局）、减少分支（循环展开/条件移动指令）和提高指令级并行（ILP），将有效 IPC 提升至目标平台能力的 60%以上。

    7.1 使用 SMT

    SMT（Simultaneous Multi-Threading，超线程）允许单个物理核心同时执行两个逻辑线程，通过交替执行来隐藏内存延迟。

    适用性分析（针对蜂窝数据平面）：

    SMT 在以下条件下收益显著：

    • 存在大量独立的内存访问（无数据依赖），可让两个线程互相填充对方的等待气泡。
    • 工作集适合 L3 Cache（避免两个线程竞争导致 Cache 抖动）。
    • 处理大量独立流（如 SDN 场景）。

    然而，在蜂窝数据平面中：

    • 包处理存在严格的顺序依赖（解析→分类→处理→发送），独立性较低。
    • 两个 SMT 线程共享 L1/L2 Cache，会相互驱逐对方的数据，加剧 Cache Miss。
    • SMT 引入的资源竞争（重命名寄存器、执行端口分配）会降低单线程的确定性。

    结论： 对于蜂窝数据平面，SMT 的复杂性和潜在收益不匹配，通常不推荐使用。循环展开是更适合的 IPC 提升手段。

    7.2 使用循环展开

    循环展开（Loop Unrolling）是提升数据平面 IPC 的主要手段，其核心是通过手动或编译器将循环体展开多倍，增加循环体内的独立指令数量，从而提高指令级并行度（ILP）。

    LU-1：为准备循环展开，BPS 中使用的所有辅助函数必须是 static inline 函数（或宏）

    要求： BPS 批量循环中调用的所有辅助函数必须声明为 static inline（或等效宏），确保编译器能将其内联展开，为后续的循环展开和指令调度提供足够的代码可见性。

    详细说明：

    循环展开和编译器自动向量化的前提是编译器能看到循环体内的完整指令序列。若辅助函数是普通外部函数，编译器无法跨函数边界进行优化（LTO 链接时优化可部分解决，但不如 inline 可靠）。

    static 属性还确保该函数不产生独立的符号，避免链接时的符号冲突和代码重复。

    LU-2：展开 BPS 批量循环时，生成的代码大小仍必须能装入 L1 指令缓存，否则必须将 BPS 拆分为更小的处理步骤

    要求： 循环展开后，BPS 的总代码尺寸（包括展开后的多个循环体副本）必须仍然满足 BATH-2 的 L1 I-Cache 约束；若超出，必须将 BPS 拆分。

    详细说明：

    循环展开会成倍增加代码尺寸。展开 4 倍意味着代码量变为原来的 4 倍（减去循环控制开销）。必须在 IPC 提升和 I-Cache 压力之间取得平衡。

    展开后 IPC 提升的原因：

    • 4 个独立的 process() 调用之间无数据依赖，CPU 可以同时在多个执行端口调度这些指令。
    • 即使某条指令因内存延迟停顿，其他独立指令仍可继续执行，填充停顿气泡。

    若展开后代码超出 L1 I-Cache（通常以 objdump -d | grep size 或编译器输出检验），优先考虑将 BPS 拆分为两个串联步骤，每个步骤单独展开并独立满足 I-Cache 约束。

    8. 总结

    高性能数据平面软件的设计是一个系统工程，需要在多个层面协同优化。以下是本文所有设计准则的核心要点汇总：

    优化优先级建议

    在资源有限的情况下，建议按以下优先级顺序实施优化：

    1. 架构层面（最高优先级）：批处理化（BATCH-1/2）、Kernel Bypass（KCS-1）、协作式调度（COOP-1/2）——这些是基础架构决策，事后改造代价极高。
    2. 数据布局层面：消除链表（APDL-1/2）、防止 False Sharing（APDL-8）、大页（MMU-1）——相对低成本，收益显著。
    3. 拥塞控制层面：识别拥塞点（CC-1）、部署 AQM（CC-2）——影响系统稳定性，需要在系统测试阶段完成。
    4. 微优化层面：预取（PF-2）、SoA 布局（APDL-4）、循环展开（LU-1/2）、索引替换指针（APDL-3）——在性能测量（FP-1、PF-1）指导下，有针对性地实施。

    3. 3.0.3. 2.3 事件开销——IRQ 上下文切换
      1. 3.0.3.1. EVT-1：所有高吞吐量接口必须支持中断合并（Interrupt Mitigation）
  • 4. 3. 多核组织
    1. 4.0.0.1. MC-1：多核用户面实现必须考虑流水线（Pipeline）组织，跨核分配处理阶段，任何偏离必须有充分理由
    2. 4.0.0.2. MC-2：设计必须使核间队列拥塞不可能发生（瓶颈在流水线头部），或对可能的瓶颈队列实现拥塞控制算法
    3. 4.0.0.3. MC-3：软件应具备跟踪每个批处理步骤处理成本的仪表盘，以支持核间负载均衡
  • 5. 4. 拥塞控制
    1. 5.0.0.1. CC-1：数据平面设计必须识别所有资源拥塞可能性，并为每种情况提供关联的拥塞管理方案
    2. 5.0.0.2. CC-2：数据平面必须支持内部拥塞缓解 AQM（主动队列管理）作为默认方案，可选支持背压
    3. 5.0.0.3. CC-3：数据平面必须对输入饱和和 CPU 饱和具有鲁棒性，并在饱和结束后能优雅恢复
    4. 5.0.0.4. CC-4：在软件饱和时，检测到饱和的 BPS 应使用背压通知流水线头部丢弃数据包，直到饱和结束
    5. 5.0.0.5. CC-5：AQM 可支持 ECN 作为丢包的替代方案，当丢包概率超过可配置阈值（默认 10%）时，应回退到丢包模式
    6. 5.0.0.6. CC-6：设计必须基于峰值流量吞吐量、流量类型和 RTT 规划缓冲容量
  • 6. 5. 最小化线程间共享
    1. 6.0.0.1. SHARE-1：设计必须识别共享数据，评估其访问和同步成本，并将此成本控制在可接受水平

    2. 7.0.0.2. HSD-2：热点共享数据，若不可避免，每批最多访问一次
    3. 7.0.0.3. HSD-3：热点共享数据若不可避免，应尽量放入一个机器字，并用 C11 原子操作更新，以最小化同步开销
