    无锁并发编程：CAS 操作与 RCU 模式解析

    Alex Wu
    2019-06-02
    内存模型

    本文介绍原子操作的局限性、Compare-and-Swap（CAS）的核心原理，以及如何在 RCU 模式和无锁链表中实际应用这些技术。

    1. 问题背景：并发写大型数据结构的挑战

    现代 CPU 提供的原子操作（如 load、store、CAS 等）通常只保证对单个内存字（word） 的操作是原子的。在 64 位系统上，这个”字”的大小是 8 字节。

    这意味着：

    • 你无法直接原子地读写一个包含多个字段的大型结构体（例如几百字节的配置对象）。
    • 如果强行对大对象做非原子操作，在多线程环境下会出现数据撕裂（data tearing） 或状态不一致。

    核心解题思路：既然不能原子地修改整个大对象，就只原子地修改指向它的指针（或索引）。所有线程通过这个共享指针访问当前数据，而指针本身只有 8 字节，可以被原子地读取和更新。

    这一思路衍生出两类经典模式：

    • RCU（Read-Copy-Update）：适用于”一个全局指针指向大型配置/数据结构”的场景，以极低代价支持大量并发读者。
    • 无锁链表：适用于需要原子地修改数据结构拓扑（插入/删除节点）的场景，如对象池的空闲链表。

    两者都依赖同一个底层原语：Compare-and-Exchange（CAS）。

    2. 理论基础：Compare-and-Exchange（CAS）

    CAS（Compare-and-Swap / Compare-and-Exchange）是无锁编程的基础原语，其语义如下：

    “如果内存地址中的当前值等于预期值（expected），则将其原子地更新为新值（desired），并返回 true；否则不做任何修改，返回 false，并将 expected 更新为当前实际值。”

    CAS 的核心价值在于：它将”读-判断-写”三步合并为一个不可分割的原子操作，消除了并发窗口中的竞态条件。

    典型用法是配合重试循环（retry loop）：

    如果 CAS 失败（说明另一个线程抢先修改了 ptr），则重新读取最新值并重试，直到成功为止。这种”乐观并发”策略在低竞争场景下性能极高，在高竞争场景下会产生重试开销。

    3. 应用一：RCU 模式（读-拷贝-更新）

    RCU（Read-Copy-Update）是一种高性能无锁同步机制，由 Paul E. McKenney 引入 Linux 内核，专为读多写少的并发场景设计，是内核中处理共享链表、路由表、设备驱动配置等数据结构的关键技术。

    3.1 核心思想与工作流程

    RCU 的工作流程分为三步：

    1. Read（读）：读者无锁、无原子操作地直接通过全局指针访问共享数据，开销接近于零。
    2. Copy（拷贝）：写者不直接修改原数据，而是在另一个内存位置创建一份完整副本。
    3. Update（更新）：在副本上完成修改后，原子地切换全局指针指向新数据，令后续读者看到新版本。

    关键约束：旧数据不能在指针切换后立即释放。必须等待所有可能仍持有旧指针的读者完成访问，才能安全回收。这一等待过程称为宽限期（Grace Period）。

    3.3 宽限期机制：何时安全回收旧数据

    宽限期是 RCU 设计的核心难题：系统需要确认所有 CPU 上的活跃读者都已退出读临界区，才能安全释放旧数据。

    Linux 内核采用静默点（Quiescent State）来检测：当一个 CPU 发生以下事件时，可以确定它不再持有任何 RCU 读临界区：

    • 上下文切换（context switch）
    • 切换到用户态执行
    • 进入 CPU 空闲状态（idle）

    一旦所有 CPU 都至少经历了一次静默点，则宽限期结束，注册的回调函数（即内存释放）可以安全执行。

    在大型多核系统（1000+ CPU）中，Linux 采用分层树状结构（Tree RCU）来高效汇总各 CPU 的静默状态，避免在单一全局变量上产生同步瓶颈。

    3.4 Linux 内核的具体实现

    Linux 内核提供了完整的 RCU API：

    API 说明
    rcu_read_lock() 进入 RCU 读临界区。在不可抢占内核中可能仅为编译器屏障；在可抢占内核中会记录状态，阻止该 CPU 上报静默点。
    rcu_read_unlock() 退出 RCU 读临界区。
    rcu_dereference(ptr) 安全读取 RCU 保护的指针，插入数据依赖屏障，防止 CPU 将指针解引用乱序提前到指针加载之前。
    rcu_assign_pointer(ptr, new) 安全发布新指针，插入写内存屏障，确保新数据的所有初始化对读者可见后，指针才对外可见。
    synchronize_rcu() 同步等待宽限期结束（阻塞写者线程，直到所有已开始的读者退出临界区）。
    call_rcu(head, callback) 异步注册回调，宽限期结束后由 RCU 子系统自动调用（不阻塞调用者）。

    rcu_dereference() 与 rcu_assign_pointer() 必须配对使用，前者保证读端安全加载，后者保证写端安全发布，共同构成 RCU 的内存序保证。

    3.6 RCU 的优缺点

    优点：

    • 读者路径零锁、零原子写操作，读延迟极低，可线性扩展到任意核心数
    • 无死锁风险，读者永远不会被写者阻塞
    • 适合实时系统，读延迟低且可预测

    缺点：

    • 写者开销较大（需拷贝副本 + 延迟回收）
    • 新旧数据短暂共存，内存占用暂时增加
    • 仅适用于通过指针访问的动态数据结构（不适用于原地修改的场景）
    • 实现复杂，调试困难

    5. 已知陷阱：ABA 问题

    使用 CAS 操作无锁链表时，存在一个经典陷阱——ABA 问题。基础 CAS 只比较值是否相等，无法区分”值一直未变”和”值变过又变回来了”这两种情况。

    常见解决方案：

    方案 原理 特点
    带版本号的指针（Tagged Pointer） 将指针的部分位用作版本计数器，每次修改递增；CAS 同时比较地址和版本号 实现简单，但在 32 位平台上可用位有限
    Hazard Pointer（危险指针） 每个线程访问节点前先声明”持有”该指针；回收系统释放前扫描所有危险指针，跳过仍在使用的节点 通用，无 ABA 问题，实现较复杂
    Epoch-Based Reclamation（基于时代的回收） 将内存释放推迟到没有任何线程引用旧数据的时代（epoch）之后，与 RCU 宽限期思想相通 适合读多写少场景，与 RCU 思路一致

    在实践中，建议优先使用经过充分验证的无锁库（如 C++ 的 libcds、folly::AtomicLinkedList），而非自行实现以规避这类陷阱。

    6. 技术对比与选型建议

    特性 互斥锁（Mutex） CAS 自旋重试 RCU
    读性能 中（每次读都需获取锁） 高（单次原子读） 极高（无任何同步开销）
    写性能 中 中（低竞争时高，高竞争时因重试而下降） 低（拷贝副本 + 延迟回收）
    实现复杂度 低 中 高
    死锁风险 有 无 无
    内存开销 低 低 暂时较高（新旧版本共存）
    适用数据规模 任意 单指针/小结构 通过指针访问的动态结构
    典型场景 通用临界区保护 计数器、标志位、无锁栈/对象池 配置热更新、路由表、内核数据结构

    选型建议：

    • 写多读少 → 互斥锁，简单可靠
    • 低竞争的短临界区（计数器、状态位、对象池） → CAS 原子操作
    • 读远多于写（配置对象、路由表、订阅列表） → RCU
    • 对象频繁分配/回收 → 无锁对象池（CAS 链表）
    • 不确定时 → 先用互斥锁，通过 profiling 确认瓶颈后再引入无锁结构

    7. 总结

    本文围绕一个核心问题展开：如何在不使用互斥锁的情况下，安全地在多线程间共享和更新数据结构？

    关键结论如下：

    1. 原子操作只能覆盖自然字长：大型结构必须通过原子指针间接访问，而非试图原子地修改整个结构体。
    2. CAS 是无锁编程的基础原语：它将”读-判断-写”合并为不可分割的原子操作，通过重试循环应对竞争失败，保证”只有当我看到的仍是旧值时，我才提交修改”。
    3. RCU 将读写路径彻底分离：读者以接近零的开销访问数据；写者拷贝副本后用 CAS 原子切换指针；通过宽限期机制安全延迟释放旧数据，实现极低且可预测的读延迟。
    4. CAS 无锁链表支撑了无锁栈、无锁对象池等高性能数据结构，在低竞争场景下性能优于加锁方案。
    5. ABA 问题是无锁编程的固有陷阱：需根据场景选择版本号指针、Hazard Pointer 或 Epoch-Based Reclamation 加以应对。

    这些技术共同构成了现代高性能并发系统（Linux 内核、数据库引擎、网络服务器、游戏引擎）底层的基石。理解它们的原理，有助于在正确的场景做出正确的设计选择，而不是无差别地追求”无锁”。
