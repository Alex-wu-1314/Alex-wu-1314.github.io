    1. 背景：为什么需要内存模型

    现代计算机体系结构中，CPU 执行指令的速度远远大于访问内存的速度。为了弥补这一差距，硬件引入了多级 Cache、分支预测和指令预取等机制，极大提升了 CPU 的执行效率。然而，随着 SMP（对称多处理器）和多核处理器的普及，多线程编程模型被广泛使用，多个线程对共享变量的并发访问成为一个极其复杂的问题。

    问题的根源在于两点：

    1. 指令重排（Reorder）：编译器和 CPU 为了优化性能，会在保证单线程语义正确的前提下对指令进行重排。这在单线程中完全没有问题，但在多线程场景下可能破坏程序员的预期。

    2. 缓存可见性（Cache Visibility）：一个线程对内存的写操作，不一定立即对其他 CPU 核心（其他线程）可见，因为写入的数据可能还停留在本地 Cache 中。

    为了在语言层面解决这些问题，C11/C++11 标准引入了统一的内存模型和 <atomic> 库，让程序员能够以跨平台、跨编译器的方式精确控制多线程间的内存访问顺序。

    2. 什么是内存模型

    关于内存模型（Memory Model），可以从两个角度理解：

    • 角度 A（多线程视角）：内存模型描述了编程语言在支持多线程编程时，对共享内存访问顺序的规定。
    • 角度 B（硬件视角）：内存模型的本质是指在单线程情况下，CPU 指令在多大程度上允许发生重排（reorder）。

    两种说法都正确，它们只是从不同层面描述同一个概念。更准确的表达是：内存顺序模型。

    一个完整的内存模型定义包含三个方面：

    方面 说明
    Atomic Operations（原子操作） 操作要么完全执行，要么完全不执行，不存在中间状态
    Partial order of operations（操作的偏序关系） 规定哪些操作之间存在先后顺序约束
    Visible effects of operations（操作的可见性） 规定一个线程的写操作何时对其他线程可见

    内存模型与三个层面密切相关：CPU 体系结构、编译器实现和编程语言规范。

    3. CPU 体系结构与内存模型

    不同的 CPU 体系结构有着不同的内存顺序模型，大致分为两类：

    架构 内存模型 类型
    x86_64 Total Store Order (TSO) 强顺序模型
    Sparc Total Store Order (TSO) 强顺序模型
    ARMv8 Weakly Ordered 弱内存模型
    PowerPC Weakly Ordered 弱内存模型
    MIPS Weakly Ordered 弱内存模型

    强顺序模型（TSO）：内存上的写操作存在全局顺序，所有 CPU 核心看到的写操作顺序是一致的，且与程序顺序相关。这对程序员友好，但对芯片实现者不友好，会牺牲一定的并发执行效率。

    弱内存模型（WMO, Weak Memory Ordering）：CPU 不保证内存访问顺序（除非同一 CPU 上存在数据依赖），而是将顺序保证的责任交给程序员，程序员需要主动插入内存屏障指令（Memory Barrier） 来强化”可见性”。弱内存模型实现简单、执行效率高，但对程序员要求更高。

    正是因为不同架构的内存模型差异，C++11 在语言层面提供了统一的抽象，让开发者无需关心底层架构细节，即可编写正确的并发程序。

    4. 4 种关键关系术语

    C++11 内存模型使用以下 4 种关系术语来精确描述操作之间的顺序和可见性。

    4.1 sequenced-before

    定义：在同一个线程内，若语句 A 的执行顺序在语句 B 之前，则称 A sequenced-before B。

    它不仅表示执行顺序，还表示可见性：A sequenced-before B 意味着 A 的执行结果对 B 可见。

    sequenced-before 是单线程内最基本的顺序保证，是构建更复杂跨线程关系的基础。

    4.2 happens-before

    定义：happens-before 描述的是不同线程之间操作的先后顺序。若 A happens-before B，则 A 的内存状态在 B 执行之前就已对 B 可见。

    happens-before 关系具有以下性质：

    • 传递性：若 A happens-before B，B happens-before C，则 A happens-before C
    • 非自反性：A 不能 happens-before 自身
    • 非对称性：若 A happens-before B，则 B 不能 happens-before A

    happens-before 包含了两种子关系：

    • sequenced-before（同线程内）
    • synchronizes-with（跨线程）

    4.3 synchronizes-with

    定义：synchronizes-with 强调的是变量被修改后的跨线程传播关系。若线程 A 修改某变量的结果能被线程 B 可见，则两者之间的操作满足 synchronizes-with 关系。

    synchronizes-with 可以理解为跨线程版本的 happens-before。满足 synchronizes-with 的操作，一定满足 happens-before。

    4.4 Carries Dependency

    定义：在同一线程内，若表达式 A sequenced-before 表达式 B，且 B 的值依赖于 A 的结果，则称 A carries dependency 到 B。

    语句 C 的值直接依赖于 *a 和 *b，因此 A→C 和 B→C 之间存在 carries dependency 关系。

    carries dependency 与 memory_order_consume 密切相关：consume 语义仅对具有依赖关系的操作链提供顺序保证，而不像 acquire 那样对所有后续操作都提供保证，因此理论上开销更小。

    5.1 memory_order_relaxed

    规则：仅保证操作本身的原子性，不提供任何跨线程的顺序或可见性保证。编译器和 CPU 可以在该操作前后自由重排其他操作。

    适用场景：只需要保证原子性、不需要同步语义的场景，典型例子是计数器。

    5.2 memory_order_consume

    规则：针对有依赖关系的操作链提供同步保证。使用 consume 加载的值，其所有数据依赖链上的操作都在 consume 之后可见，但没有依赖关系的操作不受保证。

    consume 必须与 release 配合使用，构成 dependency-ordered-before 关系。

    📝 实践说明：由于 memory_order_consume 的语义非常难以在编译器中正确实现（几乎所有主流编译器都将其提升为 acquire 处理），C++17 已将其标记为”暂不鼓励使用”。在实际工程中，建议用 acquire 替代。

    5.3 memory_order_acquire

    规则：acquire 语义用于读操作（load）。在 acquire load 之后的所有读写操作，都不能被重排到 acquire 之前。与 release 写配合，建立 synchronizes-with 关系：release 之前的所有写操作，在 acquire 之后都可见。

    关键点：release（写端）和 acquire（读端）是一对，共同建立跨线程的同步关系。sender 中 C 之前的所有写入，对 receiver 中 D 之后的代码完全可见。

    5.4 memory_order_release

    规则：release 语义用于写操作（store）。在 release store 之前的所有读写操作，都不能被重排到 release 之后。

    release 单独无法使用，必须与 acquire（建立 synchronizes-with）或 consume（建立 dependency-ordered-before）配合。

    参见 5.2 和 5.3 节的示例，release 总是作为写端出现。

    5.5 memory_order_acq_rel

    规则：acq_rel 是 acquire 和 release 的叠加，专用于读-改-写（RMW, Read-Modify-Write）操作，例如 fetch_add、compare_exchange 等。

    对于一个采用 acq_rel 的 RMW 操作：

    • 它既是一个 acquire 操作（之后的读写不能重排到它前面）
    • 也是一个 release 操作（之前的读写不能重排到它后面）
    • 其他线程在该变量上的 release 写操作及其之前的写入，对本操作开始的代码可见
    • 本操作之前的所有写入，对其他线程在该变量上的 acquire 读操作及其之后的代码可见

    典型场景是多线程中间人传递（relay）：

    5.6 memory_order_seq_cst

    规则：顺序一致性（Sequential Consistency）是约束最强的内存序，也是原子操作的默认内存序。在此模型下：

    • 每个 seq_cst 操作都有一个全局的执行顺序
    • 所有线程观察到的 seq_cst 操作顺序是完全一致的
    • 不仅同一线程内的执行结果与程序顺序一致，各线程间互相看到的执行结果也与程序顺序一致

    seq_cst 是最安全的选择，但也是性能开销最大的，因为它要求底层插入最强的内存屏障指令。

    若将 seq_cst 换成 acquire/release，上面的 assert 可能会触发，因为 acquire/release 不保证全局一致的操作顺序。

    5.7 强度对比与选用建议

    6 种内存序的约束强度从弱到强排列如下：

    内存序 适用操作 典型场景
    relaxed load / store / RMW 计数器、统计量，不需要同步
    consume load 指针跟随（不推荐，用 acquire 替代）
    acquire load 消费者端读取标志/数据
    release store 生产者端写入标志/数据
    acq_rel RMW（fetch_add, CAS 等） 中间人传递、无锁数据结构
    seq_cst load / store / RMW 需要全序保证时（性能要求不高时的安全选择）

    实践建议：

    • 在确定性能瓶颈之前，优先使用默认的 seq_cst（即不传第二个参数）
    • 仅在性能敏感的路径上，经过充分测试和分析后，才降低内存序约束
    • 宁可保守（seq_cst）也不要为了性能写出有数据竞争的代码

    7.4 atomic_compare_exchange_strong / _explicit

    功能：CAS（Compare-And-Swap）强版本。原子地比较 *Atom 与 *Exp：

    • 若相等：用 Value 替换 *Atom，返回 true
    • 若不等：用 *Atom 的当前值更新 *Exp，返回 false

    Strong 版本保证：若 *Atom == *Exp，一定会成功写入（不存在伪失败）。

    示例：无锁地更新最大值。

    7.5 atomic_compare_exchange_weak / _explicit

    功能：CAS 弱版本。与 strong 版本的区别在于，即使 *Atom == *Exp，也可能伪失败（spurious failure），返回 false 而不写入。

    因此弱版本必须在循环中使用，但在某些硬件（如 LL/SC 架构的 ARM）上比 strong 版本效率更高。

    示例：无锁栈入栈操作。

    strong vs weak 选择原则：

    • 若需要在循环中重试直至成功 → 用 weak（在 LL/SC 架构上可避免额外开销）
    • 若 CAS 失败后不重试，或需要在 if 中使用 → 用 strong

    7.13 atomic_thread_fence

    功能：插入一个线程间内存屏障（独立屏障，不与特定原子变量绑定）。强制对加载/存储操作进行排序，但本身没有关联的原子操作。

    Order 值 效果
    memory_order_relaxed 无效果
    memory_order_consume acquire fence
    memory_order_acquire acquire fence：之后的读写不能重排到 fence 前
    memory_order_release release fence：之前的读写不能重排到 fence 后
    memory_order_acq_rel acquire + release fence
    memory_order_seq_cst 全序的 acquire + release fence

    示例：用独立 fence 替代原子操作上的内存序标记。

    独立 fence 的同步条件比原子操作上的内存序更严格：需要 release fence + relaxed store 与 relaxed load + acquire fence 配合，才能建立同步关系。通常直接在原子操作上指定内存序更简洁。

    7.14 atomic_signal_fence

    功能：插入一个信号处理程序内存屏障。仅在同一线程内的信号处理程序与普通代码之间建立内存顺序，不提供跨线程同步。

    与 atomic_thread_fence 的区别：

    • atomic_thread_fence：跨线程同步，在多线程间建立 happens-before
    • atomic_signal_fence：仅同线程内（信号处理程序场景），对 CPU 来说是无操作（no-op），仅阻止编译器重排

    若返回 true，该类型的原子操作可以安全地在信号处理程序中使用（信号处理程序不能使用互斥锁）。

    7.16 atomic_init

    功能：对 atomic 对象执行非原子初始化。注意：这不是线程安全的，只能在对象被其他线程访问之前调用。

    9. 总结

    概念 核心要点
    内存模型 描述多线程共享内存访问顺序的规范，涵盖原子性、偏序关系、可见性三方面
    sequenced-before 同线程内操作的先后顺序，且前者结果对后者可见
    happens-before 跨线程操作先后顺序（含 sequenced-before 和 synchronizes-with）
    synchronizes-with release 写与 acquire 读之间建立的跨线程可见性关系
    carries-dependency 同线程内表达式间的数据依赖传递，与 consume 语义相关
    relaxed 仅原子性，无同步，性能最优
    consume 依赖链同步，理论开销小但实现难，实践中用 acquire 替代
    acquire 读端屏障，与 release 配合建立同步
    release 写端屏障，与 acquire 或 consume 配合建立同步
    acq_rel acquire + release，用于 RMW 操作
    seq_cst 全局全序，最安全、最强约束，默认选择
    volatile 不保证原子性和跨线程同步，不能替代 atomic

    C++11 内存模型为并发程序提供了强大而精细的控制能力。在实际开发中，建议遵循以下原则：

    1. 优先使用高级同步原语（mutex、condition_variable），仅在性能关键路径才考虑 atomic
    2. **默认使用 seq_cst**，只有在充分分析和测试后才降低内存序约束
    3. 避免使用 volatile 进行线程间同步
    4. 充分利用工具（ThreadSanitizer、Valgrind）检测数据竞争
