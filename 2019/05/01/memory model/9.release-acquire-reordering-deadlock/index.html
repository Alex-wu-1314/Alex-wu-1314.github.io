    Release/Acquire 操作的重排序会引入死锁吗？

    Alex Wu
    2019-05-01
    内存模型

    原文：Can Reordering of Release/Acquire Operations Introduce Deadlock?

    背景与问题的来源

    这个问题源自 Herb Sutter 在 2012 年的演讲 Atomic Weapons。在视频的 44:35 处，他提到了一个涉及自旋锁（spinlock）的例子，并警告说：release/acquire 操作的重排序可能会引入死锁。这个警告令人警觉，但它是否真的正确呢？

    本文将深入分析这个问题，并给出一个有据可查的结论。

    基础回顾：Acquire 与 Release 语义

    在正式讨论之前，先简要回顾两个核心概念：

    Acquire 语义（读-获取）

    acquire 语义只能应用于读取共享内存的操作（无论是普通 load 还是 read-modify-write 操作）。该操作被称为 read-acquire。

    Acquire 语义保证：read-acquire 操作不会与程序顺序中其后的任何读或写操作发生重排序。

    Release 语义（写-释放）

    release 语义只能应用于写入共享内存的操作（无论是普通 store 还是 read-modify-write 操作）。该操作被称为 write-release。

    Release 语义保证：write-release 操作不会与程序顺序中其前的任何读或写操作发生重排序。

    关键点

    Acquire/release 语义像一个”内存屏障三明治”，将中间的所有内存操作夹在中间，阻止它们跨越边界重排序。但需要特别注意：这两种语义并不阻止 acquire 和 release 操作本身彼此之间的相互重排序。

    编译器是否被允许做这种重排序？

    这才是问题的核心。Acquire/release 语义本身没有禁止上述重排序，但 C++ 标准中有另一条规则可以阻止它。

    C++ 标准 N4659 §4.7.2:18

    An implementation should ensure that the last value (in modification order) assigned by an atomic or synchronization operation will become visible to all other threads in a finite period of time.

    翻译：实现必须确保，一个原子操作或同步操作所赋的最终值，在有限时间内对所有其他线程可见。

    为何这条规则阻止了重排序？

    分析 Thread 1 的执行流程：

    1. A.store(0, std::memory_order_release)（Unlock A）执行后，按照标准，该写入操作的结果必须在有限时间内对其他线程可见。
    2. 如果编译器将 Lock B（acquire on B）提前到 Unlock A（release on A）之前，那么 Unlock A 的写入结果可能会被无限期推迟，因为 Lock B 可能永远等不到 B 被释放（死锁）。

    一旦执行进入 Lock B 的 while 循环，Thread 1 就可能永远卡在这里，再也没有机会执行 A.store(0,...)。这意味着 A 的解锁写入永远不会对其他线程可见，直接违反了标准的 §4.7.2:18 条款。

    因此，作者的结论是：编译器不应该做这种重排序，死锁不应该发生。

    注意（Post 更新）：这并非铁板钉钉的保证。§4.7.2:18 使用了”should”而非”shall”，这意味着它是一个建议性（非强制性）条款。部分专家（如 davmac 博客中所讨论的）认为该条款原本是为了覆盖更简单的场景（一个线程存储，另一个线程持续读取），并非专门针对这个死锁场景而设计。因此，这是一个存在不确定性的领域。

    附加场景：更简单的信号等待示例

    为了进一步说明同样的问题，作者提供了一个更简洁的示例：

    正常执行流程：

    1. Thread 1 将 A 设置为 1，然后等待 B 变为 1。
    2. Thread 2 检测到 A 为 1 后，将 B 设置为 1。
    3. Thread 1 检测到 B 为 1，退出循环。程序正常终止。

    如果编译器将 A.store(1,...) 移到 Thread 1 的末尾：

    • Thread 2 的 while (A.load(...) == 0) 永远为真，永远不会执行 B.store(1,...)。
    • Thread 1 的 while (B.load(...) == 0) 也永远为真，两个线程永远不会终止。

    这个例子同样依赖 §4.7.2:18：一旦进入循环，A.store 就无法执行，A 的写入将永远对 Thread 2 不可见，违反”有限时间内可见”的要求。

    关于 SC（顺序一致性）acquire/release 与普通 acquire/release

    Herb Sutter 在演讲中区分了两种语义：

    语义类型 说明
    Plain acquire/release 即 C++11 的 memory_order_acquire / memory_order_release
    SC acquire/release 顺序一致性语义，对应 ARMv8 的 ldar / stlr 指令

    ARMv8 引入 stlr（store-release）和 ldar（load-acquire）指令，是为了让 C++ 顺序一致性原子操作在 ARM 上有更高效的实现。这两条指令之间隐含了一个 #StoreLoad 屏障。

    然而，作者指出：要阻止本文讨论的死锁场景，既不需要 #StoreLoad 硬件屏障，也不需要顺序一致性原子操作。仅需阻止前面分析的那种编译器重排序即可，而 C++ 标准（通过 §4.7.2:18）已经（可能）做到了这一点。

    社区讨论与延伸观点

    • Anthony Williams（《C++ Concurrency in Action》作者）在评论区表示，他也认为上述示例不会死锁。
    • davmac（博客 Understanding the C/C++ memory model – part 2）持不同看法，认为 §4.7.2:18 的措辞（”should”）并不构成强制保证，且可能不是为此场景设计的。他建议使用 memory_order_seq_cst 来获得更强的保证。
    • Just Software Solutions 博客（2021 年）对同一问题进行了更详细的分析，结论与 preshing 相近，但也承认标准措辞存在模糊性。

    结论

    问题 结论
    Acquire/release 语义是否阻止 release-acquire 间的重排序？ 不阻止
    如果发生重排序，是否可能引入死锁？ 理论上可能
    C++ 标准是否禁止此类重排序？ 可能禁止（§4.7.2:18，但措辞非强制）
    主流编译器实际上是否会做此重排序？ 目前不会（经 Compiler Explorer 验证）
    是否需要 seq_cst 来防止死锁？ 不是必须的，但使用它更为保险

    作者的最终判断：memory_order_acquire 和 memory_order_release 并不比其他内存序更容易引发死锁。标准 §4.7.2:18 应当能够阻止上述危险的重排序，尽管这不是铁板钉钉的保证。

    实践建议：如果你在使用多个自旋锁或互斥量，始终以固定的、全局一致的顺序获取它们，才是避免死锁的根本之道。内存序是性能和正确性的工具，但不能替代良好的锁策略。
